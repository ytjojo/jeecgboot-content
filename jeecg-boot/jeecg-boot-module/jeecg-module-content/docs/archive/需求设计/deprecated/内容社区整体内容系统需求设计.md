# 内容系统需求设计

## 1. 概述

本设计文档面向 JeecgBoot 内容社区的整体内容相关能力，覆盖发布、存储、浏览、互动、管理、推荐与安全合规等模块，严格遵循项目分层架构（Controller -> Service -> Mapper）与 VO/DTO/Entity 分离原则，并与原始需求《内容社区系统需求文档》保持一致与可追溯。

## 2. 范围与目标

- 范围：所有“帖子”相关功能与数据结构，含内容类型、通用发布能力、附加组件、浏览与推荐、互动评论、草稿与定时、频道与转发、审核与版本、通知与禁言、举报与风控、缓存与并发、权限安全、性能与监控等。
- 目标：
  - 提供一致、可扩展的帖子模型与接口，支撑多内容类型混合发布与渲染。
  - 保证并发安全、统一事务与缓存策略，提升性能与一致性。
  - 明确验收标准与测试策略，确保功能质量与可维护性。

## 3. 术语定义

- 内容（Content）：社区中可发布、浏览、互动的内容实体，含文章、笔记、动态、视频、问答提问、问答回答等。
- 附件（Attachment/Media）：图片、视频、音频、文档等二进制资源，仅存储外部 URL 与元数据。
- 频道（Channel）：内容发布的组织与归档维度，可订阅与管理。
- 可见性（Visibility）：公开、仅自己、好友可见等访问控制策略。
- 状态（Status）：草稿、已发布、审核中、已删除、屏蔽等。
- TipTap：富文本编辑器 TipTap 的 JSON 格式。

## 4. 角色与权限（RBAC）

- 普通用户：发布、编辑、删除自己的帖子；点赞、收藏、评论；举报；关注频道；转发；参与投票；@提及。
- 提问者：在问答场景下采纳最佳答案；管理问题与回答的互动。
- 频道管理员：管理频道内内容，置顶、设精华、移除不符合主题的内容。
- 系统管理员/审核员：内容审核、封禁与屏蔽、处理举报与风控。
- 权限控制：
  - URL 级：基于 Spring Security/Shiro；统一鉴权与授权。
  - 方法级：注解基于角色与权限点；资源级能力校验（如作者本人才能编辑）。

## 5. 内容类型（Content Types）

统一采用单实体 + 类型字段（`content_type`）的策略，减少跨表复杂度，并为特殊字段通过扩展表或 JSON 元数据实现。类型枚举：

1. 文章（ARTICLE）
   - 富文本（TipTap JSON）与 Markdown；默认富文本。
   - 支持单张封面。
2. 笔记（NOTE）
   - 头部轮播（最多15张图片）。
   - 富文本/Markdown；默认富文本。
3. 动态（POST）
   - 文字在上、媒体在下；支持纯文本与图片/视频九宫格混排。
4. 视频（VIDEO）
   - 限制上传 1 条视频；附标题与简介。
5. 问答提问（QA_QUESTION）
   - 问题标题与描述（富文本可选）。
   - 其他用户提交回答；提问者采纳最佳答案。
6. 问答回答（QA_ANSWER）
   - 回答支持无限级回复；支持@提及；赞同/反对互斥。

## 6. 通用发布能力（General Publishing）

- 标签与分类（Tags & Categories）：支持话题/标签与自定义分类，便于组织与检索。
- 可见性（Visibility）：公开、仅自己、好友可见等；精细控制阅读范围。
- 预览（Preview）：发布前可预览最终渲染效果。
- 转发（Repost）：支持任意内容类型被转发并附加评论；保留原始引用关系。
- 频道设置（Channel）：支持发布到一个或多个频道（受频道规则限制）。

## 7. 附加组件（Attachable Components）

- 投票组件（Poll）：PK（二选一）、单选、多选；附着在主内容下方；独立投票记录与统计。
- @提及（Mention）：在所有文本输入区支持@用户，结构化存储用于通知与渲染。
- 位置信息（Location）：所有内容类型可附加地理位置信息（坐标 + 文本）。
- 话题/标签（Topic/Tag）：与内容类型无关的通用能力；用于聚合与搜索。
- Emoji：支持标准 Emoji 选择与显示；兼容移动端原生键盘。

## 8. 特殊元素标记格式（TipTap 扩展）

基于原始需求的 5.1/5.2 节，统一定义富文本 JSON 标记：text、mention、topic、tag、stock、link、document、image、video、audio、card 等。存储原则：

- 富文本源（TipTap JSON）存储于主表或扩展字段；避免富文本内嵌二进制。
- 渲染 HTML 需净化（XSS 防护）；编辑时返回原始 TipTap 以恢复编辑器状态。
- 特殊元素需结构化索引以支撑通知、推荐与搜索（如@mention 用户ID、话题ID）。

## 9. 附件支持（Media/Attachment）

- 附件最多 15 个；每个附件可添加描述。
- 图片/视频/音频/文档等二进制统一上传 OSS/CDN；数据库仅存外部 URL 与元数据（类型、时长、大小、分辨率等）。
- 统一媒体表 `content_media` 与主内容关联；维持展示顺序。

## 10. 内容存储策略

- 必存原始格式（source）：Markdown 文本 / 富文本 TipTap JSON / 纯文本。
- 可选缓存渲染 HTML（提升展示性能）。
- 生成纯文本版本：用于搜索与摘要。
- 元数据：作者、时间、隐私、回复关系、标签、状态、频道等。
- 编辑历史记录：保留版本；支持回溯。
- 安全：渲染 HTML 统一净化；编辑返回原始内容以恢复状态。

## 11. 浏览与推荐

- 最新内容列表：按发布时间倒序、分页加载；支持类型筛选、刷新与自动刷新；展示摘要与互动数据。
- 关注用户列表：仅展示关注对象发布内容；按时间倒序；支持分组与类型筛选。
- 精选内容列表：管理员标记或算法推荐；显示精选标识与推荐理由；支持专题化展示。
- 热门内容排行：支持 24h/7d/30d 热度；热度规则可配置（浏览、点赞、评论、分享等）。
- 统一列表体验：筛选与搜索、排序（时间/热度/推荐）、布局切换（列表/网格）、无限滚动或分页、缓存与离线、夜间模式。
- 智能推荐：基于行为、兴趣标签、社会关系与协同过滤；支持实时调整、多样性与新颖性；冷启动处理与反馈机制。

## 12. 高级互动功能

- 点赞（Like）：帖子与回答均支持；回答支持赞同/反对互斥。
- 收藏（Favorite）：收藏与稍后阅读。
- 评论（Comment）：楼层与回复、@提及、匿名、仅楼主可见、历史版本保留、发布后自动关注帖子。
- 分享（Share）：外部平台分享。
- 关注（Follow）：关注其他创作者；关注后信息流优先展示。
- 通知（Notification）：互动事件触达；@提及、点赞、评论、采纳等。
- 拉黑（Block）：屏蔽对方内容与互动。
- 举报（Report）：不当内容与行为处理与审核。

## 13. 草稿与定时发布

- 草稿：编辑中自动/手动保存；支持本地存储辅助防丢失；草稿可继续编辑并发布。
- 定时发布：指定未来时间自动发布；考虑并发安全与唯一触发（分布式任务/XXL-JOB）。

## 14. 审核与版本

- 审核：人工审核与结果记录；支持驳回与理由；审核通过进入已发布。
- 版本：编辑历史记录，支持查看差异与回滚。

## 15. 数据模型设计（Entity/DTO/VO 分离）

遵循“禁止在 Controller 直接使用 Entity；DTO 与实体转换在 Service；VO 封装请求/响应”。核心建议：

- Entity（示意字段）：
  - `content`：id、content_type、title、source_type（TipTap/MARKDOWN/TEXT）、source_payload（JSON/Text）、html_cache、text_cache、cover_url、visibility、status、author_id、channel_ids（JSON）、location（lng/lat/address）、mention_user_ids（JSON）、created_time、updated_time、deleted_flag。
  - `content_media`：id、content_id、media_type（IMAGE/VIDEO/AUDIO/DOC）、url、meta（JSON，如时长、分辨率）、order_index、created_time。
  - `content_tag_rel`：id、content_id、tag_id、created_time。
  - `poll`/`poll_option`/`poll_vote`：投票主体、选项与投票记录。
  - `interaction_like`/`favorite`/`comment`：互动表（互斥约束见并发与事务）。
  - （可选）`content_audit`、`content_version` 等。

- DTO（示意）：
  - `ContentBaseDTO`：通用字段与转换方法（toEntity/fromRequest/toResponse）。
  - `ContentCreateDTO`/`ContentUpdateDTO`/`ContentQueryDTO`/`ContentResultDTO`。
  - 禁止在 Controller 使用 Entity；禁止直接使用 BeanUtils.copyProperties 进行转换。

- VO（req/resp 示例）：
  - req：`ContentCreateVO`、`ContentUpdateVO`、`ContentQueryVO`（优先使用枚举参数）。
  - resp：`ContentListVO`、`ContentDetailVO`、`ContentResultVO`。
  - VO 中可以包含 DTO 字段；禁止使用 Entity 作为 VO。

## 16. API 设计（RESTful）

统一前缀：`/api/v1/content`，资源小写与连字符；示例：

- 发布帖子：`POST /api/v1/content/posts`
  - 请求：`ContentCreateVO`
  - 响应：`Result<ContentResultVO>`
- 编辑帖子：`PUT /api/v1/content/posts/{id}`
  - 请求：`ContentUpdateVO`
  - 响应：`Result<ContentResultVO>`
- 删除帖子：`DELETE /api/v1/content/posts/{id}`
  - 响应：`Result<Void>`
- 获取详情：`GET /api/v1/content/posts/{id}`
  - 响应：`Result<ContentDetailVO>`
- 最新内容：`GET /api/v1/content/feeds/latest`
  - 查询参数：`ContentQueryVO`
  - 响应：`Result<Page<ContentListVO>>`
- 关注信息流：`GET /api/v1/content/feeds/following`
- 精选内容：`GET /api/v1/content/feeds/featured`
- 热门排行：`GET /api/v1/content/feeds/hot`
- 点赞：`POST /api/v1/content/interactions/{id}/like`
- 反对（回答）：`POST /api/v1/content/interactions/{id}/downvote`
- 收藏：`POST /api/v1/content/interactions/{id}/favorite`
- 评论：`POST /api/v1/content/comments`
- 投票：`POST /api/v1/content/polls/{pollId}/vote`
- 转发：`POST /api/v1/content/posts/{id}/repost`

版本控制：URL 前缀与请求头 `API-Version: v1` 双轨支持。

## 17. 状态机与枚举

- 帖子状态：`DRAFT`、`PUBLISHED`、`REVIEW_PENDING`、`BLOCKED`、`DELETED`。
- 可见性：`PUBLIC`、`PRIVATE`、`FRIENDS_ONLY`。
- 内容类型：`ARTICLE`、`NOTE`、`POST`、`VIDEO`、`QA_QUESTION`、`QA_ANSWER`。
- 投票类型：`PK`、`SINGLE`、`MULTI`。
- 互动类型：`LIKE`、`DOWNVOTE`（仅回答）、`FAVORITE`、`COMMENT`。

## 18. 并发与事务

- 发布写入：使用事务保证主内容与媒体关联一致性；定时发布任务幂等控制（唯一键 + 分布式锁）。
- 点赞/反对互斥：基于唯一约束（user_id + content_id + type）与事务；回答的赞同/反对互斥校验。
- 计数一致性：写扩散与读扩散结合，Redis 原子计数 + 异步落库（消息队列）；失败重试与对账任务。
- 投票幂等：同一用户同一投票的唯一性约束；多选选项批量写入原子性。
- 评论发布：事务内写入评论与通知事件；编辑历史独立记录。

## 19. 缓存策略

- 键设计：统一前缀 `content:`；如 `content:detail:{id}`、`content:feed:latest:{page}`、`content:hot:24h`。
- 过期策略：详情缓存短期（60–300s）、列表缓存更短（30–120s）；热门榜与精选可适度长缓存（5–15m）。
- 失效与一致性：更新/删除时精确失效；计数写入后延时双删；热点内容采用逻辑过期 + 后台重建。

## 20. 权限与安全

- 鉴权：JWT 无状态；Spring Security/Shiro 配合；接口权限点明确。
- 输入安全：统一 XSS 净化；SQL 注入防护；文件白名单与大小限制；链接安全校验。
- 速率限制：Sentinel 或网关限流；防刷与机器人识别。
- 内容审核：敏感词与违规检测（可接入外部服务）；人工审核流程与记录。

## 21. 性能目标与 Benchmark 建议

- 目标：
  - 发布接口 P95 < 150ms（不含 OSS 上传耗时）。
  - 最新信息流查询 P95 < 120ms（分页 20 条，冷热混合）。
  - 热门榜单生成任务在 1e6 级内容下 5 分钟内完成。
  - 点赞/收藏/评论原子写入耗时 P95 < 80ms。
- 数据规模：模拟 1e6 内容、1e7 互动；并发 500–2000 req/s；冷热分布 80/20。
- 基准方法：
  - 离线：JMH 针对转换/渲染/计数聚合；SQL 与索引评估。
  - 在线：压测（如 Gatling/JMeter），观测 p95/p99、错误率与资源占用。

## 22. 日志与监控

- 日志：统一格式，关键业务打点（发布、编辑、删除、互动、审核、举报）。
- 监控：接口耗时、错误率、数据库慢 SQL、缓存命中率、队列堆积、任务执行情况；告警与自愈。

## 23. 验收标准（摘要）

1. 系统支持多内容类型发布与编辑（文章/笔记/动态/视频/问答）。
2. 富文本 TipTap 与 Markdown 源存储正确；编辑返回原始格式；渲染 HTML 净化安全。
3. 所有内容类型支持标签/话题、频道选择、预览、转发与位置信息。
4. 附件统一走 OSS，数据库仅存 URL 与元数据；最多 15 个。
5. 浏览：最新/关注/精选/热门列表可用，支持筛选、排序与统一交互。
6. 智能推荐具备基础能力与反馈机制；冷启动与多样性策略。
7. 互动：点赞、反对（互斥）、收藏、评论（回复、@）、分享、关注、通知、拉黑、举报。
8. 草稿与定时发布可用，且并发安全；定时发布幂等与唯一触发。
9. 审核与版本记录完整；可回溯与屏蔽。
10. 并发与事务方案落地；计数一致性与对账任务。
11. 缓存键与失效策略统一；性能指标达到目标或给出优化方案与基准数据。
12. 权限与安全合规；防注入与 XSS；速率限制与风控。

## 24. 风险与边界

- 富文本扩展过多导致渲染复杂与性能负担；需分层渲染与缓存。
- 推荐质量与数据稀疏风险；需多策略融合与反馈闭环。
- 定时发布与分布式环境一致性问题；需全链路幂等方案与任务观测。
- 热门榜与大规模计数一致性；需离线归档与在线近实时折中。
