# 内容版本管理技术指导文档

## 1. 概述

本文档指导如何实现高效的内容版本管理系统，采用差异存储策略节省存储空间，通过版本链管理内容历史。

## 2. 核心设计原则

### 2.1 存储策略
- **差异存储**：基础版本存储完整内容，增量版本仅存储差异补丁
- **版本链结构**：BaseSnapshot → Snapshot1 → Snapshot2 → LatestSnapshot
- **空间优化**：相比完整存储，可节省60-90%存储空间

### 2.2 并发控制
- **乐观锁机制**：通过版本号检测冲突，避免数据库锁
- **版本冲突处理**：检测到冲突时提示用户刷新后重试

### 2.3 技术选型
- **差异算法**：使用java-diff-utils库处理文本差异(已经对其封装成PatchUtils.java)
- **存储格式**：差异补丁以JSON格式存储
- **恢复策略**：基础版本+补丁链重建完整内容


## 3. 架构设计

### 3.1 数据模型设计

#### 核心实体关系
- **Content**：内容主表，存储最新内容
- **ContentDraft**：草稿主表，存储草稿最新内容
- **ContentVersion**：版本快照表，存储版本差异和元数据
- **关联关系**：
- Content(1) → ContentVersion(N)，通过contentId建立链式关系
- ContentDraft(1) → ContentVersion(N)，通过contentId建立链式关系
- versionType：版本类型，区分内容版本和草稿版本

#### 关键字段设计
```


ContentVersion表：(真实字段名称为下划线风格)
- version_type  DEFAULT 'DRAFT' 版本类型：CONTENT-内容版本, DRAFT-草稿版本,
- parent_snapshot_id：父版本快照ID（构建版本链）
- is_base_snapshot：是否为基础快照标识
- content_patch：原始内容差异补丁
- rendered_patch：渲染内容差异补丁
- version_number：版本号（乐观锁）
- is_active :是否当前激活/生效的版本
- is_base_snapshot :是否基础快照
- is_head :是否最新版本
```

### 3.2 核心业务流程

#### 内容版本创建流程
1. 在草稿ContentDraft更新 内容Content更新
2. 判断content_version表是否有对应记录，没有则根据查询content_draft或contents表中内容创建基础快照
4. 创建基础ContentVersion（is_base_snapshot=true,is_head=true,is_active=true，存储完整内容）


#### 内容更新流程
1. 先判断ContentVersion是否有对应记录，没有则创建,走内容版本创建流程
1. 根据contentId查询is_active为true的ContentVersion信息
2. 检查版本冲突（乐观锁）
3. 存储草稿表完整内容/内容表完整内容
4. 计算与基础版本的差异补丁
5. 创建增量ContentVersion（is_base_snapshot=false，存储差异）
6. 更新parent_snapshot_id为查询的版本id
7. 更新version_number为查询的版本号+1
8. 更新is_head为true
9. 更新is_active为true
#### 内容发布流程
1. 验证内容状态（确保内容已审核通过）
2. 获取当前激活的版本（is_active=true）
3. 更新Content内容为最新版本内容
4. 记录发布日志和发布时间
5. 触发发布后置处理（如缓存更新、搜索索引更新等）

#### 版本恢复流程
1. 获取目标版本快照
2. 如果是基础快照，更新草稿表/内容表为基础版本内容,更新is_active为true
3. 如果是增量快照，获取基础快照并应用差异补丁
4. 更新草稿表/内容表内容为应用补丁内容
5. 更新其他相同content_id其他is_active=true改为false


## 4. 核心组件实现

### 4.1 实体层设计要点

#### ContentVersion实体关键字段
- `isBaseSnapshot`：布尔标识，区分基础快照和增量快照
- `isActive`：布尔标识，是否是当前激活/生效的版本
- `isHead`：布尔标识，是否是分叉分支最新版本,会有多个head
- `contentPatch`：存储原始内容或差异补丁（JSON格式）
- `renderedPatch`：存储渲染内容或差异补丁
- `versionNumber`：版本号，实现乐观锁机制
- `parentSnapshotId`：构建版本链关系

### 4.2 差异处理工具设计

#### 技术选型
- **推荐库**：java-diff-utils（轻量级、性能好）
- **备选方案**：Google diff-match-patch（功能更强大）

#### 核心方法设计
```java
@Component
public class PatchUtils {
    /**
     * 计算文本差异
     * @param original 原始文本
     * @param revised 修改后文本
     * @return JSON格式的差异补丁
     */
    public static String computeDiff(String original, String revised)
    
    /**
     * 应用差异补丁
     * @param original 原始文本
     * @param patchJson 差异补丁JSON
     * @return 应用补丁后的文本
     */
    public static String applyPatch(String original, String patchJson)
}
```

#### 实现要点
1. **文本分割**：按行分割文本，便于差异计算
2. **序列化**：将差异对象序列化为JSON存储
3. **异常处理**：补丁应用失败时的回退机制
4. **性能优化**：大文本处理时的内存管理


#### 索引优化建议
1. **复合索引**：(content_id, version DESC) - 快速查询版本历史
2. **唯一索引**：(content_id, base_snapshot) WHERE base_snapshot=true - 确保基础快照唯一性
3. **普通索引**：parent_snapshot_id - 构建版本链查询
4. **分区索引**：按时间分区提升大数据量查询性能




#### 关键业务规则
1. **版本冲突处理**：使用乐观锁机制，版本号不匹配时抛出异常
2. **差异计算策略**：始终与基础快照计算差异，避免链式依赖
3. **快照清理策略**：保留最近N个版本，定期清理历史数据
4. **事务边界**：确保快照创建和内容更新在同一事务中

#
#### 字段作用区别
- **base_snapshot_id**：用于差异计算的基准，所有增量版本都与此基础版本计算差异
- **parent_snapshot_id**：仅用于构建版本链关系，方便追溯版本历史，不参与差异计算

#### 设计优势
这种设计避免了链式依赖问题：

- 如果基于 parent_snapshot_id 计算差异，恢复某个版本需要从基础版本开始逐个应用所有补丁
- 基于 base_snapshot_id 计算差异，任何版本都可以通过 "基础版本 + 单个补丁" 直接恢复，性能更好



#### 获取内容版本历史流程
1. 根据versionType确定是草稿content_drafts版本还是正式contents版本
2. 如果是草稿版本，查询content_drafts表
3. 如果是正式版本，查询contents表
4. **验证内容存在性**：根据内容ID查询对应的表，验证内容是否存在
5. **查询版本快照列表**：从ContentVersion表中查询该内容的所有快照记录，按版本号排序
6. **构建版本信息列表**：遍历快照记录，提取关键信息构建ContentVersio对象
   - 快照ID
   - 版本号
   - 最后修改时间
   - 所有者信息
   - 是否为基础快照标识
   - 是否为最新
   - 是否为激活
7. **返回版本历史**：将构建的版本信息列表返回给调用方
8. **异常处理**：捕获并记录异常信息，抛出业务异常

## 6. 数据库设计要点

### 6.1 纯关联表设计方案

#### 设计理念
采用纯关联表方案，将快照管理完全交由content_versions表处理，contents表只保留核心内容信息，通过关联表实现版本控制。

#### 优化后的表结构
```sql


-- 内容版本表
CREATE TABLE content_versions (
    id VARCHAR(32) PRIMARY KEY,
    content_id VARCHAR(32) NOT NULL,
    parent_snapshot_id VARCHAR(32),           -- 父快照引用
    is_base_snapshot TINYINT(1) DEFAULT 0,      -- 是否基础快照
    is_active TINYINT(1) DEFAULT 0,           -- 是否为当前激活/生效的版本
    is_head TINYINT(1) DEFAULT 0,            -- 是否最新版本
    content_patch LONGTEXT,                  -- 原始内容/差异
    rendered_patch LONGTEXT,                 -- 渲染内容/差异
    version_number BIGINT NOT NULL,                 -- 版本号
    owner VARCHAR(32) NOT NULL,              -- 修改者
    last_modified_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    -- 唯一性约束
    UNIQUE KEY uk_content_active (content_id, is_active),
    UNIQUE KEY uk_content_version_number (content_id, version_number)
);
```

#### 关键约束说明
1. **发布唯一性**：每个content_id只能有一条is_active=1的记录
2. **最新版本唯一性**：每个content_id只能有一条is_head=1的记录
3. **版本唯一性**：每个content_id的version_number必须唯一

#### 查询优化索引
```sql
-- 发布版本查询
INDEX idx_content_active (content_id, is_active)
-- 最新版本查询
INDEX idx_content_head (content_id, is_head)
-- 基础快照查询
INDEX idx_content_base (content_id, base_snapshot)
-- 版本历史查询
INDEX idx_content_version_number (content_id, version_number)
-- 时间排序查询
INDEX idx_modified_time (content_id, last_modified_at)
```

#### 业务逻辑实现
```sql
-- 获取发布版本
SELECT * FROM content 
WHERE content_id = ? AND is_active = 1;

-- 获取最新版本
SELECT * FROM content_versions 
WHERE content_id = ? AND is_head = 1;

-- 获取版本历史
SELECT * FROM content_versions 
WHERE content_id = ? 
ORDER BY version DESC;
```

#### 方案优势
1. **设计纯粹**：contents表职责单一，只管理内容基本信息
2. **扩展性强**：可轻松添加更多状态字段（如审核状态、草稿状态等）
3. **数据规范**：符合数据库设计第三范式，减少冗余
4. **维护简单**：状态变更只需更新content_versions表

#### 性能考虑
1. **查询性能**：通过合理索引设计，常用查询性能可接受
2. **缓存策略**：可针对发布版本和最新版本进行缓存优化
