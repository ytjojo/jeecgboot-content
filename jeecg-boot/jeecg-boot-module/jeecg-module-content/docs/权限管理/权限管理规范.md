内容社区权限管理设计文档

1. 背景

内容社区需要支持精细化权限控制，例如：
	•	系统管理员：可编辑所有频道、所有文章、处理所有频道禁言。
	•	频道管理员 / 版主：只能管理自己负责的频道，能编辑频道内所有文章，但不能跨频道操作。
	•	普通用户：只能编辑自己发布的文章，不能修改他人文章或频道信息。
	•	临时限制：如禁止发帖 3 天、禁止评论 7 天、禁止编辑等。

传统 RBAC（基于角色的权限控制）难以覆盖「某用户对某频道/文章」的细粒度规则，因此需要 RBAC + ABAC（基于属性的访问控制）组合模式。

⸻

2. 总体设计
	•	RBAC：负责静态角色和全局权限（如 system:all、channel:manage、content:post）。
	•	ABAC：负责基于资源属性的动态校验（如文章作者是否为本人、频道管理员是否属于该频道）。
	•	Ban 限制：独立维护，优先于权限检查。

⸻

3. 分层职责

3.1 Shiro Realm
	•	负责认证（JWT Token 验证、Subject 构建）。
	•	负责加载用户的角色和基础权限集合。
	•	不负责资源级别的动态判断。

3.2 Shiro 注解
	•	用于基础拦截（粗粒度）。
	•	例如：
	•	@RequiresRoles("system_admin")
	•	@RequiresPermissions("channel:manage")

3.3 PermissionService
	•	负责精细化业务校验（ABAC）。
	•	根据 userId + channelId/articleId，结合角色、资源属性，判断是否允许操作。
	•	提供统一 API：
	•	canManageChannel(userId, channelId)
	•	canEditArticle(userId, articleId)
	•	canMuteUser(userId, channelId)

3.4 Controller / Service
	•	接收请求后，先经过 Shiro 基础拦截。
	•	再调用 PermissionService 执行资源级别校验。
	•	最终决定是否执行业务逻辑。

4. 权限判定流程图

 ┌──────────────────────────────────────────────────┐
 │                  用户请求 API                     │
 │   (携带 JWT, 例如 PUT /channels/123/update )      │
 └──────────────────────────────────────────────────┘
                          │
                          ▼
 ┌──────────────────────────────────────────────────┐
 │              Shiro JWT Filter 拦截                │
 │  - 验签、解析 JWT，得到 userId/roles/claims        │
 │  - 构造 Subject 并交给 Shiro                      │
 └──────────────────────────────────────────────────┘
                          │
                          ▼
 ┌──────────────────────────────────────────────────┐
 │                AuthorizingRealm 授权              │
 │  - 加载静态角色/权限集合 (RBAC)                   │
 │    例：system_admin → {system:all, channel:manage}│
 │        channel_admin → {channel:manage}           │
 │        user         → {content:post, edit:own}    │
 └──────────────────────────────────────────────────┘
                          │
                          ▼
 ┌──────────────────────────────────────────────────┐
 │             Shiro 注解基础权限校验                 │
 │   ✔ 通过 → 进入 Controller                        │
 │   ✘ 无权限 → 抛出 UnauthorizedException           │
 └──────────────────────────────────────────────────┘
                          │
                          ▼
 ┌──────────────────────────────────────────────────┐
 │         Controller / Service 业务逻辑入口          │
 │  - 取 userId + channelId/articleId                │
 │  - 调用 PermissionService 校验资源级别权限         │
 └──────────────────────────────────────────────────┘
                          │
                          ▼
 ┌──────────────────────────────────────────────────┐
 │            PermissionService 精细判断              │
 │   - SystemAdmin → 全部允许                        │
 │   - ChannelAdmin → 仅限所属频道                   │
 │   - 普通用户 → 仅能操作自己资源                   │
 │   - 结合 BanService 检查是否临时禁用              │
 └──────────────────────────────────────────────────┘
                          │
                          ▼
 ┌──────────────────────────────────────────────────┐
 │       结果：允许则继续执行 Service 更新逻辑         │
 │             否则抛出 AuthorizationException        │
 └──────────────────────────────────────────────────┘

5. 权限判定规则示例

5.1 频道管理
	•	系统管理员：可管理所有频道。
	•	频道管理员 / 版主：仅可管理自己负责的频道。
	•	普通用户：无权限。

5.2 文章编辑
	•	系统管理员：可编辑所有文章。
	•	频道管理员 / 版主：可编辑所属频道内的文章。
	•	作者本人：可编辑自己的文章。
	•	其他用户：无权限。

5.3 用户禁言
	•	系统管理员：可禁言所有频道用户。
	•	频道管理员 / 版主：仅可禁言自己负责频道内的用户。
	•	普通用户：无权限。

⸻

6. 设计总结
	1.	RBAC 管理全局角色/权限（用 Shiro Realm 加载）。
	2.	ABAC 管理资源级别权限（PermissionService 动态判断）。
	3.	Shiro 注解负责粗粒度拦截，业务层代码负责精细化控制。
	4.	BanService 独立实现临时禁用（禁止发帖/评论/编辑），优先于 RBAC/ABAC。
	5.	通过分层设计，保证权限控制既清晰可维护，又能支持动态业务场景。
