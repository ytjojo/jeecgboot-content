# 内容发布历史版本管理设计文档

## 1. 概述

本文档详细介绍了基于现有`content_versions`表实现内容发布历史版本管理功能的设计方案。该方案为正式发布的内容提供完整的版本历史追踪、版本对比、版本回滚等功能。

## 2. 设计原则

1. **重用现有架构**：充分利用现有的`content_versions`表结构和版本管理机制
2. **发布版本追踪**：记录每次内容发布的与基础版本的差异
3. **版本对比功能**：支持任意两个版本之间的内容对比
4. **版本回滚机制**：支持将内容回滚到历史版本
5. **发布流程集成**：与现有内容发布流程无缝集成
6. **性能优化**：采用增量存储和缓存策略优化查询性能
7. **权限控制**：版本管理操作需要相应权限验证
8. **节省存储**：仅第一次更新才根据已发布内容创建基础快照版本,发布后无更新content_versions就不会包含对应内容版本信息

## 3. 架构设计


**重用策略**：
- 通过`version_type = 'CONTENT'`区分草稿还是正式版本
- `content_id`字段关联到`contents.id`
- 利用现有的版本号和变更类型机制
- 扩展变更原因字段记录发布说明
- base_snapshot_id**：用于差异计算的基准，所有增量版本都与此基础版本计算差异
- parent_snapshot_id**：仅用于构建版本链关系

### 3.2 内容发布版本管理架构

```
内容发布版本管理架构：

Contents (正式内容表)
    ↓
ContentVersions (版本表)
    ├── version_type = 'CONTENT'
    ├── content_id = content.id
    └── 发布版本存储机制
        ├── 第一次更新先创建基础版本 (change_type = 'create')
        ├── 更新基于基础版本创建增量版本 (change_type = 'update')
        └── 删除版本 (change_type = 'delete')
```

### 3.3 版本存储策略

#### 3.3.1 创建版本存储
- **触发时机**：内容第一次更新时
- **存储内容**：完整的内容数据
- **设置为基础快照**：is_base_snapshot=true
- **版本号**：1
- **变更类型**：create
- **存储格式**：JSON格式存储完整内容

#### 3.3.2 更新版本存储
- **触发时机**：内容更新时
- **存储内容**：与base_snapshot_id记录对比变更补丁以JSON格式存储
- **版本号**：递增（2, 3, 4...）
- **变更类型**：update
- **存储格式**：JSON格式存储变更补丁
- **更新字段**: 设置is_active=true,is_head=true

### 3.4 发布版本管理流程

#### 3.4.1 内容发布流程
```
一定要和普通内容发布流程区分开,此流程仅适用于发布后进行更新的情况
1. 内容发布前验证（权限、内容完整性）
2. 判断是否携带明确版本号
    - 携带版本号：判断版本号是否存在,查询is_active版本是否是同一版本,如果相同,直接更新contents表的发布状态
    非同一版本走版本回滚流程
    - 不携带版本号：判断参数中携带的完整内容,
    - 判断参数携带内容和content_versions is_active=true内容是否有差异
    - 有差异走更新版本流程
    - 无差异不创建版本,仅更新contents表的发布状态
7. 更新contents表的发布状态
8. 记录发布日志和变更说明
9. 触发发布后处理（通知、索引更新等）
```

#### 3.4.2 内容更新流程
```
1. 内容更新前验证（权限、版本冲突检查）
2. 创建更新版本记录
3. 存储变更补丁
4. 更新contents表内容
5. 记录更新日志和变更说明
6. 触发更新后处理
```

#### 3.4.3 版本回滚流程
```
1. 权限验证（管理员权限）
2. 获取目标版本内容
3. 验证回滚操作的合法性
4. 创建回滚版本记录
5. 更新contents表内容为目标版本
6. 记录回滚操作日志
```



## 9. 配置和优化

### 9.1 版本管理配置

```yaml
# 内容版本管理配置
content:
  version:
    # 最大保留版本数
    max-versions: 100
    # 自动清理间隔（小时）
    cleanup-interval: 168  # 7天
    # 版本差异缓存时间（分钟）
    diff-cache-time: 60
    # 是否启用版本管理
    enabled: true
    # 版本导出格式
    export-formats: ["json", "xml", "csv"]
    # 版本回滚权限
    rollback-roles: ["admin", "content_manager"]
```

### 9.2 性能优化策略

1. **索引优化**：
   ```sql
   -- 内容版本查询优化索引
   CREATE INDEX IF NOT EXISTS idx_versions_content_type ON content_versions(content_id, version_type, version_number DESC) WHERE version_type = 'CONTENT';
   CREATE INDEX IF NOT EXISTS idx_versions_publish_time ON content_versions(publish_time DESC) WHERE version_type = 'CONTENT';
   CREATE INDEX IF NOT EXISTS idx_versions_creator ON content_versions(create_by, create_time DESC) WHERE version_type = 'CONTENT';
   ```

2. **缓存策略**：
   - 缓存最新版本内容
   - 缓存版本历史列表
   - 缓存版本差异计算结果
   - 缓存版本统计信息

3. **异步处理**：
   - 异步清理历史版本
   - 异步计算版本差异
   - 异步生成导出数据




## 11. 总结

本设计方案通过重用现有的`content_versions`表，为正式发布的内容提供了完整的版本管理功能：

1. **架构重用**：充分利用现有版本管理架构，降低开发和维护成本
2. **完整版本追踪**：记录内容从创建到删除的完整版本历史
3. **版本对比功能**：支持任意版本间的差异对比和可视化展示
4. **版本回滚机制**：支持管理员将内容回滚到任意历史版本
5. **权限控制**：版本管理操作需要相应权限验证
6. **性能优化**：通过索引、缓存、异步处理等策略保证系统性能
7. **集成友好**：与现有发布流程和草稿箱版本管理无缝集成

该方案为内容管理系统提供了企业级的版本控制能力，满足内容发布历史追踪、版本对比、版本回滚等高级需求。
