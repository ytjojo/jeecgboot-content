# MyBatis 复杂关联查询最佳实践

## 1. 方案选择指南

### 1.1 方案一：分两次查询（最佳平衡）
**适用场景**：
- 数据量较大，关联表数据多
- 需要分页的列表查询
- 对性能要求高但可接受2次查询

**不适用**：
- 简单对象关联
- 实时性要求极高的场景

### 1.2 方案二：嵌套结果映射（最优雅）
**适用场景**：
- 详情页查询
- 关联数据量适中
- 需要完整对象结构的场景

**不适用**：
- 超大数据量关联
- 需要深度分页的场景

## 2. 分两次查询实现规范

### 2.1 Entity设计
```java
@Data
@TableName("contents")
public class Content {
    private Long id;
    private String title;
    private String cover;
    private Long authorId;
    
    @TableField(exist = false)
    private List<Topic> topics;
    
    @TableField(exist = false)
    private List<Tag> tags;
}
```

### 2.2 Mapper实现
```java
public interface ContentMapper extends BaseMapper<Content> {
    
    // 第一次查询：基础数据
    @Select("SELECT * FROM contents WHERE id = #{id}")
    Content selectBasicById(Long id);
    
    // 第二次查询：批量关联数据
    @Select({"<script>",
        "SELECT t.*, ct.content_id",
        "FROM topics t",
        "JOIN content_topics ct ON t.id = ct.topic_id",
        "WHERE ct.content_id IN",
        "<foreach item='id' collection='ids' open='(' separator=',' close=')'>",
        "#{id}",
        "</foreach>",
        "</script>"
    })
    @Results({
        @Result(property = "id", column = "id"),
        @Result(property = "name", column = "name")
    })
    List<Topic> selectTopicsByContentIds(@Param("ids") List<Long> ids);
}
```

### 2.3 Service层实现
```java
@Service
@RequiredArgsConstructor
public class ContentService {
    private final ContentMapper contentMapper;
    
    public Content getContentDetail(Long id) {
        // 第一次查询
        Content content = contentMapper.selectBasicById(id);
        
        // 第二次查询（批量）
        List<Long> contentIds = Collections.singletonList(id);
        List<Topic> topics = contentMapper.selectTopicsByContentIds(contentIds);
        
        // 组合数据
        content.setTopics(topics);
        return content;
    }
    
    // 批量处理版本
    public List<Content> batchGetContents(List<Long> ids) {
        List<Content> contents = contentMapper.selectBasicByIds(ids);
        List<Topic> allTopics = contentMapper.selectTopicsByContentIds(ids);
        
        // 按contentId分组
        Map<Long, List<Topic>> topicsMap = allTopics.stream()
            .collect(Collectors.groupingBy(Topic::getContentId));
        
        // 设置关联数据
        contents.forEach(content -> 
            content.setTopics(topicsMap.get(content.getId())));
        
        return contents;
    }
}
```

## 3. 嵌套结果映射实现规范

### 3.1 Mapper实现
```java
public interface ContentMapper extends BaseMapper<Content> {
    
    @Select({"<script>",
        "SELECT",
        "c.id, c.title, c.cover, c.author_id,",
        "t.id as topic_id, t.name as topic_name, t.description as topic_desc",
        "FROM contents c",
        "LEFT JOIN content_topics ct ON c.id = ct.content_id",
        "LEFT JOIN topics t ON ct.topic_id = t.id",
        "WHERE c.id = #{id}",
        "</script>"
    })
    @Results(id = "contentDetailMap", value = {
        @Result(property = "id", column = "id"),
        @Result(property = "title", column = "title"),
        @Result(property = "cover", column = "cover"),
        @Result(property = "authorId", column = "author_id"),
        
        // 嵌套集合映射
        @Result(property = "topics", column = "id", 
                many = @Many(resultMap = "topicResultMap"))
    })
    Content selectContentWithTopics(Long id);
    
    // 可复用的ResultMap
    @ResultMap("topicResultMap")
    @Results({
        @Result(property = "id", column = "topic_id"),
        @Result(property = "name", column = "topic_name"),
        @Result(property = "description", column = "topic_desc")
    })
    List<Topic> mapTopics(ResultSet rs) throws SQLException;
}
```

### 3.2 多对多关联处理
```java
@Select({"<script>",
    "SELECT",
    "c.id, c.title,",
    "a.id as author_id, a.name as author_name,",
    "t.id as tag_id, t.name as tag_name",
    "FROM contents c",
    "LEFT JOIN authors a ON c.author_id = a.id",
    "LEFT JOIN content_tags ct ON c.id = ct.content_id",
    "LEFT JOIN tags t ON ct.tag_id = t.id",
    "WHERE c.id = #{id}",
    "</script>"
})
@Results({
    @Result(property = "id", column = "id"),
    @Result(property = "title", column = "title"),
    @Result(property = "author.id", column = "author_id"),
    @Result(property = "author.name", column = "author_name"),
    @Result(property = "tags", column = "id", 
            many = @Many(resultMap = "tagResultMap"))
})
Content selectContentWithAssociations(Long id);
```

## 4. 性能优化建议

### 4.1 索引策略
```sql
-- 关联表必须建立索引
CREATE INDEX idx_content_topics ON content_topics(content_id, topic_id);
CREATE INDEX idx_content_tags ON content_tags(content_id, tag_id);
```

### 4.2 查询优化
- 使用`LIMIT`控制关联数据量
- 避免`SELECT *`，明确指定字段
- 使用`LEFT JOIN`替代子查询

### 4.3 缓存策略
```java
@Cacheable(value = "content", key = "#id")
public Content getContentDetail(Long id) {
    return contentMapper.selectContentWithTopics(id);
}
```

## 5. 代码规范

### 5.1 命名约定
- 查询方法：`select[Entity]With[Association]s`
- ResultMap：`[entity]ResultMap`
- 关联字段：使用复数形式（`topics`, `tags`）

### 5.2 异常处理
```java
try {
    return contentMapper.selectContentWithTopics(id);
} catch (DataAccessException e) {
    log.error("查询内容详情失败, id: {}", id, e);
    throw new BusinessException("查询失败");
}
```

## 6. 选择矩阵

| 场景 | 推荐方案 | 理由 |
|------|----------|------|
| 内容列表分页 | 分两次查询 | 避免大数据量关联查询 |
| 内容详情页 | 嵌套结果映射 | 代码简洁，一次查询 |
| 管理后台 | 分两次查询 | 灵活控制查询字段 |
| 实时性要求高 | 嵌套结果映射 | 减少网络往返时间 |

## 7. 监控指标

- 查询响应时间：< 100ms
- 数据库连接数：< 最大连接数80%
- N+1查询次数：0（通过代码审查保证）

---

**总结**：根据具体场景选择方案，列表查询用方案一，详情查询用方案二，始终保持代码的可维护性和性能平衡。
