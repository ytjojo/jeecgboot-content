## 一、架构漂移检测

### 漂移1: 文件命名架构漂移 ⚠️ 中
**设计约定**: [tasks.md, design.md File Structure] 页面文件为index.vue
**实际实现**: 
- mutual-follow/MutualFollowList.vue (非index.vue)
- fan/FanDashboard.vue + FanList.vue + FanTrend.vue + FanProfile.vue (拆分多文件，入口为FanDashboard)
- invite/InviteShare.vue (非index.vue)
**漂移影响**:
- 与项目其他模块（如settings）使用index.vue的惯例不一致
- 路由正确引用了实际文件名，不影响功能
- 但文档描述与实际不符，增加维护成本

**修复建议**: 
- 方案A: 更新design.md和tasks.md中的文件命名描述
- 方案B: 重命名为index.vue（工作量较大）
- 建议采用方案A，补充文档说明组件拆分原因

### 漂移2: Store依赖隐式API ⚠️ 高
**设计约定**: 未明确提及relation API
**实际实现**: mutualFollow store动态导入`/@/api/content/relation`的getMutualStatus方法
**漂移影响**:
- 隐式依赖，该API文件是否存在未验证
- 如果文件不存在会导致运行时错误
- design.md中未提及此API依赖

**修复建议**:
- 立即验证`/src/api/content/relation.ts`是否存在及导出getMutualStatus
- 如缺失需补充该API文件
- 在design.md中补充此依赖说明

### 漂移3: 粉丝页面组件拆分 ⚠️ 低
**设计约定**: [design.md:162-163] FanList.vue + FanTrend.vue + FanProfile.vue独立文件
**实际实现**: 额外增加FanDashboard.vue作为Tab入口容器
