## Context

JeecgBoot_sass 内容社区模块（`jeecg-module-content`）前端基于 Vue 3 + Vite + Ant Design Vue + TypeScript，使用 `defHttp` 封装 API 请求，`useTable`/`useForm`/`useModal` 等 Hooks 管理页面状态。当前社区已有关注体系（EPIC-04）、个人资料（EPIC-02）、积分体系（EPIC-03）等基础能力，但缺少社交关系扩展功能。

前端代码位于 `jeecgboot-vue3/src/views/content/` 目录，API 封装位于 `src/api/content/`，Store 位于 `src/store/modules/`。

## Goals / Non-Goals

**Goals:**
- 实现互关标识展示（用户主页 + 评论区），互关好友列表页，支持搜索和取消互关
- 实现"仅互关可见"私密内容的发布与展示（Feed 流、用户主页、搜索结果的可见性控制）
- 实现粉丝管理页：粉丝列表（分页+搜索）+ 粉丝趋势图表（ECharts，日/周/月维度）
- 实现邀请分享页：邀请码生成、复制链接、邀请记录、收益统计
- 实现邀请链接落地页：被邀请者注册引导，含转化漏斗设计
- 实现社区角色标签组件（评论区展示创作者/版主/管理员标识）
- 实现版主/管理员评论区管理操作（删除评论、警告用户）+ 跳转 EPIC-09
- 实现审计日志页（管理员专用）

**Non-Goals:**
- 粉丝画像（兴趣分布、地域分布、活跃时段）— 降级至二期
- 封禁用户、禁言用户、撤销处罚 — 统一在 EPIC-09 实现
- 创作者认证申请页面 — 本期自动审核
- 自定义角色扩展 — 后续迭代
- 微信/QQ/微博 SDK 对接 — 仅复制链接
- 独立 IM 即时通讯

## Decisions

### 1. 互关状态缓存策略：评论接口自带字段为主 + Store 缓存为辅

**选择**: 评论列表接口和用户主页接口附带 `mutualFollow` 字段，前端直接使用；仅在动态加载评论（增量场景）时调用批量查询接口，结果缓存到 `useMutualFollowStore`。

**替代方案**: 每次渲染评论都调用互关状态接口 → N+1 请求，性能差。

**理由**: 评论区互关标识渲染无额外网络开销，缓存策略覆盖增量场景，会话内有效。

### 2. ECharts 按需引入：使用 `echarts/core` + tree-shaking

**选择**: 使用 `echarts/core` 按需注册图表类型和组件，不引入完整 echarts 包。

**替代方案**: 直接 `import * as echarts from 'echarts'` → 包体积大（~800KB）。

**理由**: 本期仅需折线图（趋势），按需引入可将 echarts 体积控制在 ~200KB 以内。

### 3. 私密内容可见性控制：前端展示层控制 + 后端接口过滤

**选择**: 后端接口根据当前用户与内容作者的互关状态过滤私密内容，前端仅负责展示"仅互关可见"标识和权限提示。

**替代方案**: 前端获取全量内容后自行过滤 → 安全性差，私密内容可能泄露。

**理由**: 安全性要求高，私密内容数据不应传递给非互关用户。

### 4. 邀请落地页独立路由：`/invite/:inviteCode`

**选择**: 邀请落地页使用独立路由 `/invite/:inviteCode`，不嵌套在内容社区布局中。

**替代方案**: 嵌套在内容社区布局中 → 被邀请者看到完整社区界面，干扰注册引导。

**理由**: 落地页面向未注册用户，需要简洁的注册引导体验，独立布局更合适。

### 5. 角色标签数据来源：评论接口附带 `communityRole` 字段

**选择**: 评论列表接口在返回数据中附带评论者的 `communityRole` 字段，前端直接渲染 CommunityRoleBadge 组件。

**替代方案**: 前端单独请求每个评论者的角色信息 → N+1 请求。

**理由**: 与互关标识策略一致，避免额外网络开销。

### 6. 管理操作权限控制：前端按钮显隐 + 后端接口校验双重保障

**选择**: 前端通过 `usePermission` Hook 控制管理操作按钮的显隐，后端接口做最终权限校验。

**替代方案**: 仅前端控制 → 用户可通过 DevTools 绕过；仅后端控制 → 用户体验差。

**理由**: 前端提供良好 UX（不显示无权限按钮），后端保障安全性。

### 7. 审计日志页路由归属：系统管理模块

**选择**: 审计日志页路由路径为 `/system/audit-log`，归入系统管理模块菜单。

**替代方案**: 归入内容社区模块 → 与内容社区混在一起，管理员不便查找。

**理由**: 审计日志是系统级管理功能，归入系统管理更符合管理员心智模型。

## Risks / Trade-offs

- **[ECharts 包体积]** → 使用 `echarts/core` 按需引入，监控打包体积，必要时启用动态 import
- **[私密内容安全性]** → 依赖后端正确过滤，前端不做数据层面的安全保障；需与后端明确接口契约
- **[邀请码防刷]** → 后端实现 IP 限制、冷却期等策略，前端仅做展示；需确认后端防刷能力
- **[互关状态缓存一致性]** → 取关/关注操作后需清除缓存，可能存在短暂不一致；通过操作后主动清除缓存缓解
- **[EPIC-09 依赖]** → 管理操作的"前往用户管理"跳转依赖 EPIC-09 用户管理页已实现；如 EPIC-09 未完成，该入口暂时隐藏
- **[粉丝画像降级]** → Tab 3 画像功能延后至二期，需在 UI 上做好降级提示，避免用户困惑
