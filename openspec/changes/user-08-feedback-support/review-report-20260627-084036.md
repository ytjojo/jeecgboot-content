## 三、问题详情

### FLAG 级问题 (8个)

**FLAG-01 [接口契约]**: 所有用户接口使用@RequestParam("userId")存在越权风险（CRITICAL级别）
- **位置**: ContentUserSupportController:68,78,89,101,118,129,139,168,199,209,219,228,240
- **描述**: 用户A可以传入userId=B查看/操作B的举报和申诉
- **影响**: 严重越权漏洞
- **修复建议**: 从SecurityContext获取当前用户ID，增加所有权校验

**FLAG-02 [接口契约]**: 管理端接口权限校验注解需确认
- **位置**: ContentUserSupportAdminController
- **描述**: 需要确认是否有@PreAuthorize或类似权限控制
- **建议**: 补充权限校验注解或AOP切面

**FLAG-03 [边界覆盖]**: 同一用户对同一目标重复举报24h限制的实现需验证并发安全
- **位置**: tasks.md 1.4
- **描述**: 防重复逻辑在并发场景下可能绕过
- **建议**: 使用数据库唯一索引或分布式锁保证

**FLAG-04 [边界覆盖]**: 用户30天内申诉3次的计数口径未明确
- **位置**: tasks.md 1.5
- **描述**: 30天是自然月还是滚动窗口未明确
- **建议**: 明确计数规则并补充边界测试

**FLAG-05 [API命名规范]**: /appeal/create路径风格不RESTful
- **位置**: tasks.md 2.1
- **描述**: 建议POST /appeals（资源复数形式）
- **建议**: 统一API路径风格，考虑资源复数

**FLAG-06 [API命名规范]**: /report/create同样风格问题
- **位置**: tasks.md 2.2
- **建议**: POST /reports

**FLAG-07 [跨端一致性]**: 前端配对change user-08-feedback-support-frontend完成率0/50
- **描述**: 后端完成但前端完全未开发，跨端联调存在风险
- **建议**: 协调前后端进度

**FLAG-08 [依赖分析]**: 申诉关联举报ID需要确认content_user_report表结构
- **位置**: tasks.md 1.6
- **建议**: 验证report_id外键约束或逻辑关联
