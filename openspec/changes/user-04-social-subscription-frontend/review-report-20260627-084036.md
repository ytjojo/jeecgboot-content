## 二、BLOCK 级别问题 (1个)

#### B-001: design.md中API路径前缀表述自相矛盾
- **位置**: `design.md:33`(Non-Goals第4条) vs `design.md:180,212`(API Endpoints)
- **严重级别**: BLOCK
- **描述**: Non-Goals第4条说"不迁移 `/api/v1/*` 路径到 `/content/user/*`（前端 PRD 已统一使用 `/api/v1/content/user/*` 路径）"，这句话语义混乱：
  - 前半句"不迁移 `/api/v1/*` 到 `/content/user/*`"暗示保留旧路径
  - 后半句"已统一使用 `/api/v1/content/user/*`"暗示使用新路径
  - 实际API Endpoints章节使用 `/api/v1/content/user/relation/*` 和 `/api/v1/content/user/subscription/*`，是正确的
- **影响**: 开发者阅读Non-Goals时会困惑到底用哪个前缀
- **修复建议**: 删除或改写Non-Goals第4条，明确写"所有社交API统一使用 `/api/v1/content/user/relation/*` 和 `/api/v1/content/user/subscription/*` 前缀"

## 三、FLAG 级别问题 (7个)

#### F-001: 分组创建/重命名/移动参数字段名与后端VO不一致
- **位置**: `design.md:282-294`(VO定义) vs 代码实际实现
- **严重级别**: FLAG
- **描述**: 
  - `ContentRelationGroupReq`定义字段`groupName`，代码使用`name`
  - `ContentRelationGroupMoveReq`定义字段`relationGroupId`，代码使用`groupId`
  - renameFollowGroup/removeFromGroup也用`groupId`而非`relationGroupId`
- **影响**: 前后端字段名不匹配会导致400参数校验错误
- **修复建议**: 统一以后端VO字段名为准：`groupName`、`relationGroupId`

#### F-002: 批量订阅操作使用sourceIds而非subscriptionIds
- **位置**: `design.md:310-313`(ContentSubscriptionBatchReq) vs 代码
- **严重级别**: FLAG
- **描述**: `ContentSubscriptionBatchReq`定义字段为`subscriptionIds: string[]`，代码使用`sourceIds: string[]`
- **影响**: 批量暂停/恢复/取消订阅时后端无法识别参数
- **修复建议**: 改为`subscriptionIds: string[]`

#### F-003: 通知偏好字段命名与后端VO完全不同
- **位置**: `design.md:318-324,461-468` vs 代码
- **严重级别**: FLAG
- **描述**: 
  - 后端VO：`notificationChannels: ('IN_APP'|'PUSH'|'EMAIL')[]`、`notificationFrequency`、`dndStartTime`、`dndEndTime`
  - 前端代码：`channelInApp: boolean`、`channelPush: boolean`、`channelEmail: boolean`、`frequency`、`quietStart`、`quietEnd`
- **影响**: 通知偏好保存和读取完全无法与后端对接
- **修复建议**: 前端请求体字段名统一以后端VO为准，可在组件内部用boolean做UI映射，但API层必须传数组格式

#### F-004: 分页参数命名不一致（page/size vs pageNo/pageSize）
- **位置**: `design.md:247-248`(分页策略) vs 代码
- **严重级别**: FLAG
- **描述**: design明确说"所有分页接口统一使用 `pageNo` + `pageSize` 参数"，但以下接口代码使用`page`/`size`：
  - getRecommendations: page/size
  - getFollowingFeed: page/size
  - getSubscribeFeed: page/size
  - getSubscribePlaza: page/size
- **影响**: 分页参数不匹配，后端无法正确解析分页请求
- **修复建议**: 所有分页接口统一使用`pageNo`/`pageSize`

#### F-005: 订阅请求缺少sourceName、notificationChannels、notificationFrequency字段
- **位置**: `design.md:299-306`(ContentSubscriptionReq) vs 代码subscribeSource/subscribeFromPlaza
- **严重级别**: FLAG
- **描述**: ContentSubscriptionReq定义了`sourceName`(必填)、`notificationChannels?`、`notificationFrequency?`，但代码只传`sourceId`和`sourceType`
- **影响**: 后端订阅接口因缺少必填字段sourceName返回400
- **修复建议**: subscribeSource和subscribeFromPlaza请求体中补充sourceName字段

#### F-006: 关注流/订阅流缺少userId参数和特别关注置顶结构
- **位置**: `design.md:71-76`(D3决策) vs 代码getFollowingFeed/getSubscribeFeed
- **严重级别**: FLAG
- **描述**: 
  - getFollowingFeed/getSubscribeFeed的params中无userId参数（查询自己的流需要）
  - D3决策明确说后端返回`{ priorityItems: [...], items: [...] }`结构，但代码未标注返回类型，无法验证是否处理了置顶分区
- **影响**: 无法正确查询指定用户的feed流；特别关注置顶功能可能无法正确渲染
- **修复建议**: 添加userId参数；定义FeedResponse类型包含priorityItems和items

#### F-007: File Structure中stores路径与项目实际不符
- **位置**: `design.md:552-555`
- **严重级别**: FLAG
- **描述**: File Structure写`stores/follow.ts`、`stores/subscribe.ts`、`stores/feed.ts`，但项目实际Pinia store目录为`src/store/modules/`（非stores/）
- **影响**: 按文档创建文件会放错位置
- **修复建议**: 更正为`src/store/modules/social/`或`src/store/modules/follow.ts`等正确路径

## 五、审核结论

**整体评价**: C+ (71/100) — design.md文档非常详尽（API Endpoints、VO字段、File Structure、Test Strategy质量很高），但代码实现与design/后端VO存在大量字段名不匹配问题，是5个change中接口契约一致性最差的。

**关键问题**:
1. 通知偏好字段命名完全不同（BLOCK级别功能问题）
2. 分组CRUD参数名与后端VO不匹配
3. 批量订阅参数名错误（sourceIds vs subscriptionIds）
4. 分页参数page/size与design规定的pageNo/pageSize不一致
5. 订阅请求缺少必填的sourceName字段

**亮点**: design.md的完整性和详细程度是5个change中最好的，API Endpoints表格、VO/Req TypeScript定义、Test Strategy、边界场景覆盖都非常出色。问题主要出在代码实现未严格遵循这些定义。

**阻塞项**: B-001(文档矛盾)需澄清；多个FLAG级字段名问题阻塞前后端联调。
