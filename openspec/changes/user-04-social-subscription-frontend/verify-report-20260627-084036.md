## 二、CRITICAL 级别问题 (2个)

#### C-001: 社交模块页面路由完全未注册
- **位置**: `router/routes/modules/contentSocial.ts`
- **严重级别**: CRITICAL
- **描述**: contentSocial.ts路由文件仅注册了4个路由（/content/mutual-follow、/content/fan、/content/invite、/invite/:inviteCode），**完全没有注册**以下9个社交模块核心页面：
  - /social/follow（关注列表）
  - /social/follow/special（特别关注）
  - /social/follow/group（分组管理）
  - /social/follow/recommend（关注推荐）
  - /social/feed（关注流）
  - /social/subscribe/feed（订阅流）
  - /social/subscribe/square（订阅广场）
  - /social/subscribe/detail（订阅源详情）
  - /social/subscribe/manage（订阅管理）
  - /social/subscribe/notification（通知配置）
- **影响**: 所有社交页面无法通过URL访问，用户无法进入关注列表、关注流、订阅广场等任何核心功能页面。这是阻塞级问题
- **修复建议**: 在contentSocial.ts中添加所有social页面路由，path建议使用`/content/social/*`前缀与其他content模块保持一致，组件指向`/src/views/social/*`

#### C-002: 通知偏好API字段名与后端VO完全不匹配
- **位置**: `api/content/subscribe.ts:72-76`
- **严重级别**: CRITICAL
- **描述**: saveNotificationPreference发送的字段为`channelInApp/channelPush/channelEmail/frequency/quietStart/quietEnd`，但后端VO定义为`notificationChannels: string[]/notificationFrequency/dndStartTime/dndEndTime`。字段名和数据结构完全不同
- **影响**: 通知偏好保存和读取功能完全无法工作，后端无法识别请求参数
- **修复建议**: 
  ```typescript
  // API层应传：
  data: {
    notificationChannels: string[],  // ['IN_APP', 'PUSH', 'EMAIL']
    notificationFrequency: 'REALTIME' | 'DAILY',
    dndStartTime?: string,  // 'HH:mm'
    dndEndTime?: string,
  }
  // 组件内部可使用boolean做UI映射，但API层必须转换为后端VO格式
  ```

## 三、WARNING 级别问题 (6个)

#### W-001: 分组操作字段名与后端VO不匹配
- **位置**: `api/content/relation.ts:58-75`
- **严重级别**: WARNING
- **描述**: 
  - createFollowGroup传`name`应为`groupName`
  - renameFollowGroup/moveFollowGroup/removeFromGroup传`groupId`应为`relationGroupId`
- **影响**: 分组创建、重命名、移动功能后端无法正确接收参数
- **修复建议**: 统一以后端VO字段名为准

#### W-002: 批量订阅参数名错误
- **位置**: `api/content/subscribe.ts:56-65`
- **严重级别**: WARNING
- **描述**: batchPauseSubscribe/batchResumeSubscribe/batchCancelSubscribe传`sourceIds: string[]`，后端VO定义为`subscriptionIds: string[]`
- **影响**: 批量订阅操作后端无法识别参数
- **修复建议**: 改为`subscriptionIds: string[]`

#### W-003: 分页参数不一致
- **位置**: `api/content/relation.ts:86,98`、`api/content/subscribe.ts:40,44`
- **严重级别**: WARNING
- **描述**: getRecommendations、getFollowingFeed、getSubscribeFeed、getSubscribePlaza使用`page/size`分页参数，design规定统一使用`pageNo/pageSize`
- **影响**: 分页查询可能无法正确翻页
- **修复建议**: 统一为`pageNo/pageSize`

#### W-004: 订阅请求缺少必填字段sourceName
- **位置**: `api/content/subscribe.ts:20,52`
- **严重级别**: WARNING
- **描述**: subscribeSource和subscribeFromPlaza只传sourceId和sourceType，缺少必填字段sourceName
- **影响**: 订阅请求可能因缺少必填字段被后端拒绝
- **修复建议**: data中补充`sourceName: string`字段

#### W-005: Feed接口缺少userId参数
- **位置**: `api/content/relation.ts:98`、`api/content/subscribe.ts:40`
- **严重级别**: WARNING
- **描述**: getFollowingFeed和getSubscribeFeed参数中无userId
- **影响**: 无法正确查询指定用户的feed流；查看他人的feed流场景无法支持
- **修复建议**: 添加`userId?: string`参数，不传时默认查当前用户

#### W-006: API函数缺少TypeScript泛型返回类型
- **位置**: `api/content/relation.ts`和`api/content/subscribe.ts`大部分函数
- **严重级别**: WARNING
- **描述**: 除getMutualStatus外，所有API函数均未标注`defHttp.get<Type>`或`defHttp.post<Type>`泛型返回类型
- **影响**: TypeScript类型安全丢失，IDE无法提供自动补全，调用方无法获得类型提示
- **修复建议**: 为所有API函数添加正确的泛型返回类型，参考design.md中定义的VO接口
