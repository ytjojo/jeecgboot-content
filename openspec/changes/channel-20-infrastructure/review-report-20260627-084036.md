## 三、问题详情

### BLOCK 级问题 (6个)

**BLOCK-01 [一致性]**: plan.md SQL脚本主键类型与实际JeecgBoot框架不匹配
- **位置**: plan.md 第29行 (SQL DDL) vs jeecg-boot-base-core/JeecgEntity.java
- **描述**: plan.md定义`id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT`，但实际JeecgBoot使用`ASSIGN_ID`(雪花算法)，主键类型为VARCHAR(32)
- **影响**: 按plan.md实现会与JeecgBoot框架主键策略冲突，导致插入失败
- **修复建议**: 更新plan.md中的SQL为VARCHAR(32)主键，使用JeecgEntity基类

**BLOCK-02 [接口契约]**: design.md未定义任何具体API路径和请求/响应结构
- **位置**: design.md 第88-142行 (File Structure章节)
- **描述**: design.md仅列出了Controller文件名，但未定义具体API端点、HTTP方法、请求参数、响应VO结构、错误码
- **影响**: 前后端无法对齐接口，Controller实现缺乏契约约束
- **修复建议**: 在design.md中补充API契约表格，包含路径、方法、请求体、响应体、错误码

**BLOCK-03 [边界覆盖]**: 缺少null/空值/超长字符串/非法格式输入的边界Scenario
- **位置**: specs/channel-creation/spec.md
- **描述**: Scenario未覆盖name=null/空串/超过100字符、description超过500字符、iconUrl/coverUrl非法URL格式等
- **影响**: 恶意或误操作输入可能导致数据库异常或XSS风险
- **修复建议**: 补充输入校验相关Scenario

**BLOCK-04 [边界覆盖]**: 缺少并发创建频道的竞态条件处理
- **位置**: specs/channel-creation/spec.md
- **描述**: 未描述两个用户同时创建同名频道时的并发控制策略（数据库唯一约束 vs 分布式锁 vs 乐观锁）
- **影响**: 高并发下可能绕过名称唯一性校验，产生重名频道
- **修复建议**: 明确并发控制策略，补充Scenario

**BLOCK-05 [数据表审核]**: SQL脚本缺少del_flag软删除字段，但spec要求支持软删除
- **位置**: V_channel_infrastructure.sql (plan.md 第28-52行)
- **描述**: spec要求"支持软删除（del_flag）"，但SQL脚本三张表均无del_flag字段
- **影响**: 无法实现软删除，删除操作直接物理删除，违反数据保留要求
- **修复建议**: 添加del_flag TINYINT DEFAULT 0字段，或说明使用JeecgBoot租户隔离机制

**BLOCK-06 [数据表审核]**: ChannelReview实体@TableName与SQL表名不一致
- **位置**: entity/ChannelReview.java 第14行 `@TableName("channel_review")` vs SQL `content_channel_review`
- **描述**: 实体映射表名为"channel_review"，但Flyway脚本创建的表为"content_channel_review"
- **影响**: 运行时SQL报错，表不存在
- **修复建议**: 修正@TableName注解为"content_channel_review"

### FLAG 级问题 (15个)

**FLAG-01 [完整性]**: proposal.md缺少错误码定义和鉴权矩阵
- **位置**: proposal.md 全文
- **描述**: 未定义错误码区间、未明确哪些接口需要管理员权限、哪些需要频道主权限
- **建议**: 补充错误码规范和权限矩阵

**FLAG-02 [一致性]**: plan.md使用LocalDateTime，但实际代码使用java.util.Date
- **位置**: plan.md 多处示例代码 vs entity/Channel.java
- **描述**: plan.md示例使用LocalDateTime，JeecgEntity使用Date类型
- **建议**: 统一时间类型，或在plan.md中说明适配JeecgBoot基类

**FLAG-03 [一致性]**: plan.md中ownerId/categoryId/organizationId为Long类型，实际为String
- **位置**: plan.md 实体定义 vs entity/Channel.java
- **描述**: plan.md定义ID字段为Long，实际因雪花算法使用String类型
- **建议**: 更新plan.md中所有ID类型为String

**FLAG-04 [可实现性]**: design.md D3提到使用Redis缓存已占用名称，但未定义缓存key和失效策略
- **位置**: design.md 第58行
- **描述**: 提及Redis但无具体缓存设计（key格式、TTL、更新/失效策略）
- **建议**: 补充Redis缓存设计细节

**FLAG-05 [可实现性]**: "内容安全审核服务"接口未定义，创建个人/组织频道时调用方式不明
- **位置**: design.md Open Questions #1
- **描述**: Open Question未关闭即进入实现，内容安全审核是同步/异步、接口协议不明
- **建议**: 明确审核服务接口，或先Mock实现

**FLAG-06 [可测试性]**: 边界条件Scenario不足，TDD测试无法覆盖负面路径
- **位置**: specs/ 各spec.md
- **描述**: Scenario以Happy Path为主，负面路径覆盖不足
- **建议**: 每个Requirement补充至少2-3个异常Scenario

**FLAG-07 [API命名规范]**: Controller中使用/create、/cancel-delete等RPC风格路径
- **位置**: ChannelController.java 第57行 `/create`、第181行 `/cancel-delete`
- **描述**: RESTful规范下创建应使用POST /，撤销删除应使用POST /{id}/delete:cancel
- **建议**: 统一为RESTful风格，使用资源子路径表达动作

**FLAG-08 [API命名规范]**: 更新操作使用@PutMapping但AGENTS.md要求所有写操作用POST
- **位置**: ChannelController.java 第141行 `@PutMapping("/{id}")`
- **描述**: 项目AGENTS.md硬规则："All writes use POST (save/update)"
- **建议**: 改为@PostMapping("/update")或遵循项目规范

**FLAG-09 [API命名规范]**: 删除操作使用@DeleteMapping但AGENTS.md要求DELETE用于逗号分隔ID批量删除
- **位置**: ChannelController.java 第173行 `@DeleteMapping("/{id}")`
- **描述**: 项目规范删除使用DELETE + 逗号分隔IDs，但这里是单资源删除且包含冷静期逻辑
- **建议**: 统一为POST /{id}/delete-request

**FLAG-10 [跨端一致性]**: plan.md的DTO/VO结构与前端change未对齐字段命名
- **位置**: design.md File Structure vs channel-20-infrastructure-frontend/
- **描述**: 后端pinWeight vs 前端pin_weight，iconUrl vs icon_url等命名风格需确认
- **建议**: 前后端对齐字段命名，后端使用camelCase，JSON序列化自动转snake_case

**FLAG-11 [依赖分析]**: ChannelReview实体字段与plan.md定义差异大（多了reviewId/reviewType/status/applicantId等）
- **位置**: entity/ChannelReview.java vs plan.md
- **描述**: 实际ChannelReview有reviewId、reviewType、applicantId、submitTime、reviewTime等字段，是EPIC-24审核流程扩展的结果，但未同步到EPIC-20文档
- **建议**: 同步更新EPIC-20的design.md和specs以反映实际实体结构

**FLAG-12 [Plan审核]**: Plan分层顺序不完全符合Entity→Mapper→Service→Biz→Controller
- **位置**: plan.md 整体
- **描述**: Task 6开始Biz层，但此时Controller DTO（CreateChannelDTO）在Task 6 Step 3才创建，顺序略乱
- **建议**: 调整Plan顺序，确保底层依赖先创建

**FLAG-13 [Plan审核]**: Plan中测试使用@MybatisPlusTest但项目可能未引入mybatis-plus-test-autoconfigure
- **位置**: plan.md 第343行
- **描述**: 需确认测试依赖是否可用
- **建议**: 验证pom.xml中测试依赖

**FLAG-14 [可实现性]**: organizationId在个人频道创建时未强制为null
- **位置**: ChannelBizManageService.java createPersonalChannel方法
- **描述**: 个人频道创建未清理organizationId字段，可能导致个人频道意外绑定组织
- **建议**: 在buildChannelFromDTO中根据channelType设置差异化字段

**FLAG-15 [边界覆盖]**: 名称唯一性校验未覆盖ReadonlyFrozen/Hidden/Archived状态
- **位置**: ChannelServiceImpl.java NAME_OCCUPIED_STATUSES
- **描述**: spec要求校验范围包含ReadonlyFrozen/Hidden/Archived，但代码仅校验PENDING_REVIEW/ACTIVE/DELETE_COOLING
- **影响**: 冻结/隐藏/归档频道的名称可能被占用
- **建议**: 补充状态到校验列表（需确认这些状态是否在EPIC-24定义）
