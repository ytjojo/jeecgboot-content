## Context

内容社区模块位于 `jeecg-boot/jeecg-boot-module/jeecg-module-content/`，后端遵循 `controller / biz / service / mapper / entity / req / vo / dto` 分层，接口统一返回 `Result<T>`，写操作使用 POST，删除使用 DELETE。当前变更需要在内容社区内提供完整用户认证与账号安全能力，并复用 JeecgBoot 现有 Spring Boot 3、MyBatis-Plus、Redis、Flyway、Spring Authorization Server 基础设施。

认证能力涉及高安全风险和多外部通道：短信、邮件、第三方 OAuth、IP 地理位置、人机验证。实现必须保证验证码、密码、会话、风控、注销状态都有可审计记录，并且在数据库迁移、Redis 失效、第三方服务失败时显式降级或失败。

## Goals / Non-Goals

**Goals:**

- 提供手机号、邮箱、第三方账号的注册登录能力，并统一产生内容社区用户会话。
- 提供设备会话管理、Token 黑名单、设备信任、异常登录提醒。
- 提供手机号、邮箱、第三方账号的绑定、换绑、解绑，并保证账号仍有可用登录方式。
- 提供密码重置、密码历史校验、高风险找回补充校验。
- 提供登录失败、批量注册等基础风控规则和审计日志。
- 提供账号注销申请、冷静期取消、最终匿名化或删除处理。
- 使用 Flyway 管理表结构变更，并提供可执行的回滚方案。

**Non-Goals:**

- 不实现生物识别登录和两步验证。
- 不实现企业组织通讯录集成。
- 不建设独立机器学习训练平台；本期风控使用规则引擎和预留模型评分字段。
- 不重构 JeecgBoot 系统级 OAuth2 基础设施；仅在必要位置通过适配层对接。

## Decisions

### 1. 在内容社区模块内新增认证子域

新增 `org.jeecg.modules.content.auth` 包，按既有模块规范拆分：

- `controller`: `ContentAuthController`、`ContentAccountSecurityController`、`ContentAccountCancellationController`
- `biz`: `ContentAuthBizService`、`ContentAccountSecurityBizService`、`ContentRiskBizService`、`ContentAccountCancellationBizService`
- `service`: 单表服务，如用户、凭证、第三方绑定、设备会话、风险事件、注销申请
- `req / vo / dto`: 按接口入参、响应、内部流转拆分

理由：认证是内容社区的基础能力，但不应直接改造系统模块。通过内容社区子域实现业务策略，再以适配层复用已有认证基础设施。

替代方案：直接修改 `jeecg-module-system` 的用户与授权逻辑。放弃原因是影响范围过大，容易破坏后台管理和平台级登录。

### 2. 用户主体与登录凭证分离

数据库采用用户主体表加多凭证表的结构：

- `content_user_account`: 内容社区账号主体，保存账号状态、注销状态、基础安全状态。
- `content_user_credential`: 手机号、邮箱、密码哈希等登录凭证，唯一约束保证手机号/邮箱只能绑定一个账号。
- `content_user_social_binding`: 微信、Apple、Google 等第三方 provider 绑定。
- `content_user_password_history`: 最近密码哈希历史，用于禁止复用最近 3 次密码。

理由：手机号、邮箱、第三方账号都可能绑定/解绑，分离后能避免在用户主表堆积大量可空字段，也便于唯一约束和审计。

替代方案：把所有凭证字段放在用户表。放弃原因是换绑、解绑、第三方扩展和历史密码校验会变得脆弱。

### 3. 验证码和短期计数使用 Redis，安全结果落库

Redis 保存短期状态：

- `content:auth:code:{scene}:{target}`，TTL 按场景设置，短信验证码 300 秒。
- `content:auth:code:fail:{scene}:{target}`，记录验证码失败次数。
- `content:auth:pwd_fail:{accountId}`，记录密码错误次数，TTL 900 秒。
- `content:auth:lock:{scope}:{value}`，记录临时锁定。
- `content:auth:token_blacklist:{jti}`，记录被下线或注销的 token。
- `content:risk:counter:{scope}:{value}`，记录 IP、设备、账号维度的风控计数。

长期事实落库到风险事件、设备会话、注销申请等表。理由是 Redis 适合高频短期判断，数据库负责审计和最终一致性。

替代方案：全部落库。放弃原因是验证码、失败计数和风控计数写频高，会增加数据库压力。

### 4. 会话以设备会话表为准，Token 黑名单即时失效

新增 `content_user_device_session` 记录设备类型、操作系统、IP、地理位置、设备指纹、最后登录时间、是否信任、会话状态。主动下线、异常登录否认、账号注销时，将对应会话置为失效，并把 token jti 写入 Redis 黑名单。

理由：数据库可展示设备列表和审计状态，Redis 黑名单可快速拦截已签发但被撤销的 Token。

替代方案：只依赖 JWT 过期时间。放弃原因是无法满足主动下线和异常设备立即失效。

### 5. 风控先做规则引擎，预留扩展字段

本期实现确定性规则：登录失败次数、同 IP 注册数量、设备指纹异常、验证码失败次数、冷却时间、账号锁定。风险事件表保留 `risk_score`、`risk_reason`、`decision` 字段，便于后续接入模型评分。

理由：PRD 的核心验收可以用规则满足，且比一开始引入机器学习模型更可控。

替代方案：直接集成模型评分服务。放弃原因是缺少训练数据和评估基线，容易影响正常用户。

### 6. 外部服务通过接口适配，业务层只依赖抽象

定义短信、邮件、第三方 OAuth、IP 地理位置、人机验证的适配接口，由配置选择实际实现。业务层只处理发送结果、验证结果和失败原因。

理由：外部服务不稳定，抽象接口便于测试、替换供应商和做降级。

替代方案：在业务服务里直接调用 SDK。放弃原因是测试困难且供应商切换成本高。

### 7. 注销采用状态机和定时任务

账号注销状态为 `ACTIVE -> CANCELLING -> CANCELLED`，申请时检查未完成事项，冷静期内允许登录后取消，冷静期结束由定时任务执行最终匿名化或删除，并保留审计日志 3 年。

理由：状态机能避免重复申请、并发取消和最终注销冲突。

替代方案：申请时立即物理删除。放弃原因是不满足冷静期和误操作恢复要求。

## Risks / Trade-offs

- [短信或邮件服务失败] → 通过适配层返回明确失败原因，记录发送日志，预留备用通道配置。
- [第三方 OAuth 平台返回字段变更] → Provider 解析集中在适配层，绑定表保存 provider、openId、unionId 和原始摘要，避免污染业务层。
- [风控误判影响正常用户] → 风控决策默认可解释，支持人机验证解锁、申诉白名单和风险事件复盘。
- [Token 黑名单 Redis 丢失] → 设备会话状态落库；Redis 恢复后可按失效会话重建短期黑名单，敏感操作同时校验会话状态。
- [账号注销误删数据] → 冷静期内仅打标，不物理删除；最终执行前再次校验账号状态和冷静期截止时间。
- [并发绑定/解绑导致最后登录方式被移除] → 绑定相关写操作使用事务和唯一约束，解绑前在同一事务内重新计算可用登录方式数量。

## Migration Plan

1. 新增 Flyway migration，创建用户认证相关表、唯一索引和审计字段。
2. 先发布实体、Mapper、Service 和空路由保护，不开放入口。
3. 发布注册登录、验证码、密码登录和第三方登录接口。
4. 发布设备会话、绑定解绑、密码找回、异常登录、风控、注销接口。
5. 配置短信、邮件、第三方 OAuth、人机验证、IP 地理位置服务参数。
6. 灰度开放注册登录入口，监控登录成功率、验证码发送成功率、风控误判率和异常登录通知延迟。

**Rollback strategy:**

- 应用回滚：回滚到上一版本后，新表不影响旧代码；关闭新入口和外部回调地址。
- 数据回滚：Flyway 回滚脚本按创建顺序反向删除新增索引和新增表。若已有线上数据，先导出认证相关表备份，再执行删除或保留空表。
- Redis 回滚：删除 `content:auth:*` 与 `content:risk:*` 相关键，不影响旧系统会话。
- 配置回滚：撤销短信、邮件、OAuth、人机验证相关新增配置或置为 disabled。

## Open Questions

- 内容社区账号是否需要与 Jeecg 系统用户 `sys_user` 建立一对一映射，还是独立账号体系仅在内容社区内使用？
- 第三方登录在 Web、移动端、小程序的 provider 配置是否共用同一套 App ID？
- 账号注销最终处理采用匿名化优先还是物理删除优先，需要结合合规和业务审计要求确认。
- 高风险找回中的“安全问题或身份证明”具体产品形态尚未确定，本设计预留补充校验接口和人工审核状态。
