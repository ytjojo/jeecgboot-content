## Why

内容社区当前缺少清晰的拉黑、屏蔽和屏蔽规则能力，用户遇到骚扰、冲突或信息噪音时无法有效管理个人空间。
EPIC-05 要求补齐“拉黑是双向切断、屏蔽是单向降噪”的产品边界，并让信息流、主页、互动和通知在同一套关系规则下保持一致。

## What Changes

- 新增拉黑能力：用户可拉黑、解除拉黑和查看黑名单；拉黑后双方内容、主页、评论、动态和互动能力双向隔离。
- 新增拉黑后的关系处理：已有互关自动解除，评论、点赞、私信、@ 通知等互动统一拦截，且被拉黑方只看到模糊失败提示。
- 新增屏蔽用户动态能力：用户可屏蔽某用户在推荐流和关注流中的内容，但不影响访问主页、关注关系和对方访问自己内容。
- 新增“不感兴趣”与规则屏蔽：记录反馈，并支持按内容类型、话题/Tag、临时话题和屏蔽词过滤信息流。
- 新增黑名单和屏蔽列表管理：黑名单、屏蔽用户、屏蔽话题、屏蔽类型、屏蔽词独立管理，并支持恢复或取消屏蔽。
- 新增操作说明和确认文案：在拉黑、屏蔽和解除操作中明确行为边界，减少误用和纠纷。
- 不包含内容审核系统、自动风控拦截、反小号骚扰识别和推荐算法完整重构。

## Capabilities

### New Capabilities

- `blocking-muting`: 内容社区降噪与内容保护能力，覆盖拉黑双向切断、屏蔽单向降噪、不感兴趣反馈、屏蔽规则、列表管理和行为边界说明。

### Modified Capabilities

- 无。

## Impact

- 影响内容社区后端模块 `jeecg-boot/jeecg-boot-module/jeecg-module-content`，重点涉及用户关系、信息流查询、主页查询、互动校验、通知触发和用户设置。
- 需要新增拉黑关系表、屏蔽关系表、屏蔽规则表、不感兴趣反馈表，并同步 entity、mapper、Flyway SQL 和相关测试。
- 需要在评论、点赞、私信、@ 通知、关注关系、推荐流和关注流查询前统一调用关系边界判断。
- 需要新增或扩展隐私设置相关 API，提供黑名单、屏蔽列表、屏蔽词、屏蔽话题/类型和临时屏蔽管理接口。
- 需要前端在用户主页、内容卡片、评论区、隐私设置和帮助说明中展示拉黑/屏蔽入口、确认框和状态文案。
- 依赖 EPIC-01 的注册登录用户身份；与 EPIC-04 关注关系、EPIC-09 通知设置存在行为联动。
