## Context

EPIC-05 面向内容社区的用户保护与信息流降噪。现有内容社区模块按 `controller / biz / service / mapper / entity / req / vo / dto` 分层，跨聚合编排应放在 `biz`，单表读写留在 `service`。本变更会同时影响用户关系、内容流过滤、主页可见性、评论点赞私信等互动边界和通知触发，因此需要在实现前明确统一的关系判断入口。

本设计默认用户身份来自 EPIC-01 已登录主体；若实现时 EPIC-01 尚未完全落地，则先复用现有内容社区用户 ID 获取方式。拉黑优先级高于屏蔽和关注关系：一旦双方任意一方存在拉黑关系，相关可见性与互动均按双向切断处理。

## Goals / Non-Goals

**Goals:**

- 建立拉黑、屏蔽、不感兴趣和屏蔽规则的持久化模型。
- 在信息流、主页、评论、点赞、私信、@ 通知和关注关系中统一执行拉黑/屏蔽边界。
- 支持黑名单、屏蔽列表、屏蔽词、话题/类型屏蔽和临时屏蔽的查询与恢复。
- 保证拉黑操作静默，不向被拉黑用户暴露具体原因。
- 提供清晰的操作确认与解除提示，帮助用户理解“屏蔽是单向降噪、拉黑是双向切断”。
- 使用 Flyway 管理表结构变更，并提供回滚方案。

**Non-Goals:**

- 不实现内容审核系统、自动风控拦截和反小号骚扰识别。
- 不重构完整推荐算法，只提供规则过滤和“不感兴趣”反馈记录。
- 不自动恢复解除拉黑前的关注关系。
- 不实现管理后台的批量治理能力。

## Decisions

### 1. 关系事实分表存储

新增独立表保存不同生命周期的关系事实：

- `content_user_block`: 拉黑关系，包含 `user_id`、`blocked_user_id`、`block_time`、`status`、审计字段和唯一索引。
- `content_user_mute`: 屏蔽用户关系，包含 `user_id`、`muted_user_id`、`mute_time`、`status`。
- `content_user_filter_rule`: 屏蔽规则，按 `WORD`、`REGEX`、`TOPIC`、`CONTENT_TYPE` 存储，支持 `expires_at` 表达临时屏蔽。
- `content_user_not_interested`: 不感兴趣反馈，记录 `user_id`、`content_id`、`content_type`、`topic`、`reason`。

理由：拉黑、屏蔽、过滤规则和反馈的业务语义不同，分表更容易维护唯一约束、过期清理和查询索引。

替代方案：使用单张用户偏好 JSON 表。放弃原因是难以做唯一约束、批量取消、TTL 清理和高频信息流过滤。

### 2. 拉黑判断集中在 biz 编排层

新增关系边界编排服务，提供 `isBlockedEitherWay`、`assertCanInteract`、`filterVisibleAuthorIds` 等方法。Controller 不直接判断拉黑，评论、点赞、私信、@ 通知、主页和信息流调用该服务完成统一裁决。

理由：拉黑跨内容、关系、互动和通知多个聚合，属于跨领域编排；集中判断可以避免不同入口提示和行为不一致。

替代方案：各业务 service 自行查询拉黑表。放弃原因是容易漏掉入口，也不利于模糊错误提示统一。

### 3. 拉黑写入与解除关注放入同一事务

拉黑用户时先幂等写入拉黑关系，再解除双方关注关系，最后清理相关缓存。解除拉黑只移除拉黑关系，不恢复关注关系，并返回需要手动重新关注的提示。

理由：EPIC-05 明确要求拉黑后已有互关自动解除，解除拉黑不自动恢复关注关系。

替代方案：拉黑只保存状态，不处理关注关系。放弃原因是关注列表会继续暴露关系状态，与“双向切断”不一致。

### 4. 信息流过滤按不同规则分层处理

推荐流和关注流查询需要排除：

1. 当前用户拉黑或被拉黑的作者。
2. 当前用户屏蔽的作者。
3. 当前用户屏蔽的话题、内容类型和未过期临时规则。
4. 命中屏蔽词或正则的内容。

主页主动访问只受拉黑影响，不受屏蔽用户关系影响。屏蔽词命中内容默认折叠或隐藏，返回可展开状态由接口响应字段表达。

理由：屏蔽是单向降噪，不应改变主页访问和关注关系；拉黑才是双向切断。

替代方案：把屏蔽也应用到主页访问。放弃原因是违背 EPIC-05 对屏蔽的单向边界定义。

### 5. 静默与模糊提示作为安全默认

被拉黑用户访问主页时返回“不存在”或空白等模糊结果；互动失败统一提示“操作失败”或对应业务的通用失败文案，不返回“你已被拉黑”。拉黑和解除拉黑均不发送通知。

理由：减少对抗和隐私暴露，符合 EPIC-05 的静默拉黑要求。

### 6. 屏蔽词正则先受限实现

支持关键词和正则规则，但正则规则必须限制长度和数量，并在保存时预编译校验。信息流过滤先采用数据库候选内容 + 服务层匹配；若性能不足，再引入 Aho-Corasick 或专门的规则索引。

理由：先满足可测试的产品需求，避免为低优先级屏蔽词提前引入复杂依赖。

## Risks / Trade-offs

- [多个入口遗漏拉黑校验] → 以关系边界编排服务作为唯一调用入口，任务中覆盖评论、点赞、私信、@、主页和信息流测试。
- [信息流过滤影响查询性能] → 拉黑和屏蔽用户使用索引快速排除，话题/类型规则优先下推查询，屏蔽词和正则只对候选内容执行。
- [正则规则造成性能风险] → 限制规则长度、数量和保存时校验；匹配超时或异常时降级为不应用该规则并记录告警。
- [拉黑解除关注与已有关注逻辑冲突] → 拉黑事务中调用关注领域服务解除双方关系，并以集成测试固定行为。
- [静默提示导致用户困惑] → 对发起拉黑者提供清晰确认和状态说明，对被拉黑者保持模糊提示。
- [缓存导致可见性延迟] → 拉黑、解除拉黑、屏蔽和取消屏蔽后清理关系缓存与信息流缓存，新请求必须重新评估关系。

## Migration Plan

1. 新增 Flyway migration，创建拉黑、屏蔽、屏蔽规则和不感兴趣反馈表及唯一索引。
2. 新增 entity、mapper、service 基础类，保持新增表字段允许安全默认值。
3. 新增关系边界 biz 服务，先接入主页和信息流查询，再接入互动和通知入口。
4. 新增黑名单、屏蔽列表和屏蔽规则 API，并补充前端调用所需 VO。
5. 发布后观察拉黑/屏蔽操作耗时、信息流过滤耗时、互动拦截次数和失败率。

**Rollback strategy:**

- 应用回滚：回滚旧版本后，旧代码不会读取新增表；新增表保留不影响旧流程。
- 数据回滚：如需撤销 migration，先导出新增表数据，再按 `content_user_not_interested`、`content_user_filter_rule`、`content_user_mute`、`content_user_block` 顺序删除索引和表。
- 关系回滚：若拉黑事务已解除关注，回滚应用不会自动恢复关注关系；需要业务确认后通过导出的操作日志人工恢复。
- 缓存回滚：删除 `content:user:block:*`、`content:user:mute:*`、`content:feed:filter:*` 等关系与信息流缓存键。

## Open Questions

- 私信模块是否已经在内容社区内实现，还是需要预留外部消息服务适配点？
- 被拉黑用户访问主页时最终采用“不存在”还是空白页面，需要前端产品文案确认。
- 屏蔽词命中内容默认隐藏还是折叠展示，本设计按“可由接口表达折叠/隐藏状态”预留。
- 不感兴趣反馈是否需要同步给独立推荐服务，本期先记录数据并在本地查询中降噪。
