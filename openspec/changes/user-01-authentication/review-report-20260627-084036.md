## 三、问题详情

### BLOCK 级问题 (8个)

**BLOCK-01 [Plan审核]**: plan.md文件完全缺失
- **位置**: openspec/changes/user-01-authentication/ 目录
- **描述**: 其他change均有plan.md指导TDD实现，但本change仅tasks.md标记完成状态，缺少按分层顺序的实现计划、测试计划、代码结构说明
- **影响**: 无法追溯实现顺序是否遵循Entity→Mapper→Service→Biz→Controller的TDD流程，缺少代码审查的参照基线
- **修复建议**: 补充plan.md，至少包含：包结构说明、实体映射表、API端点清单、分层实现顺序、测试策略

**BLOCK-02 [一致性]**: design.md定义的表名与实际Flyway脚本不一致
- **位置**: design.md 第47行 vs V3.9.1_61__content_auth_foundation.sql
- **描述**: design.md定义`content_user_social_binding`表用于第三方绑定，但实际使用`content_user_third_party_auth`表（通过ALTER TABLE扩展字段）
- **影响**: 按design.md实现会找不到对应表，或重复建表
- **修复建议**: 更新design.md中第三方绑定的表名和实体说明，或确认使用扩展现有表的方案

**BLOCK-03 [数据表审核]**: ContentUserAccount实体riskLevel字段类型与SQL定义不匹配
- **位置**: entity/ContentUserAccount.java:58 `private String riskLevel;` vs V3.9.1_61.sql:16 `risk_level int NOT NULL DEFAULT 0`
- **描述**: 实体中riskLevel是String类型，但SQL中是int类型(0-100)，运行时MyBatis类型映射会报错
- **影响**: 查询/插入时类型转换错误，导致账号风险等级功能无法使用
- **修复建议**: 将实体riskLevel改为Integer类型，或更新SQL为VARCHAR类型（需与风控规则对齐）

**BLOCK-04 [接口契约]**: design.md未定义具体API请求/响应结构
- **位置**: design.md 全文
- **描述**: design.md列出了Controller类名，但未定义每个API的请求Req类字段、响应VO结构、错误码、鉴权规则
- **影响**: 前后端无法对齐接口，Controller实现缺乏契约约束
- **修复建议**: 在design.md中补充API契约表格，包含路径、方法、请求体字段、响应体字段、错误码、权限要求

**BLOCK-05 [接口契约]**: 缺少统一错误码定义
- **位置**: proposal.md + design.md
- **描述**: proposal.md提到"新增认证相关接口"但未定义错误码区间（如AUTH_001=验证码错误，AUTH_002=账号锁定等）
- **影响**: 前端无法根据错误码做差异化处理，错误提示不统一
- **修复建议**: 定义AuthErrorCodeConstant，明确错误码区间和含义

**BLOCK-06 [依赖分析]**: 外部服务适配接口定义不完整
- **位置**: design.md Decision 6
- **描述**: 提到短信、邮件、第三方OAuth、IP地理位置、人机验证的适配接口，但未定义接口方法签名、参数、返回值、降级策略
- **影响**: 外部服务集成缺乏契约，Mock实现与真实实现可能不一致
- **修复建议**: 补充每个Port接口的方法定义，明确失败时的降级策略

**BLOCK-07 [数据表审核]**: content_user_device_session设备会话表缺少关键字段
- **位置**: V3.9.1_50.sql:278-295 (原始表) vs V3.9.1_62.sql:4-13 (扩展字段)
- **描述**: design.md要求记录设备指纹、操作系统、地理位置、是否信任，但V50原始表字段不足，依赖V62 ALTER TABLE添加。两脚本版本跨度大（V50到V61/V62），中间缺少对应关系说明
- **影响**: 数据库迁移时如果顺序执行错误可能导致字段缺失
- **修复建议**: 明确V50到V62之间的依赖关系，或合并认证相关表到同一migration版本

**BLOCK-08 [一致性]**: 账号状态枚举值不统一
- **位置**: design.md Decision 7 vs SQL vs 实体
- **描述**: design.md注销状态机为ACTIVE→CANCELLING→CANCELLED，但SQL中account_status默认值是ACTIVE，cancellation_status未明确枚举值，实体中两个状态字段都是String无枚举约束
- **影响**: 状态流转无类型安全，可能写入非法状态值
- **修复建议**: 明确状态枚举值，使用Java Enum类约束，补充状态转换校验逻辑

### FLAG 级问题 (16个)

**FLAG-01 [完整性]**: proposal.md缺少高风险找回补充校验的具体产品形态
- **位置**: proposal.md Non-Goals + Open Questions #4
- **描述**: Open Question提到"高风险找回补充校验"产品形态未定，但spec中已写入Scenario，属于未关闭问题进入实现
- **建议**: 明确高风险找回的补充校验方式（如人脸、人工审核、安全问题），或先实现基础短信/邮箱找回

**FLAG-02 [完整性]**: 缺少与Jeecg系统用户sys_user的映射关系说明
- **位置**: design.md Open Questions #1
- **描述**: Open Question未关闭，未明确内容社区账号content_user_account是否与sys_user一对一/一对多/独立
- **建议**: 明确用户体系关系，否则用户ID来源和权限校验会混乱

**FLAG-03 [一致性]**: 密码失败锁定次数和时长与spec不一致
- **位置**: tasks.md 6.1 vs spec.md Scenario "Password failure lockout"
- **描述**: tasks.md说"30分钟锁定"，spec说"15分钟锁定"；tasks.md说"10次失败挑战，20次失败锁定"，spec说"5次连续失败锁定"
- **建议**: 统一锁定阈值和时长，更新spec或tasks

**FLAG-04 [一致性]**: 设备上限数量spec与tasks一致（5台），但Redis TTL不统一
- **位置**: design.md Decision 3 vs tasks.md
- **描述**: design.md说短信验证码TTL 300秒，密码失败计数TTL 900秒，但tasks.md测试用例未明确这些配置值
- **建议**: 在design.md中明确所有Redis key的TTL配置

**FLAG-05 [可实现性]**: Redis Token黑名单丢失后的降级策略未明确
- **位置**: design.md Risks "Token黑名单Redis丢失"
- **描述**: 提到"敏感操作同时校验会话状态"但未明确哪些是敏感操作，如何降级
- **建议**: 明确Redis不可用时的降级策略：是拒绝所有请求？还是仅校验DB会话状态？

**FLAG-06 [可实现性]**: 第三方登录在多端的App ID共用问题未关闭
- **位置**: design.md Open Questions #2
- **描述**: Web/移动端/小程序的provider配置是否共用App ID未确定
- **建议**: 明确第三方登录配置策略，预留appName/channel字段区分

**FLAG-07 [可实现性]**: 账号注销最终处理（匿名化vs物理删除）未确定
- **位置**: design.md Open Questions #3 （旁边标注"匿名化优先"但未正式决策）
- **描述**: 未正式确认匿名化方案，代码中anonymized字段存在但具体匿名化逻辑未定义
- **建议**: 正式决策并补充匿名化字段处理规则

**FLAG-08 [可测试性]**: Scenario缺少并发场景测试
- **位置**: specs/user-authentication/spec.md
- **描述**: 未覆盖并发换绑、并发注销、并发登录（同账号多端竞态）等场景
- **建议**: 补充并发Scenario，明确乐观锁/分布式锁策略

**FLAG-09 [可测试性]**: 缺少第三方OAuth授权失败的具体Mock场景
- **位置**: spec.md Scenario "Third-party authorization fails"
- **描述**: 列出了"cancellation, null identity, empty identity, invalid signature, API error"但未定义具体触发条件和测试数据
- **建议**: 补充每个失败场景的具体触发条件

**FLAG-10 [边界覆盖]**: 密码强度规则不完整
- **位置**: spec.md "password without a number" vs tasks.md
- **描述**: spec要求密码"至少8位且包含数字"，但未提及是否需要大小写字母、特殊字符、最大长度
- **建议**: 明确完整的密码强度规则

**FLAG-11 [边界覆盖]**: 缺少特殊手机号/邮箱格式的校验规则
- **位置**: spec.md "invalid mobile format"
- **描述**: 未明确支持哪些国家区号、是否支持虚拟运营商号段、邮箱最大长度
- **建议**: 补充手机号/邮箱格式校验规则

**FLAG-12 [边界覆盖]**: 设备指纹生成规则未定义
- **位置**: design.md多处提到device_fingerprint但未说明生成规则
- **描述**: 如何计算设备指纹？哪些字段参与？指纹冲突如何处理？
- **建议**: 补充设备指纹生成算法说明

**FLAG-13 [跨端一致性]**: 前端specs字段命名与后端可能不一致
- **位置**: user-01-authentication-frontend/specs/ vs 后端Req/VO
- **描述**: 需检查前端specs中的字段名（camelCase vs snake_case）与后端JSON序列化配置是否一致
- **建议**: 前后端对齐字段命名风格，确认Jackson配置

**FLAG-14 [API命名规范]**: 部分API路径使用RPC风格动词
- **位置**: design.md vs 实际Controller
- **描述**: /sms/send、/email/send、/captcha/verify等是RPC风格，但RESTful风格可能更好
- **建议**: 考虑统一为资源子路径风格，如POST /auth/sms-code（发送验证码）

**FLAG-15 [数据表审核]**: 密码历史表缺少唯一约束，可能插入重复记录
- **位置**: V3.9.1_61.sql:57-68 content_user_password_history
- **描述**: 只有普通索引，没有(user_id, password_hash)唯一约束，并发重置密码可能插入重复密码记录
- **建议**: 添加唯一约束或在Service层做去重判断

**FLAG-16 [依赖分析]**: Flyway迁移脚本版本号跨度大且分散
- **位置**: V3.9.1_50（用户域初始化）、V61（认证基础）、V62（会话/风控/注销）
- **描述**: 认证相关表分散在3个版本（50/61/62），版本号不连续，中间可能有其他模块脚本穿插
- **建议**: 梳理迁移脚本依赖关系，确保执行顺序正确
