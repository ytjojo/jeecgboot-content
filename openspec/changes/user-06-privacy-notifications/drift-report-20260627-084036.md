## 四、架构分层审核

### 分层依赖方向检查

```

```

| 层次 | 文件 | 依赖注入 | 跳层调用 |
|------|------|---------|---------|
| Controller | ContentUserSettingsController.java | IContentUserProfileService, IContentUserVisibilityPolicyService, IContentUserNotificationSettingService, IContentUserFeedSettingService, IContentUserSecuritySettingService | ✅ 正确注入Service层 |
| Controller | ContentUserThirdPartyAuthController.java | IContentUserThirdPartyAuthService | ✅ 正确注入 |
| Service | (IContentUser*Service接口) | - | ✅ |
| Mapper | ContentUser*Mapper.java | extends BaseMapper | ✅ |

### 架构问题

**CRITICAL**:
- **严重安全漏洞**: 所有Controller接口使用@RequestParam("userId")接收用户ID，未从SecurityContext获取当前登录用户
  - 位置: 所有Controller方法
  - 影响: 水平越权漏洞，任意用户可查看/修改其他用户的隐私设置、撤销其他用户的第三方授权
  - 建议: 
    1. 删除所有@RequestParam("userId")参数
    2. 使用`SecureUtil.getLoginUser().getId()`或`SecurityUtils.getSubject().getPrincipal()`获取当前用户ID
    3. 如确实需要管理员查询其他用户，增加权限注解@RequiresPermissions或@PreAuthorize

**WARNING**:
- Controller接口路径末尾斜杠不一致
  - 位置: ContentUserThirdPartyAuthController.java:29 `@GetMapping("/")`
  - 建议: 统一为`@GetMapping("")`

- Redis缓存key需确认是否使用常量统一管理
  - 位置: ContentUserSettingsCacheService
  - 建议: 使用ContentUserCacheConstant中定义的常量，避免硬编码
