## 架构审核

### defHttp封装规范审核
- ✅ 所有API统一使用`defHttp`封装，符合项目规范
- ✅ 正确使用`defHttp.get`/`defHttp.post`区分请求类型（除getCaptchaImage外）
- ✅ 参数传递使用对象形式，符合项目惯例

- ⚠️ 部分文件位置不一致：auth相关API在`api/content/auth/`，但account security在`api/content/account/`，与后端包名`account-security`不完全对应，建议统一目录结构

### Pinia Store规范审核

## WARNING 级别漂移/问题 (4个)

#### W-001: 账号安全API前缀系统性使用/auth/而非/account-security/
- **位置**: `api/content/account/security.ts:42-55`
- **严重级别**: WARNING
- **描述**: 10个账号安全接口错误使用`/api/v1/content/auth/`前缀而非设计规定的`/api/v1/content/account-security/`前缀
- **影响**: 后端路由匹配失败，所有绑定/设备管理/密码重置功能404
- **架构影响**: 违反模块边界划分（auth模块处理认证，account-security处理账号安全）
- **修复建议**: 批量修正为`/api/v1/content/account-security/`前缀

#### W-002: 字段命名phone/mobile系统性不一致
- **位置**: `api/content/auth/index.ts`和`api/content/account/security.ts`所有参数字段
- **严重级别**: WARNING
- **描述**: 前端统一使用`phone`作为字段名，后端design和spec使用`mobile`
- **影响**: 后端无法正确反序列化请求参数，注册/登录/绑定功能异常
- **修复建议**: 将所有参数字段从`phone`改为`mobile`，包括：
  - SmsCodeParams.phone → mobile
  - RegisterMobileParams.phone → mobile
  - LoginSmsParams.phone → mobile
  - bindPhone/rebindPhone参数名同步修正

#### W-003: getCaptchaImage错误使用POST方法
- **位置**: `api/content/auth/captcha.ts:19`
- **严重级别**: WARNING
- **描述**: 获取图片资源使用POST不符合RESTful语义，也与design.md定义的GET不一致
- **影响**: 可能被后端拦截或返回405 Method Not Allowed
- **修复建议**: 改为`defHttp.get<CaptchaImage>({ url: Api.image })`

#### W-004: cancelCancellation路径使用/cancel而非/revoke
- **位置**: `api/content/account/cancellation.ts:21`
- **严重级别**: WARNING
- **描述**: 取消注销接口路径名为`/cancel`但design定义为`/revoke`
- **影响**: 取消注销功能404
- **修复建议**: 修正路径为`/api/v1/content/account-cancellation/revoke`
