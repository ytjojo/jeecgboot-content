## 三、FLAG 级别问题 (2个)

#### F-001: batch-delete接口参数传递方式与design约定不一致
- **位置**: `design.md:70`（D3：参数风格）vs `api/content/filterRule.ts:20-21`
- **严重级别**: FLAG
- **描述**: design明确约定"所有写操作使用 POST，参数通过 @RequestParam（查询参数）传递"，但batchDeleteFilterRules将ruleIds放在body中（`data: ruleIds`），userId放在params中。数组参数用@RequestParam传递需要逗号分隔字符串，用@RequestBody传递数组更自然，但与design约定不一致
- **影响**: 若后端严格按@RequestParam实现，批量删除功能会失败
- **修复建议**: 两种方案选一：
  1. 将ruleIds转为逗号分隔字符串通过params传递：`params: { userId, ruleIds: ruleIds.join(',') }`
  2. 更新design.md D3，明确"批量操作的数组参数通过@RequestBody传递"

#### F-002: MuteListPageVO缺少屏蔽类型/过期时间等字段的明确分组定义
- **位置**: `api/content/mute.ts:28-35`（MuteListItemVO）
- **严重级别**: FLAG
- **描述**: tasks 6.5-6.9要求屏蔽列表包含4个Tab（用户/话题/内容类型/临时屏蔽），但MuteListItemVO仅定义了userId/nickname/avatar/mutedAt/muteType/expiresAt字段。话题屏蔽和内容类型屏蔽需要不同的字段（如topicId/topicName、typeCode/typeName），当前VO无法支持
- **影响**: 屏蔽话题Tab和屏蔽内容类型Tab无法正确渲染数据
- **修复建议**: 扩展MuteListItemVO使用联合类型或添加可选字段，如：
  ```typescript
  interface MuteListItemVO {
    muteId: string;
    muteType: 'USER' | 'TOPIC' | 'CONTENT_TYPE' | 'TEMP_USER';
    userId?: string; nickname?: string; avatar?: string;
    topicId?: string; topicName?: string; topicCover?: string;
    contentTypeCode?: string; contentTypeName?: string;
    mutedAt: string;
    expiresAt?: string;
  }
  ```

## 四、ADVISORY 级别问题 (3个)

#### A-001: getMuteList接口缺少分页以外的筛选参数
- **位置**: `api/content/mute.ts:18`
- **严重级别**: ADVISORY
- **描述**: getMuteList仅有pageNo/pageSize参数，4个Tab切换需要按muteType筛选，当前API未支持muteType参数。后端可能默认返回全部分组数据，但前端需要按Tab过滤
- **修复建议**: 添加可选的muteType参数：`getMuteList(userId, pageNo, pageSize, muteType?)`

#### A-002: getFilterRuleList接口分页参数与其他接口不一致
- **位置**: `api/content/filterRule.ts:24-25`
- **严重级别**: ADVISORY
- **描述**: getFilterRuleList的pageNo/pageSize是可选参数（其他接口如getBlacklist/getMuteList为必填），且规则列表可能不需要后端分页（数据量小，前端分页即可）
- **修复建议**: 若规则列表预期数据量不大（如上限100条），可明确标注pageNo/pageSize不传时返回全量；若需要分页则保持与其他接口一致设为必填

#### A-003: API函数名添加Filter前缀与tasks声明略有差异
- **位置**: tasks.md 1.3声明函数名addRule/deleteRule/batchDeleteRules/getRuleList vs 代码addFilterRule/deleteFilterRule/batchDeleteFilterRules/getFilterRuleList
- **严重级别**: ADVISORY
- **描述**: 代码中函数名加了"Filter"前缀（addFilterRule而非addRule），更清晰避免命名冲突，属于改进
- **修复建议**: 更新tasks.md中的函数名以与代码一致
