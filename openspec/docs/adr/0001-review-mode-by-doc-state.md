# ADR-0001: 审核模式条件基于文档完成状态

- **日期**: 2026-06-28
- **状态**: accepted
- **决策者**: @jiulongteng

## 背景

`openspec-review-change` skill 的 `review-gate-rules.md` 中审核模式 A/B/C 的条件原为"前后端开发进度"（前后端均未开发 / 后端部分完成 / 后端全部完成）。

但在 review 阶段，审核者面对的是规范文档（proposal.md、design.md、specs/），而非代码。用"开发进度"作为判定条件会导致歧义——review 关心的是**文档**是否完成，而非代码是否写完。

## 决策

审核模式 A/B/C 的条件改为基于 **change 文档完成状态**（文件系统实际存在的 change 目录）：

| 模式 | 条件 | 审核范围 |
|------|------|---------|
| A | 后端 change 文档完成，前端 change 不存在 | 后端全量审核，跳过衔接审计 |
| B | 后端 change 文档完成，前端 change 文档也存在 | 后端全量审核 + 衔接审计 |
| C | 仅前端 change 完成（纯 UI 改动） | 前端独立审核 |
| — | 仅前端 change 完成（需后端对接） | 不审核，提示先完成后端 change |

## 后果

- 审核模式判定逻辑从"推测开发进度"变为"检查文件系统"，确定性更高
- 纯 UI 前端 change 可独立审核（通过 proposal.md Impact 章节是否有 API 依赖清单识别）
- 衔接审计的触发条件更精确——仅在配对 change 文档实际存在时执行

## 备选方案

保留"开发进度"语言但增加"文档完成状态"补充说明。被拒绝原因：两套语言并行会造成更大歧义。
