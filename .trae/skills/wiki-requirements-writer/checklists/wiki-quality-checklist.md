# Wiki 需求质量检查清单

用于审查以下对象：

- 新写的 Wiki 风格需求文档
- 从线性 PRD 重写出的分层文档
- AI Agent 自动生成的需求输出
- 现有 Confluence / Notion / 飞书 / 语雀需求页

---

## 1. 结构检查

- [ ] 是否有清晰的总览页或索引页
- [ ] 是否明确采用了轻量层级或 `BR / UR / SR / DR` 四层结构
- [ ] 是否每页只承载一个主要主题，未把多个主题混在同一页
- [ ] 是否存在明确的 `In Scope` 和 `Out of Scope`
- [ ] 是否所有页面命名风格一致
- [ ] 是否所有需求 ID 唯一、连续、不可复用

---

## 2. 追溯检查

- [ ] 上层页面是否列出 `下级实现`
- [ ] 下层页面是否声明 `上级来源`
- [ ] 是否使用统一的 Wiki 内链格式 `[[ID|标题]]`
- [ ] 是否建立了全量或关键路径追溯矩阵
- [ ] 是否能从需求追到设计、接口、测试或变更记录
- [ ] 是否存在孤儿页面或孤儿需求条目

---

## 3. 内容完整性检查

- [ ] 是否写清背景、痛点和目标价值
- [ ] 是否给出可量化成功指标
- [ ] 是否写明目标用户、角色或影响对象
- [ ] 是否包含主流程、备选流程、异常流程
- [ ] 是否列出明确业务规则
- [ ] 是否覆盖关键依赖、冲突和影响分析

---

## 4. 验收与可测试性检查

- [ ] 是否为核心能力提供了验收标准
- [ ] 验收标准是否使用 `Given / When / Then` 或同等可测试表达
- [ ] 是否能直接判断通过或失败
- [ ] 是否覆盖正常流、异常流和边界条件
- [ ] 是否避免了“尽量”“快速”“友好”“支持多场景”等模糊表达

---

## 5. 非功能需求检查

- [ ] 是否定义性能指标，如 `P95`、吞吐量、并发量
- [ ] 是否定义可用性或可靠性目标
- [ ] 是否定义安全、权限、审计、合规要求
- [ ] 是否定义可观测性要求，如日志、监控、告警
- [ ] 是否使用可量化阈值，而不是主观描述

---

## 6. Agent 友好性检查

- [ ] 是否提供关键实体、Schema 或字段定义
- [ ] 是否提供状态机或生命周期规则
- [ ] 是否提供错误码 / 异常矩阵
- [ ] 是否声明不可违反的业务断言
- [ ] 是否引用了已有模块、接口或存量代码上下文
- [ ] 是否减少了歧义，避免让 Agent 猜业务规则

---

## 7. 管理信息检查

- [ ] 是否包含状态、版本、负责人
- [ ] 是否记录变更历史
- [ ] 是否记录评审结论或待办项
- [ ] 是否有风险项和缓解措施
- [ ] 是否列出开放问题和下一步行动

---

- 轻微：命名不一致 / 标签不规范 / 链接不完整
