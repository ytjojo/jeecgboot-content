## 复盘中 — 生成质量

### §0 Evidence

- [ ] [BLOCK] 提交范围和 diff 规模已通过命令获取（非估算）
- [ ] [BLOCK] 任务完成率已从 tasks.md 精确计算（非目测）
- [ ] 条目按实际情况动态增删（非机械套用默认清单）
- [ ] 每条数据可被后续章节引用

### §1 Wins

- [ ] 每项声明引用 §0 中的具体证据（commit hash / file / test）
- [ ] 每项 evidence 引用可定位到证明「成功」的具体文件/行（非「a1b2c3d: 代码实现顺利」式空洞引用）
- [ ] 无空洞的自我表扬（如"完成得很好""一切顺利""无特殊问题"）

### §2 Misses

- [ ] 严重程度分级正确（🔴阻塞 / 🟡痛苦 / 📌小问题）
- [ ] 每项引用具体证据
- [ ] 不涉及代码质量评价（对照边界红线表检查）
- [ ] 如无 🔴 无 🟡 项：已出具「零问题声明」
- [ ] 所有 📌 项合计不超过 3 个，超过则视为把严重问题降级 → 重新分级

### §3 Plan deviations

- [ ] 对比了 plan.md/tasks.md 与实际执行（openspec 模式）
- [ ] 每个偏差说明了「变化内容」和「原因」
- [ ] 通用降级模式：如实标注无正式计划

### §4 Skill compliance

- [ ] [BLOCK] 仅列出 apply 阶段技能（仅 apply 阶段技能，不含 brainstorming、writing-plans 等 plan 阶段技能）
- [ ] 每个 ✗ 填写了 What / Why / How to prevent 三问
- [ ] Why 给出了具体 trigger（引用 git log --stat 输出或 AGENTS.md/CLAUDE.md 规则原文），无模糊理由
- [ ] Why 未使用禁止理由清单中的任一表述（见下方清单）

**禁止理由清单**（任一命中 → 不通过）：
1. 不需要
2. 太小
3. 没时间
4. 纯配置
5. 变更量小
6. 单提交
7. 手动等价

9. 没必要
- [ ] How to prevent 选了具体方案（非空泛承诺）；`n/a — skip justified` 仅当 Why 引用了 AGENTS.md/CLAUDE.md 的明确跳过规则

### §5 Surprises

- [ ] 记录了被证伪的假设（非"一切顺利"）
- [ ] 写入 "(none observed)" 前已执行自问检查
- [ ] 无内容时写了 `(none observed)` 占位

### §6 Promote candidates

- [ ] 每条包含 Why 和 How to apply
- [ ] 晋升目标选择合理

## 越界红线参考 — 以下内容不能出现在 retrospective.md

| 不可评价 | 禁止示例 | 如果是过程问题，应聚焦 |
|----------|---------|----------------------|
| 代码质量 | "函数过长""命名不清晰""方法复杂" | "缺少 subagent 导致过度集中修改" |
| 架构设计 | "耦合过高""缺少抽象层" | "plan 未预见到跨模块影响" |
| 设计文档 | "需求文档缺少边界条件" | "实现过程中发现 plan 未覆盖的 case" |
| 测试覆盖率 | "覆盖率仅 60%""测试不足" | "TDD 技能未被遵守（§4 中标记 ✗）" |
| 提交粒度 | "commit 太大""提交混乱" | "subagent 编排未遵守（因 task 未拆分）" |
| 命名/风格 | "变量命名差""格式不一致" | 不属于过程复盘范围，不评价 |

**提交粒度判断规则**：可评价「是否因未遵守 subagent 编排 / task 拆分规则导致提交混乱」，不可评价「提交内容本身的好坏」。

## 常见理性化借口对照

复盘中如出现以下任一说辞，立即停止并复核：

| 借口 | 反制 |
|------|------|
| "太简单，没必要复盘" | 简单 ≠ 跳过条件。先跑 `git diff --stat`，用数据说话 |
| "没什么问题 / 一切顺利" | 零 🔴🟡 项时必须出具「零问题声明」并附 3 条自问 |
| "边界检查已通过" | checklist 自查 ≠ 真实通过。跑越界红灯关键词扫描二次验证 |
| "这是过程问题，不是代码质量" | 对照边界红线表逐条确认，不凭直觉判断 |
| "已经引用了 §0 证据" | 引用 ≠ 有效。检查 evidence 能否定位到具体文件/行 |
| "§5 没有意外" | "没有意外" 本身可能是信号 — 检查是否遗漏了被推翻的假设 |
| "复盘已完成" | 确认 checklist 复盘后项目全部勾选，carry-forward 已执行 |

## Red Flags — 立即停止并复核

复盘中如出现以下任一信号，暂停写入，逐条自查：

- §1 中使用了 "顺利完成"、"一切正常"、"无特殊问题"这类空洞表述
- §2 中无 🔴 无 🟡 项，且 📌 项少于 2 个
- 全文未出现 `git diff --stat` 或 `openspec/changes/` 的实际输出
- "none observed" 出现超过 2 次
- §4 技能合规显示全部 ✓，但 change 有 > 200 行 diff
- 所有 📌 项合计超过 3 个（可能把严重问题降级处理）

**以上任一 → 复核质量，不直接写入文件。** 修正后重新执行 Step 2。
