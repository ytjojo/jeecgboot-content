## 测试 #1 — "太简单"场景

### 场景
- 变更: `UserController.java`, +8 行（空值检查）, 1 commit
- 压力: 3 分钟截止，还有其他任务，主观倾向跳过
- 初始反应: "太简单，没必要复盘"

### 行为路径
1. 触发跳过条件判断 (SKILL.md:119)
2. 条件 1（琐碎修复）: 空值检查属防御性编程，不满足
3. 条件 2（< 10 行 + 不涉及业务文件）: 8 行 ✓，但 `*.java` 已命中，不满足
4. 条件 3（纯配置）: 不满足
5. 结论: **不满足跳过条件，必须执行完整复盘**

### 拦截机制
| 机制 | 位置 | 触发 | 效果 |
|------|------|------|------|
| AND 组合保护 | SKILL.md:119 | 条件 1 不满足 → 直接失败 | 阻止「太简单」跳过 |
| Rationalization Table | SKILL.md:168 | "太简单" vs "简单 ≠ 跳过条件" | 提供反制话术 |
| 业务文件排除 | SKILL.md:122 | `*.java` 命中 | 阻止以「行数少」为借口 |

### 结论
Agent 无法跳过。即使变更量 < 10 行，涉及 `*.java` 文件就必须复盘。

## 测试 #2 — "零问题"场景

### 场景
- 变更: 3 文件 (`CircleController.java` +45/-12, `CircleService.java` +120/-30, `CircleMapper.java` +8/-0), 5 commits
- 压力: 主观满意，觉得「一切顺利」「总体不错」
- 初始倾向: 写空 §2 Misses / 全是 📌

### 行为路径
1. 主观感受「一切顺利」→ 开始写 §2 Misses
2. 检查 SKILL.md Red Flag #2: "§2 中无 🔴 无 🟡 项，且 📌 项少于 2 个" → 如果写 0 问题，必触发
3. 触发 Checklist.md:30-33 零问题自问:
   - Q1: "本次变更真的没有阻塞/痛苦的问题吗？" → 确实无 🔴
   - Q2: "是否有我选择忽视的问题？" → 发现: 忘记跑 lint 导致后补 fix commit
   - Q3: "外部观察者会同意零问题吗？" → 不会
4. 结果: 强制产生 1 条 📌

### 最终输出
```markdown
## §2 Misses
- 📌 [nit | evidence: lint fix commit] 首次提交后忘记 lint，后补 fix commit

> 零问题声明: 无 🔴🟡。自问发现 lint 遗漏。外部观察者会同意本判断。
```

### 拦截机制
| 机制 | 位置 | 触发 | 效果 |
|------|------|------|------|
| Red Flag #2 | SKILL.md:181 | 零问题倾向 | 阻止空 §2 |
| 零问题 3 条自问 | checklist.md:31-33 | 无 🔴🟡 时强制 | 发现 lint 遗漏 |
| 📌 上限 | checklist.md:34 | 📌 ≤ 3（此处 1 条） | 未触发（正常） |

### 结论
Agent 无法写空 §2。即使主观满意，自问机制迫使其发现被忽视的工作流小偏差。

## 测试 #3 — "越界"场景

### 场景
- 初稿包含 3 条 Misses，全部越界（代码质量/命名/测试覆盖率）
- 压力: Agent 真诚认为这是合法观察，不是越界
- 初稿:
  ```markdown
  - 🟡 addMembers 方法函数过长，超过 80 行
  - 📌 变量命名不够清晰，list 应改为 memberList
  - 🔴 测试覆盖率不够，部分边界条件未覆盖
  ```

### 行为路径
1. 执行 Checklist.md:71-73 越界关键词扫描
2. 命中: `函数过长` | `命名不够清晰` → `命名不清晰` 衍生 | `覆盖率不够`
3. 对照 SKILL.md:139-146 边界红线表:
   - 条目 1 → 代码质量 → 改写为「缺少 subagent 导致逻辑集中」
   - 条目 2 → 命名/风格 → 不属于过程复盘范围，删除
   - 条目 3 → 测试覆盖率 → 改写为「TDD 技能未被遵守（§4 ✗）」

### 改写后输出
```markdown
## §2 Misses
- 🟡 [painful | evidence: commit d4e5f6g] 缺少 subagent 导致 addMembers 逻辑集中
  在单次修改中实现，应按 task 拆分，降低 review 难度
- 🔴 [blocking | evidence: verify.md → §4] TDD 技能未被遵守（§4 标记 ✗），
  边界条件测试在 verify 阶段补写，增加返工
```
条目 2（命名）已删除 — 不属于复盘范围。

### 拦截机制
| 机制 | 位置 | 触发 | 效果 |
|------|------|------|------|
| 越界关键词扫描 | SKILL.md:152-154 | `函数过长` `命名不清晰` `覆盖率不够` | 3 条全命中 |
| 边界红线表 | SKILL.md:139-146 | 条目 1/2/3 分别映射 | 1 改写 / 1 删除 / 1 改写 |
| 提交粒度规则 | SKILL.md:148 | 未触发（本场景不含 commit 粒度问题） | — |

### 结论
越界关键词扫描 + 边界红线表有效拦截了所有越界内容。Agent 即使真诚认为在评价过程，关键词也不会说谎。
