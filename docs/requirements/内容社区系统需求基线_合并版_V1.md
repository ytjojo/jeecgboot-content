# 内容社区系统需求基线（V1 合并版）

## 0. 说明

- 文档定位：合并并去重 `docs/requirements/` 目录内需求文档，统一口径，形成 V1 可落地、可追踪、可验收的需求基线。
- 实现状态：当前为需求设计阶段，所有功能均未实现。
- 技术基线：后端以 Spring Boot + MyBatis-Plus 为主；数据库生产基线为 MySQL 8；媒体文件使用对象存储（OSS/CDN）。
- 范围约束：本文为需求基线，不直接输出数据库 DDL 与 API 细节到字段级；实现阶段另行输出技术设计与数据设计文档。

## 1. Background

内容社区系统支持用户创建与分享多种类型内容（文章、笔记、动态、视频、问答），并提供浏览、互动、治理与后台运营能力。

## 2. Goals and Non-Goals

### 2.1 Goals

- 支持多内容类型的统一发布与编辑（富文本为主、Markdown 备选）。
- 支持统一的互动体系（点赞/收藏/分享/关注/举报），并对问答回答提供互斥的赞同/反对。
- 支持基础评论能力（MVP）与内容治理（审核、违禁词、反垃圾、举报处理）。
- 支持频道（V1 单频道发布）与后台运营管理。
- 形成可测试的验收标准（Given-When-Then），便于拆解与联调验收。

### 2.2 Non-Goals（V1 不做）

- 多频道发布（V2 扩展，需补齐权限与重复曝光策略）。
- 评论增强能力（匿名评论、仅楼主可见、@机器人等，V2 再做）。
- 社区币/积分/邀请码/创作者激励等“缺库模块”（进入需求池/后续迭代）。
- UI 高保真稿与交互细节稿、算法训练实现细节、支付/清结算合规条款细化（不在本基线内展开）。

## 3. Scope

### 3.1 In Scope（V1）

- 内容：文章、笔记、动态、视频、问答问题、问答回答、转发。
- 编辑：TipTap 富文本（结构化 JSON）+ Markdown（保留原文）。
- 媒体与附件：统一总数上限、视频主视频规则。
- 频道：频道创建/编辑/订阅/内容管理；发布时仅允许选择 1 个频道。
- 浏览：最新、关注、精选、热门等列表；内容详情页。
- 搜索：综合搜索 + 类型/时间/标签等筛选。
- 互动：点赞、收藏、分享、关注、举报；@提及通知。
- 评论（MVP）：发表评论、回复、编辑/删除、自身可见；支持 @ 提及用户。
- 治理：内容审核、违禁词管理、基础反垃圾。
- 后台：用户管理、内容审核/管理、举报处理、评论管理、统计与配置。

### 3.2 Out of Scope（V2/Backlog）

来自 `docs/requirements/需求池.md` 的需求池：

- 打赏
- 角色管理
- 广告位管理
- 活动
- 类似 Twitter 的列表功能
- 问答：追加问题、悬赏现金/悬赏积分
- 会员：会员等级/会员角色/登录日志/私信/系统通知等
- 会员卡管理（会员卡列表、订单）
- 问题反馈

## 4. Personas and Roles

- 游客：浏览公开内容、搜索、查看频道/话题。
- 注册用户：发布/编辑/删除自己的内容；互动与评论；关注频道与用户。
- 创作者（扩展角色）：关注内容表现、粉丝增长与收益（若启用创作者中心）。
- 频道创建者/管理者：管理频道信息与频道内内容（置顶/精华/移除）。
- 审核员/管理员：审核内容与评论、处理举报、治理与运营配置；具备审计追溯责任。
- 权限原则：以 RBAC 为主，数据权限按角色与资源边界收敛。

## 5. User Stories（骨架）

- As a 用户, I want 发布多类型内容, so that 分享观点并获得互动反馈。
- As a 用户, I want 在内容中添加 @ / 话题 / 标签 / 附件 / 卡片, so that 表达更完整并提高传播效率。
- As a 用户, I want 浏览推荐与搜索结果, so that 快速找到感兴趣内容。
- As a 提问者, I want 采纳最佳答案, so that 形成高质量问答沉淀。
- As an 管理员, I want 审核与反垃圾处理, so that 维护社区质量与合规。
- As an 运营人员, I want 查看统计与报表, so that 支撑运营决策。

## 6. Conflict Matrix（已冻结为 V1 决议）

- C-01 编辑器选型冲突：同时支持 TipTap（结构化 JSON）与 Markdown（保留原文）。
- C-02 富文本格式冲突：以 TipTap JSON/Markdown 为源数据；HTML 为渲染缓存；mention 关系独立入表。
- C-03 笔记图片上限冲突：统一 18 张（并计入附件总数上限）。
- C-04 数据库方言冲突：生产基线以 MySQL 8 为主，JSON 字段按 MySQL 能力实现。
- C-05 主键策略冲突：统一使用字符串主键（UUID/Snowflake 字符串）并在全域保持一致。
- C-06 问答采纳角色冲突：仅提问者可采纳最佳答案。
- C-07 功能实现状态冲突：当前为需求设计阶段，所有功能均未实现。
- C-08 发布到频道冲突：V1 限单频道发布，V2 再扩展多频道并补充权限与重复曝光策略。
- C-09 评论能力冲突：拆分为“基础评论能力（MVP）”与“增强评论能力（V2）”。
- C-10 附件与视频规则冲突：Video 类型仅允许 1 个主视频；其他类型附件按总数 18 限制。

## 7. Business Rules

### 7.1 主键与审计字段

- 所有业务主键统一为字符串（UUID/Snowflake 字符串）。
- 全表统一审计字段：create_by、create_time、update_by、update_time、del_flag。

### 7.2 内容状态流与发布策略

- 内容状态流：草稿 -> 待审核 -> 已发布 -> 已下架/已删除
- 发布策略：支持“先审后发”与“先发后审”两种策略，V1 默认先审后发（可配置）。

### 7.3 编辑器与存储策略

- 原始内容（source）必须可回放：TipTap JSON 或 Markdown 原文（以及纯文本/摘要）。
- 渲染 HTML 作为缓存（可重建），需要进行安全净化（XSS 防护）。
- @ 提及关系单独入表，不从 HTML/JSON 反解析作为唯一来源。

### 7.4 媒体与附件规则（统一口径）

- 附件总数上限：每条内容最多 18 个附件（图片/视频/音频/文档等总和）。
- 视频内容（Video 类型）：必须且仅能包含 1 个主视频附件（file_type=VIDEO，is_primary=1）；其余附件不得为 VIDEO，且总数不超过 18。
- 笔记（Note 类型）头部媒体：仅允许图片；最多 18 张；计入附件总数。

### 7.5 频道发布规则

- V1：发布时仅允许选择 1 个频道（channel_id 单值）。
- V2：扩展多频道发布（channel_ids 多值），并补齐权限与重复曝光策略。

### 7.6 问答采纳规则

- 仅提问者可采纳最佳答案。
- 每个问题最多存在 1 个最佳答案；可取消或更换采纳（需记录审计日志）。

### 7.7 评论能力拆分

- 评论能力（MVP）：登录用户可发表评论、回复、编辑/删除自己的评论；支持 @ 提及用户；支持审核与举报。
- 评论增强能力（V2）：匿名评论、仅楼主可见、@机器人等。

## 8. Functional Requirements

### 8.1 内容发布（通用）

- 系统 SHALL 支持用户创建内容：选择内容类型、填写标题/正文、选择频道、添加标签/话题、添加附件、预览与提交。
- 系统 SHALL 支持保存草稿、继续编辑与删除草稿。
- 系统 SHALL 支持发布后编辑与删除（软删），并保留版本与审计记录（V1 可先保留“最近一次修改版本”作为最小实现）。
- 系统 SHALL 在发布/编辑时进行校验：内容类型合法、附件规则合法、标题/正文长度合法、敏感词策略生效。

### 8.2 内容类型（Content Types）

#### 8.2.1 文章（Article）

- 编辑器：TipTap（默认）或 Markdown（可切换）。
- 支持设置单张封面图（可复用附件表或独立字段）。

#### 8.2.2 笔记（Note）

- 头部媒体：最多 18 张图片轮播展示。
- 正文：TipTap（默认）或 Markdown。

#### 8.2.3 动态（Post）

- 布局：文字在上，媒体在下；支持纯文本。
- 媒体：图片/视频混合（遵循附件总数上限与类型校验）。

#### 8.2.4 视频（Video）

- 主视频：必须且仅 1 条。
- 可选：标题与简介（正文支持 TipTap/Markdown 之一）。
- 可选：附加非视频附件（如封面图、文档等），总数不超过 18。

#### 8.2.5 问答问题（QA-Question）

- 字段：问题标题 + 问题描述（描述支持 TipTap/Markdown）。
- 支持话题/标签、关注问题、邀请回答（V1 可先预留数据结构，具体功能按排期）。

#### 8.2.6 问答回答（QA-Answer）

- 支持发布回答（正文支持 TipTap/Markdown）。
- 支持赞同/反对互斥、收藏、评论、分享等通用互动。

#### 8.2.7 转发（Repost）

- 系统 SHALL 支持用户转发任意内容类型并附带评论。
- 转发内容需保留对原内容的引用关系，并具备独立的互动数据统计。

### 8.3 通用组件与特殊元素

#### 8.3.1 @提及（Mention）

- 所有文本输入区支持 `@` 提及用户。
- 发布成功后写入提及关系表，并向被提及用户发送通知。

#### 8.3.2 标签/话题（Tag/Topic）

- 系统 SHALL 支持创建/选择标签与话题，并统计使用次数与关注数（可选）。
- 系统 SHALL 支持按标签/话题筛选内容。

#### 8.3.3 卡片（Card）

- 系统 SHOULD 支持块级卡片元素：投票卡片、视频卡片、音频卡片、内容卡片、链接卡片、文档卡片、商品卡片、活动卡片、位置卡片、广告卡片等。
- V1 最小闭环：投票卡片 + 链接卡片（其余作为可配置扩展点）。

### 8.4 内容浏览

- 系统 SHALL 提供内容列表：最新、关注、精选、热门。
- 系统 SHALL 支持按内容类型筛选列表，并支持分页/无限滚动（由前端实现策略决定）。
- 系统 SHALL 提供内容详情页：正文渲染、附件展示、互动区、评论区、推荐区（可选）。

### 8.5 搜索

- 系统 SHALL 支持综合搜索：标题、正文（纯文本）、标签、作者等。
- 系统 SHALL 支持筛选：内容类型、时间范围、标签/话题、作者、频道等。
- 系统 SHOULD 支持搜索建议与热门词（后台可配置）。

### 8.6 互动体系

- 系统 SHALL 支持点赞、收藏、分享、关注用户、关注频道、举报。
- 对问答回答：赞同/反对互斥；执行反对时应撤销赞同（反之亦然），并记录行为日志。
- 系统 SHOULD 记录互动统计：点赞数、评论数、收藏数、转发数、浏览量等。

### 8.7 评论（MVP）

- 系统 SHALL 支持用户发表评论与回复评论（至少 2 层：评论/回复）。
- 系统 SHALL 支持用户编辑与删除自己的评论（软删）。
- 系统 SHALL 支持评论审核与举报处理。
- 系统 SHALL 支持评论内 @ 提及用户并触发通知。

### 8.8 频道（Channel）

- 系统 SHALL 支持频道创建、编辑、删除（需满足约束，如频道下无内容或具备权限）。
- 系统 SHALL 支持频道订阅/取消订阅，并在信息流中体现订阅权重（算法细节可后置）。
- 系统 SHALL 支持频道内内容管理：置顶、精华、移除不当内容（需权限）。
- 系统 SHOULD 支持频道搜索与推荐（可配置）。

### 8.9 治理与运营

#### 8.9.1 内容审核

- 系统 SHALL 支持审核队列：按内容类型、优先级、时间排序。
- 系统 SHALL 支持审核操作：通过、拒绝、需修改；记录审核理由与审计日志。
- 系统 SHOULD 支持多级审核流程配置（V1 可先单级）。

#### 8.9.2 违禁词

- 系统 SHALL 支持违禁词库增删改查、分类、策略（拦截/转审核/替换）。
- 系统 SHALL 在发布与编辑时执行违禁词策略，并提供命中记录（可选）。

#### 8.9.3 反垃圾（基础）

- 系统 SHOULD 支持重复内容检测与频率限制策略（V1 先提供接口与配置入口）。

### 8.10 后台管理（Admin）

- 用户管理：查询/编辑/启用禁用/角色设置/行为查看。
- 内容管理：审核、列表、编辑基础信息、置顶/精华、删除。
- 举报处理：查看、处理、追溯、批量操作。
- 评论管理：评论审核、删除、查看回复链路。
- 圈子管理：V1 仅保留管理入口与数据结构预留；详细功能进入 V2。
- 系统配置：分类、标签、话题、违禁词、搜索配置、通知配置。
- 数据统计：实时指标与历史趋势，支持导出。

## 9. Workflow / State Transitions

- 内容状态流：草稿 -> 待审核 -> 已发布 -> 已下架/已删除。
- 举报状态流：待处理 -> 处理中 -> 已处理 -> 已申诉 -> 申诉结论。
- 问答状态流：问题已发布 -> 回答提交 -> 提问者采纳/未采纳 -> 结案/继续互动。
- 约束：若开启“先发后审”，需在风险评审后定义灰度策略与回滚策略。

## 10. Data and Interfaces（概念模型）

### 10.1 主要实体

- contents：内容主表（id、content_type、title、author_id、channel_id、status、统计字段、审计字段）。
- content_sources：内容源表（content_id、tiptap_json(JSON)、markdown_text(TEXT)、plain_text(TEXT)、render_html_cache(MEDIUMTEXT)、version）。
- content_attachments：附件表（content_id、file_type、file_url、file_id、描述、排序、is_primary）。
- content_mentions：提及关系表（content_id、mentioned_user_id、位置/范围可选）。
- tags / topics：标签与话题主表。
- content_tag_relations / content_topic_relations：关联表。
- comments：评论表（content_id、parent_id、author_id、正文 source/渲染缓存、状态、审计字段）。
- reactions：互动表（target_type、target_id、user_id、reaction_type、状态、审计字段），支持互斥规则扩展。
- channels：频道表（id、owner_id、名称、描述、权限/可见性、统计）。
- audits / reports：审核记录与举报记录表。

### 10.2 统一约束

- 标识策略：主键策略全域统一（UUID/Snowflake 字符串），不得在同一域混用多套主键风格。
- 文件策略：媒体文件 SHALL 存对象存储，业务库仅存 ID/URL/元数据。
- 接口约束：核心操作需具备幂等策略、权限校验、审计追踪。

### 10.3 MySQL 8 约束

- JSON 字段使用 MySQL JSON 类型；如需索引，采用生成列（generated column）或冗余字段方案。
- 不使用 PostgreSQL JSONB / GIN 索引语法。

## 11. Non-Functional Requirements

- 可用性：核心服务目标可用性 SHOULD >= 99.9%。
- 性能：核心读链路（列表/详情）P95 响应时间 SHOULD < 200ms（需结合部署模式与缓存策略分环境校准）。
- 并发：系统 SHOULD 支撑 10 万级并发访问（分阶段达成）。
- 安全：系统 SHALL 具备 XSS/CSRF 防护、鉴权、审计日志与敏感数据保护；HTML 渲染缓存必须进行安全净化；上传文件需类型校验与大小限制。
- 可观测性：系统 SHALL 提供监控、告警、链路日志和关键业务指标看板。
- 可扩展性：系统 SHALL 支持服务与存储横向扩展；写链路（发布/评论/互动）应支持限流与降级。

## 12. Acceptance Criteria（关键链路样例）

- AT-01 内容发布  
  Given 用户已登录并具备发布权限  
  When 提交一条文章并通过校验  
  Then 系统返回发布成功并进入待审核或已发布状态（按审核策略）

- AT-02 @提及  
  Given 用户在正文输入 @ 并选择目标用户  
  When 发布成功  
  Then 系统写入提及关系并向被提及用户发送通知

- AT-03 问答采纳  
  Given 问题下存在多个回答  
  When 提问者执行采纳  
  Then 仅一个回答被标记为最佳答案，且状态可追溯

- AT-04 互动互斥（问答回答）  
  Given 用户已对回答赞同  
  When 对同一回答执行反对  
  Then 系统自动撤销赞同并记录反对

- AT-05 审核流  
  Given 内容命中“需审核”规则  
  When 用户提交内容  
  Then 内容进入审核队列并可被审核员处理

- AT-06 搜索筛选  
  Given 用户输入关键词并勾选类型筛选  
  When 执行搜索  
  Then 返回结果包含高亮关键词且满足筛选条件

## 13. Milestones and Plan（建议节奏）

- M1（1-2周）：冲突决议冻结、统一术语、范围基线确认。
- M2（2-4周）：核心链路落地（发布/互动/审核/搜索）。
- M3（2-3周）：后台运营与统计完善、通知与治理能力收口。
- M4（持续）：缺库模块（社区币/积分/邀请码/创作者中心）按优先级迭代。

## 14. Risks and Dependencies

- R-01 需求冲突未冻结导致反复返工。
- R-02 数据库方言与主键策略不统一导致实现割裂。
