# 用户状态与生命周期治理

## 史诗概览

**史诗ID**: EPIC-09
**标题**: 建立完善的用户状态机与生命周期管理机制，实现精细化治理与审计追溯
**背景**: 内容社区需要对不同阶段的用户施加不同的规则限制，包括禁言、冻结、封禁、注销等状态。当前缺少统一的状态管理模型和审计机制，导致处罚不一致、无法追溯、恢复困难
**用户**: 所有注册用户、版主/管理员、风控系统
**成功标准**: 
- 状态变更准确率 100%
- 审计日志完整率 100%
- 自动解禁准确率 >95%
- 申诉恢复成功率 >70%
**范围**: 
- ✅ 包含：用户状态定义（游客/未完善/正常/禁言/限制推荐/冻结/封禁/注销中/已注销）、状态触发与流转、功能限制策略、审计留痕、自动/人工解禁、申诉恢复
- ❌ 不包含：独立风控引擎、机器学习异常检测
**预计持续时间**: 6 周
**优先级**: 高

---

## 特性列表

### 特性 1: 用户状态定义与管理

**描述**: 定义并管理多种用户状态，由用户操作、风控策略、审核处罚、管理员处理等事件触发
**关键交互**: 触发事件 → 状态变更 → 应用限制 / 查看状态历史
**非功能需求**: 状态变更即时生效，状态查询响应时间 <100ms
**验收条件**: 平台能正确定义和管理所有用户状态，状态变更有明确触发条件

#### 故事列表

##### 故事 9.1.1: 定义与管理用户状态

**作为** 平台，  
**我希望** 定义并管理用户状态：游客、已注册未完善资料、正常、禁言、限制推荐、冻结、封禁、注销中、已注销，  
**以便** 对不同阶段施加不同规则。

**验收标准**:
1. **给定** 新用户访问平台，**当** 未登录，**则** 系统标记为"游客"状态，仅允许浏览公开内容
2. **给定** 用户完成注册但未完善资料，**当** 首次登录，**则** 系统标记为"未完善资料"状态，引导完善资料
3. **给定** 用户正常使用，**当** 无违规，**则** 系统保持"正常"状态，所有功能可用
4. **给定** 用户被处罚禁言，**当** 处罚生效，**则** 系统标记为"禁言"状态，禁止评论/私信/发动态
5. **给定** 用户被限制推荐，**当** 处罚生效，**则** 系统标记为"限制推荐"状态，内容不进入公共推荐流
6. **给定** 用户账号存在风险，**当** 风控触发，**则** 系统标记为"冻结"状态，暂时禁止登录
7. **给定** 用户严重违规，**当** 处罚生效，**则** 系统标记为"封禁"状态，禁止登录与互动
8. **给定** 用户申请注销，**当** 进入冷静期，**则** 系统标记为"注销中"状态
9. **给定** 冷静期结束，**当** 执行注销，**则** 系统标记为"已注销"状态，数据匿名化或删除

**附加说明**:
- **优先级**: 高
- **依赖项**: EPIC-01（用户需先注册）
- **技术说明**: 状态存储于用户表的状态字段；使用枚举类型；状态变更触发事件

##### 故事 9.1.2: 状态触发与流转

**作为** 平台，  
**我希望** 用户状态由用户操作、风控策略、审核处罚、管理员处理等事件触发，  
**以便** 覆盖全场景。

**验收标准**:
1. **给定** 用户发布违规内容，**当** 审核团队处罚，**则** 系统自动将用户状态变更为"禁言"或"封禁"
2. **给定** 风控系统检测到异常行为，**当** 触发规则，**则** 系统自动将用户状态变更为"冻结"
3. **给定** 用户申请注销，**当** 提交申请，**则** 系统将状态变更为"注销中"
4. **给定** 禁言期限到期，**当** 定时任务检查，**则** 系统自动将状态恢复为"正常"
5. **给定** 管理员手动调整状态，**当** 执行操作，**则** 系统立即变更状态并记录操作人

**附加说明**:
- **优先级**: 高
- **依赖项**: 故事 9.1.1
- **技术说明**: 状态流转通过状态机实现；支持自动和手动触发；所有变更写入审计日志

### 特性 2: 状态对应的功能限制策略

**描述**: 为每种用户状态定义明确的功能限制，确保处罚精准且可预期
**关键交互**: 用户尝试操作 → 系统检查状态 → 允许/拒绝 → 提示原因
**非功能需求**: 权限检查延迟 <50ms，不影响正常用户体验
**验收条件**: 不同状态的用户受到正确的功能限制，提示信息清晰

#### 故事列表

##### 故事 9.2.1: 禁言状态的功能限制

**作为** 平台，  
**当我用户被禁言时，我希望禁止其评论/私信/回答/发动态等互动，但允许登录与浏览，以便精准处罚。

**验收标准**:
1. **给定** 用户处于"禁言"状态，**当** 我尝试发表评论，**则** 系统阻止并提示"您已被禁言，禁言期限：XXX，原因：XXX"
2. **给定** 用户处于"禁言"状态，**当** 我尝试发送私信，**则** 系统阻止并提示相同信息
3. **给定** 用户处于"禁言"状态，**当** 我尝试发布动态，**则** 系统阻止并提示相同信息
4. **给定** 用户处于"禁言"状态，**当** 我登录并浏览内容，**则** 我可以正常浏览、点赞、收藏（被动互动）
5. **给定** 用户处于"禁言"状态，**当** 我查看个人主页，**则** 我看到禁言提示和剩余时间

**附加说明**:
- **优先级**: 高
- **依赖项**: 故事 9.1.1
- **技术说明**: 权限检查在接口层统一执行；禁言期限存储于用户表

##### 故事 9.2.2: 限制推荐状态的内容分发限制

**作为** 平台，  
**当我用户被限制推荐时，我希望内容仍可见但不进入公共推荐流或降低曝光，以便治理但不完全封禁。

**验收标准**:
1. **给定** 用户处于"限制推荐"状态，**当** 该用户发布内容，**则** 内容对粉丝和关注者可见
2. **给定** 用户处于"限制推荐"状态，**当** 系统生成推荐流，**则** 该内容不进入公共推荐流
3. **给定** 用户处于"限制推荐"状态，**当** 其他用户搜索该内容，**则** 搜索结果中降低排序或不显示
4. **给定** 用户处于"限制推荐"状态，**当** 该用户查看自己的内容，**则** 他能看到所有内容正常显示
5. **给定** 限制推荐期限到期，**当** 定时任务检查，**则** 系统自动恢复正常推荐

**附加说明**:
- **优先级**: 中
- **依赖项**: 故事 9.1.1
- **技术说明**: 推荐算法读取用户状态并调整权重；搜索排序也考虑状态

##### 故事 9.2.3: 冻结与封禁状态的严格限制

**作为** 平台，  
**当我用户被冻结时，我希望暂时禁止登录或大部分功能并引导安全核验，以便处理风险账号。当我用户被封禁时，我希望禁止登录与互动，并记录处罚原因与期限，以便可审计与可复核。

**验收标准**:
1. **给定** 用户处于"冻结"状态，**当** 我尝试登录，**则** 系统阻止登录并提示"账号已冻结，请进行安全核验"
2. **给定** 用户处于"冻结"状态，**当** 我完成安全核验（如手机验证），**则** 系统解冻并恢复"正常"状态
3. **给定** 用户处于"封禁"状态，**当** 我尝试登录，**则** 系统阻止登录并提示"账号已被封禁，封禁期限：XXX，原因：XXX"
4. **给定** 用户处于"封禁"状态，**当** 封禁期限到期，**则** 系统自动解封并恢复"正常"状态
5. **给定** 用户处于"封禁"状态，**当** 我尝试通过 API 访问，**则** 系统返回 403 Forbidden 错误

**附加说明**:
- **优先级**: 高
- **依赖项**: 故事 9.1.1
- **技术说明**: 冻结和封禁在认证层拦截；封禁期限可配置（临时封禁/永久封禁）

### 特性 3: 审计留痕与追溯

**描述**: 每次状态变更记录操作人、触发原因、依据规则、开始/结束时间与备注，支持审计追溯
**关键交互**: 状态变更 → 自动生成审计日志 / 查看审计历史
**非功能需求**: 审计日志不可篡改，保留至少 3 年
**验收条件**: 所有状态变更都有完整的审计记录，能追溯到操作人和原因

#### 故事列表

##### 故事 9.3.1: 状态变更审计日志

**作为** 平台，  
**我希望** 每次状态变更记录操作人、触发原因、依据规则、开始/结束时间与备注，  
**以便** 审计追溯。

**验收标准**:
1. **给定** 用户状态发生变更，**当** 变更发生，**则** 系统自动生成审计日志
2. **给定** 审计日志，**当** 我查看，**则** 我看到变更时间、操作用户 ID、原状态、新状态、操作人（系统/管理员 ID）、触发原因、依据规则、开始时间、结束时间、备注
3. **给定** 管理员手动变更状态，**当** 操作完成，**则** 审计日志记录管理员 ID 和操作 IP
4. **给定** 系统自动变更状态，**当** 操作完成，**则** 审计日志记录触发规则 ID 和风控评分
5. **给定** 我想查看某用户的状态历史，**当** 我进入"状态历史"页面，**则** 我看到按时间倒序排列的所有状态变更记录

**附加说明**:
- **优先级**: 高
- **依赖项**: 故事 9.1.1
- **技术说明**: 审计日志存储于独立的 audit_log 表；使用 WORM（Write Once Read Many）存储防止篡改

##### 故事 9.3.2: 敏感操作留痕

**作为** 平台，  
**我希望** 账号安全、状态处罚、认证审核、积分扣减、勋章回收等敏感操作必须留痕，  
**以便** 合规审计。

**验收标准**:
1. **给定** 执行敏感操作（如封禁、积分扣减），**当** 操作完成，**则** 系统生成审计日志
2. **给定** 审计日志，**当** 合规团队查看，**则** 他们能导出审计报告
3. **给定** 审计日志保留 3 年，**当** 超过 3 年，**则** 系统归档到冷存储或删除（根据合规要求）
4. **给定** 有人尝试篡改审计日志，**当** 检测到，**则** 系统告警并记录安全事件

**附加说明**:
- **优先级**: 高
- **依赖项**: 故事 9.3.1
- **技术说明**: 敏感操作定义配置化；审计日志加密存储；访问审计日志需特殊权限

### 特性 4: 解禁与恢复机制

**描述**: 支持自动解禁、人工解禁与申诉恢复，恢复后保留历史记录用于审计
**关键交互**: 期限到期/申诉成功 → 状态恢复 → 通知用户 / 管理员手动解禁
**非功能需求**: 解禁即时生效，恢复后功能立即可用
**验收条件**: 用户能在处罚结束后或被证明无辜时恢复正常状态

#### 故事列表

##### 故事 9.4.1: 自动解禁与人工解禁

**作为** 平台，  
**我希望** 支持自动解禁、人工解禁，恢复后保留历史记录用于审计，  
**以便** 平衡治理与公平。

**验收标准**:
1. **给定** 用户禁言期限到期，**当** 定时任务检查，**则** 系统自动将状态恢复为"正常"并通知用户
2. **给定** 管理员决定提前解禁，**当** 管理员执行"解禁"操作，**则** 系统立即恢复用户状态并记录操作人
3. **给定** 用户状态恢复，**当** 恢复完成，**则** 用户所有功能立即可用
4. **给定** 用户被多次处罚，**当** 查看历史记录，**则** 系统保留所有处罚和恢复记录
5. **给定** 自动解禁失败，**当** 检测到，**则** 系统告警并转人工处理

**附加说明**:
- **优先级**: 高
- **依赖项**: 故事 9.1.1 和 9.3.1
- **技术说明**: 自动解禁通过定时任务执行；人工解禁需权限控制

##### 故事 9.4.2: 申诉恢复

**作为** 平台，  
**我希望** 支持申诉恢复，恢复后保留历史记录用于审计，  
**以便** 平衡治理与公平。

**验收标准**:
1. **给定** 用户申诉成功，**当** 审核团队批准，**则** 系统撤销处罚并恢复用户状态
2. **给定** 处罚被撤销，**当** 恢复完成，**则** 系统通知用户"申诉成功，您的账号已恢复正常"
3. **给定** 申诉恢复，**当** 查看审计日志，**则** 日志记录申诉编号、审核人、恢复时间和原因
4. **给定** 用户被错误处罚，**当** 恢复后，**则** 系统可选择补偿用户（如返还积分、延长会员）
5. **给定** 申诉被驳回，**当** 审核团队驳回，**则** 系统维持原处罚并告知用户原因

**附加说明**:
- **优先级**: 高
- **依赖项**: EPIC-08（申诉系统）
- **技术说明**: 申诉恢复与 EPIC-08 的申诉系统集成；补偿机制可选

## 风险与假设

- **风险**: 
  - 状态变更逻辑复杂可能导致 bug
  - 审计日志存储成本高
  - 自动解禁失败可能导致用户投诉
- **假设**: 
  - 状态机能覆盖所有业务场景
  - 审计日志能满足合规要求
  - 定时任务稳定可靠，能准时执行解禁
