# 隐私设置与通知管理

## 史诗概览

**史诗ID**: EPIC-06
**标题**: 提供精细化的隐私控制与通知管理能力，平衡用户隐私保护与信息获取需求
**背景**: 内容社区需要为用户提供完善的隐私保护和通知管理机制。当前缺少细粒度的隐私设置、免打扰模式、在线状态控制和第三方授权管理，导致用户隐私边界模糊且通知打扰过多
**用户**: 所有注册用户
**成功标准**: 
- 隐私设置使用率 >70%
- 通知关闭率降低 30%
- 用户对隐私控制的满意度 >85%
- 通知点击率提升 20%
**范围**: 
- ✅ 包含：互动通知开关与渠道、免打扰时段、动态/在线状态/搜索引擎索引可见性、第三方授权管理、账户安全设置入口
- ❌ 不包含：数据导出功能、GDPR 合规工具（可选）
**预计持续时间**: 4 周
**优先级**: 高

---

## 特性列表

### 特性 1: 通知偏好设置

**描述**: 允许用户分别设置各类互动通知的开关和推送渠道，并配置免打扰时段
**关键交互**: 进入通知设置 → 选择通知类型 → 配置开关和渠道 → 保存
**非功能需求**: 设置调整后即时生效，安全类通知不受普通通知开关影响
**验收条件**: 用户能精确控制每类通知的接收方式和时间，设置即时生效

#### 故事列表

##### 故事 6.1.1: 分别设置互动通知开关与渠道

**作为** 用户，  
**我希望** 能够分别设置点赞、评论、关注、收藏、@我、私信等通知开关，并选择推送渠道（App内/短信/邮件），  
**以便** 控制消息打扰并获得重要通知。

**验收标准**:
1. **给定** 我进入"通知设置"页面，**当** 我查看通知类型列表，**则** 我看到所有可配置的通知类型（点赞、评论、关注、收藏、@我、私信等）
2. **给定** 我想关闭点赞通知，**当** 我将"点赞"通知开关关闭，**则** 系统不再发送点赞相关的通知
3. **给定** 我想只通过 App 接收评论通知，**当** 我关闭短信和邮件渠道，仅保留 App 内通知，**则** 系统仅在 App 内推送评论通知
4. **给定** 我想接收所有渠道的 @我 通知，**当** 我开启所有渠道开关，**则** 系统通过 App、短信、邮件三种方式通知我
5. **给定** 我修改了通知设置，**当** 保存成功，**则** 新设置立即生效，后续通知按新规则发送

**附加说明**:
- **优先级**: 高
- **依赖项**: EPIC-01（用户需先注册登录）
- **技术说明**: 通知配置存储于用户偏好表；通知发送服务读取配置决定发送渠道

##### 故事 6.1.2: 设置免打扰时段

**作为** 用户，  
**我希望** 能够设置免打扰时段，  
**以便** 在休息时减少打扰。

**验收标准**:
1. **给定** 我想在夜间免打扰，**当** 我设置免打扰时段为 23:00-07:00，**则** 系统在该时段内不发送非紧急通知
2. **给定** 在免打扰时段内有新通知，**当** 时段结束，**则** 系统汇总发送摘要通知（如配置了摘要模式）
3. **给定** 我有多个免打扰时段，**当** 我设置工作日和周末不同时段，**则** 系统按不同规则执行
4. **给定** 我想临时关闭免打扰，**当** 我点击"暂时关闭"，**则** 系统在接下来 1 小时内正常发送通知
5. **给定** 安全类通知（如异地登录提醒），**当** 在免打扰时段触发，**则** 系统仍然发送（白名单豁免）

**附加说明**:
- **优先级**: 中
- **依赖项**: 故事 6.1.1
- **技术说明**: 免打扰逻辑在通知发送层判断；支持时区适配

### 特性 2: 隐私可见性控制

**描述**: 允许用户控制动态可见性、在线状态和搜索引擎索引，管理隐私边界
**关键交互**: 进入隐私设置 → 选择可见性选项 → 保存 → 生效
**非功能需求**: 隐私设置调整后对新请求即时生效，缓存页面在 5 分钟内失效
**验收条件**: 用户能成功设置各类隐私选项，设置即时生效且不会泄露隐私

#### 故事列表

##### 故事 6.2.1: 设置动态可见性

**作为** 用户，  
**我希望** 能设置"我的动态（浏览记录、点赞动态、收藏夹）"的可见性，  
**以便** 管理隐私边界。

**验收标准**:
1. **给定** 我想隐藏我的浏览记录，**当** 我将"浏览记录"设置为"仅自己可见"，**则** 其他用户无法在我的主页看到我的浏览历史
2. **给定** 我想让关注者看到我的点赞动态，**当** 我将"点赞动态"设置为"仅关注者可见"，**则** 只有关注我的人能看到我点赞了哪些内容
3. **给定** 我想公开我的收藏夹，**当** 我将"收藏夹"设置为"公开"，**则** 任何用户都能查看我的收藏夹
4. **给定** 我修改了可见性设置，**当** 其他用户刷新页面，**则** 他们在 5 分钟内看到更新后的可见性效果
5. **给定** 我将某项设置为"仅自己可见"，**当** 我查看自己的主页，**则** 我仍能看到该内容（对自己始终可见）

**附加说明**:
- **优先级**: 高
- **依赖项**: EPIC-01
- **技术说明**: 可见性判定在查询接口层统一执行；使用 Redis 缓存，变更后 5 分钟内失效

##### 故事 6.2.2: 设置在线状态可见性

**作为** 用户，  
**我希望** 能设置在线状态对他人是否可见，  
**以便** 控制即时存在感。

**验收标准**:
1. **给定** 我想隐藏在线状态，**当** 我将"在线状态"设置为"隐藏"，**则** 其他用户无法看到我是否在线
2. **给定** 我设置为隐藏在线状态，**当** 我实际在线，**则** 系统对我显示"在线"，但对其他人显示"离线"或隐藏
3. **给定** 我想仅对互关好友显示在线状态，**当** 我设置为"仅互关可见"，**则** 只有与我互关的用户能看到我的在线状态
4. **给定** 我设置为公开在线状态，**当** 我在线，**则** 所有用户都能看到我的在线标识
5. **给定** 我修改了在线状态设置，**当** 保存成功，**则** 新设置立即生效

**附加说明**:
- **优先级**: 中
- **依赖项**: EPIC-01
- **技术说明**: 在线状态通过 WebSocket 或心跳机制维护；可见性在查询时判定

##### 故事 6.2.3: 设置搜索引擎索引权限

**作为** 用户，  
**我希望** 能设置是否允许被搜索引擎索引，  
**以便** 满足隐私与曝光需求。

**验收标准**:
1. **给定** 我不想被搜索引擎收录，**当** 我将"允许搜索引擎索引"设置为"否"，**则** 系统在我的个人主页添加 noindex meta 标签
2. **给定** 我禁止搜索引擎索引，**当** 搜索引擎爬虫访问我的主页，**则** 系统返回 noindex 指令，搜索引擎不收录该页面
3. **给定** 我想被搜索引擎收录以提升曝光，**当** 我设置为"允许"，**则** 系统移除 noindex 标签，允许搜索引擎抓取
4. **给定** 我修改了索引设置，**当** 保存成功，**则** 新设置在下次爬虫访问时生效
5. **给定** 我是认证用户或创作者，**当** 我查看设置，**则** 系统默认建议我开启索引以提升曝光

**附加说明**:
- **优先级**: 低
- **依赖项**: EPIC-01
- **技术说明**: 通过 HTTP header 或 meta 标签控制；已收录的页面需等待搜索引擎重新抓取

### 特性 3: 第三方授权管理

**描述**: 允许用户查看和管理第三方应用/服务的访问授权，并可撤销授权
**关键交互**: 进入授权管理 → 查看授权列表 → 选择撤销 → 确认 → 生效
**非功能需求**: 撤销授权后立即生效，相关 Token 立即失效
**验收条件**: 用户能清晰看到所有授权，能随时撤销授权，撤销后立即生效

#### 故事列表

##### 故事 6.3.1: 查看与管理第三方授权

**作为** 用户，  
**我希望** 能查看第三方应用/服务访问授权列表并撤销授权，  
**以便** 控制数据访问。

**验收标准**:
1. **给定** 我进入"第三方授权"页面，**当** 页面加载，**则** 我看到所有已授权的应用列表（应用名称、授权时间、授权范围）
2. **给定** 我想了解某个应用的权限，**当** 我点击查看详情，**则** 我看到该应用可以访问的数据范围（如个人资料、发布内容等）
3. **给定** 我想撤销某个应用的授权，**当** 我点击"撤销授权"并确认，**则** 系统立即使该应用的 Access Token 失效
4. **给定** 我撤销了授权，**当** 该应用尝试访问我的数据，**则** 系统拒绝访问并返回"授权已撤销"错误
5. **给定** 我想重新授权，**当** 我再次使用该应用，**则** 系统引导我重新进行 OAuth 授权流程

**附加说明**:
- **优先级**: 中
- **依赖项**: EPIC-01
- **技术说明**: 授权信息存储于 OAuth 授权表；撤销时清除 Token 并通知第三方应用（可选）

### 特性 4: 账户安全设置入口

**描述**: 提供统一的账户安全设置入口，集中管理设备管理、密码修改、两步验证、登录提醒等功能
**关键交互**: 进入账户安全 → 选择安全功能 → 配置 → 保存
**非功能需求**: 安全设置调整后即时生效，敏感操作需二次验证
**验收条件**: 用户能方便地访问所有安全功能，能统一管理账户安全

#### 故事列表

##### 故事 6.4.1: 统一账户安全设置入口

**作为** 用户，  
**我希望** 拥有账户安全设置入口（设备管理、密码修改、两步验证、登录提醒），  
**以便** 提升账号安全。

**验收标准**:
1. **给定** 我进入"账户与安全"页面，**当** 页面加载，**则** 我看到所有安全功能的入口（设备管理、密码修改、两步验证、登录提醒）
2. **给定** 我想查看登录设备，**当** 我点击"设备管理"，**则** 我跳转到设备管理页面（EPIC-01 已实现）
3. **给定** 我想修改密码，**当** 我点击"密码修改"，**则** 我进入密码修改流程，需输入原密码和新密码
4. **给定** 我想启用两步验证，**当** 我点击"两步验证"，**则** 系统引导我绑定手机或邮箱作为第二步验证方式
5. **给定** 我想设置登录提醒，**当** 我开启"登录提醒"，**则** 系统在每次新设备登录时发送通知

**附加说明**:
- **优先级**: 中
- **依赖项**: EPIC-01（设备管理、密码修改已在 EPIC-01 实现）
- **技术说明**: 本史诗主要提供统一入口和导航；具体功能在其他史诗中实现

### 特性 5: 通知退订规则与合规

**描述**: 确保用户关闭某类通知后，系统不得继续通过同一渠道发送非必要提醒（安全类通知除外）
**关键交互**: 用户关闭通知 → 系统停止发送 / 安全通知始终发送
**非功能需求**: 通知退订即时生效，符合 GDPR 等合规要求
**验收条件**: 用户关闭通知后确实不再收到，安全类通知例外

#### 故事列表

##### 故事 6.5.1: 严格执行通知退订规则

**作为** 平台系统，  
**我希望** 用户关闭某类通知后，系统不得继续通过同一渠道发送非必要提醒（安全类通知除外），  
**以便** 符合合规与体验要求。

**验收标准**:
1. **给定** 用户关闭了营销类通知，**当** 系统准备发送营销通知，**则** 系统检查用户偏好并跳过发送
2. **给定** 用户关闭了所有推送渠道，**当** 系统准备发送非安全通知，**则** 系统不发送任何通知
3. **给定** 有安全类通知（如异地登录、密码修改），**当** 即使用户关闭了通知，**则** 系统仍然发送（白名单机制）
4. **给定** 用户投诉收到已关闭的通知，**当** 系统收到投诉，**则** 系统检查日志并确认是否违规发送
5. **给定** 系统发现违规发送，**当** 确认后，**则** 系统修复 bug 并向用户道歉

**附加说明**:
- **优先级**: 高
- **依赖项**: 故事 6.1.1
- **技术说明**: 通知发送前必须检查用户偏好；安全通知白名单配置化；定期审计通知发送日志

## 风险与假设

- **风险**: 
  - 隐私设置过于复杂导致用户困惑
  - 缓存失效不及时可能导致隐私泄露
  - 用户关闭所有通知后错过重要信息
- **假设**: 
  - 用户重视隐私控制并愿意花时间设置
  - 通知渠道（短信、邮件）送达率高
  - 第三方应用遵守 OAuth 协议，撤销授权后立即停止访问
