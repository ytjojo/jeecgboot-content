## 1. 需求说明

### 需求目标
- **需求名称**: 社交关系与内容订阅（social-subscription）
- **一句话概述**: 构建用户关注体系与多维度内容订阅能力，促进社交互动与内容发现
- **要解决的问题**: 内容社区缺少完善的关注管理、分组、特别关注和多源订阅能力，用户难以高效管理信息流
- **期望达成的结果**: 平均关注数提升 40%、关注流点击率 >30%、订阅转化率 >25%、用户留存率提升 20%

#### 成功指标定义

| 指标 | 定义公式 | 基线值 | 目标值 | 衡量方式 | 数据来源 | 达成时间窗口 |
|------|----------|--------|--------|----------|----------|-------------|
| 平均关注数 | 总关注关系数 / 活跃用户数 | 待采集（上线前统计当前均值） | 基线值 × 1.4 | 后端定时任务每日统计，按周聚合 | user_follow 表 COUNT | 上线后 4 周 |
| 关注流点击率 | 关注流内容点击次数 / 关注流内容曝光次数 | 待采集 | >30% | 前端埋点：曝光事件 + 点击事件，按日聚合 | 前端埋点 SDK | 上线后 2 周 |
| 订阅转化率 | 新增订阅数 / 订阅广场页面 UV | 待采集 | >25% | 前端埋点：广场页面 PV/UV + 订阅按钮点击事件 | 前端埋点 SDK + 后端订阅表 | 上线后 2 周 |
| 用户留存率 | 次周仍活跃的用户数 / 本周活跃用户数 | 待采集 | 基线值 × 1.2 | 后端用户活跃度统计（登录或产生行为视为活跃），按周计算 | 用户行为日志 | 上线后 8 周 |

**埋点关键事件清单**:
- `follow_click` — 关注按钮点击（携带 targetUserId、来源页面）
- `unfollow_click` — 取消关注点击
- `feed_expose` — 关注流内容曝光（携带 contentId、动态类型）
- `feed_click` — 关注流内容点击
- `subscribe_click` — 订阅按钮点击（携带 sourceId、来源类型）
- `subscribe_square_expose` — 订阅广场页面曝光
- `batch_operation` — 批量操作执行（携带操作类型、数量、成功/失败数）

### 目标用户与使用场景
- **主要用户**: 所有注册内容社区用户
- **次要用户**: 内容创作者（被关注方、被订阅方）
- **使用场景**: 桌面端 + 移动端跨端适配
- **入口路径**:

## 3. 功能详细说明

### 3.1 关注与取消关注

#### 核心用户流程
1. **触发条件**: 用户浏览到其他用户主页、内容详情页或推荐列表
2. **操作步骤**: 点击用户卡片上的"关注"按钮
3. **系统反馈**: 按钮即时变为"已关注"（灰色描边），关注数 +1
4. **完成条件**: 关注关系建立成功

#### 交互要求
- **主操作**: "关注"/"已关注"切换按钮
- **已关注态 hover**: 按钮文字变为"取消关注"（红色描边）
- **取消关注**: 点击后弹出 Popconfirm 二次确认："确定取消关注该用户？"，确认后恢复"关注"状态
- **禁止自关注**: 当前用户与目标用户相同时，按钮不展示或展示为禁用态
- **防抖**: 所有关注操作 500ms 内不重复请求
- **乐观更新**: 先更新 UI 状态，API 失败时回滚并提示错误

#### 布局与结构要求
- **用户主页**: 关注按钮位于用户名右侧或头像下方，作为主操作
- **用户卡片（列表/推荐）**: 关注按钮位于卡片右侧
- **内容详情页**: 内容作者头像旁显示关注按钮

#### 状态与边界情况
- **加载中**: 按钮显示 loading 图标，禁止重复点击
- **操作成功**: 按钮状态切换 + 全局消息提示"关注成功"/"已取消关注"
- **网络错误**: 保持原状态，Toast 提示"操作失败，请重试"
- **目标用户已注销/被封禁**: 按钮禁用，Tooltip 提示"该用户暂不可关注"
- **已被对方拉黑**: 按钮禁用，提示"无法关注该用户"

### 3.2 特别关注与强提醒

#### 核心用户流程
1. **触发条件**: 用户已关注某用户，在其主页或关注列表中操作
2. **操作步骤**: 点击星标图标 → 弹出确认"设为特别关注后，该用户动态将置顶显示并开启强提醒" → 确认
3. **系统反馈**: 星标变为实心金色，特别关注数 +1
4. **完成条件**: 特别关注标记设置成功

#### 交互要求
- **未关注时**: 星标按钮禁用，Tooltip 提示"请先关注该用户"
- **已关注未特别关注**: 空心星标图标，点击后二次确认
- **已特别关注**: 实心金色星标，点击后二次确认取消
- **特别关注标识**: 在关注流动态卡片中，特别关注用户的卡片左侧显示金色竖线或星标角标

#### 布局与结构要求
- **用户主页**: 星标按钮位于关注按钮旁边
- **关注列表**: 每个用户卡片的操作区显示星标按钮
- **关注流**: 特别关注用户的动态在列表顶部独立分区展示

#### 状态与边界情况
- **空状态**: 特别关注列表为空时，展示引导图 + "设置特别关注，重要动态不再错过" + 关注列表入口按钮

### 3.3 关注分组管理

#### 核心用户流程
1. **触发条件**: 用户进入"分组管理"页面
2. **操作步骤**: 点击"新建分组" → 填写分组名和排序值 → 保存
3. **系统反馈**: 分组列表刷新，新增一项
4. **完成条件**: 分组创建成功

#### 交互要求
- **新建分组**: 点击按钮弹出 Modal 表单
  - 分组名: 必填，最多 20 字符，实时校验唯一性（同用户名下不可重复）
  - 排序值: 可选，数字输入，默认 0
  - 底部按钮: "取消" + "确定"
- **重命名**: 点击分组项的"重命名"操作 → 弹出 Modal，同新建表单
- **删除**: 点击"删除" → Popconfirm 二次确认："删除后该分组内的关注用户将回到默认分组"
  - 默认分组的"删除"按钮不展示
- **移动用户进组**: 在用户卡片点击"调整分组" → Dropdown 展示所有分组列表 → 选择后即时更新
- **移出分组**: 在用户卡片点击"移出分组" → 用户回到默认分组

#### 布局与结构要求
- **PC 端**: 左侧分组列表面板（宽度 280px）+ 右侧成员列表区域
- **移动端**: 全屏分组列表，点击某分组进入成员列表（导航式页面跳转）
- **分组列表项**: 分组名（左）+ 成员数 badge（中）+ 操作按钮组（右）
- **默认分组**: 列表第一位，图标标记"默认"标签，不可删除不可重命名

#### 状态与边界情况
- **分组名重复**: 输入框下方红色提示"该分组名已存在"
- **分组名为空**: 输入框下方红色提示"请输入分组名"
- **删除含成员的分组**: 确认弹窗中额外提示"当前分组有 N 位关注用户，删除后将回到默认分组"

### 3.6 关注流

#### 数据流架构

**聚合模式**: 读扩散（实时查询聚合），不引入写扩散或消息扇出。

**接口调用时序**:
1. 前端请求 `GET /api/v1/content/user/relation/feed?pageNo=1&pageSize=20&types=post,like,favorite`
2. 后端根据当前用户的关注列表，实时聚合查询各关注对象的动态
3. 后端返回分页数据，包含动态列表 + 游标/页码 + 是否有更多数据

**关键设计决策**:
- **特别关注置顶**: 后端分两个区域返回数据。接口返回 `{ priorityItems: [...], items: [...] }`，`priorityItems` 为特别关注用户的最新动态（最多展示最近 24 小时内的 5 条），`items` 为常规关注流数据。前端拼接展示，无需二次排序。
- **动态类型筛选**: 后端支持 `types` 参数过滤。前端切换筛选条件后重新请求接口，不进行前端过滤。
- **关注量性能边界**: 当关注数超过 500 时，后端仅聚合最近 7 天的动态；超过 2000 时，仅聚合最近 3 天的动态。前端对用户无感知，仅影响数据时效性。
- **超时降级**: 单次聚合查询时间预算 2s，超时后降级返回最近 N 小时的动态（N 根据关注量动态调整），前端展示提示"部分内容加载中"。

**前端状态管理**:
- 关注流数据使用 Pinia store 管理，支持缓存和增量更新
- 切换动态类型筛选时清空缓存重新请求
- 下拉刷新时请求最新数据，有新数据替换列表，无新数据提示"暂无新内容"

#### 核心用户流程
1. **触发条件**: 用户进入首页"关注"标签页
2. **操作步骤**: 浏览动态 → 可选配置动态类型 → 滚动加载更多
3. **系统反馈**: 展示关注对象的动态内容

#### 交互要求
- **动态类型筛选**: 顶部三个开关 Toggle：发布（默认开）、点赞（默认开）、收藏（默认开）
  - 至少保留一个开启，全部关闭时提示"至少选择一种动态类型"
  - 切换后立即重新请求数据
  - 配置自动保存
- **动态卡片点击**: 点击内容区域跳转到内容详情
- **用户头像点击**: 跳转到用户主页
- **滚动加载**: 距底部 200px 自动加载下一页，200ms 节流
- **下拉刷新**: 移动端支持下拉刷新，有新数据替换列表，无新数据提示"暂无新内容"

#### 布局与结构要求
- **页面结构**: 标签栏（关注/推荐/订阅）→ 动态筛选栏 → 动态内容列表
- **动态卡片组成**: 用户头像 + 昵称 + 动态类型标签（发布/点赞/收藏）+ 内容摘要 + 时间（相对时间）
- **特别关注区**: 列表顶部独立分区，卡片左侧金色竖线标识
- **PC 端**: 单列宽卡片，最大宽度 680px 居中
- **移动端**: 全宽卡片，隐藏内容摘要，仅显示标题和时间

#### 状态与边界情况
- **加载中**: 骨架屏（3 个动态卡片占位）
- **空状态（无关注）**: 引导图 + "关注感兴趣的创作者，精彩内容将在这里展示" + 推荐关注入口
- **无新内容**: 下拉刷新后 Toast 提示"暂无新内容"
- **加载失败**: 列表底部"加载失败，点击重试"按钮

### 3.7 关注推荐

#### 接口设计

**推荐列表接口**:
- 请求: `GET /api/v1/content/user/relation/recommendations?pageNo=1&pageSize=20`
- 响应字段:
  - `id`: 用户 ID
  - `nickname`: 昵称
  - `avatar`: 头像
  - `bio`: 简介
  - `reasonType`: 推荐理由类型（`similar` | `popular` | `mutual_follow` | `interest_tag`）
  - `reasonText`: 后端返回的推荐理由文案（如"与您关注的 XXX 相似"、"共同关注 XX 人"）
  - `mutualFollowCount`: 共同关注数（`reasonType = mutual_follow` 时返回）
  - `similarUserName`: 相似用户昵称（`reasonType = similar` 时返回）
- 分页: 基于游标分页，返回 `hasMore` + `nextCursor`
- 数量上限: 单次最多返回 20 条，前端滚动加载最多累计加载 200 条
- 排除逻辑: **后端实现**，排除当前用户已关注、已拉黑、已注销的用户

**用户反馈接口**:
- 请求: `POST /api/v1/content/user/relation/recommendations/feedback`，body: `{ userId, action: 'not_interested' }`
- 用户点击"不感兴趣"后，该用户不再出现在推荐列表中

### 3.8 批量管理

#### 核心用户流程
1. **触发条件**: 用户在关注列表/特别关注列表点击"批量管理"
2. **操作步骤**: 多选用户 → 选择操作 → 确认执行 → 查看结果
3. **系统反馈**: 弹窗展示操作结果明细

#### 交互要求
- **多选模式**: 每个用户卡片左侧出现 Checkbox，顶部出现"全选"
- **操作栏**: 底部固定操作栏，显示"已选 N 人" + 操作按钮组：
  - 批量取消关注
  - 批量取消特别关注
  - 批量移动分组（弹出分组选择 Dropdown）
- **执行确认**: 点击操作后弹出 Modal 二次确认："确定对 N 位用户执行 XXX 操作？"

- **失败分类与重试**:
  - **可重试失败**（网络超时、服务端 500 等临时性错误）: 失败项标橙色，显示"可重试"标签，弹窗底部展示"重试失败项"按钮

  - 最多重试 2 轮，第 2 轮仍有失败则全部标记为不可重试

#### 布局与结构要求
- **底部操作栏**: 固定在页面底部，白色背景 + 顶部阴影，包含选中计数和操作按钮
- **结果弹窗**: 成功/失败统计顶部，失败项列表下方，可重试项在前、不可重试项在后，底部"重试失败项"按钮（仅存在可重试项时展示）

#### 状态与边界情况
- **未选择任何用户**: 操作按钮禁用
- **部分失败**: 结果弹窗中可重试项标橙色，不可重试项标红色并显示具体原因
- **重试中**: "重试失败项"按钮显示 loading 状态，禁用重复点击

  - 数据列表 → `Table`（`src/components/Table/`）或 `JVxeTable`
  - 新增/编辑弹窗 → `Modal` + `Form`（`src/components/Modal/` + `src/components/Form/`）

  - 页面容器 → `Page`（`src/components/Page/`）
  - 卡片展示 → `CardList`（`src/components/CardList/`）
  - 按钮 → `Button`（`src/components/Button/`，支持权限控制）
  - 图标 → `Icon`（`src/components/Icon/`，Iconify + SVG）
  - 加载状态 → `Loading`（`src/components/Loading/`）
  - 下拉菜单 → `Dropdown`（`src/components/Dropdown/`）
- **Hooks 优先复用**:
  - `useTable` → 表格逻辑封装
  - `useForm` → 表单逻辑封装
  - `useModal` → 弹窗逻辑封装
  - `useMessage` → 消息提示
  - `usePermission` → 权限判断

### 主题色
- 主色: `--j-global-primary-color`（动态设置）
- 预设色板: `#0960bd`, `#1890ff`, `#009688`, `#536dfe` 等
- 特别关注标识色: 金色（建议 `#faad14`）
- 危险操作色: 红色（Ant Design Vue 默认 danger 色）

**关注流/订阅流**
- PC 端: 单列宽卡片，最大宽度 680px 居中
- 平板端: 单列布局，卡片宽度自适应
- 移动端: 全宽卡片，隐藏内容摘要仅显示标题

- PC 端: 表格布局，展示完整信息列
- 平板端: 表格布局，隐藏次要列（关注时间、订阅时间）

**订阅广场**
- PC 端 (>=1200px): 4 列卡片网格

### 通用交互要求

以下交互适用于所有页面：

- **加载状态**: 数据请求期间显示骨架屏（Skeleton）或 Spin 加载指示
- **空状态**: 无数据时显示空状态插图 + 引导文案 + 操作入口按钮
- **错误反馈**: 操作失败时 Toast 提示明确错误信息，保留用户已输入内容
- **表单校验**: 必填字段实时校验（失焦触发），提交时触发全量校验，错误提示显示在对应字段下方红色文字
- **防重复提交**: 提交类按钮在请求期间显示 loading 并禁用
- **危险操作确认**: 取消关注、取消订阅、删除分组等不可逆操作需 Popconfirm 或 Modal 二次确认
- **成功反馈**: 操作成功后右上角显示成功消息提示（useMessage）
- **防抖**: 搜索输入 300ms 防抖
- **节流**: 滚动加载 200ms 节流
- **乐观更新**: 关注/订阅等高频操作先更新 UI，失败时回滚
- **触摸目标**: 移动端可点击元素最小 44px x 44px

### 待确认问题
1. **关注流中的点赞/收藏动态是否需要受内容作者隐私或内容可见范围二次过滤？** 当前假设：必须过滤，不展示被屏蔽内容
2. **订阅源（专题、合集、话题、栏目）当前是否都有稳定的内容主表？** 当前假设：需要先定义轻量目录表承载广场展示。**前置依赖**：订阅广场的分类浏览和搜索功能依赖统一的订阅源目录表数据模型。在本 PRD 落地前，必须先完成后端数据模型设计评审，确认目录表结构（至少包含 sourceId、sourceType、name、category、subscriberCount、updatedAt 等字段），否则前端分类浏览和搜索功能无法对接。
3. **邮件摘要由内容社区自建定时任务发送，还是接入系统现有消息中心统一发送？** 当前假设：暂不在前端处理，后端统一调度
4. **关注列表和订阅列表是否需要同步展示粉丝数、订阅人数等计数快照？** 当前假设：列表 VO 聚合返回，前端直接展示

### 默认假设
1. **用户身份**: 当前阶段继续使用显式 `userId` 参数，不在前端强制读取登录态上下文
2. **推荐算法**: 本期使用可解释规则（热门创作者、共同关注、兴趣标签），不接入机器学习模型
3. **关注流实现**: 读扩散（实时查询聚合），不引入写扩散或消息扇出
4. **批量操作**: 后端返回成功数、失败数和失败原因明细，前端按明细展示结果
5. **通知配置层级**: 订阅级配置覆盖全局默认值，未配置时回退全局设置
6. **付费订阅**: 本期不实现，仅在订阅状态字段保留扩展点
7. **语言**: 默认中文界面，暂不实现国际化
8. **分组数量上限**: 假设单用户最多 20 个自定义分组
9. **批量操作上限**: 假设单次批量操作最多 100 条
10. **关注数上限**: 单用户最大关注数 5000 人。达到上限后，"关注"按钮点击时 Toast 提示"关注数已达上限（5000），请先取消部分关注"。特别关注无额外上限（从已有关注中设置）。
11. **大关注量用户降级**: 关注数超过 500 时，关注流仅聚合最近 7 天动态；超过 2000 时，仅聚合最近 3 天动态。降级对用户无感知。
