# EPIC-11 圈子内容与互动 - 前端 PRD

## 1. 概述

### 需求目标
- **需求名称**: 圈子内容与互动（circle-content-interaction）
- **概述**: 在 circle-core MVP 基础上，补齐内容管理增强（置顶/精华）、圈子互动（公告/@成员）和审核机制（加入申请/举报处理）
- **要解决的问题**: MVP 圈子缺乏内容质量管控、成员互动方式和违规内容处理能力
- **期望达成结果**: 圈子运营者可突出优质内容、传达重要信息；成员可互动提及；管理员可审核申请和处理举报

### 术语定义

本文档中使用以下统一称谓：

| 术语 | 定义 | 说明 |
|------|------|------|
| **圈子管理员** | 版主 + 创建者的统称 | 具备圈子内管理权限的角色，包括置顶/精华、公告发布、申请审核、举报处理等操作 |
| **创建者** | 圈子创建者 | 圈子的最高权限拥有者，拥有所有圈子管理员权限，额外拥有禁言用户的权限 |
| **版主** | 圈子版主 | 由创建者授予管理权限，可执行置顶/精华、公告发布、申请审核、举报处理（不含禁言） |
| **普通成员** | 非管理角色的圈子成员 | 可浏览内容、@其他成员、提交举报，无管理操作权限 |
| **平台管理员** | 系统级管理员 | 本 Epic 不涉及平台管理员的操作，如有需要由后续 PRD 定义 |

> 全文统一使用"圈子管理员"指代具有管理权限的角色，不再交替使用"版主/创建者"和"管理员"。

### 目标用户与使用场景
| 角色 | 使用场景 | 入口 |
|------|---------|------|
| 圈子创建者 | 管理圈子内容、发布公告、审核申请/举报 | 圈子管理后台 |
| 圈子版主 | 置顶/精华内容、发布公告、审核申请/举报 | 圈子管理后台 |
| 普通成员 | 浏览置顶/精华内容、@其他成员、提交举报 | 圈子内容列表/详情页 |
| 申请用户 | 提交加入申请 | 圈子首页 |

### 范围定义

#### 本期范围
- 内容置顶与精华标记（列表排序、标识展示、权限控制）
- 圈子公告发布与展示（单条生效、有效期、顶部展示）
- 圈子内 @成员（候选列表、提及解析、通知触发）
- 加入申请审核（申请列表、批准/拒绝、超时提醒）
- 内容举报处理（举报提交、处理操作、结果通知）

#### 非本期范围
- 数据统计与分析面板
- 推荐算法
- 成长激励体系
- AI 自动内容审核

## 3. 页面设计

### 3.1 圈子内容列表页（增强）

**页面职责**: 展示圈子内内容列表，支持置顶排序和精华标识

**布局结构**:
```
┌─────────────────────────────────────────────┐
│ 圈子公告栏（条件展示，仅当有生效公告时）        │
├─────────────────────────────────────────────┤
│ 内容列表                                      │
│  ┌─ 置顶内容卡片（带置顶标识，按 pinned_at 排序）─┐
│  │  [置顶] [精华] 标题 / 作者 / 时间             │
│  └──────────────────────────────────────────┘
│  ┌─ 普通内容卡片 ────────────────────────────┐
│  │  [精华] 标题 / 作者 / 时间                  │
│  └──────────────────────────────────────────┘
│  ...分页加载                                    │
└─────────────────────────────────────────────┘
```

**交互规则**:
- 置顶内容始终排列在非置顶内容之前，多条置顶按 `pinned_at` 倒序
- 精华内容在卡片左上角展示精华标识（金色徽章）
- 版主/创建者可见操作菜单（更多按钮），包含"置顶/取消置顶""精华/取消精华""举报"选项
- 普通成员仅可见"举报"选项
- 点击操作后即时更新列表状态，无需刷新页面

**状态与边界**:
- 无内容时：展示空状态插图 + "暂无内容"文案
- 加载中：骨架屏占位
- 权限不足操作：弹出 Toast 提示"权限不足"

### 3.3 公告发布弹窗

**触发方式**: 版主/创建者在圈子管理区域点击"发布公告"按钮

**布局结构**:
```
┌─────────────────────────────────────┐
│ 发布公告                         [X] │
├─────────────────────────────────────┤
│ 公告内容 *                           │
│ ┌─────────────────────────────────┐ │
│ │ 富文本编辑器（Tinymce）           │ │
│ └─────────────────────────────────┘ │
│                                     │
│ 有效期 *                             │
│ ┌─────────────────────────────────┐ │
│ │ 日期时间选择器（至）              │ │
│ └─────────────────────────────────┘ │
│                                     │
│ 提示：发布新公告将替换当前生效公告    │
├─────────────────────────────────────┤
│              [取消]  [发布]          │
└─────────────────────────────────────┘
```

**交互规则**:
- 公告内容为必填，使用 Tinymce 富文本编辑器
- 有效期为必填，仅设置截止时间（expireAt），不得早于当前时间；公告起始时间为发布时间（createdAt），展示格式为"有效期至：YYYY-MM-DD"
- 点击"发布"时，若已有生效公告，弹出确认框："当前已有生效公告，发布新公告将替换旧公告，是否继续？"
- 发布成功后关闭弹窗，圈子顶部展示新公告
- 普通成员无"发布公告"按钮入口

**状态与边界**:
- 内容为空时提交：字段下方显示红色错误提示"请输入公告内容"
- 有效期已过时提交：字段下方显示"有效期不得早于当前时间"
- 发布中：按钮显示 loading 并禁用

### 3.5 @成员选择器

**触发方式**: 用户在内容发布框或评论输入框中输入 `@` 字符

**布局结构**:
```
┌─────────────────────────────────────┐
│ @成员（搜索）                    [X] │
├─────────────────────────────────────┤
│ 🔍 [搜索成员...]                     │
├─────────────────────────────────────┤
│ 👤 张三 (版主)                       │
│ 👤 李四                              │
│ 👤 王五                              │
│ ...                                  │
│ ── 已选成员 ──                        │
│ 👤 张三 [×]                          │
└─────────────────────────────────────┘
```

**交互规则**:
- 输入 `@` 后弹出浮层，展示当前圈子成员列表
- 支持搜索过滤，按昵称/姓名模糊匹配
- 仅展示当前圈子内成员，已退出成员不展示
- 点击成员项插入 @提及标记，浮层关闭
- 支持键盘上下选择 + Enter 确认
- 已选成员显示在输入框下方，可点击 [×] 移除
- 浮层在输入框上方或下方自动定位（空间不足时翻转）

**@提及标记数据格式约定**:

前端在富文本/纯文本中插入 @提及标记，需遵循以下格式规范：

```typescript
// 前端插入格式（纯文本场景）：@{userId:xxx}昵称
// 示例：@{userId:u001}张三

// 前端插入格式（富文本场景 Tinymce）：
// 使用自定义标签：<span class="mention" data-user-id="xxx">@昵称</span>

// 传给后端 API 的内容格式：
// 纯文本：@{userId:xxx}昵称
// 富文本：<span class="mention" data-user-id="xxx">@昵称</span>

// 后端返回内容格式：同上，前端需解析为可点击链接
// 渲染逻辑：正则匹配 @{userId:xxx}昵称 或解析 mention 标签，渲染为 <a> 标签并绑定点击事件（跳转用户主页）
```

> 评论中的 @提及格式与内容中保持一致。后端负责在发布内容/评论时解析 @提及标记并触发异步通知。

**状态与边界**:
- 圈子无成员时：浮层显示"暂无可提及成员"
- 搜索无结果时：显示"未找到匹配成员"
- 加载中：成员列表区域显示 Loading

### 3.6 加入申请审核页

**页面职责**: 展示待处理的加入申请列表，支持批准/拒绝操作

**布局结构**:
```
┌─────────────────────────────────────────────┐
│ 加入申请审核           [待处理] [已处理] [全部] │
├─────────────────────────────────────────────┤
│ 筛选区：申请时间范围 / 状态筛选               │
├─────────────────────────────────────────────┤
│ ┌─ 申请卡片 ────────────────────────────┐   │
│ │ 👤 用户头像  用户昵称                    │   │
│ │ 申请时间：2026-06-01 10:00              │   │
│ │ 申请理由：...                           │   │
│ │                    [拒绝]  [批准]       │   │
│ └────────────────────────────────────────┘   │
│ ...分页                                      │
└─────────────────────────────────────────────┘
```

**交互规则**:
- 默认展示"待处理"标签页，按申请时间排序
- 超过 3 天未处理的申请卡片显示橙色警告标识
- 超时提醒机制说明：
  - **前端展示层**：超过 3 天未处理的申请卡片显示橙色警告标识（前端根据 `applyTime` 判断是否超过 3 天）
  - **管理入口角标**：圈子管理入口展示红点/角标，数字来源于 `pendingJoinRequestCount`（复用 Store 中的状态）
  - **通知层**：超时不触发后端推送通知（仅前端视觉提示），如需站内信/消息通知由后续 PRD 定义
- 点击"批准"：弹出确认框"确认批准该用户的加入申请？"，确认后调用 API
- 点击"拒绝"：弹出输入框要求填写拒绝原因（必填），确认后调用 API
- 操作成功后，该卡片从"待处理"列表移除，Toast 提示"已批准"或"已拒绝"
- 支持批量操作：勾选多条申请后批量批准

**状态与边界**:
- 无待处理申请时：展示空状态"暂无待处理申请"
- 拒绝原因为空时提交：输入框下方显示"请填写拒绝原因"
- 网络错误时：Toast 提示"操作失败，请重试"，卡片保留在列表中

### 3.7 举报处理页

**页面职责**: 展示举报列表，支持圈子管理员处理举报

**访问权限与操作权限矩阵**:

| 操作 | 创建者 | 版主 | 普通成员 |
|------|--------|------|----------|
| 查看举报列表 | 是 | 是 | 否 |
| 查看被举报内容详情 | 是 | 是 | 否 |
| 删除被举报内容 | 是 | 是 | 否 |
| 忽略举报 | 是 | 是 | 否 |
| 禁言用户 | 是 | 否 | 否 |

> 说明：禁言用户操作仅限创建者，版主不具有禁言权限。举报处理页入口仅对圈子管理员（创建者+版主）可见，普通成员无入口。

**布局结构**:
```
┌─────────────────────────────────────────────┐
│ 举报处理              [待处理] [已处理] [已忽略] │
├─────────────────────────────────────────────┤
│ 筛选区：举报时间范围 / 状态筛选               │
├─────────────────────────────────────────────┤
│ ┌─ 举报卡片 ────────────────────────────┐   │
│ │ 举报者：张三  →  被举报内容：标题        │   │
│ │ 举报原因：违规广告                       │   │
│ │ 举报时间：2026-06-01 10:00              │   │
│ │ [查看内容]     [忽略] [禁言用户] [删除内容]│   │
│ └────────────────────────────────────────┘   │
│ ...分页                                      │
└─────────────────────────────────────────────┘
```

**交互规则**:
- 默认展示"待处理"标签页
- 点击"查看内容"：以抽屉形式展示被举报内容详情
- 点击"删除内容"：确认框"确认删除该内容？删除后举报者将收到通知"，确认后调用 API
- 点击"忽略举报"：确认框"确认忽略该举报？举报者将收到通知"，确认后调用 API
- 点击"禁言用户"：弹窗选择禁言时长（1小时/1天/7天/30天/永久），确认后调用 API
- 操作成功后，卡片状态更新，Toast 提示"已删除"/"已忽略"/"已禁言"（根据操作类型）

**状态与边界**:
- 无待处理举报时：展示空状态"暂无待处理举报"
- 被举报内容已被删除：卡片显示"该内容已被删除"，仅展示忽略选项
- 禁言操作中：按钮显示 loading

## 4. 组件设计

### 4.1 圈子公告栏组件 (CircleAnnouncementBar)

**Props**:
| 属性 | 类型 | 说明 |
|------|------|------|
| circleId | string | 圈子ID |
| editable | boolean | 是否显示编辑/删除操作 |

**内部状态**:
- `announcement`: 当前生效公告对象（null 表示无公告）
- `expanded`: 是否展开完整内容

**行为**:
- 组件挂载时查询当前圈子生效公告
- 无公告时不渲染整个组件
- 公告过期后自动隐藏（前端定时检查或依赖接口返回）

---

### 4.2 内容操作菜单组件 (ContentActionMenu)

**Props**:
| 属性 | 类型 | 说明 |
|------|------|------|
| contentId | string | 内容ID |
| isPinned | boolean | 是否已置顶 |
| isFeatured | boolean | 是否已精华 |
| isModerator | boolean | 当前用户是否为版主/创建者 |

**行为**:
- 根据 `isModerator` 动态渲染菜单项
- 版主菜单：置顶/取消置顶、精华/取消精华、举报
- 普通成员菜单：举报
- 点击菜单项调用对应 API，成功后触发 `@update` 事件通知父组件刷新

```typescript
  contentId: string;      // 被举报内容ID
  reasonType: string;     // 举报原因类型：AD/PORNO/ATTACK/OTHER
  description?: string;   // 补充说明
}
// Response: { code: 200, result: null, success: boolean }

// GET /circle-report/list?circleId={id}&status={status}&page=&size=
// Response: { code: 200, result: { records: ReportVO[], total: number, page: number, size: number }, success: boolean }

interface ReportVO {
  id: string;
  reporterId: string;       // 举报者用户ID
  reporterName: string;     // 举报者昵称

  contentTitle: string;     // 被举报内容标题摘要
  reasonType: 'AD' | 'PORNO' | 'ATTACK' | 'OTHER'; // 举报原因类型
  reasonLabel: string;      // 举报原因中文文案
  description?: string;     // 补充说明
  status: 'PENDING' | 'PROCESSED' | 'IGNORED';
  createdAt: string;        // 举报时间 ISO8601
  processedAt?: string;     // 处理时间 ISO8601
  muteDuration?: string;    // 禁言时长（如有）
}

// PUT /circle-report/{id}/delete-content
// 无请求体
// Response: { code: 200, result: null, success: boolean }

// PUT /circle-report/{id}/ignore
// 无请求体
// Response: { code: 200, result: null, success: boolean }

// PUT /circle-report/{id}/mute-user

  duration: '1h' | '1d' | '7d' | '30d' | 'permanent';

```

## 8. 交互设计

### 8.1 通用交互规则

以下交互行为适用于本 Epic 所有页面和组件：

| 场景 | 反馈方式 |
|------|---------|
| 数据加载中 | 骨架屏（列表页）或 Spin（弹窗/浮层） |
| 无数据 | Empty 组件 + 引导文案 |
| 操作成功 | `message.success()` 全局提示 |
| 操作失败 | `message.error()` 全局提示，保留用户输入 |
| 权限不足 | `message.warning("权限不足")` |
| 防重复提交 | 按钮 loading + disabled |
| 危险操作 | 二次确认弹窗（Modal.confirm） |
| 表单校验 | 必填字段实时校验，提交时全量校验 |

**操作成功文案规范**（格式：动词+结果）:

| 操作 | 成功文案 |
|------|---------|
| 置顶内容 | 已置顶 |
| 取消置顶 | 已取消置顶 |
| 标记精华 | 已标记精华 |
| 取消精华 | 已取消精华 |
| 发布公告 | 公告已发布 |
| 删除公告 | 公告已删除 |
| 批准加入申请 | 已批准 |
| 拒绝加入申请 | 已拒绝 |
| 删除被举报内容 | 已删除 |
| 忽略举报 | 已忽略 |
| 禁言用户 | 已禁言 |
| 提交举报 | 举报已提交，管理员将尽快处理 |

### 8.2 置顶/精华操作交互

```
用户点击"置顶" → 调用 API → 成功 → 列表重新排序 → Toast "已置顶"
                                              → 失败 → Toast "操作失败"
用户点击"取消置顶" → 确认框"取消置顶后该内容将按普通排序展示" → 调用 API → 成功 → 列表重新排序
```

### 8.3 公告发布交互

```
点击"发布公告" → 打开弹窗 → 填写内容+有效期 → 点击"发布"
  → 已有生效公告 → 确认框"将替换当前公告" → 确认 → 调用 API → 成功 → 关闭弹窗 → 公告栏更新
  → 无生效公告 → 直接调用 API → 成功 → 关闭弹窗 → 公告栏展示
```

### 8.4 @成员交互

```
用户在输入框输入 "@" → 触发浮层展示 → 加载成员列表
  → 输入关键词 → 防抖搜索（300ms）→ 更新列表
  → 点击/Enter 选中 → 插入 @昵称 标记 → 关闭浮层
  → 点击外部/Esc → 关闭浮层
```

### 8.5 加入申请审核交互

```
管理员点击"批准" → 确认框 → 确认 → 调用 API → 成功 → 卡片移出列表 → Toast "已批准"
管理员点击"拒绝" → 弹窗（拒绝原因必填）→ 确认 → 调用 API → 成功 → 卡片移出列表 → Toast "已拒绝"
```

### 8.6 举报处理交互

```
管理员点击"删除内容" → 确认框"删除后举报者将收到通知" → 确认 → 调用 API → 成功 → 卡片状态更新
管理员点击"忽略" → 确认框 → 确认 → 调用 API → 成功 → 卡片状态更新
管理员点击"禁言用户" → 弹窗选择时长 → 确认 → 调用 API → 成功 → 卡片状态更新
```
