# EPIC-10 圈子基础能力 - 前端 PRD

## 1. 概述

### 需求目标
- **需求名称**: 圈子基础能力（circle-core）
- **概述**: 构建圈子 MVP 基础设施，支持圈子创建与设置、成员加入/退出/角色管理/禁言移除、圈子搜索三大核心能力
- **要解决的问题**: 内容社区缺乏基于兴趣聚合的垂直社区能力，用户无法围绕特定主题进行深度交流
- **期望达成结果**: 用户可创建圈子、通过多种方式加入圈子、搜索发现公开圈子；圈子创建者可管理成员角色和执行治理操作

### 目标用户与使用场景

| 角色 | 使用场景 | 入口 |
|------|---------|------|
| 圈子创建者 | 创建圈子、设置隐私与加入方式、管理成员角色、禁言/移除成员 | 圈子列表页「创建圈子」按钮 / 圈子管理页 |
| 圈子版主 | 禁言/移除普通成员 | 圈子成员管理页 |
| 普通成员 | 搜索圈子、加入/退出圈子 | 圈子搜索页 / 圈子详情页 |
| 游客（未登录） | 浏览公开圈子搜索结果（不可创建/加入） | 圈子搜索页 |

### 范围定义

#### 本期范围
- 圈子创建表单（名称、简介、图标、封面图、分类标签、隐私类型、加入方式）
- 圈子名称唯一性校验与敏感词检测反馈
- 圈子详情页（展示基础信息、成员数、加入操作）
- 加入圈子流程（直接加入、申请审核、密码输入、邀请限制）
- 退出圈子流程
- 成员列表与角色管理（设置/取消版主）
- 成员禁言（1小时/24小时/7天/永久）与解除禁言
- 成员移除
- 圈子搜索（关键词模糊匹配、结果列表、空状态）
- 治理操作日志查看

#### 非本期范围
- 内容置顶/精华、公告、@成员（EPIC-11）
- 审核机制（人工审核流程）
- 数据统计与分析面板
- 推荐算法与成长激励（EPIC-12/13）

### 成功指标（Success Metrics）

| 指标名称 | 定义 | 统计周期 | 目标值（参考） |
|---------|------|---------|-------------|
| 圈子创建量 | 上线后新建的圈子总数 | 30 天 | 待运营确认 |
| 圈子活跃率 | 有成员互动（发布内容/评论/点赞）的圈子占比 | 7 天滚动 | 待运营确认 |
| 成员留存率 | 加入圈子后 7 日 / 30 日仍活跃的成员占比 | 7 日 / 30 日 | 待运营确认 |
| 搜索转化率 | 搜索结果页点击「加入」按钮的用户占比 | 7 天滚动 | 待运营确认 |
| 加入成功率 | 各加入方式（直接/密码/申请）的完成率 | 实时 | 直接加入 > 95%；密码加入 > 80% |
| 退出率 | 成员主动退出圈子的比率 | 30 天 | 待运营确认 |

> **说明**: 以上指标将直接影响后续 EPIC-11/12 的优先级排序。前端需在关键操作节点（创建成功、加入成功、退出、搜索点击）埋点上报，具体埋点方案在开发阶段确定。目标值需运营团队确认后补充。

## 3. 页面设计

### 3.1 圈子列表页

**页面职责**: 展示用户已加入的圈子和推荐公开圈子，提供创建入口和搜索入口

**页面路径**: `/circle/list`

**布局结构**:
```
┌─────────────────────────────────────────────┐
│ 顶部操作栏                                    │
│  [搜索框（关键词输入）]     [+ 创建圈子]        │
├─────────────────────────────────────────────┤
│ Tab 切换: [已加入] [发现]                      │
├─────────────────────────────────────────────┤
│ 圈子卡片网格（2列PC / 1列移动端）               │
│  ┌─────────────┐ ┌─────────────┐             │
│  │ 封面图/图标   │ │ 封面图/图标   │             │
│  │ 圈子名称      │ │ 圈子名称      │             │
│  │ 简介（2行截断）│ │ 简介（2行截断）│             │
│  │ 成员数 / 分类  │ │ 成员数 / 分类  │             │
│  │ [已加入] 标识  │ │ [加入] 按钮   │             │
│  └─────────────┘ └─────────────┘             │
│  ...分页/加载更多                               │
└─────────────────────────────────────────────┘
```

**交互规则**:
- 「已加入」Tab 展示当前用户已加入的圈子列表，按加入时间倒序
- 「发现」Tab 展示公开圈子列表，按成员数倒序
- 搜索框输入关键词后按 Enter 或点击搜索图标触发搜索，跳转搜索结果页
- 「创建圈子」按钮：未登录时点击跳转登录页；已登录时打开创建圈子弹窗/页面
- 圈子卡片点击进入圈子详情页
- 卡片上的「加入」按钮：公开+直接加入的圈子点击后直接加入成功；其他类型跳转详情页

**状态与边界**:
- 无圈子时：空状态插图 + "还没有加入任何圈子"文案 + 「发现圈子」引导按钮
- 加载中：骨架屏卡片占位（6个）
- 网络错误：Toast 提示 "加载失败，请重试" + 重试按钮

### 3.2 创建圈子页面

**页面职责**: 引导用户完成圈子创建流程，包含基础信息填写、隐私与加入方式设置

**页面路径**: `/circle/create`（独立页面）或 Modal 弹窗

**布局结构**:
```
┌─────────────────────────────────────────────┐
│ 步骤条: [基础信息] → [隐私与加入方式] → [完成]   │
├─────────────────────────────────────────────┤
│ Step 1: 基础信息                               │
│  ┌──────────────────────────────────────────┐│
│  │ 圈子名称 *     [________________]         ││
│  │              （实时唯一性校验状态提示）       ││
│  │ 圈子简介 *     [________________]         ││
│  │              [________________] (多行)     ││
│  │ 圈子图标       [上传区域 1:1]              ││
│  │ 封面图         [上传区域 16:9]             ││
│  │ 分类标签       [选择/输入]                  ││
│  └──────────────────────────────────────────┘│
│  [取消]                          [下一步 →]    │
├─────────────────────────────────────────────┤
│ Step 2: 隐私与加入方式                          │
│  ┌──────────────────────────────────────────┐│
│  │ 隐私类型 *     ○ 公开  ○ 私有  ○ 密码保护   ││
│  │ 加入方式 *     ○ 直接加入  ○ 申请审核       ││
│  │                ○ 邀请加入  ○ 密码加入       ││
│  │ 访问密码       [________________] (条件展示)││
│  └──────────────────────────────────────────┘│
│  [← 上一步]                       [提交创建]   │
├─────────────────────────────────────────────┤
│ Step 3: 创建成功                               │
│  ✅ 圈子创建成功                                │
│  [进入圈子]  [返回圈子列表]                      │
└─────────────────────────────────────────────┘
```

**交互规则**:
- 步骤条展示当前进度，已完成步骤可点击回退
- Step 1 → Step 2 时校验必填项（名称、简介），校验不通过高亮对应字段
- 圈子名称输入框失焦或输入停顿 500ms 后触发唯一性校验，校验中显示 Loading，结果以行内提示展示
- 敏感词检测在提交时触发，命中后 Toast 提示 "内容包含敏感词，需人工审核"
- 敏感词服务降级策略：若敏感词服务不可用，**不阻断创建**，采用「降级放行 + 后端标记待复核」策略。前端无感知，创建流程正常完成。后端将该圈子标记为「待复核」状态，由运营在管理后台审核
- 「需人工审核」的含义：创建请求正常提交成功，圈子进入「待复核」状态（仅运营可见，普通用户不可搜索/加入），前端 Toast 提示 "创建成功，内容将在审核后公开"
- 图标上传：支持 JPG/PNG/WebP，比例 1:1，建议 >= 200x200，单张 <= 2MB，上传前裁剪
- 封面图上传：支持 JPG/PNG/WebP，比例 16:9，建议 >= 750x422，单张 <= 5MB，上传前裁剪
- 选择「密码保护」隐私类型时，展示密码输入框（必填），输入时显示密码强度指示器（弱/中/强）
- 选择「密码加入」方式时，展示密码输入框（必填），输入时显示密码强度指示器（弱/中/强）
- 密码强度规则：纯数字 = 弱；字母+数字 = 中；字母+数字+特殊字符或长度 >= 12 = 强
- 密码错误次数限制由后端控制，前端在收到「密码错误次数过多」错误码时，Toast 提示 "密码错误次数过多，请稍后再试" 并禁用密码输入框
- 隐私类型与加入方式的联动规则：选择「私有」时，加入方式默认为「申请审核」；选择「密码保护」时，加入方式固定为「密码加入」且不可切换
- 提交按钮点击后显示 Loading 并禁用，防止重复提交
- 创建成功后跳转 Step 3 成功页

**字段校验规则**:

| 字段 | 校验规则 | 错误提示 |
|------|---------|---------|
| 圈子名称 | 必填，2-30字符，不可与已有名称重复，不含敏感词 | "请输入圈子名称"/"该圈子名称已存在，请修改"/"内容包含敏感词" |
| 圈子简介 | 必填，10-500字符 | "请输入圈子简介"/"简介长度需在10-500字之间" |
| 圈子图标 | 可选，JPG/PNG/WebP，<= 2MB，1:1 比例 | "图标格式不支持"/"图标大小不能超过2MB" |
| 封面图 | 可选，JPG/PNG/WebP，<= 5MB，16:9 比例 | "封面图格式不支持"/"封面图大小不能超过5MB" |
| 隐私类型 | 必填，默认「公开」 | - |
| 加入方式 | 必填，默认「直接加入」 | - |
| 访问密码 | 密码保护/密码加入时必填，6-20位，不允许纯数字，至少包含字母 | "请设置访问密码"/"密码长度需在6-20位之间"/"密码不能为纯数字，请包含字母" |

**状态与边界**:
- 名称校验中：输入框右侧显示 Loading 图标
- 名称重复：输入框下方红色提示 "该圈子名称已存在，请修改"
- 敏感词命中：Toast 提示 "内容包含敏感词，需人工审核"
- 提交失败：Toast 提示具体错误信息，保留已填写内容
- 网络中断：Toast 提示 "网络异常，请检查网络后重试"

### 3.3 圈子详情页

**页面职责**: 展示圈子完整信息，提供加入/退出操作入口，供创建者/版主进入成员管理

**页面路径**: `/circle/:id`

**布局结构**:
```
┌─────────────────────────────────────────────┐
│ 封面图区域（16:9 比例，无封面时显示主题色背景）    │
│  圈子图标（左下角叠加）                         │
├─────────────────────────────────────────────┤
│ 圈子名称                                      │
│ 简介（最多3行，超出展开/收起）                    │
│ 分类标签 · 成员数/上限 · 隐私类型标识               │
├─────────────────────────────────────────────┤
│ 操作栏                                         │
│  非成员: [加入] / [申请加入] / [输入密码加入]      │
│  成员:   [退出] [成员列表]                       │
│  创建者: [编辑] [成员管理] [治理日志]              │
│  版主:   [成员管理] [退出]                       │
├─────────────────────────────────────────────┤
│ 内容区（Tab: 动态 / 成员）                       │
│  动态Tab: 圈子内容列表（后续EPIC-11实现）          │
│  成员Tab: 成员列表（头像/昵称/角色/状态）          │
└─────────────────────────────────────────────┘
```

**交互规则**:
- 加入按钮根据圈子设置动态变化：
  - 公开+直接加入 → 「加入」按钮，点击直接加入
  - 申请审核 → 「申请加入」按钮，点击后弹出确认 Modal "确定要申请加入该圈子吗？"，确认后调用接口，成功后按钮变为「申请中」灰色禁用状态，Toast 提示 "申请已提交，请等待审核"
  - 密码加入 → 「密码加入」按钮，点击弹出密码输入 Modal
  - 邀请加入 → 非邀请用户显示「仅限邀请加入」禁用按钮 + 文字提示 "该圈子仅限受邀用户加入"
- **申请状态持久化规则**:
  - 用户提交加入申请后，后端记录申请状态（PENDING），前端通过圈子详情接口返回的 `applyStatus` 字段判断按钮展示
  - 再次访问详情页时：若 `applyStatus === 'PENDING'`，按钮显示「申请中」灰色禁用状态
  - 申请被拒绝后：按钮恢复为「申请加入」，用户可重新申请
  - 申请通过后：按钮变为「已加入」，可执行退出操作
- **邀请加入判断逻辑**:
  - 前端通过圈子详情接口返回的 `isInvited` 字段判断当前用户是否为受邀用户
  - 邀请关系数据由后端维护，前端仅根据接口返回值展示对应 UI
  - 若 `isInvited === true`，按钮显示为「加入」（直接加入行为）
  - 若 `isInvited === false`（或未登录），按钮显示为「仅限邀请加入」禁用状态
- 已加入成员显示「退出」按钮（创建者不显示）
- 创建者可见「编辑」「成员管理」「治理日志」操作
- 版主可见「成员管理」操作
- 简介超过3行时显示「展开/收起」切换
- 成员 Tab 展示成员列表，支持按角色筛选

**状态与边界**:
- 圈子已满员：加入按钮禁用，Tooltip 提示 "圈子已满员，无法加入"。成员上限默认值为 500 人（所有隐私类型统一），可由后端配置覆盖。详情页展示格式为「成员数/上限」（如「128/500」），列表页卡片展示「成员数」
- 用户已被禁言：在内容区显示禁言提示条 "您已被禁言，解除时间：{time}"
- 用户已被移除：详情页正常展示，加入按钮可用（可重新申请）
- 圈子不存在：404 页面 "圈子不存在或已被删除"
- 密码错误：Modal 内显示 "密码错误，请重新输入"，清空输入框

### 3.4 成员管理页

**页面职责**: 管理圈子成员，支持角色变更、禁言、移除操作

**页面路径**: `/circle/:id/members`

**布局结构**:
```
┌─────────────────────────────────────────────┐
│ 成员管理                                      │
│  筛选: [全部角色 ▼] [全部状态 ▼] [搜索成员]     │
├─────────────────────────────────────────────┤
│ 成员列表（Table）                               │
│ ┌──────┬──────┬──────┬──────┬──────────────┐│
│ │ 成员  │ 角色  │ 状态  │ 加入时间│ 操作         ││
│ ├──────┼──────┼──────┼──────┼──────────────┤│
│ │ 头像  │ 创建者 │ 正常  │ 日期  │ -            ││
│ │ 昵称  │ 版主  │ 正常  │ 日期  │ [降级][禁言]  ││
│ │      │ 成员  │ 禁言中│ 日期  │ [设为版主]    ││
│ │      │      │      │      │ [禁言][移除]   ││
│ └──────┴──────┴──────┴──────┴──────────────┘│
│  分页: < 1 2 3 ... >                          │
└─────────────────────────────────────────────┘
```

**交互规则**:
- 筛选区：角色下拉（全部/创建者/版主/成员）、状态下拉（全部/正常/禁言中/已移除）、成员搜索（昵称关键词）
- 操作列根据当前用户角色和目标成员角色动态展示：
  - 创建者对普通成员：「设为版主」「禁言」「移除」
  - 创建者对版主：「取消版主」「禁言」「移除」
  - 创建者对自己：无操作
  - 版主对普通成员：「禁言」「移除」
  - 版主对其他版主：无操作（提示 "权限不足"）
- 禁言操作：点击「禁言」弹出 Modal，选择时长（1小时/24小时/7天/永久）+ 填写原因（可选），确认后执行
- 移除操作：点击「移除」弹出确认 Modal "确定要移除该成员吗？移除后可重新申请加入"

**状态与边界**:

- 列表为空时：空状态 "暂无成员"
- 操作失败：Toast 提示具体错误原因

## 6. 状态管理

### 6.1 新增 Store: `useCircleStore`

```typescript
// src/store/modules/circle.ts
interface CircleStore {
  // 当前查看的圈子详情
  currentCircle: CircleVO | null;
  // 当前用户在当前圈子的角色
  currentRole: 'CREATOR' | 'MODERATOR' | 'MEMBER' | null;
  // 当前用户在当前圈子的状态
  currentMemberStatus: 'ACTIVE' | 'MUTED' | 'REMOVED' | null;
  // 搜索关键词
  searchKeyword: string;
}
```

### 6.2 数据流设计

```
圈子列表页
  ├── useCircleStore.searchKeyword → 搜索结果页
  └── 圈子详情页
        ├── currentCircle → 详情展示
        ├── currentRole → 按钮/操作权限判断
        ├── currentMemberStatus → 禁言提示/操作限制
        └── 成员管理页
              ├── 成员列表（useTable 管理）
              └── 操作结果 → 刷新列表
```

### 6.2.1 Store 数据生命周期管理

| 数据字段 | 加载时机 | 失效/清理时机 | 刷新策略 |
|---------|---------|-------------|---------|
| `currentCircle` | 进入圈子详情页时请求 | 离开详情页（路由变更）时清空；执行退出操作后清空 | 切换 Tab（动态/成员）时不重新加载；执行加入/退出操作后重新加载 |
| `currentRole` | 随 `currentCircle` 一起从详情接口返回 | 离开详情页时清空；退出圈子后清空为 `null` | 角色变更（设为版主/取消版主）后重新加载详情 |
| `currentMemberStatus` | 随 `currentCircle` 一起从详情接口返回 | 离开详情页时清空；退出圈子后清空为 `null` | 禁言/解除禁言操作后重新加载详情 |
| `searchKeyword` | 搜索时写入 | **不持久化**：从搜索结果页进入详情页后返回时，从 URL 参数 `q` 重新读取 | 用户修改搜索框内容时实时更新 |

### 6.2.2 列表页缓存策略

- 「已加入」Tab 和「发现」Tab 的数据**不共享缓存**，各自独立管理
- Tab 切换时保留已加载数据（`keep-alive`），切换回时检查数据是否过期（超过 5 分钟则自动刷新）
- 执行加入/退出操作后，同时刷新两个 Tab 的数据以保持一致性

### 6.3 权限判断逻辑

```typescript
// 前端权限判断（配合后端校验）
const canManageMember = computed(() => {
  return currentRole.value === 'CREATOR' || currentRole.value === 'MODERATOR';
});

const canManageRole = computed(() => {
  return currentRole.value === 'CREATOR';
});

const canMute = (targetRole: string) => {
  if (currentRole.value === 'CREATOR') return true;
  if (currentRole.value === 'MODERATOR' && targetRole === 'MEMBER') return true;
  return false;
};
```

## 8. 交互设计

### 8.1 表单交互

**圈子创建表单**:
- 名称输入框：失焦 + 防抖 500ms 触发唯一性校验，校验中右侧显示 Spin，结果行内提示
- 简介输入框：多行 textarea，显示字数统计 `{current}/500`
- 图标/封面图上传：点击上传区域 → 选择文件 → 裁剪弹窗 → 确认上传 → 展示预览
- 隐私类型 Radio 切换时，联动更新加入方式选项和条件字段

**成员管理筛选**:
- 角色/状态下拉选择后立即刷新列表（无需点搜索）
- 搜索成员输入框 Enter 触发搜索

### 8.2 操作反馈

| 操作 | 成功反馈 | 失败反馈 |
|------|---------|---------|
| 创建圈子 | Toast "圈子创建成功" + 跳转成功页 | Toast 具体错误信息 |
| 加入圈子 | Toast "加入成功" + 按钮变为"已加入" | Toast 具体错误（满员/密码错误/仅邀请等） |
| 退出圈子 | Toast "已退出圈子" + 清空 store 中的 `currentRole` 和 `currentMemberStatus` + 返回圈子列表 | Toast "退出失败，请重试" |
| 设置版主 | Toast "已设置为版主" + 列表刷新 | Toast 具体错误（权限不足等） |
| 禁言成员 | Toast "已禁言" + 列表刷新 | Toast 具体错误 |
| 移除成员 | Toast "已移除" + 列表刷新 | Toast 具体错误 |

### 8.3 危险操作确认

以下操作需二次确认弹窗：
- 退出圈子："确定要退出该圈子吗？退出后需重新申请加入"
- 移除成员："确定要移除 {nickname} 吗？移除后可重新申请加入"
- 角色变更："确定要将 {nickname} 设为/取消 版主吗？"
- 禁言成员：禁言 Modal 本身即为确认，点击「确认禁言」执行

### 8.4 加载与防重

- 所有提交按钮在请求期间显示 Loading 并 disabled
- 加入/退出按钮点击后立即 disabled，请求完成后更新状态
- 列表加载使用骨架屏，首次加载和翻页时展示

**治理日志页**:
- PC: Table 完整展示
- 移动端: Table 转为时间线/卡片列表

| 图标/封面图上传 | 单张 < 5s | 压缩上传 + 进度条 |
| 表单提交 | < 1s | Loading 状态 + 防重复提交 |
| 页面切换 | < 300ms | 路由缓存 + keep-alive |

**前端优化措施**:
- 圈子卡片列表使用虚拟滚动（当列表超过 100 条时）
- 搜索输入防抖 300ms，减少无效请求
- 图片懒加载：列表中的图标和封面图
- 成员列表使用 `useTable` 的内置分页，避免一次性加载全部数据
- 圈子详情使用 `keep-alive` 缓存，避免反复请求

### 10.5 并发与竞态处理

以下场景前端需做降级处理，后端需保证操作幂等性：

| 竞态场景 | 前端降级策略 | 后端要求 |
|---------|------------|---------|
| 两个管理员同时对同一成员执行禁言/移除 | 操作失败时 Toast 提示 "操作失败，该成员状态已变更"，自动刷新成员列表 | 接口幂等，重复操作返回明确错误码 |
| 用户在详情页停留期间，圈子隐私类型被修改 | 切换 Tab 或执行加入操作时重新拉取圈子详情，若隐私类型已变更则刷新页面状态 | 详情接口返回最新隐私类型 |
| 成员列表分页浏览时有成员被移除或新增 | 列表翻页时重新拉取数据；执行操作后强制刷新当前页 | 分页接口返回实时数据，支持游标分页避免偏移量不一致 |
| 用户点击「加入」时圈子已满员 | 接口返回满员错误码，前端 Toast 提示 "圈子已满员，无法加入"，按钮变为禁用状态 | 并发加入时以实际成员数为准，超限拒绝 |

## 12. 待确认问题 / 默认假设

### 待确认问题

1. **圈子分类标签**: 是否需要预定义枚举列表？还是允许用户自由输入？默认假设为预定义枚举（如：技术、生活、兴趣、学习等），支持自定义输入
2. **搜索服务**: 后端搜索是否已有 Elasticsearch 支持？默认假设 MVP 阶段使用 MySQL LIKE 查询
3. **敏感词检测**: 是否已有敏感词服务？默认假设调用平台已有服务，降级方案为本地词库
4. **圈子图标默认值**: 用户不上传图标时，是否使用系统默认图标？默认假设使用首字母+主题色背景的默认图标
5. **邀请加入流程**: 「邀请加入」模式下，邀请功能是否在本期实现？默认假设本期仅做入口限制，邀请功能在后续 EPIC 实现。前端通过详情接口 `isInvited` 字段判断当前用户是否受邀，邀请关系由后端维护
6. **申请审核流程**: 「申请审核」模式下，审核功能是否在本期实现？默认假设本期仅做申请提交，审核功能在 EPIC-11 实现。前端通过详情接口 `applyStatus` 字段展示申请状态（PENDING → 申请中；APPROVED → 已加入；REJECTED → 可重新申请），申请中状态持久化在后端
7. **密码修改**: 密码保护圈子的密码是否支持修改？默认假设本期不支持修改，后续迭代
8. **圈子解散**: 创建者是否可以解散圈子？默认假设本期不实现解散功能

### 默认假设

1. 所有页面基于 Vue 3 + TypeScript + Ant Design Vue 4 技术栈
2. API 统一使用 `defHttp` 封装，响应格式为 `{ code: 200, result, message, success }`
3. 圈子列表默认分页大小为 20 条，成员列表为 50 条
4. 未登录用户可访问圈子搜索和详情页（只读），但创建/加入/管理操作需登录
5. 圈子名称全局唯一（非同类型内唯一）
6. 图标裁剪在前端完成，上传裁剪后的图片到后端
7. 禁言到期解除通过后端查询时检查实现，前端无需定时轮询
8. 治理日志保留 180 天，前端不做本地缓存
9. 国际化（i18n）：当前版本仅面向中文用户，文案硬编码中文。后续迭代需国际化时，组件层面需预留 `t()` 翻译函数接口，避免大规模改造。本期开发时所有用户可见文案统一使用常量文件管理（`src/locales/lang/zh-CN/circle.ts`），不散落在模板中
10. 无障碍（a11y）：圈子图标需设置 `alt` 文本（使用圈子名称）；操作按钮需支持键盘 Tab 导航和 Enter 触发；状态标签（已加入/禁言中/申请中）使用 `aria-label` 属性；颜色对比度遵循 WCAG 2.1 AA 标准（主色与背景对比度 >= 4.5:1）
