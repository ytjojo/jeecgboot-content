## 1. 概述

### 需求目标
- **需求名称**: 社交关系扩展与互动增强（social-extensions）
- **一句话概述**: 为内容社区提供互关标识、粉丝增长分析、邀请传播和社区角色展示等社交扩展能力
- **要解决的问题**: 当前社区缺少互关关系标识、粉丝数据分析、邀请机制和社区角色展示，导致用户间关系不清晰、创作者无法了解粉丝增长趋势优化内容、社区治理缺少角色透明度
- **期望达成的结果**: 互关率提升 35%、邀请转化率 >15%、粉丝管理功能使用率 >40%、社区角色识别准确率 >95%

### 目标用户与使用场景

| 用户角色 | 使用场景 | 入口 |
|---------|---------|------|
| 普通用户 | 查看互关标识、发布私密内容、使用邀请功能 | 用户主页、内容发布页、邀请好友页 |
| 创作者 | 查看粉丝列表/趋势、了解增长情况 | 个人中心 > 粉丝管理 |
| 版主 | 评论区管理（删评论、警告用户） | 评论区操作菜单 |
| 管理员 | 用户治理（评论区快捷操作 + 跳转 EPIC-09 用户管理页）、审计日志 | 评论区操作菜单、审计日志页 |

### 范围定义

#### 本期范围
- 互关标识展示（用户主页 + 评论区）
- 互关好友列表

- ~~粉丝画像（兴趣分布、地域分布、活跃时段）~~ → **降级为 P3，延后至二期迭代**（见用户故事 US-7.2.2 调整说明）

- 审计日志查看

#### 非本期范围

## 3. 页面设计

### 3.1 互关好友列表页

**页面路径**: `/content/mutual-follow`
**页面入口**: 个人中心 > 互关好友 / 用户主页 > 互关列表链接

#### 页面结构
```
Page
  ├── 顶部信息栏
  │   ├── 标题: "互关好友"
  │   └── 互关好友总数
  ├── 搜索栏 (输入框，按昵称/用户名搜索)
  └── 好友列表 (卡片列表，每项包含)
      ├── 头像
      ├── 昵称 + 互关标识徽章
      ├── 互关时间
      └── 操作按钮 (取消关注 / 发消息)
```

#### 交互要求
- 列表默认按最近互动时间倒序排列
- 搜索为实时防抖搜索（300ms），输入清空时恢复全量列表
- 点击头像或昵称跳转用户主页
- 取消关注需二次确认弹窗："确定取消与 [昵称] 的互关关系？"
- 取消关注成功后，列表项移除并提示"已取消互关"

#### 状态与边界情况
- **空状态**: 显示空状态插图 + "还没有互关好友，去发现更多有趣的人吧" + 引导按钮
- **加载中**: 骨架屏（3 列卡片占位）
- **搜索无结果**: "未找到匹配的好友"
- **网络错误**: 错误提示 + 重试按钮

### 3.3.1 邀请完整转化漏斗设计

> **补充说明**: PRD 原版仅覆盖"发起邀请"端，以下补充被邀请者完整用户旅程，确保邀请转化率 >15% 指标可衡量、可优化。

#### 被邀请者完整旅程

```
被邀请者点击邀请链接
  → 落地页展示（见下方设计）
  → 点击"立即注册"按钮
  → 跳转注册/登录页（自动携带 inviteCode 参数）
  → 注册成功 → 自动关联邀请人（后端通过 inviteCode 绑定）
  → 引导完成首次互动（发布内容/评论/关注）
  → 首次互动完成 → 触发奖励（双方获得积分）
  → 邀请人收到通知："您邀请的 [昵称] 已成功注册并完成首次互动，获得 [X] 积分"
```

#### 邀请链接落地页设计

**页面路径**: `/invite/:inviteCode`
**页面入口**: 被邀请者通过分享链接访问

**页面结构**:
```
Page
  ├── 顶部 Banner
  │   ├── 平台 Logo + 名称
  │   └── 邀请人头像 + 昵称（"[昵称] 邀请你加入"）
  ├── 核心价值展示区
  │   ├── 平台特色功能简介（3-4 个卖点）
  │   └── 社区内容精选（2-3 条热门内容预览）
  ├── 注册引导区
  │   ├── "立即注册"主按钮
  │   ├── "已有账号？立即登录"链接
  │   └── 注册奖励说明（"注册即享 [X] 积分"）
  └── 底部说明
      ├── 邀请规则简介
      └── 用户协议 + 隐私政策链接
```

**交互要求**:
- 落地页自动识别 inviteCode 参数，无效码显示"邀请链接已失效"
- 注册按钮点击后跳转注册页，URL 携带 inviteCode 参数
- 已登录用户点击邀请链接 → 直接跳转首页，不重复注册
- 落地页需支持移动端优先的响应式设计

#### 奖励触发规则

| 触发条件 | 奖励对象 | 奖励内容 | 说明 |
|---------|---------|---------|------|
| 被邀请者完成注册 | 被邀请者 | 注册积分（如 50 分） | 即时发放 |
| 被邀请者完成首次互动 | 邀请人 | 邀请积分（如 50 分） | 首次发布内容/评论/关注后触发 |
| 被邀请者完成首次互动 | 被邀请者 | 互动积分（如 20 分） | 引导新用户活跃 |

> **"成功邀请"定义**: 被邀请者完成注册 **且** 完成首次互动（发布内容/评论/关注任一），仅注册不计入邀请人奖励。

#### 防刷与安全策略

| 策略 | 规则 | 说明 |
|------|------|------|
| 邀请码有效期 | 30 天（可配置） | 过期后链接失效，需重新生成 |
| 单用户邀请上限 | 100 人（可配置） | 防止恶意批量邀请 |
| 同设备/IP 限制 | 同一 IP 24 小时内最多注册 3 个被邀请账号 | 防止刷量 |
| 奖励冷却期 | 同一邀请人 1 小时内最多获得 5 次奖励 | 防止集中刷奖励 |
| 异常检测 | 后端检测注册后立即注销等异常行为 | 冻结可疑奖励 |

#### 邀请链接失效处理

- **邀请码过期**: 落地页显示"邀请链接已过期，请联系邀请人获取新链接"
- **邀请人数达上限**: 落地页显示"该邀请人邀请名额已满"
- **邀请人账号异常**: 落地页显示"邀请链接已失效"
- 以上情况均引导用户直接注册（无邀请奖励）

### 3.4 社区角色标签（评论区内嵌组件）

**展示位置**: 评论区评论项内，用户昵称右侧

#### 组件结构
```
CommentItem
  ├── 用户头像
  ├── 用户昵称
  ├── CommunityRoleBadge (角色标签)
  │   ├── 普通用户: 无标签
  │   ├── 创作者: "创作者" + 认证图标 (蓝色)
  │   ├── 版主: "版主" 标签 (绿色)
  │   └── 管理员: "管理员" 标签 (橙色)
  └── 评论内容
```

#### 交互要求
- 角色标签支持 hover 显示角色说明 Tooltip
- 创作者标签带认证图标（复用现有 `ContentUserVerificationBadge` 样式）
- 角色标签颜色区分：创作者蓝色、版主绿色、管理员橙色
- 点击角色标签弹出 Popover 显示角色说明（如"版主负责维护社区秩序"）

#### 角色体系演进规则

**创作者认证**:
- **申请条件**: 粉丝数 >= 100 且发布内容 >= 10 篇且近 30 天活跃天数 >= 15 天
- **申请流程**: 用户在个人中心提交申请 → 后端自动审核条件 → 通过后自动授予创作者角色
- **本期实现**: 仅实现自动审核逻辑（后端），前端不单独开发申请页面，满足条件的用户自动获得创作者标签
- **降级规则**: 连续 90 天未发布内容 → 自动降级为普通用户，需重新申请

**版主任命**:
- **任命方式**: 由管理员通过平台 RBAC 系统手动任命，不开放自助申请
- **权限范围**: 评论区删除评论 + 警告用户（本 PRD 定义的操作）
- **撤销规则**: 管理员可随时通过 RBAC 系统撤销版主角色
- **本期实现**: 版主角色通过后端 RBAC 管理，前端仅做 UI 层面的角色展示和按钮显隐

**自定义角色**:
- 本期不支持自定义角色，仅支持上述 4 级角色体系
- 后续可根据社区规模扩展角色类型（如"资深创作者"、"话题主持人"）

### 3.5 版主/管理员管理操作（评论区内嵌）

**展示位置**: 评论区评论项的操作菜单中

> **职责边界说明**: 本 PRD 仅负责评论区内的**轻量管理能力**（删除评论、警告用户）。封禁用户、禁言用户、撤销处罚等重量级治理操作统一在 **EPIC-09（用户状态与治理）** 的用户管理页中实现，避免功能重复开发和用户体验割裂。评论区操作菜单中提供"前往用户管理"快捷入口，跳转至 EPIC-09 的用户详情页。

#### 操作菜单结构
```
评论操作区
  ├── 普通用户可见: 点赞 / 回复 / 举报
  ├── 版主额外可见: 删除评论 / 警告用户
  ├── 管理员额外可见: 删除评论 / 警告用户 / 前往用户管理（跳转 EPIC-09）
  └── 注: 封禁/禁言/撤销操作在 EPIC-09 用户管理页中统一处理
```

#### 交互要求
- "删除评论"点击后弹出确认弹窗，需填写删除原因（必填）
- "警告用户"点击后弹出 Modal，包含警告原因输入框（必填）和确认按钮
- 管理员"前往用户管理"点击后跳转 EPIC-09 的用户详情页（`/system/user-governance/:userId`），可在该页面执行封禁/禁言/撤销操作
- 所有管理操作成功后显示全局消息提示，并刷新评论列表
- 撤销处罚：在 EPIC-09 用户管理页操作，不在评论区内处理

## 6. 状态管理

### 新增 Store

| Store | 文件 | 用途 |
|-------|------|------|
| `useMutualFollowStore` | `src/store/modules/mutualFollow.ts` | 缓存互关状态 Map（userId → boolean），避免重复请求 |

| Store | 复用内容 |
|-------|---------|
| `useUserStore` | 当前用户信息、角色权限判断（是否版主/管理员） |

### 互关状态缓存策略
- **评论区（首选方案）**: 评论列表接口自带 `mutualFollow` 字段，前端直接使用，无需额外请求或缓存
- **动态加载评论（增量场景）**: 使用 `mutual-status` 批量查询接口，结果缓存到 `mutualFollowStore`
- 用户主页单独查询互关状态（使用接口自带字段或 `mutual-status` 接口）
- 缓存有效期：当前会话内有效，取关/关注操作后清除对应缓存

## 8. 交互设计

### 8.1 通用交互规则

| 场景 | 交互行为 |
|------|---------|
| 数据加载中 | 骨架屏（列表）/ Spin（图表） |
| 空数据 | 空状态插图 + 引导文案 + 可选引导按钮 |
| 操作成功 | `message.success()` 全局提示，2 秒自动关闭 |
| 操作失败 | `message.error()` 显示错误信息，保留用户输入 |
| 网络错误 | 错误提示 + 重试按钮 |
| 防重复提交 | 提交按钮 loading + disabled |
| 危险操作 | 二次确认弹窗（Modal.confirm） |

### 8.2 关键交互流程

#### 互关标识渲染流程
```
评论列表加载 → 评论接口自带 mutualFollow 字段
  → 直接渲染 MutualFollowBadge 组件（无额外请求）

动态加载评论（增量场景）:
  → 收集评论者 userId 列表
  → 批量请求互关状态 API
  → 缓存结果到 store
  → 渲染 MutualFollowBadge 组件
```

#### 私密内容发布流程
```
用户点击发布 → 内容编辑表单
  → 可见性下拉选择 (公开 / 仅互关可见)
  → 选择"仅互关可见" → 显示提示: "仅互关好友可以看到此内容"
  → 提交 → API 保存 visibility=MUTUAL_FOLLOW_ONLY
```

#### 私密内容展示策略（补充）

**用户主页展示**:
- 用户主页内容列表默认仅展示公开内容
- 当前用户与主页用户互关时，内容列表自动包含私密内容（无需 Tab 切换）
- 私密内容在列表中显示"仅互关可见"标识（锁形图标 + 文案），与公开内容视觉区分

**非互关用户访问包含私密内容的用户主页**:
- 私密内容完全隐藏，不显示占位符或模糊卡片
- 内容列表仅展示公开内容，分页总数基于公开内容计算
- 不展示"该用户有 X 条私密内容"等暗示性文案

**Feed 流中的私密内容**:
- 互关用户的私密内容正常出现在 Feed 流中
- 非互关用户在 Feed 流中不会看到私密内容
- 如果 Feed 流中出现私密内容（互关用户发布），显示"仅互关可见"标识

**取消互关后的内容可见性**:
- 取消互关后，私密内容从 Feed 流中不再展示
- 用户主页内容列表自动切换为仅展示公开内容
- 已打开的私密内容详情页：刷新后显示"该内容仅对互关好友可见"提示
- 浏览历史中的私密内容：再次访问时显示权限提示

**搜索结果中的私密内容**:
- 私密内容不进入公共搜索索引
- 仅互关用户的私密内容可被搜索到（搜索结果中标注"仅互关可见"）

**私密内容的转发与收藏**:
- 私密内容不支持转发（转发按钮隐藏）
- 私密内容支持收藏，但取关后收藏列表中显示"内容已不可见"
- 收藏的私密内容详情页：非互关用户访问时显示权限提示

#### 版主删除评论流程
```
版主点击评论"更多" → 下拉菜单出现"删除评论"
  → 点击 → 弹出 Modal
  → 表单: 删除原因 (必填, textarea)
  → 点击确认 → 调用 delete-comment API
  → 成功 → message.success + 刷新评论列表
  → 失败 → message.error + 保留弹窗
```

#### 粉丝趋势查看流程
```
进入粉丝管理页 → 默认显示列表 Tab
  → 切换到趋势 Tab → 请求日维度趋势数据
  → 渲染折线图
  → 切换维度 (周/月) → 重新请求 → 图表更新
  → 点击某天数据点 → 弹出 Modal 显示当日新增粉丝列表
```

## 10. 性能要求

| 指标 | 要求 | 说明 |
|------|------|------|
| 互关标识加载 | < 50ms 额外开销 | 评论接口自带字段（首选），无额外请求；动态加载时批量查询 + 缓存 |
| 粉丝列表加载 | < 1s | 分页查询，每页 20 条 |
| 粉丝趋势图表 | < 2s | ECharts 按需加载，数据量可控（最多 30 个数据点） |
| 粉丝画像图表 | < 2s | 3 个图表并行请求 **[延后至二期]** |
| 邀请码生成 | < 500ms | 首次生成，后续缓存 |
| 管理操作响应 | < 1s | 单次写操作 |
| 评论区角色标签 | < 50ms | 从评论接口附带返回，无额外请求 |
| 页面首屏 | < 3s | 3G 网络条件下 |

### 优化策略
- ECharts 按需引入（`echarts/core`），使用 tree-shaking 减少包体积
- 互关状态批量查询，避免 N+1 请求
- 粉丝画像数据 T+1 更新，前端缓存当日结果
- 管理操作按钮使用 `usePermission` 按需渲染，减少 DOM 节点

### 12.2 核心转化漏斗

**邀请转化漏斗**:
```
邀请链接点击 → 落地页浏览 → 注册点击 → 注册完成 → 首次互动 → 奖励触发
```
- 各环节转化率需单独统计，定位流失环节
- 目标：邀请链接点击 → 注册完成 转化率 >15%

**互关转化漏斗**:
```
互关标识展示 → 点击标识 → 查看主页 → 关注/回关 → 互关建立
```
- 目标：互关率（互关用户数/总关注关系数）提升 35%

**粉丝管理使用漏斗**:
```
个人中心入口 → 粉丝列表查看 → 趋势图表查看 → 画像查看（二期）
```
- 目标：粉丝管理功能使用率 >40%

4. **邀请码自动生成**: 假设后端在首次请求时自动生成，前端无需调用单独的生成接口
5. **管理操作权限**: 假设后端接口已做权限校验，前端仅做 UI 层面的按钮显隐控制

7. **评论区批量互关查询**: 假设后端提供批量互关状态查询接口（传入 userId 列表，返回互关状态 Map）
8. **审计日志页路由**: 假设归入系统管理模块，路由路径为 `/system/audit-log`
