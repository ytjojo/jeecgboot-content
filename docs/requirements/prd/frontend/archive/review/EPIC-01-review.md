# EPIC-01 用户认证与账号安全 — 前端 PRD 审核报告

> **审核人**: 📋 John (Product Manager)
> **审核日期**: 2026-06-02
> **审核视角**: 产品管理 — 用户价值、优先级、业务对齐

## 总体评价

这份 PRD 质量较高，功能范围覆盖了认证体系的核心闭环：注册、登录、会话管理、账号安全、密码找回、风控交互和账号注销。文档结构清晰，交互细节、边界情况、响应式适配、API 对接均有详细描述，开发团队可据此直接进入实现阶段。但在用户价值量化、部分用户旅程完整性、以及若干产品决策上仍有改进空间。

## 优点

- **功能边界清晰**：本期范围与非本期范围划分明确，生物识别、两步验证、企业通讯录等功能合理排除，避免了范围蔓延
- **异常处理设计全面**：每个功能模块都详细列出了状态与边界情况，包括网络错误、风控拦截、并发冲突、验证码过期等，体现了成熟的产品思维
- **安全设计到位**：密码错误隐藏具体原因（不暴露账号是否存在）、最后联系方式不可解绑、注销冷静期机制、异常登录实时通知等设计均符合安全最佳实践
- **API 对接规范**：接口路径、请求方法、响应格式均有明确定义，且提供了 TypeScript 封装示例，降低了前后端联调成本
- **响应式设计完整**：覆盖了 PC、平板、移动端三种场景，弹窗在移动端改为 Drawer 的设计考虑周到
- **Jobs-to-be-Done 覆盖合理**：从"我要注册"到"我要注销账号"，用户生命周期的关键节点均有对应功能

## 问题与建议

1. 🔴 **注册转化率提升 25% 的目标缺乏基线和测量方案。** PRD 提出"注册转化率提升 25%"，但未说明当前基线是多少、如何定义转化率（访问注册页到完成注册？还是点击注册按钮到完成注册？）、以及前端需要埋哪些事件来追踪。建议在 PRD 中补充埋点需求清单，明确漏斗各环节的事件名称和参数，否则上线后无法验证目标是否达成。

2. 🔴 **缺少"登录成功后跳转来源页"的完整用户旅程。** 第 999 行提到"登录成功后跳转到首页或来源页"，但这是在"待确认问题"中以假设形式出现。对于内容社区场景，用户很可能是在浏览某篇文章时被要求登录，登录后回到原文是核心体验。建议将来源页跳转作为默认行为写入功能规格，而非留作假设。

3. 🔴 **邮箱注册后用户处于"未验证"状态的产品定义缺失。** F2 邮箱注册成功后发送验证邮件，但 PRD 未说明：未验证邮箱的用户能否正常使用社区功能？如果能，验证的意义是什么？如果不能，未验证状态下的限制策略是什么？这是一个关键的产品决策，直接影响用户体验和注册转化。

4. 🟡 **第三方登录"引导完善资料"弹窗可跳过，但缺少后续触发策略。** 第 181 行提到用户可跳过引导弹窗，后续在个人中心完善。但未说明：跳过后是否有其他时机再次提醒（如发布内容时）？个人中心是否有视觉提示（如头像为空的红点）？建议补充资料完善度的渐进式引导策略。

5. 🟡 **设备管理的"信任设备"功能在 F6 中未体现。** F10 异常登录通知中有"信任设备"概念（点击"是我本人操作"后标记为信任），但 F6 设备管理页的页面结构中虽然有"信任状态标签"，却未说明用户能否手动标记/取消信任、信任设备是否跳过异常检测。建议在 F6 中补充信任设备的管理交互。

6. 🟡 **注销检查项中"未结清积分"的处理入口缺失。** 第 516 行列出注销检查项包括"未结清积分"，未通过时显示红色并附带处理入口链接。但积分系统属于哪个模块、处理入口指向哪里未做说明。如果积分系统尚未上线，这个检查项可能导致用户永远无法注销。

7. 🟡 **缺少注册引导与内容消费的衔接设计。** 用户注册完成后跳转首页，但内容社区的核心价值是内容消费。建议设计注册后的个性化推荐引导（如选择感兴趣的频道/话题），帮助新用户快速找到有价值的内容，提升留存。

8. 🟡 **密码找回流程中"新密码不能与前 3 次密码相同"的前端校验不可行。** 第 407 行要求前端校验新密码不能与前 3 次相同，但前端不掌握历史密码数据。这应由后端校验，前端只负责展示后端返回的错误提示。建议将此条移至 API 对接说明中，明确由后端返回错误码。

9. 🟢 **第 10 行入口路径缺少"从内容页面触发登录"的场景。** 当前入口路径仅列出登录页、个人中心等，但内容社区最常见的场景是：用户在浏览帖子/评论时被要求登录。建议补充"内容浏览 → 点赞/评论/收藏 → 弹出登录引导 → 登录后回到原位置"的入口路径。

10. 🟢 **管理员角色在功能列表中无对应功能。** 第 26 行定义了管理员角色（查看风控日志、处理误判申诉），但功能列表 F1-F12 中没有任何管理员专属功能。如果管理员功能不属于本期范围，建议在非本期范围中明确标注。

## 缺失用户旅程

- **内容消费触发登录**：用户在浏览内容时触发点赞/评论/收藏，需要登录后才能操作。这个高频场景的登录引导交互（底部弹窗登录 vs 跳转登录页）未在 PRD 中体现。
- **登录态过期后的无感刷新**：用户在浏览过程中 Token 过期，PRD 第 978 行仅提到"自动刷新 Token 或跳转登录页"，但未定义优先策略。对于内容社区，应优先无感刷新，仅在 refresh_token 也过期时才跳转登录页。
- **多账号切换**：部分用户可能有个人账号和工作账号，当前设计不支持账号切换，每次需要退出再登录。虽然本期可不做，但建议在非本期范围中提及。
- **无障碍访问**：PRD 未提及 WCAG 合规要求。验证码输入框、弹窗焦点管理、键盘导航等对于视障用户至关重要，建议至少在通用交互要求中加入基础无障碍规范。

## 优先级建议

- **F3 第三方登录注册当前为 P0，建议评估实际优先级。** 第三方登录需要对接微信/Apple/Google 的 OAuth 流程，涉及 App 申请、回调配置、多端适配（尤其是微信在移动端的跳转逻辑），开发成本显著高于手机号和密码登录。如果内容社区初期以国内用户为主，微信登录是 P0，Apple 和 Google 可降为 P1。
- **F10 异常登录通知为 P0，但依赖 F6 设备管理。** 两者在功能列表中的依赖关系标注为 F10 依赖 F6，但 F6 本身是 P1。建议将 F6 的基础能力（设备列表展示、下线操作）提升为 P0，或调整 F10 的优先级。
- **F12 账号注销为 P1，建议确认是否需要配合合规要求。** 根据《个人信息保护法》，用户有权注销账号，平台不得设置不合理障碍。如果平台面向 C 端用户，注销功能可能有合规时间要求，建议确认法务意见。

## 总结

这是一份详尽且可执行的前端 PRD，核心认证流程覆盖完整，边界情况考虑周全。主要改进方向是：补充数据埋点方案以支撑业务目标验证、明确邮箱注册未验证状态的产品策略、以及补充内容消费场景下的登录引导设计。建议优先解决标记为 🔴 的 3 个高优先级问题后再进入开发。
