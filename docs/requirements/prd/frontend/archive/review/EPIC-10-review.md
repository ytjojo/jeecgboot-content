# EPIC-10 圈子基础能力 — 前端 PRD 审核报告

> **审核人**: 📊 Mary (Business Analyst)
> **审核日期**: 2026-06-02
> **审核视角**: 业务分析 — 需求完整性、指标、数据流

## 总体评价

PRD 整体质量较高，功能范围清晰，页面结构详尽，交互规则和边界状态覆盖较好。作为 MVP 阶段的前端 PRD，文档在页面设计、组件复用、API 对接和响应式策略方面均有明确描述，可直接指导前端开发。但在业务需求完整性、成功指标定义、部分边界场景和数据流设计方面存在若干需要补充或澄清的问题。

## 优点

1. **范围界定清晰**: 本期/非本期范围划分明确，第 1 节和功能列表中的优先级标注（P0/P1）有助于开发排期。
2. **页面设计详尽**: 6 个页面均有 ASCII 布局图、交互规则、状态与边界三要素，覆盖了主要的异常状态（404、网络错误、空状态、权限不足）。
3. **权限模型明确**: 创建者/版主/成员/游客四角色的操作权限矩阵在成员管理页（3.4）中有详细描述，前端权限判断逻辑（6.3）直接可用。
4. **组件复用意识强**: 4.2 和 4.3 列出了复用现有组件和 Hooks，避免重复造轮子，与项目既有架构一致。
5. **待确认问题诚实**: 第 12 节列出了 8 个待确认问题并给出了默认假设，降低了阻塞风险。
6. **隐私类型与加入方式的联动规则**: 创建表单中「私有→申请审核」「密码保护→密码加入且不可切换」的联动逻辑设计合理，减少了用户配置错误。

## 问题与建议

### 🔴 1. 缺少成功指标（Success Metrics）

**问题**: PRD 未定义任何可量化的业务成功指标。对于一个 MVP 功能，应明确回答"如何衡量圈子基础能力是否成功"。

**建议**: 补充核心指标，例如：
- 圈子创建量：上线 30 天内新建圈子数（目标值）
- 圈子活跃率：7 天内有成员互动的圈子占比
- 成员留存率：加入圈子后 7 日/30 日留存
- 搜索转化率：搜索结果页点击加入的比例
- 加入成功率：各加入方式（直接/密码/申请）的完成率

这些指标将直接影响后续 EPIC-11/12 的优先级排序。

### 🔴 2. 「申请审核」和「邀请加入」在本期仅做入口，但缺少明确的状态流转

**问题**: 第 12 节第 5、6 点提到「邀请加入」本期仅做入口限制，「申请审核」本期仅做申请提交，审核功能在 EPIC-11 实现。但以下问题未回答：
- 用户点击「申请加入」后，前端状态如何展示？是停留在「申请中」还是直接提示「请等待审核」？申请中的状态是否持久化？
- 如果用户申请后再次访问详情页，按钮应该显示什么？「申请中」还是仍然显示「申请加入」？
- 邀请加入模式下，非邀请用户看到的是禁用按钮+文字提示，但如果该用户确实是被邀请的，前端如何判断？邀请关系数据从哪个接口获取？

**建议**: 补充申请状态的前端展示规则和接口说明，至少定义「申请中」状态的 UI 表现和生命周期。

### 🔴 3. 缺少并发与竞态场景的业务规则

**问题**: 以下竞态场景未定义业务规则：
- 两个管理员同时对同一成员执行禁言/移除操作
- 用户在详情页停留期间，圈子被创建者修改了隐私类型（从公开改为私有）
- 用户正在填写密码加入表单时，圈子密码被修改（虽然后续迭代才支持修改，但应考虑边界）
- 成员列表分页浏览时，有成员被移除或新增，导致分页数据不一致

**建议**: 至少明确前端对这些竞态的降级策略（如操作失败时的 Toast 提示、列表自动刷新等），并标注后端需保证的幂等性要求。

### 🟡 4. 圈子成员上限未定义

**问题**: 第 3.3 节提到了「圈子已满员」的 UI 状态，但 PRD 中未定义：
- 默认成员上限是多少？
- 是否可配置？
- 不同隐私类型的圈子是否有不同的上限？

**建议**: 明确成员上限的默认值和配置规则，前端需要在详情页和列表页展示当前成员数/上限。

### 🟡 5. 密码安全策略不完整

**问题**: 访问密码仅定义了 6-20 位长度限制，缺少以下安全考量：
- 是否允许纯数字密码？（常见弱密码风险）
- 是否需要前端做密码强度提示？
- 密码输入错误是否有次数限制或锁定机制？
- 密码传输是否需要前端加密（即使有 HTTPS）？

**建议**: 至少增加密码复杂度的最低要求（如不允许纯数字），并确认后端是否有错误次数限制。前端可增加密码强度指示器。

### 🟡 6. 数据流设计过于简化

**问题**: 6.2 节的数据流设计仅展示了页面间的导航关系，未体现关键数据的生命周期：
- `currentCircle` 数据何时加载、何时失效、何时刷新？
- 成员执行退出操作后，`currentRole` 和 `currentMemberStatus` 如何清空？
- 搜索关键词 `searchKeyword` 是否需要持久化（如用户从搜索结果进入详情页后返回）？
- 圈子列表页的「已加入」Tab 和「发现」Tab 的数据是否共享缓存？

**建议**: 补充 store 数据的生命周期管理策略，特别是页面切换时的数据保留与清理规则。

### 🟡 7. 敏感词检测的降级策略不明确

**问题**: 第 12 节第 3 点提到敏感词检测默认假设调用平台已有服务，降级方案为本地词库。但 PRD 未定义：
- 如果敏感词服务不可用，是阻断创建还是放行？
- 「需人工审核」的提示意味着创建请求进入了什么状态？是待审核队列还是直接失败？
- 前端是否需要展示审核进度？

**建议**: 明确敏感词服务降级时的行为策略。建议 MVP 阶段采用「降级则放行+后端标记待复核」的策略，避免阻断用户创建。

### 🟡 8. 治理日志的数据保留与导出

**问题**: 默认假设第 8 条提到治理日志保留 180 天，但：
- 180 天后的日志是自动归档还是删除？
- 创建者是否有导出日志的需求（如用于纠纷处理）？
- 日志量较大时（高频治理的圈子），前端分页和筛选的性能保障策略是什么？

**建议**: MVP 阶段可暂不实现导出，但应在 PRD 中标注为后续需求，并确认后端日志查询接口有合适的索引支持。

### 🟡 9. 缺少国际化（i18n）和无障碍（a11y）考量

**问题**: PRD 中所有文案均为硬编码中文，未提及国际化方案。同时未涉及无障碍设计：
- 圈子图标的 alt 文本如何处理？
- 操作按钮是否支持键盘导航？
- 颜色对比度是否满足 WCAG 标准？
- 屏幕阅读器如何朗读「已加入」「禁言中」等状态标签？

**建议**: 如果当前版本仅面向中文用户，可标注为后续迭代需求。但建议至少在组件层面预留 i18n 接口，避免后续大规模改造。

### 🟢 10. 圈子分类标签的数据来源未明确

**问题**: 第 12 节第 1 点提到分类标签默认假设为预定义枚举+支持自定义输入，但：
- 预定义枚举列表由谁维护？是否有管理后台？
- 自定义输入是否需要审核？
- 一个圈子可以选几个分类标签？
- 分类标签是否有敏感词检测？

**建议**: MVP 阶段建议限制为单选预定义枚举，降低复杂度。明确枚举列表的数据来源（配置表/前端常量）。

### 🟢 11. 搜索结果的排序规则未定义

**问题**: 第 3.5 节仅提到搜索结果展示，但未定义排序规则：
- 是按相关度排序还是按成员数排序？
- 精确匹配是否优先于模糊匹配？
- 搜索结果是否需要去重？

**建议**: MVP 阶段建议按成员数倒序（与「发现」Tab 一致），后续可引入相关度排序。

### 🟢 12. 圈子详情页的「动态」Tab 缺少占位设计

**问题**: 第 3.3 节提到动态 Tab 内容在 EPIC-11 实现，但未定义本期该 Tab 的占位表现：
- 是否显示空白 Tab？
- 是否显示「即将上线」提示？
- 是否隐藏该 Tab？

**建议**: 建议本期隐藏「动态」Tab，仅展示「成员」Tab，避免用户困惑。

## 业务建议

1. **补充业务流程图**: 建议增加核心业务流程的状态流转图，特别是「加入圈子」的四种方式各自的状态机（未加入→申请中→已加入/被拒绝；未加入→输入密码→已加入/密码错误），这将帮助开发和测试更准确地理解业务逻辑。

2. **定义 MVP 验收标准**: 当前 PRD 有测试要点但缺少明确的 MVP 验收标准。建议定义最小可发布集合（如：创建圈子+直接加入+搜索 三个核心路径必须完全可用）。

3. **考虑灰度发布策略**: 圈子作为社区基础能力，建议定义灰度发布的范围（如先开放给特定用户群），PRD 中未涉及此策略。

4. **数据迁移与兼容性**: 如果系统已有其他形式的用户分组或社区功能，需考虑数据迁移方案。PRD 未提及与现有功能的关系。

5. **运营干预能力**: MVP 阶段虽然不实现审核机制，但应考虑运营侧是否需要紧急关闭/冻结圈子的能力（如违规内容处理），建议至少预留接口。

## 总结

EPIC-10 前端 PRD 在页面设计、交互规则和组件复用方面完成度很高（约 85%），可作为前端开发的基础输入。主要缺失集中在**业务层面**：成功指标未定义、部分状态流转不完整、竞态场景未覆盖。建议在开发启动前优先补充成功指标和申请/邀请状态的展示规则（问题 1、2），其余问题可在开发过程中逐步澄清。

**审核结论**: 有条件通过，需补充问题 1（成功指标）和问题 2（状态流转）后方可进入开发阶段。
