# EPIC-04 社交关系与内容订阅 — 前端 PRD 审核报告

> **审核人**: 📊 Mary (Business Analyst)
> **审核日期**: 2026-06-02
> **审核视角**: 业务分析 — 需求完整性、指标、数据流

## 总体评价

PRD 文档结构完整、功能覆盖全面，包含 14 个子功能模块、详细的交互说明、响应式适配规则和验收标准。需求目标清晰（关注数提升 40%、点击率 >30% 等），范围边界明确（非本期范围单独列出）。但从业务分析角度看，在成功指标可衡量性、数据流闭环、边界场景覆盖和依赖风险方面仍有改进空间。

## 优点

- **范围界定清晰**: "本期范围"与"非本期范围"划分明确，付费订阅仅保留扩展点、不引入 ML 推荐等决策务实合理
- **交互细节充分**: 每个功能均包含触发条件、操作步骤、系统反馈、完成条件四要素，覆盖了防抖、乐观更新、二次确认等关键交互模式
- **响应式规则完整**: 断点定义清晰，每个页面在 PC/平板/手机三种场景下均有具体布局规则，移动端操作收进 Dropdown 等决策一致
- **边界情况覆盖较好**: 关注了已注销用户、被拉黑、分组名重复、批量操作部分失败等常见边界场景
- **假设显式声明**: 第 6 节列出 9 条默认假设，便于后续验证和追溯
- **组件复用策略明确**: 指定了 Ant Design Vue 4 + 项目已有组件/hooks 的复用路径，减少重复开发

## 问题与建议

1. **🔴高 — 成功指标缺乏基线数据和衡量方式。** 第 1 节提出"平均关注数提升 40%、关注流点击率 >30%、订阅转化率 >25%、用户留存率提升 20%"，但未说明：当前基线值是多少？"点击率"的分母是什么（曝光次数？UV？）？"订阅转化率"如何定义（订阅广场曝光→订阅？）？留存率按日/周/月计算？建议补充指标定义公式、基线数据来源、数据采集埋点方案和目标达成的时间窗口。

2. **🔴高 — 缺少关注流数据流架构说明。** 假设 3 明确"读扩散（实时查询聚合），不引入写扩散或消息扇出"，但 PRD 未描述关注流的聚合逻辑：当用户关注 500+ 人时，实时聚合查询的性能边界在哪里？"特别关注置顶"是在前端排序还是后端分两个接口返回？动态类型筛选（发布/点赞/收藏）是前端过滤还是后端支持？这些直接影响接口设计和前端性能方案，建议补充数据流示意图或接口调用时序。

3. **🔴高 — 关注数上限未定义。** 假设中仅定义了分组上限（20 个）和批量操作上限（100 条），但未定义单用户最大关注数。如果平台允许关注数千人，关注流实时聚合查询将成为性能瓶颈。建议明确关注数上限（如 5000），超出后的行为策略（如"关注已满"提示），以及大关注量用户关注流的降级方案。

4. **🟡中 — 订阅源目录表设计存在前置依赖风险。** 待确认问题 2 提到"订阅源（专题、合集、话题、栏目）当前是否都有稳定的内容主表？"并假设"需要先定义轻量目录表承载广场展示"。这是一个关键前置依赖——如果各内容源的数据模型不统一，订阅广场的分类浏览和搜索功能将无法实现。建议在此 PRD 落地前，先完成订阅源目录表的数据模型设计评审。

5. **🟡中 — 订阅通知配置的"全局默认值"数据来源未说明。** 3.10 节提到"页面顶部展示当前全局通知配置，订阅级配置覆盖全局"，但未说明全局默认值由谁维护、从哪个接口获取、管理员是否可配置。如果全局默认值硬编码在前端，后续调整需要发版。建议明确全局通知配置的数据来源和管理入口。

6. **🟡中 — 批量操作结果展示缺少失败重试机制。** 3.8 节定义了批量操作结果展示（成功 X 个、失败 Y 个，失败项展开显示原因），但未说明：用户能否对失败项重新执行操作？失败原因是否可操作（如"该用户已注销"vs"网络超时"，后者应可重试）？建议区分可重试和不可重试的失败类型，并提供"重试失败项"操作。

7. **🟡中 — 关注推荐的"推荐理由"与推荐算法细节不足。** 3.7 节提到推荐理由标签如"与您关注的 XXX 相似"、"共同关注 XX 人"，但未说明：推荐理由由后端返回还是前端拼接？推荐列表是否有数量上限？用户能否对推荐结果反馈（"不感兴趣"）？"排除已关注、已拉黑用户"的验收标准 5.5 已列出，但未说明排除逻辑在前端还是后端实现。建议明确推荐接口的入参/出参设计。

8. **🟡中 — 关注流与订阅流的"取消关注/取消订阅"后的列表刷新策略不一致。** 验收标准 5.4 写"取消关注后刷新，该用户动态不再出现"，但交互要求中使用乐观更新（先更新 UI，失败回滚）。如果取消关注后需要"刷新"列表，意味着要重新请求全量数据，与乐观更新的增量策略矛盾。建议统一策略：取消关注后在前端移除该用户的所有动态卡片（乐观更新），而非重新请求。

9. **🟢低 — 订阅广场分类标签（科技、娱乐、体育等）的分类体系未定义。** 3.11 节提到分类标签但未给出完整分类列表，也未说明分类由后端动态返回还是前端硬编码。建议明确分类数据来源，避免后续对齐问题。

10. **🟢低 — 邮件通知的用户邮箱采集入口缺失。** 3.10 节配置了"邮件通知"渠道开关，但未说明如果用户未绑定邮箱，开启邮件通知时的处理逻辑。建议增加"未绑定邮箱"的引导提示和邮箱绑定入口。

11. **🟢低 — 缺少数据埋点规划。** 成功指标（点击率、转化率、留存率）的衡量依赖前端埋点，但 PRD 未列出需要埋点的关键事件（如关注按钮点击、订阅广场曝光、关注流滚动深度等）。建议补充埋点事件清单，与指标定义形成闭环。

## 业务建议

1. **分阶段交付**: 14 个功能模块工作量较大，建议按优先级分两个迭代交付：第一迭代（P0）= 关注/取消关注 + 关注列表 + 关注流 + 订阅内容源 + 订阅流；第二迭代（P1）= 特别关注 + 分组管理 + 推荐 + 批量管理 + 订阅广场 + 通知配置 + 订阅管理。

2. **关注流性能预案**: 读扩散方案在关注量大时存在性能风险，建议在技术方案评审时明确：关注流分页大小、单次聚合查询的时间预算、超时降级策略（如仅展示最近 N 小时的动态）。

3. **用户引导策略**: 文档中多处定义了空状态引导（"还没有关注任何人"等），但未设计新用户首次进入时的引导流程。建议增加新用户首次进入关注流时的引导弹窗或推荐卡片，降低冷启动门槛。

4. **数据一致性**: 乐观更新策略可能导致短暂的数据不一致（如关注数 +1 但后端实际失败），建议在关注数等计数展示上增加"正在同步"的中间态提示，避免用户困惑。

## 总结

PRD 功能定义扎实、交互规格详尽，具备直接进入技术方案评审的条件。主要风险集中在成功指标缺乏可操作的衡量方案、关注流数据流架构未明确、以及订阅源目录表的前置依赖。建议优先补齐指标定义和数据流说明后再启动开发。
