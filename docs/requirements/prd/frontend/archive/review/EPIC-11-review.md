1. **结构层次分明** — 从概述 → 功能列表 → 页面设计 → 组件设计 → API → 状态管理 → 交互设计 → 响应式 → 性能 → 测试，逻辑递进清晰，开发者可按章节逐步理解需求。

2. **ASCII 布局图实用** — 每个页面/弹窗均配有 ASCII 线框图（如 3.1、3.3、3.5、3.7 等），直观传达布局意图，减少设计师与开发者之间的理解偏差。

3. **交互流程伪代码清晰** — 第 8 节使用 `→` 箭头串联的伪代码描述交互流程（如 8.2 置顶操作、8.4 @成员交互），比纯文字描述更易理解。

4. **组件设计包含 Props/Events/行为** — 第 4 节为每个组件定义了 Props 表、Events 表和行为描述，前端开发可直接据此编码，无需反复确认。

5. **待确认问题与默认假设显式列出** — 末尾的"待确认问题"和"默认假设"章节体现了文档的诚实性，避免隐性假设导致返工。

6. **响应式设计有断点表** — 第 9 节给出了明确的断点定义和各页面的移动端适配规则，覆盖了 PC/移动端两端。

## 问题与建议

### 1. 🔴 "版主/创建者"与"管理员"称谓混用

**位置**: 全文多处

**问题**: 文档中对"有管理权限的角色"存在三种不同称谓：
- 3.1 节第 74 行："版主/创建者可见操作菜单"
- 功能列表第 45 行："所有成员/管理员"
- 3.7 节第 242 行："支持管理员处理举报"
- 3.6 节第 229 行："管理员点击'批准'"

"管理员"在此语境下指代不明 — 是指"圈子创建者+版主"这个集合，还是指平台级管理员？功能列表中"管理员"与"版主/创建者"并列出现，暗示它们是不同角色，但交互描述中又将二者混用。

**建议**: 统一术语。若"管理员"等同于"版主+创建者"，建议在文档开头定义一个角色术语表，例如：
- **圈子管理员** = 版主 + 创建者（统称）
- **普通成员** = 非管理角色的圈子成员
- **平台管理员** = 系统级管理员（如有）

后续全文统一使用"圈子管理员"而非交替使用"版主/创建者"和"管理员"。

功能列表第 45 行写"所有成员/管理员"，但这是指"所有成员可举报，管理员可处理"还是"所有成员都可处理"？

### 3. 🔴 API 响应格式不完整

**位置**: 5.1-5.5 API 对接章节（第 415-526 行）

**问题**: 大部分 API 仅定义了请求格式，缺少完整的响应格式定义。例如：
- 5.1 节的 `togglePin` 有响应格式，但 `toggleFeatured` 没有
- 5.2 节的 `CreateAnnouncementReq` 有请求格式，但未给出响应格式
- 5.4 节的批准接口 `PUT /circle-join-request/{id}/approve` 无请求/响应格式
- 5.5 节的 `CreateReportReq` 有请求格式，但 `delete-content`/`ignore` 无请求格式

虽然 `frontend-standards.md` 定义了通用响应格式 `{ code: 200, result: any, message: string, success: boolean }`，但各接口的 `result` 字段内容未定义。前端开发需要知道响应体结构才能正确处理数据。

**建议**: 为每个 API 补充完整的响应格式定义，至少包括 `result` 中的字段列表。例如：

```typescript
// PUT /circle-content/{id}/featured
// Response: { code: 200, result: { isFeatured: boolean }, success: boolean }
```

### 4. 🟡 公告有效期字段语义不清

**位置**: 3.3 公告发布弹窗（第 124-128 行）和 5.2 API（第 441-443 行）

**问题**: 公告有效期的字段命名和语义存在歧义：
- 布局图中标注"有效期 *"，选择器标签为"日期时间选择器（至）"— 暗示只有一个截止时间
- API 中字段名为 `expireAt`（截止时间）
- 但 3.4 节公告展示栏（第 158 行）显示"有效期：2026-06-01 ~ 2026-06-30"— 暗示有起止时间范围

这里存在矛盾：如果只有一个截止时间，展示栏的起始日期从何而来？是公告创建时间吗？

**建议**: 明确有效期的含义：
- 若只有截止时间，展示栏应改为"有效期至：2026-06-30"，并去掉起始日期
- 若是起止时间范围，API 请求应增加 `startAt` 字段，弹窗应增加开始时间选择器

### 6. 🟡 @成员提及标记格式未定义

**位置**: 默认假设第 4 条（第 794 行）和 5.3 API（第 454-462 行）

**问题**: 假设中写道"@成员的提及标记在前端以特殊标签形式嵌入富文本或纯文本中，具体格式与后端约定"，但：
- 前端如何在输入框中插入 @提及标记？是纯文本 `@张三` 还是特殊标签 `<mention userId="xxx">@张三</mention>`？
- 内容发布时，@标记以什么格式传给后端 API？
- 评论中 @成员的格式与内容中是否一致？
- 后端返回的内容中 @标记格式是什么？前端如何解析并渲染为可点击链接？

这是前端开发的关键实现细节，"与后端约定"不够具体。

**建议**: 补充 @提及标记的数据格式约定，例如：

```typescript
// 前端插入格式：纯文本或富文本标签
// 传给后端格式：@{userId:xxx}昵称 或自定义格式
// 后端返回格式：同上，前端需解析为可点击链接
```

### 10. 🟢 组件设计中 JoinRequestVO 和 ReportVO 未定义

**位置**: 4.4（第 379 行）、4.5（第 399 行）

**问题**: `JoinRequestCard` 的 Props 引用了 `JoinRequestVO` 类型，`ReportCard` 引用了 `ReportVO` 类型，但文档中未给出这两个类型的字段定义。前端开发需要知道 `request` 对象包含哪些字段（如 `id`、`nickname`、`avatar`、`applyTime`、`reason` 等）才能正确渲染组件。

**建议**: 补充 VO 类型定义，至少列出前端渲染所需的字段。例如：

```typescript
interface JoinRequestVO {
  id: string;
  userId: string;
  nickname: string;
  avatar: string;
  applyTime: string;   // ISO8601
  reason?: string;      // 申请理由
  status: 'PENDING' | 'APPROVED' | 'REJECTED';
}
```

## 文档改进建议

1. **增加术语表** — 在第 1 节概述后增加"术语定义"小节，统一定义"圈子管理员""版主""创建者""普通成员""平台管理员"等角色称谓，以及"置顶""精华""公告"等核心概念的精确含义。

2. **增加角色-权限矩阵** — 在功能列表之后增加一个完整的角色-功能-权限对照表，覆盖所有 6 个功能模块的读/写权限。当前文档中权限信息分散在各页面描述中，不便于全局查阅。

3. **补充数据模型章节** — 当前文档有 API 定义和组件设计，但缺少统一的数据模型（VO/DTO）定义章节。建议新增一节，集中定义 `AnnouncementVO`、`MentionMemberVO`、`JoinRequestVO`、`ReportVO` 等类型。

4. **统一 API 定义格式** — 当前各 API 的请求/响应格式详略不一。建议为每个接口统一使用三段式格式：请求参数 → 请求体（如有） → 响应体。

5. **交互流程图补充异常分支** — 第 8 节的交互流程主要描述了成功路径。建议为关键操作（如公告发布、举报处理）补充失败分支和网络异常处理流程。

## 总结

EPIC-11 前端 PRD 是一份质量较好的需求文档，在页面布局、交互流程、组件设计、响应式适配等方面提供了充分的细节。主要改进方向集中在三个方面：

1. **术语一致性** — "版主/创建者/管理员"的混用是全文最突出的问题，建议在文档开头定义术语表并全文统一。
2. **数据契约完整性** — API 响应格式、VO 类型定义、@提及标记格式等数据契约需要补充，否则前端开发需反复与后端确认。
3. **边界条件精确性** — 举报处理权限、超时提醒机制、公告有效期语义等关键边界条件需要进一步明确。

以上问题均为可快速修复的文档层面改进，不影响整体架构设计的合理性。建议在开发启动前完成修订。
