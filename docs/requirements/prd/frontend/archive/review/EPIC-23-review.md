## 问题与建议

### 🔴 高优先级

**1. "频道类型" 术语在文档内存在两种含义，易引发混淆**

文档中 "频道类型" 至少指代两个不同概念：
- 第 88 行：`频道类型标签` — 指 channelType（personal/organization/system）
- 第 110 行：`频道类型筛选（个人/组织/系统）` — 同上
- 第 224 行：`系统频道: 可选择特殊分类` — 这里 "系统" 是 channelType 的一种

但在第 298 行筛选条件配置中写的是 `频道类型: 多选（个人/组织/系统）`，与 5.3 节 `channelType: 'personal' | 'organization' | 'system'` 对应。问题在于：文档中同时存在 "频道类型" 和 "频道状态"（如第 199 行 `状态异常`、第 420 行 `status: 'active' | 'expired' | 'abnormal'`），两者在筛选条件中是否合并为同一维度未明确说明。

**建议**：在文档开头增加"术语表"小节，统一定义"频道类型"（channelType）、"频道状态"（channelStatus）、"分类"（category）、"标签"（tag）等核心概念，避免后续开发歧义。

**2. 搜索 API 的分页参数未在类型定义中体现**

第 358 行搜索 API 注释为 `搜索频道（参数: keyword, type, category, sort）`，但第 373-375 行代码示例中实际传参为 `{ keyword, type, category, sort, page, pageSize }`。API 列表表格中遗漏了 `page` 和 `pageSize` 参数。

同样，第 359 行 `分类浏览频道列表（参数: categoryId, type, sort, page）` 也缺少 `pageSize` 参数说明。

**建议**：API 列表表格的"说明"列应包含完整的请求参数列表，或在表格旁增加参数说明子表。

**3. `ChannelEditorialPickVO` 的 status 类型与页面描述不一致**

5.3 节定义 `status: 'active' | 'expired' | 'abnormal'`，但第 192 行状态筛选为 `全部 / 生效中 / 已过期 / 状态异常`。"active" 与 "生效中"、"expired" 与 "已过期" 的映射虽可推断，但 "abnormal" 的触发条件仅在第 199 行描述为 `频道变为私有/隐藏/冻结时`，未说明是否包含"限制公开曝光"和"未通过审核"这两种可见性状态。

**建议**：明确 "abnormal" 状态的完整触发条件列表，与 11.2 节可见性测试用例保持一致。

---

### 🟡 中优先级

**4. 推荐理由与匹配原因的数据结构过于简化**

第 399 行 `reason: string` 和第 405 行 `matchReason: string` 均为纯字符串，但第 98 行描述推荐卡片第三层为 `订阅数 + 推荐理由`，第 139 行搜索结果第三层为 `订阅数 + 匹配原因（如"名称匹配""标签匹配"）`。

如果推荐理由需要结构化展示（如图标+文案），纯 string 不够；如果匹配原因需要枚举值以支持多语言或前端样式差异化，也应使用联合类型而非 string。

**建议**：考虑将 `reason` 和 `matchReason` 定义为联合类型或包含 type + text 的结构体，例如 `matchReason: { type: 'name' | 'tag' | 'description'; text: string }`。

**5. 分类树最大层级 "4 级" 的约束来源不明**

第 262 行 `maxLevel: number` 默认值为 4，第 798 行假设中也提到 `平台分类最多 4 级`，但文档未说明这个限制是产品决策、技术约束还是后端限制。如果后端不限制层级，前端单独限制可能导致数据不一致。

**建议**：在 12.2 默认假设中补充说明 4 级限制的来源（产品决策 or 后端约束），以及前端是否需要做层级校验。

**6. Store 缓存策略描述不够完整**

第 467 行 `lastFetchTime: number` 注释为 `上次拉取时间，5 分钟缓存`，第 705 行也提到 `推荐数据 5 分钟缓存`。但以下问题未明确：
- 缓存过期后是自动刷新还是用户操作触发？
- 分类树缓存的失效策略是什么？（仅写 `变更后刷新`，未定义 "变更" 的检测机制）
- 搜索结果是否缓存？（`useChannelSearchStore` 中有 `results` 字段但未说明缓存策略）

**建议**：在 6.2 Store 结构或 10.2 优化策略中增加缓存失效规则表，明确每个 Store 的缓存时长、失效条件和刷新方式。

**7. "频道发现页" 入口描述存在多处但未统一**

第 92 行列出三个入口：`频道模块顶部导航 / 首页频道推荐区域 / 底部 Tab 切换`。但这三个入口是否全部实现、各自跳转行为是否一致未说明。特别是 "首页频道推荐区域" 与发现页的关系——是嵌入式模块还是跳转入口？

**建议**：在 3.1 节补充入口交互说明，明确每个入口的跳转方式（路由跳转/页面内嵌）和预期用户流。

**8. 排行榜 "新晋榜" 和 "系统榜" 的维度切换未定义**

第 151 行提到 `热门榜支持日/周/月维度切换`，但未说明新晋榜和系统榜是否支持维度切换。第 5.1 节排行榜 API 参数为 `type, dimension`，暗示所有榜单都支持 dimension，但页面设计仅对热门榜做了说明。

**建议**：明确每个榜单类型支持的维度选项，或在 3.4 节补充说明 "维度切换仅对热门榜生效"。

### 🟢 低优先级

**9. 用户故事优先级与页面设计的对应关系可加强**

U-C5（不感兴趣反馈）标记为 P1，但 8.2 节交互设计中将其描述为 `从卡片 ... 菜单中选择，二次确认后调用反馈 API`，而 12.1 待确认问题中仍在讨论操作位置。P1 功能的设计应在文档定稿前确认。

**建议**：在 12.1 节该问题旁标注为"阻塞 P1 功能开发"，提升优先级。

**10. 部分组件 Props 缺少可选/必填标注**

4.1-4.6 节的 Props 表格均未标注属性是否必填。例如 ChannelCard 的 `rank` 和 `showRank` 在非 ranking 模式下是否必填？`hotCategories` 为空时 SearchBar 如何表现？

**建议**：在 Props 表格中增加"必填"列，或在类型定义中使用 `?` 标注可选属性。

**11. 骨架屏与虚拟滚动的具体实现策略未展开**

第 708 行提到 `所有列表页首屏展示骨架屏`，第 710 行提到 `排行榜列表超过 100 条时启用虚拟滚动`。但骨架屏的形态（与实际卡片布局的对应关系）和虚拟滚动的组件选型（是否使用现有组件库的 VirtualList）未说明。

**建议**：可作为实现阶段的技术设计补充，但在 PRD 中注明骨架屏设计稿的引用链接或 Figma 地址。

**12. 搜索满意度指标缺少采集方式**

第 15 行定义 `搜索满意度 >= 85%`，第 134 行提到 `结果有帮助"反馈按钮（用于满意度统计）`。但 "满意度" 的计算公式未定义——是反馈按钮的正向点击率？还是包含其他维度（如点击搜索结果后的停留时长）？

**建议**：在 10.1 或 11.4 节补充满意度指标的计算口径，或标注为"待数据团队确认"。

## 文档改进建议

1. **增加术语表**：在第 1 章概述之后、第 2 章用户故事之前，增加"术语定义"小节，统一定义频道类型、频道状态、分类、标签、精选、推荐理由等核心概念。

2. **增加数据流图**：建议增加一张简要的数据流示意图，展示用户从发现页 → 搜索/分类浏览 → 频道主页的核心路径，以及各 Store 之间的数据关系。

3. **API 参数补全**：将 API 列表表格的"说明"列扩展为包含完整请求参数和关键响应字段，避免开发时反复查看类型定义。

4. **优先级与排期映射**：当前用户故事有优先级（P0/P1/P2），但未与 4 周开发周期做映射。建议增加简要的迭代规划表，说明哪些故事在哪个 Sprint 完成。

5. **国际化预留**：文档中多处包含中文文案（如 "不感兴趣"、"结果有帮助"、"标签名称不能为空"），但未说明是否需要支持多语言。如果需要，建议标注所有用户可见文案的 i18n key 规划。

## 总结

EPIC-23 前端 PRD 整体质量良好，结构完整、细节丰富，具备直接指导开发的能力。主要改进方向集中在三个方面：一是术语一致性（"频道类型"等核心概念需要统一定义），二是 API 参数完整性（分页参数遗漏），三是缓存与指标的计算口径需要更明确的定义。上述 12 个问题中，3 个高优先级建议在开发启动前解决，其余可在实现过程中逐步完善。
