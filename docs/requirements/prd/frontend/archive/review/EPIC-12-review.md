# EPIC-12 圈子数据与推荐 — 前端 PRD 审核报告

> **审核人**: Amelia (Senior Software Engineer)
> **审核日期**: 2026-06-02
> **审核视角**: 工程实现 — 可测试性、复杂度、API 契约

## 总体评价

PRD 整体结构清晰，页面设计、组件拆分、状态管理、交互矩阵均有覆盖，属于中等偏上质量的需求文档。两个核心特性（数据统计面板 + 推荐/榜单）在工程上是独立的，可以并行开发，里程碑划分合理。主要风险集中在 API 契约的模糊地带、图表库选型未确定、以及若干边界条件的缺失。

## 优点

1. **组件拆分合理**：StatCard、TrendChart、RankList、CircleCard 等组件职责单一，Props 定义明确，复用性好。CircleCard 复用 EPIC-10 并通过扩展 Props 增加 `source`/`rank`/`showRank` 的做法是正确的增量设计。

2. **状态管理分层清晰**：Pinia Store 区分了 `useCircleAnalyticsStore` 和 `useCircleRecommendStore`，页面级状态用 `ref`/`reactive` 管理，职责边界清楚。

3. **交互矩阵完整**：8.1 和 8.2 的交互矩阵覆盖了主要用户操作路径，包括降级模式、权限不足、空状态等异常场景。

4. **响应式设计有具体断点和策略**：9.1-9.3 给出了明确的断点定义和各页面在不同宽度下的布局策略，移动端也有针对性处理（横向滑动图表、原生日期选择器）。

5. **性能优化策略务实**：批量曝光上报延迟 500ms 合并、图表数据超过 30 天按周聚合、防抖 300ms 等策略都是合理的工程决策。

6. **TypeScript 接口定义完整**：5.3 节的核心数据结构已定义，可直接用于 API 层开发。

## 问题与建议

### 1. [高] API 契约缺少分页、排序和错误码定义

5.1 节仅列出了接口路径和方法，缺少以下关键契约信息：

- **请求参数未定义**：`GET /api/v1/content/circle/{id}/analytics` 的 `startDate`/`endDate` 格式是 `YYYY-MM-DD` 还是 ISO 8601？是否支持时区？
- **分页策略缺失**：推荐列表假设一次性返回（第 12 节假设 3），但 `CircleRecommendVO` 没有分页字段。如果推荐量增长，需要预留分页能力或明确上限。
- **错误响应格式未约定**：403 权限不足、404 圈子不存在、500 服务端错误的响应体结构是什么？前端需要统一的错误处理逻辑。
- **导出接口的响应类型**：`GET /analytics/export` 返回的是文件流还是 JSON 包含下载链接？文件名格式是什么？

**建议**：补充 API 契约文档，至少包含请求参数类型、响应状态码、错误响应体结构。推荐接口增加 `limit` 参数，即使初期不分页也便于控制返回量。

### 2. [高] 图表库选型悬而未决

第 12 节待确认问题 1 提到图表库未确定，假设用 ECharts 但 fallback 为 AntV G2。这两个库的 API 差异很大，直接影响：

- TrendChart 组件的内部实现
- 按需引入的 bundle 大小（ECharts tree-shaking 方案 vs AntV 模块化引入）
- 移动端横向滑动的实现方式
- Tooltip 交互行为

**建议**：在开发启动前确认图表库选型。如果项目已有 ECharts 依赖，直接锁定 ECharts，移除 AntV 的 fallback 假设。TrendChart 组件应封装图表库细节，对外暴露统一 Props 接口，降低后续切换成本。

### 3. [高] 曝光上报的实现细节不足

PRD 要求"延迟 500ms 合并上报推荐曝光事件"，但以下问题未澄清：

- **可见性判定**：是 IntersectionObserver 检测卡片进入视口，还是仅在推荐列表渲染完成时一次性上报所有卡片？如果是前者，虚拟滚动场景下如何处理？
- **去重策略**：用户反复滚动同一区域，同一圈子是否重复上报？
- **上报失败重试**：PRD 说"静默处理"，但是否需要本地队列 + 下次进入页面时重试？还是直接丢弃？
- **页面离开时的 flush**：用户快速离开页面时，500ms 延迟内的曝光数据是否丢失？需要 `beforeunload` 或 `visibilitychange` 事件触发立即上报。

**建议**：明确曝光上报的触发时机（IntersectionObserver 还是渲染完成），定义去重逻辑和页面离开时的 flush 策略。建议使用 `navigator.sendBeacon` 保证页面离开时的数据不丢失。

### 4. [中] CircleCard 扩展设计可能导致 Props 膨胀

CircleCard 从 EPIC-10 复用，新增了 `source`、`rank`、`showRank` 三个 Props。随着后续 Epic 增加（如 EPIC-13 成长激励），这个组件的 Props 会持续膨胀。

**建议**：考虑使用 `slots` 或 `renderProps` 模式处理榜单排名等可变展示区域，而非不断增加 Props。或者将榜单排名展示拆为 RankCircleCard（组合 CircleCard + 排名标识），遵循组合优于继承原则。

### 5. [中] 数据统计页的"与上期对比"逻辑未定义

StatCard 的 `change` 字段表示"与上期对比变化百分比"，但"上期"的计算基准未定义：

- 如果当前选择"近 7 天"，上期是前 7 天还是前一个自然周？
- 如果选择"自定义范围"（如 10 天），上期是前 10 天还是前一个自然周期？
- 百分比计算基数为零时如何处理（如上期成员数为 0，本期新增 5 人，变化百分比是无穷大还是 100%）？

**建议**：明确"上期"的计算规则（建议使用与当前选择等长的前一时间段），并在 StatCard 组件中处理基数为零的边界情况（展示"新增"而非 NaN 或 Infinity）。

### 6. [中] Pinia Store 的 clearData 生命周期管理

`useCircleAnalyticsStore` 有 `clearData()` action，PRD 说"离开页面时清除"。但实现方式未说明：

- 在 `onUnmounted` 中调用？如果用户只是切换 Tab 再切回来，数据会被清除导致重新请求。
- 如果使用路由离开钩子（`beforeRouteLeave`），SPA 内部导航和浏览器后退的行为可能不同。

**建议**：统计数据适合在 `onActivated`（keep-alive 场景）或路由进入时按需刷新，而非在离开时清除。推荐列表和榜单的"会话级缓存"应明确缓存失效条件（时间过期、用户主动刷新、还是永久缓存直到页面关闭）。

### 7. [中] 自定义时间范围"不超过 90 天"的限制仅在交互矩阵中提及

8.1 交互矩阵提到"范围不超过 90 天"，但以下位置均未体现：

- TimeRangeSelector 组件的 Props 定义中没有 `maxRange` 属性
- API 接口参数说明中没有提及
- 测试要点 11.5 提到了"限制提示"但未说明提示方式（禁用日期？Toast 提示？DatePicker 校验？）

**建议**：在 TimeRangeSelector 组件中增加 `maxRangeDays` Prop，DatePicker.RangePicker 使用 `disabledDate` 禁用超出范围的日期，而非提交后才提示错误。这是前端校验优先原则。

### 8. [中] 推荐为空时的降级逻辑需要明确状态转换

PRD 说"推荐为空时自动切换展示热门榜单作为兜底内容"，但实现上有歧义：

- 是推荐接口返回空数组后自动切换，还是推荐接口报错后也降级？
- 降级后用户手动切回"推荐"Tab 会怎样？是再次请求推荐接口还是保持降级状态？
- `fallbackMode` 状态是否持久化到 URL（如 `?tab=recommend&fallback=true`），刷新页面后是否保持降级？

**建议**：明确降级触发条件（仅空数组 vs 空数组 + 接口错误），降级状态下 Tab 切换行为，以及刷新页面后的状态恢复策略。

### 9. [中] CSV 导出缺少前端字段映射和格式规范

导出接口 `GET /analytics/export` 返回 CSV，但以下细节未定义：

- CSV 的列头是中文还是英文？是否与页面展示的指标名称一致？
- 日期格式（CSV 中的趋势数据是 `YYYY-MM-DD` 还是其他格式）？
- 数值精度（百分比保留几位小数）？
- 文件名格式（PRD 说"含圈子名和时间范围"，具体格式是什么？如 `圈子名_2026-05-01_2026-05-07.csv`）
- 大文件场景：PRD 提到"流式下载"和"下载进度"，但浏览器原生下载不支持精确进度展示，需要 Blob + `<a>` download 还是后端返回下载链接？

**建议**：补充 CSV 导出的字段映射表和文件命名规范。如果是后端直接返回文件流，前端使用 `Blob` + `URL.createObjectURL` 触发下载即可，无需展示进度条（浏览器自带下载进度）。

### 10. [低] TrendChart 的 data 类型未包含日期粒度信息

`TrendDataPoint` 只有 `date` 和 `value`，但当时间范围超过 30 天时 PRD 要求按周聚合。此时 `date` 字段代表的是周起始日还是周结束日？图表 X 轴标签如何展示（"第 X 周"还是日期范围）？

**建议**：在 `TrendDataPoint` 中增加可选的 `granularity` 字段（`'day' | 'week'`），或由后端在响应中注明聚合粒度，前端据此调整 X 轴标签格式。

### 11. [低] 游客场景的默认 Tab 逻辑需要路由层支持

PRD 说游客默认展示"热门榜"Tab，已登录用户默认展示"推荐"Tab。如果使用 URL 参数控制 Tab（如 `?tab=recommend`），游客通过分享链接进入时会加载推荐接口（可能返回 401）。

**建议**：Tab 默认值应根据登录状态动态决定，而非从 URL 参数读取。如果需要支持 URL 分享带 Tab 参数，应在路由守卫中做登录态校验，未登录时强制覆盖为 `hot`。

### 12. [低] 测试要点缺少埋点上报的验证方法

11.1 测试要点中"推荐曝光事件正确上报"和"推荐点击事件正确上报"是 checkbox 项，但没有说明如何验证：

- 是否需要 Mock 后端接口验证请求体？
- 是否需要在开发环境增加埋点日志面板？
- 批量上报的 500ms 延迟如何在测试中验证（需要 `vi.advanceTimersByTime` 或类似手段）？

**建议**：在测试要点中补充埋点验证的具体方法，建议在开发环境增加埋点调试开关，方便 QA 验证上报数据。

## 实现建议

### 技术方案要点

1. **图表封装层**：无论选 ECharts 还是 AntV，都应封装 `useChart` composable 处理实例创建、resize 监听、销毁等生命周期，TrendChart 组件仅负责数据映射和配置。

2. **曝光上报方案**：使用 `IntersectionObserver` + `Set` 去重 + `navigator.sendBeacon` 保底。伪代码：
   ```typescript
   // 可见时加入待上报集合
   // 延迟 500ms 后批量 flush
   // 页面离开时 sendBeacon 兜底
   ```

3. **权限控制**：数据统计页的权限校验应在路由守卫层完成（检查当前用户是否为圈子创建者/版主），而非进入页面后在组件内校验。避免页面闪烁。

4. **导出实现**：后端返回文件流，前端使用 `fetch` + `Blob` + `<a download>` 触发下载。无需引入额外文件下载库。

5. **移动端图表**：ECharts 移动端性能需要关注，建议关闭动画、减少数据点、使用 canvas 渲染器而非 svg。

### 开发顺序建议

1. **Week 1 前半**：确认图表库选型 + StatCard/TrendChart 通用组件开发 + API 接口联调
2. **Week 1 后半**：DataAnalyticsPanel 集成 + 权限路由守卫 + CSV 导出
3. **Week 2 前半**：CircleCard 扩展 + RankList + RecommendationPanel + 曝光/点击上报
4. **Week 2 后半**：响应式适配 + 移动端测试 + 边界场景补全
5. **Week 3**：全量联调 + 性能测试 + 兼容性测试

## 总结

PRD 在页面设计和交互层面覆盖较为完整，但在 API 契约精度、边界条件定义、以及若干工程决策（图表库选型、曝光上报方案、缓存失效策略）上存在模糊地带。建议在开发启动前优先解决 3 个高优先级问题（API 契约补全、图表库确认、曝光上报方案明确），其余中低优先级问题可在开发过程中逐步澄清。整体预估工作量为 2 名前端工程师 2-3 周，风险可控。
