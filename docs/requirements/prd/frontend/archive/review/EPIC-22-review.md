# EPIC-22 频道内容发布与治理 -- 前端 PRD 审核报告

> **审核人**: 📊 Mary (Business Analyst)
> **审核日期**: 2026-06-02
> **审核视角**: 业务分析 -- 需求完整性、指标、数据流、依赖、边界

---

## 总体评价

EPIC-22 前端 PRD 覆盖了频道内容发布与治理的完整链路，从发布权限配置到审核、治理、回收站、公告，形成闭环。文档结构清晰，页面设计配有 ASCII 线框图，组件选型贴合项目技术栈。从业务分析角度看，PRD 在需求完整性、用户故事覆盖和页面交互设计方面表现良好，但在**业务指标的前端埋点设计、跨 EPIC 依赖的数据流闭环、关键边界场景的业务规则**三方面存在需要补齐的缺口。

**总分: 7.5/10**

---

## 优点

1. **用户旅程完整且角色覆盖全面** -- 覆盖了内容创作者、频道主、频道管理员、内容编辑、平台运营、普通成员 6 种角色，4 条用户旅程从发布到治理形成闭环，与后端 EPIC-22 的 11 个用户故事完全对齐。

2. **发布结果逐频道反馈设计精良** -- PublishResult 组件为每个目标频道独立展示状态（成功/待审/失败），失败项展示具体原因和重试操作。这直接支撑了后端 EPIC-22 的成功指标"频道发布成功率 >= 99%"的可度量性。

3. **待确认问题与默认假设分离** -- 第 12 节将 6 个待确认问题和 7 个默认假设分开列出，并标注了影响范围和建议负责人，便于决策跟踪。这是良好的需求管理实践。

4. **数据保留策略明确** -- 回收站 30 天保留、治理日志 180 天保留，在前端通过"剩余天数倒计时"和"已过保留期"置灰展示，业务规则与 UI 表现一致。

5. **组件层级关系图清晰** -- 第 4.2 节的组件树准确反映了页面-组件嵌套关系，便于开发理解职责边界。

## 问题与建议

### 🔴 P1 -- 发布权限模型的"可叠加"与 RadioGroup 矛盾

**位置**: 3.3 节 发布权限配置页

PRD 交互规则写"先审后发与其他模式可叠加（先审后发优先级最高）"，但线框图使用 RadioGroup（互斥选择）。后端 design.md 的 Decision 1 也明确 `publish_permission` 是枚举字段。

**业务影响**: 开发无法确定实现方式。如果"先审后发"是独立开关，则需要额外的 Toggle 组件；如果是 RadioGroup 中的一项，则"可叠加"表述需要删除。

**建议**: 明确为以下二选一方案：
- 方案 A: RadioGroup 四选一，"先审后发"作为独立选项，删除"可叠加"描述
- 方案 B: RadioGroup 三选一 + 独立的"先审后发"开关，可与任意模式叠加

### 🔴 P2 -- 成功指标缺少前端埋点设计

**位置**: 缺失章节

后端 EPIC-22 定义了 6 项成功指标（发布成功率、权限准确率、定时发布准时率、治理可追溯率、回收站恢复有效性、频道秩序改善），但前端 PRD 没有定义对应的埋点方案。

**业务影响**: 没有前端埋点，"发布成功率"无法区分前端校验失败和后端业务拒绝；"定时发布准时率"无法从前端视角度量用户看到结果的延迟。

**建议**: 新增"埋点与度量"章节，至少覆盖：
- 发布提交事件（频道数、成功/失败/待审数、失败原因分类）
- 审核操作事件（通过/拒绝、批量/单条、耗时）
- 治理操作事件（操作类型、是否二次确认、是否撤销）
- 页面性能事件（ChannelSelector 加载耗时、发布提交响应耗时）

### 🔴 P3 -- 跨 EPIC 依赖的数据流未闭环

**位置**: 全文

PRD 在范围定义中声明依赖 EPIC-20（频道基础架构）和 EPIC-21（隐私与成员管理），但未明确以下数据流：

1. **EPIC-20 频道状态变更** -- 频道被冻结/删除时，已设定的定时发布任务如何处理？前端需要展示什么状态？
2. **EPIC-21 成员角色变更** -- 用户从管理员降为普通成员后，其待审区的审核权限如何回收？已提交的待审内容是否受影响？
3. **EPIC-21 禁言/黑名单** -- 禁言用户发布时前端展示什么错误信息？黑名单用户能否看到频道内容？

**业务影响**: 开发阶段会频繁遇到"这个逻辑 EPIC-20/21 处理了吗"的阻塞问题。

**建议**: 新增"跨 EPIC 依赖矩阵"章节，明确每个依赖点的：
- 数据流向（谁提供、谁消费）
- 状态变更时的前端响应
- 兜底策略（依赖服务不可用时的降级方案）

### 🟡 P4 -- 多频道发布的业务规则不完整

**位置**: 3.1 节 ChannelSelector、故事 22.1.1

PRD 提到"已选频道数达到上限 N 时不可继续添加"，但以下业务规则未定义：

1. **同一内容在同一频道重复发布** -- 后端 EPIC-22 明确"同一内容不可在同一频道重复展示"，但前端 ChannelSelector 是否需要展示"已在此频道"的标识？
2. **频道数上限 N 的来源** -- 是系统配置还是用户等级决定？前端如何获取这个值？
3. **部分频道发布失败后的重试** -- 重试时是否允许更换目标频道？还是只能重试原来的失败频道？

**建议**: 补充以上业务规则的前端交互定义。

### 🟡 P5 -- 待审区超时机制的业务规则模糊

**位置**: 3.4 节 ReviewQueueTable

PRD 默认假设"超时阈值为 24 小时"，但以下问题未明确：

1. **超时后自动处理还是仅标识？** -- 后端 EPIC-22 未定义超时后的自动行为（自动通过？自动拒绝？仅通知？），前端仅做高亮标识还是需要触发提醒？
2. **超时提醒的目标对象** -- 仅提醒频道主？还是所有有审核权限的人？
3. **超时计数的统计口径** -- "超时数"是待审区 badge 的一部分，但统计逻辑未定义（按提交时间？按最后提醒时间？）

**建议**: 与产品/社区治理团队确认超时策略后，在前端 PRD 中补充超时相关的 UI 反馈（如 toast 通知、badge 数字更新频率）。

### 🟡 P6 -- 编辑协助的作者知情权设计不足

**位置**: 3.5 节 ContentGovernance、故事 22.2.4

PRD 默认假设"编辑协助修改即时生效，不需作者确认"，但未定义：

1. **作者通知机制** -- 管理员修改后，作者是否收到通知？通知内容是什么（仅告知"被修改"还是展示修改前后对比）？
2. **作者申诉入口** -- 如果作者不认可修改，是否有申诉或回退入口？
3. **修订历史的可见范围** -- 修订历史仅管理员可见还是作者也可查看？

**业务影响**: 这是创作者权益保护的核心问题。如果作者无法感知和申诉编辑协助，可能导致创作者流失。

**建议**: 至少定义作者通知的交互（如消息中心通知 + 修改前后 diff 展示），并明确修订历史对作者的可见性。

### 🟡 P7 -- 公告管理缺少版本和发布策略

**位置**: 3.8 节 AnnouncementManage

PRD 定义"一个频道同一时间最多一条主公告"且"历史公告不保留"，但存在以下业务风险：

1. **公告误删后无法恢复** -- 历史公告不保留意味着删除后需要完全重写，没有回收站机制
2. **公告编辑的冲突处理** -- 多个管理员同时编辑同一公告时如何处理？
3. **公告发布的生效延迟** -- 发布后是否立即在频道顶部展示？是否有 CDN 缓存导致的延迟？

**建议**: 考虑为公告增加简单的版本保留（如最近 3 个版本），至少与回收站的"可恢复"理念一致。

### 🟡 P8 -- 定时发布的业务边界未定义

**位置**: 故事 22.1.3、3.3 节

PRD 默认假设"定时发布时间范围为当前时间到 30 天内"，但以下场景未覆盖：

1. **定时发布期间用户被禁言** -- 到达发布时间时用户已被禁言，前端展示什么状态？是否通知用户？
2. **定时发布期间频道权限变更** -- 从"所有成员可发布"变为"仅管理员可发布"，普通成员的定时发布任务如何处理？
3. **定时发布的编辑/取消** -- PRD 有编辑 API 但缺少取消 API，前端也没有取消定时发布的交互设计
4. **多个定时发布任务的管理** -- 用户可能同时有多个定时发布任务，但 `useChannelPublishStore` 仅管理单次发布状态

**建议**: 补充定时发布列表页（或在发布页增加"我的定时发布"入口），并定义取消和异常处理的前端交互。

### 🟡 P9 -- 内容类型覆盖的业务规则不一致

**位置**: 故事 22.1.1、术语定义

后端 EPIC-22 明确定义覆盖"文章、图文帖子、视频、笔记、问答问题、问答答案"6 种内容类型，但前端 PRD 的故事描述中仅提到"文章、图文帖子、视频、笔记、问答"5 种（缺少问答答案）。

**业务影响**: 问答答案的发布流程可能与问答问题不同（如答案不需要选择频道，而是跟随问题所在的频道），需要明确。

**建议**: 确认问答答案是否需要独立的频道选择流程，并在前端 PRD 中补齐。

### 🟢 P10 -- 推荐频道的排序和展示逻辑未定义

**位置**: 3.1 节 ChannelSelector

ChannelSelector 的"推荐频道"分组未定义排序逻辑。是按活跃度？用户兴趣匹配？还是运营手动配置？

**业务影响**: 影响用户的选择效率和频道的流量分配公平性。

**建议**: 明确推荐频道的排序算法或配置来源，至少定义默认排序规则（如按频道活跃度降序）。

### 🟢 P11 -- 治理日志的操作类型枚举不完整

**位置**: 3.7 节 GovernanceLog

交互规则中列出的操作类型为"置顶/精华/删除/恢复/移出/编辑协助/公告变更"，但缺少：
- 发布权限模型变更
- 发布限额变更
- 待审区审核操作（通过/拒绝）

**建议**: 补充以上操作类型的日志记录，确保治理可追溯率指标的完整性。

### 🟢 P12 -- 批量操作的确认弹窗缺少影响说明

**位置**: 3.4 节、3.5 节

PRD 提到批量操作"展示确认弹窗（含影响说明）"，但未定义影响说明的具体内容。例如：
- 批量拒绝 10 条内容：是否展示"将向 10 位提交者发送拒绝通知"？
- 批量删除 20 条内容：是否展示"20 条内容将进入回收站，30 天内可恢复"？

**建议**: 为每个批量操作定义确认弹窗的具体文案模板。

## 业务建议

### 1. 建议新增"业务规则汇总"章节

当前业务规则分散在用户故事、页面设计、交互规则和待确认问题中，建议新增一个集中的业务规则汇总表，包含：
- 规则编号
- 规则描述
- 来源（EPIC-22/EPIC-21/默认假设）
- 前端表现
- 异常处理

### 2. 建议定义"状态机"文档

频道内容从发布到最终状态存在多个中间态：
- 发布中 → 已发布 / 待审核 / 发布失败
- 待审核 → 审核通过 / 审核拒绝 / 超时
- 已发布 → 置顶 / 精华 / 删除 / 移出
- 已删除 → 回收站 → 恢复 / 过期清除

建议用状态机图明确每个状态的转换条件和前端 UI 表现，避免开发遗漏边界转换。

### 3. 建议补充"通知与消息"章节

PRD 多处提到通知（审核结果通知、编辑协助通知、超时提醒），但未定义：
- 通知渠道（站内消息？推送？）
- 通知内容模板
- 通知的前端展示位置（消息中心？toast？badge？）

建议新增独立章节或明确复用 EPIC-06（隐私与通知）的通知能力。

### 4. 建议定义"降级与兜底"策略

以下场景需要前端降级策略：
- 后端 API 超时时的重试策略（发布提交是否自动重试？）
- WebSocket 断连时的降级方案（回退到轮询？）
- 依赖服务（EPIC-20/21）不可用时的前端表现

## 总结

EPIC-22 前端 PRD 在页面设计、组件选型和交互规则方面达到了可开发的水平，覆盖了频道内容发布与治理的核心场景。主要差距集中在三个维度：

1. **业务指标落地** -- 后端定义了 6 项成功指标，但前端缺少对应的埋点方案，指标无法度量
2. **跨 EPIC 数据流闭环** -- 与 EPIC-20/21 的依赖关系仅在范围声明中提及，未定义具体的数据流和异常处理
3. **边界场景的业务规则** -- 多频道发布重复检测、定时发布异常处理、编辑协助作者知情权等关键业务规则缺失

建议在进入开发前优先解决 P1-P3 问题（权限模型矛盾、埋点设计、依赖闭环），P4-P8 问题可在开发过程中同步补齐。
