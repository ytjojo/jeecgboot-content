# EPIC-05 降噪与内容保护 — 前端 PRD 审核报告

> **审核人**: 📚 Paige (Technical Writer)
> **审核日期**: 2026-06-02
> **审核视角**: 文档质量 — 清晰度、术语一致性、结构

## 总体评价

本文档结构清晰、覆盖全面，从概述到测试要点形成了完整的闭环。术语"拉黑"与"屏蔽"的核心区别在概述部分有明确定义，且在确认弹窗文案中反复强化，这是一份质量较高的前端 PRD。但文档在术语使用的一致性、部分细节描述的歧义性以及交叉引用的完整性方面仍有改进空间。

## 优点

- **核心概念定义清晰**：概述部分用一句话点明"屏蔽是单向降噪，拉黑是双向切断"，并在确认弹窗组件（3.6 节）中以具体文案强化这一区别，降低了开发和测试的歧义风险。
- **页面结构可视化良好**：3.1-3.5 节使用树形 ASCII 图描述页面层级，比纯文字描述更直观，便于前端开发快速理解组件嵌套关系。
- **交互流程完整**：7.1 节对拉黑、屏蔽、不感兴趣、解除拉黑四个核心流程均有分步描述，且包含了成功/失败分支，逻辑链条完整。
- **响应式设计有明确断点**：8.1 节定义了三个断点及对应设备类型，8.2 节逐一列出各页面在 PC 端和移动端的差异，前端开发可直接参照实现。
- **待确认问题显式化**：11 节将未决问题与默认假设以表格形式列出，避免了隐式假设导致的返工，这是非常专业的做法。
- **测试要点覆盖全面**：10 节从功能、边界、交互、权限四个维度列出了测试矩阵，可直接作为测试用例编写的基础。

## 问题与建议

1. **🟡中 — 术语不一致："屏蔽用户"与"屏蔽该用户动态"混用**
   - 3.5.1 用户主页入口使用"屏蔽该用户动态"，而 3.5.2 内容卡片入口和 3.5.3 评论区入口使用"屏蔽该用户"。两者操作的是同一个接口（`/api/content/user/mute`），但文案含义不同——"屏蔽动态"暗示仅屏蔽内容流，"屏蔽用户"暗示全面屏蔽。建议统一为一个表述，并在确认弹窗文案中补充说明屏蔽的实际范围。

2. **🟡中 — 3.3 节屏蔽列表管理页与 3.1 节隐私设置聚合页存在结构冲突**
   - 3.1 节隐私设置聚合页中，"屏蔽列表"、"屏蔽话题/类型"、"临时屏蔽"分别是三个独立的入口卡片，跳转到各自的管理页。但 3.3 节将这三者合并为一个带 Tab 切换的屏蔽列表管理页。文档未明确说明：最终采用聚合页三个独立入口 + 各自独立子页面的方案，还是聚合页一个入口 + Tab 切换页的方案？这两者在信息架构上有本质区别。

3. **🟡中 — "不感兴趣"流程中气泡选项的触发条件未定义**
   - 7.1 节不感兴趣流程描述：点击后"显示气泡选项"，但未说明气泡是立即出现还是在接口返回后出现。此外，气泡中的"屏蔽此类内容"和"屏蔽该话题"需要知道当前内容的类型和话题信息，文档未说明这些数据从哪个接口或字段获取。

4. **🟢低 — 第 4 节组件清单中部分组件缺少 Props 定义**
   - 4.1 节仅列出了组件名、路径、用途和父组件，但未定义关键 Props。例如 `BlockMuteMenu` 需要接收 `targetUserId` 和 `source`（来源页面类型），`FilteredContentCard` 需要接收内容数据和命中关键词。建议补充核心 Props 表格，减少开发过程中的理解偏差。

5. **🟢低 — API 接口清单缺少请求参数和响应字段定义**
   - 5.1 节接口清单仅列出方法、路径和用途，缺少请求体参数（如 `getBlacklist` 的分页参数结构）和响应字段（如黑名单列表每条记录包含哪些字段）。5.2 节的代码示例虽然有 TypeScript 类型，但覆盖不完整（如 `filterRule` 的 `ruleType` 枚举值未列出）。建议补充接口契约表或链接到后端接口文档。

6. **🟡中 — 7.3 节高风险操作分类存在自相矛盾**
   - 7.3 节将"屏蔽用户"归类为"以下操作无需确认弹窗（低风险、可逆）"，但括号内又注明"但需显示确认弹窗说明行为边界"。这与 3.6 节确认弹窗组件表中明确列出的"屏蔽用户"确认弹窗（标题"确认屏蔽"）相矛盾。实际上屏蔽用户是需要弹窗的，只是弹窗目的不是"确认"而是"说明边界"。建议将分类调整为三类：强制确认弹窗、说明性弹窗（不可跳过）、无需弹窗。

7. **🟢低 — 屏蔽词正则规则的长度限制仅在待确认问题中提及**
   - 11 节问题 4 提到正则最大长度 200 字符、单用户最多 20 条正则规则，但 3.4 节屏蔽词设置页的交互规则中只提到"屏蔽词数量上限 100 个"，未区分关键词和正则的各自上限。建议在 3.4 节中补充完整限制说明。

8. **🟢低 — Pinia Store 缓存策略缺少过期机制**
   - 6.3 节缓存策略提到"关系状态查询结果缓存在内存中，页面切换时保留"，但未定义缓存过期时间。如果用户在 A 页面拉黑了某人，然后切换到 B 页面查看同一用户，缓存是否仍为旧值？虽然 6.2 节有 `clearRelationCache` 方法，但未说明调用时机。

9. **🟢低 — 移动端"上拉加载更多"与 PC 端"分页"的交互细节不对称**
   - 8.2 节提到移动端使用"上拉加载更多"替代分页器，但 3.2 节黑名单管理页的页面结构中仅提到"分页（PC 端）/ 加载更多（移动端）"，未描述加载更多的具体交互（如 Loading 状态、无更多数据提示、网络错误重试等）。

10. **🟡中 — 被拉黑方看到的文案存在两种说法**
    - 3.7 节页面展示中写"被拉黑方看到的是空白页或'该用户不存在'"，而 11 节问题 1 的默认假设写"默认采用空白页面 + '该用户不存在'文案"。"空白页"和"'该用户不存在'文案"是两种不同的用户体验——前者是完全空白，后者是有文字提示。建议统一为一种明确描述。

## 文档改进建议

- **补充术语表**：在文档开头或附录中增加术语表，明确定义"拉黑"、"屏蔽"、"不感兴趣"、"屏蔽词"、"屏蔽规则"等核心概念及其英文对应（Block、Mute、Not Interested、Keyword Filter、Filter Rule），便于后续与后端文档和代码命名保持一致。
- **增加页面跳转流程图**：当前文档以独立页面描述为主，建议增加一张页面间的跳转关系图（如从隐私设置聚合页到各子页面、从操作入口到确认弹窗到结果页），帮助开发理解完整的用户动线。
- **统一交互规则描述格式**：部分页面用"交互规则"小节描述，部分用流程图描述（7.1 节），建议统一格式——页面级交互用"交互规则"小节，跨页面流程用流程图。
- **补充国际化/文案管理说明**：当前所有提示文案均硬编码在 PRD 中（如"已拉黑"、"操作失败，请重试"），建议说明是否需要国际化支持，以及文案是否抽取到语言包中管理。

## 总结

本文档作为前端 PRD 已具备较高的完成度，核心功能描述清晰、测试覆盖全面、待决问题显式化。主要改进方向集中在：术语使用的一致性（特别是"屏蔽"相关表述）、页面信息架构的明确化（3.1 与 3.3 的结构冲突）、以及接口契约的补充。建议在进入开发前优先解决第 2、6、10 项问题，其余可在开发过程中逐步完善。
