# EPIC-07 社交关系扩展 — 前端 PRD 审核报告

> **审核人**: 📋 John (Product Manager)
> **审核日期**: 2026-06-02
> **审核视角**: 产品管理 — 用户价值、优先级、业务对齐

## 总体评价

PRD 整体结构完整，涵盖了 6 个页面/组件、7 个用户故事、详细的 API 对接和交互设计。文档质量较高，状态处理、响应式设计、性能要求都有覆盖。但从产品管理角度看，存在几个关键问题：**用户价值假设缺少验证路径**、**功能范围过大导致优先级模糊**、**缺少核心用户旅程的完整闭环设计**。

## 优点

1. **目标量化清晰**：互关率提升 35%、邀请转化率 >15% 等指标明确，便于后续衡量
2. **边界定义明确**："非本期范围"列出了 IM、企业通讯录、实时画像、社交 SDK 对接，避免范围蔓延
3. **状态与边界处理完善**：每个页面都有空状态、加载中、网络错误等边界情况设计
4. **API 设计前置**：请求参数和响应格式已定义，减少前后端联调歧义
5. **性能指标具体**：互关标识 <50ms、页面首屏 <3s 等，可量化验收
6. **组件复用意识**：复用 `useUserStore`、`ContentUserVerificationBadge` 等现有资产

## 问题与建议

### 🔴 1. 粉丝画像功能的用户价值假设未验证，建议降级或延后

PRD 将粉丝画像（兴趣分布、地域分布、活跃时段）列为 P2，但包含了 3 个 ECharts 图表（饼图、柱状图、热力图）、CSV 导出、<100 粉丝的降级逻辑。这是一个**开发成本高但用户价值存疑**的功能：

- 创作者真的需要"地域分布"来优化内容方向吗？还是更需要"哪些内容带来最多粉丝"？
- 粉丝画像 <100 人不可用，意味着绝大多数中小创作者无法使用此功能
- T+1 离线计算意味着数据延迟，降低实时决策价值

**建议**：将粉丝画像拆为独立 Story，MVP 阶段仅做粉丝列表 + 趋势图表，画像功能在验证创作者有真实需求后再迭代。或者先用简化版（仅展示粉丝总数趋势 + 新增来源），验证 PMF 后再做完整画像。

### 🔴 2. 邀请功能缺少完整的转化漏斗设计

PRD 定义了邀请码生成、分享链接、邀请记录，但**缺少邀请链路的关键环节**：

- 被邀请者点击链接后的注册/登录流程是什么？是否自动关联邀请人？
- 邀请奖励的触发条件不明确："成功注册数"还是"注册 + 完成首次互动"？
- 邀请链接的落地页设计未提及 — 用户看到什么？如何引导注册？
- 没有邀请失效机制（过期、次数上限、防刷策略）

邀请转化率 >15% 是核心指标，但当前 PRD 只覆盖了"发起邀请"这一端，**"接收邀请 → 注册 → 激活"的完整链路缺失**。

**建议**：补充被邀请者的完整用户旅程，定义"成功邀请"的业务规则（注册即算？还是需要完成特定动作？），明确邀请链接落地页设计。

### 🔴 3. 版主/管理员治理功能与 EPIC-09 存在职责重叠

PRD 中 US-7.4.2 明确依赖 EPIC-09（用户状态与治理），但本 PRD 又完整定义了封禁、禁言、撤销等治理操作的前端页面和 API。这带来两个风险：

- **功能重复开发**：EPIC-09 如果也包含用户管理页面，两处会打架
- **用户体验割裂**：管理员需要在"评论区操作"和"用户管理页"两个地方执行治理操作

**建议**：明确治理操作的唯一入口设计。评论区仅保留"快捷操作"（删评论、警告），封禁/禁言/撤销统一在 EPIC-09 的用户管理页中操作，本 PRD 只负责评论区内的轻量管理能力。

### 🟡 4. "仅互关可见"私密内容的用户旅程不完整

PRD 定义了发布流程和展示逻辑，但**缺少关键场景**：

- 用户主页上如何区分公开内容和私密内容？是否有 Tab 切换？
- 非互关用户访问包含私密内容的用户主页时，看到什么？（完全隐藏？还是显示模糊占位？）
- 取消互关后，之前看到的私密内容是否从浏览历史中消失？
- 私密内容是否支持转发、收藏？如果支持，转发后互关关系变化怎么办？

**建议**：补充私密内容在用户主页、feed 流、搜索结果中的展示策略，以及取关后的内容可见性变更逻辑。

### 🟡 5. 互关标识的性能假设过于乐观

PRD 假设"后端在评论列表接口中已附带 `mutualFollow` 字段"，但同时又定义了批量查询接口。这两个设计**存在矛盾**：

- 如果评论接口已附带互关字段，为什么还需要 `mutual-status` 批量查询接口？
- 如果是单独批量查询，100 条评论的额外请求是否会拖慢评论加载？

**建议**：明确互关数据的获取策略 — 是评论接口自带字段（最优），还是独立批量查询（需评估延迟），并在 API 对接章节统一口径。

### 🟡 6. 缺少数据驱动的功能迭代机制

PRD 定义了最终指标（互关率 35%、邀请转化率 >15%），但**没有定义如何衡量和迭代**：

- 互关率如何统计？对比基线是什么？
- 哪些埋点事件需要在本期实现？（互关标识展示、邀请链接点击、角色标签 hover 等）
- 数据看板由谁查看？频率如何？

**建议**：增加埋点需求章节，定义核心事件和转化漏斗，确保上线后能基于数据做迭代决策。

### 🟡 7. 社区角色体系缺少演进路径

当前角色定义为 4 级（普通用户、创作者、版主、管理员），但：

- 创作者如何获得认证？是否有申请流程？前端是否需要创作者认证申请页？
- 版主如何任命？PRD 待确认问题 #2 提到了这个问题，但没有给出默认方案
- 是否需要支持自定义角色（如"资深创作者"、"话题主持人"）？

**建议**：明确创作者认证和版主任命的产品规则，即使本期不实现前端页面，也需要在 PRD 中定义业务规则，否则开发时无法确定权限判断逻辑。

### 🟢 8. 审计日志的数据粒度和保留策略未定义

审计日志页设计了操作人、操作时间、操作类型、目标用户、原因、IP 地址等字段，但：

- 日志保留多久？30 天？90 天？永久？
- 是否支持导出？
- IP 地址是否需要脱敏处理（合规要求）？

**建议**：明确日志保留策略和数据敏感度处理规则。

## 缺失用户旅程

1. **邀请完整链路**：被邀请者点击链接 → 落地页 → 注册 → 关联邀请人 → 触发奖励 → 邀请人收到通知。当前仅覆盖"发起邀请"端。

2. **创作者成长路径**：普通用户 → 发布优质内容 → 积累粉丝 → 申请创作者认证 → 获得创作者标签 → 使用粉丝管理工具。缺少认证申请环节。

3. **社区治理闭环**：用户被警告 → 收到通知 → 查看处罚记录 → 申诉（如有）→ 管理员审核申诉。当前只有处罚执行，没有被处罚者的感知和申诉路径。

4. **互关发现路径**：用户 A 想找到互关好友 → 如何发现？除了互关列表页，是否有推荐机制？feed 流中是否优先展示互关内容？

5. **私密内容消费路径**：用户在 feed 流中看到"仅互关可见"的标记 → 点击 → 发现需要互关才能查看 → 如何引导用户去关注/建立互关？

## 优先级建议

| 功能 | 当前优先级 | 建议优先级 | 理由 |
|------|-----------|-----------|------|
| 互关标识展示 | P0 | P0（保持） | 核心社交体验，开发成本低，ROI 高 |
| 互关好友列表 | P0（隐含） | P1 | 互关标识的配套功能，但非核心路径 |
| 仅互关可见内容 | P1 | P2 | 用户旅程不完整，需先补充设计再开发 |
| 粉丝列表 + 搜索 | P1 | P1（保持） | 基础粉丝管理能力 |
| 粉丝趋势图表 | P1 | P1（保持） | 创作者核心需求，有数据价值 |
| 粉丝画像 | P2 | P3 | 高开发成本，低用户基数（<100粉不可用），建议延后 |
| 邀请码 + 分享 | P1 | P1 | 有价值但需补充完整链路设计 |
| 社区角色标签 | P1 | P1（保持） | 开发成本低，治理透明度提升明显 |
| 版主管理操作 | P1 | P1 | 需与 EPIC-09 对齐职责边界 |
| 审计日志 | P1 | P2 | 管理员低频使用，可延后 |

## 总结

EPIC-07 的 PRD 在页面级设计上做得扎实，组件选型、API 对接、交互细节都有覆盖。但从产品视角看，**功能范围偏大、用户价值假设偏多、核心链路不完整**。

**三个优先改进项**：

1. **砍范围**：粉丝画像降级延后，本期聚焦互关标识 + 粉丝列表/趋势 + 邀请 + 角色标签
2. **补链路**：邀请功能补充被邀请者完整旅程，私密内容补充消费端场景
3. **对齐边界**：与 EPIC-09 明确治理功能的职责划分，避免重复开发

建议在以上问题解决后再进入开发阶段，避免边开发边改需求导致的返工。
