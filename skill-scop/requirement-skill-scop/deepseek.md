# 互联网/软件产品PRD完整框架与质量检查清单

## 一、文档必备核心模块与子要素

### 1. 文档基础信息与版本控制
- 文档编号、产品名称、版本号
- 创建/修订日期、责任人
- 版本历史记录（版本号/日期/作者/变更内容/审核人）
- 评审记录表（评审日期/参与人/意见/结论）

### 2. 项目背景与目标
- **背景**：业务痛点、市场机会、用户诉求、数据驱动的洞察
- **目标**：可量化的业务目标（如转化率提升X%）、产品目标（支持XX场景）
- **范围界定**：明确本次包含/不包含的功能模块
- **成功指标**：关键绩效指标（KPI）、北极星指标、辅助衡量指标

### 3. 用户画像与场景
- **用户角色矩阵**：角色名称、权限等级、核心诉求、使用频率
- **用户画像卡片**：人口属性、行为特征、技能水平、设备偏好
- **使用场景**：时间/地点/触发条件/任务目标/情绪状态
- **用户旅程映射**：关键触点、痛点、机会点

### 4. 业务流程图
- 主业务流程泳道图（跨角色动作+系统响应+异常分支）
- 状态机图（核心实体生命周期）
- 数据流图（输入/输出/存储/流转路径）
- 异常处理流程（回滚、补偿、告警机制）

### 5. 功能需求详述（核心）
每个功能点需包含：
- 功能ID与优先级（MoSCoW法则：Must/Should/Could/Won’t）
- 用户故事（作为XX，我希望XX，以便XX）
- 前置/后置条件
- 交互流程（正常路径 + 异常路径）
- 业务规则与约束（计算公式、校验逻辑、权限控制）
- 界面文案与提示语
- **边界条件**：极限值、空状态、首屏加载、网络切换

### 6. 页面原型说明
- 页面层级结构（站点地图）
- 页面元素标注（控件类型、状态、走查规则）
- 交互逻辑（跳转、弹窗、滑动、手势、动效）
- 适配规则（响应式断点、不同分辨率、横竖屏）
- 无障碍访问要求（可选）

### 7. 非功能需求
- **性能**：响应时间（P95/P99）、吞吐量、并发用户数
- **可用性**：SLA（服务等级协议）、故障恢复时间（RTO/RPO）
- **安全性**：数据加密、权限体系、防篡改/防注入、合规要求（GDPR/PIPL/等保）
- **可扩展性**：水平扩展方案、插件化设计
- **兼容性**：浏览器/OS版本、第三方服务接口版本
- **可观测性**：日志规范、告警阈值、链路追踪要求

### 8. 数据埋点需求
- 埋点事件字典（事件ID/触发时机/参数列表/参数类型/取值示例）
- 页面浏览（PV/UV）与元素曝光/点击
- 转化漏斗节点定义
- 用户行为序列留存需求
- 埋点触发时机与去重规则

### 9. 验收标准
- **功能验收**：基于每个功能的Given-When-Then测试用例
- **体验验收**：用户操作步数、出错率、满意度基线
- **数据验收**：埋点上报准确率（≥99.5%）、数据一致性
- **性能验收**：压测通过条件下的基线数值
- **安全验收**：渗透测试、权限越权测试

### 10. 风险与依赖
- **技术风险**：技术可行性、遗留系统债、第三方服务稳定性
- **业务风险**：政策变化、用户接受度、反作弊
- **资源依赖**：人力排期、设计/前端/后端资源、外部API授权
- **缓解措施**：预案、降级方案、回滚计划
- **假设与约束**：对未知事项的预判、强制约束条件

### 11. 附录
- 术语表
- 参考资料（竞品分析、用户调研结论、数据报表）
- 变更请求流程说明

## 二、模块编写规范与质量评判标准

| 模块 | 编写规范 | 质量标准（符合性检查） |
|------|----------|------------------------|
| **功能需求** | 使用“应/必须/应当”等规范性措辞；每条需求独立且原子化 | ✅ **可测试性**：可转化为3-5条GWT测试用例<br>✅ **无歧义**：无“尽可能”“良好体验”等模糊词，数值量化<br>✅ **完整性**：异常+边界+正常路径均覆盖 |
| **用户画像** | 基于真实数据或已验证假设；每个角色至少3个具体特征 | ✅ **可验证性**：可通过访谈或问卷验证<br>✅ **可区分性**：角色间无重叠且覆盖主流场景 |
| **业务流程图** | 使用BPMN 2.0或UML标准符号；标注所有泳道和网关 | ✅ **无分支遗漏**：每个判定节点有标注所有可能出口<br>✅ **闭环性**：无“悬空”连线，终止节点明确 |
| **非功能需求** | 采用SMART原则（具体、可测量、可达成、相关、有时限） | ✅ **可度量**：配置监控工具即可验证<br>✅ **可追溯**：回连到业务目标（如“首页加载≤1.5s支撑跳出率降低”） |
| **验收标准** | GWT格式（Given/When/Then）；包含正向和异常用例 | ✅ **二义性自由**：同一用例不同人执行结论一致<br>✅ **原子性**：每条验收项对应唯一功能点 |
| **数据埋点** | 明确事件触发时机、上报条件、参数数据类型 | ✅ **可复现性**：相同操作在不同环境产生相同埋点日志<br>✅ **完备性**：覆盖所有需要分析的转化节点 |
| **风险与依赖** | 每条风险标注发生概率、影响等级、责任人 | ✅ **可缓解**：有明确预案而非仅“关注”<br>✅ **时效性**：标注监测触发条件与应对执行窗口 |

**通用质量标准**：
- **可追溯性**：每条需求可反向链接到业务目标或用户场景（建立需求追踪矩阵）
- **一致性**：全文术语统一（同一实体在不同章节同义同词）
- **优先级清晰**：区分Must/Should/Could/Won’t，无歧义冲突

## 三、常见遗漏项与低质文档规避要点

### 常见遗漏项（高频且致命）

| 遗漏类型 | 典型表现 | 后果 | 规避方法 |
|----------|----------|------|----------|
| **边界条件** | 仅描述正常流程，无空列表、长文本、网络异常 | 上线后崩溃或白屏 | 对每个输入字段/列表组件强制附加边界走查清单 |
| **权限与数据范围** | 仅写“管理员可操作”，未区分租户、组织、数据行级权限 | 越权漏洞 | 绘制权限矩阵表（角色×数据对象×操作类型） |
| **异步与回调** | 支付、导入等长耗时操作未写轮询/超时/重试策略 | 用户重复操作或数据不一致 | 强制对每个异步任务设计状态机与超时补偿 |
| **降级与限流** | 无高峰期行为说明 | 系统雪崩 | 在非功能需求中增加熔断/降级/限流阈值 |
| **国际化与本地化** | 默认所有文案单语言、单日期格式 | 海外用户困惑 | 标注哪些字段需支持多语言/时区/货币转换 |
| **向后兼容** | 未说明API版本策略或数据结构变更时的迁移方案 | 老客户端崩溃 | 增加“兼容性条款”章节，明确版本号与不兼容行为 |
| **删除行为** | “删除”定义模糊（物理/逻辑/软删除） | 数据恢复灾难 | 统一删除类型定义，并说明回收站与清理策略 |

### 低质文档典型特征与规避

| 低质特征 | 示例 | 改进方法 | 检查动作 |
|----------|------|----------|----------|
| **歧义形容词** | “页面加载不宜过长”“用户友好地提示” | 量化：≤1.5秒；提示文案见UI规范第3条 | 全文搜索“较好/合理/适当/高效”，逐条量化 |
| **需求遗漏分支** | “点击提交按钮，若成功则跳转” | 补充：失败时是否保留已填数据？重复提交如何防？断网提交提示？ | 对每个操作强制列出3类路径（成功/失败/异常中断） |
| **场景缺失** | 仅写PC Web端，未说明移动端适配 | 增加响应式断点或明确不支持移动端 | 在页面原型说明中声明适配范围 |
| **无优先级** | 所有功能标注为P0 | 使用MoSCoW方法，并注明P0以外的功能必须给出不含其的MVP定义 | 发布前核对开发排期是否匹配优先级 |
| **原型与需求脱节** | 原型无标注，需求文档单独描述 | 原型上直接标注控件ID，并与需求章节ID交叉引用 | 检查每个原型元素是否在功能需求中有对应行为说明 |
| **忽略监控告警** | 无埋点、日志、告警阈值 | 新增“可观测性”子章节，包含关键操作日志级别与告警条件 | 检查是否有指标用于上线后判断功能“是否生效” |
| **验收缺失反向用例** | 仅写“正确输入可登录” | 补充：密码错误/账号锁定/网络超时场景的预期行为 | 对每个功能写不少于2条反向验收用例 |

### 强制落地建议（检查清单）

**PRD发布前自检清单（15项）**：
1. [ ] 每条功能需求是否唯一ID并关联到业务目标？
2. [ ] 每个用户操作是否有3类路径（正常/失败/异常中断）？
3. [ ] 流程图每个判定节点是否标注所有分支走向？
4. [ ] 列表/分页/输入框是否写明空状态与极限值？
5. [ ] 异步操作是否写明超时值、重试次数、失败提示？
6. [ ] 权限需求是否包含角色-数据范围矩阵？
7. [ ] 非功能需求是否包含具体可测量数值？
8. [ ] 删除操作是否注明物理/逻辑及恢复机制？
9. [ ] 是否包含降级/限流策略（如为大促/突增高并发）？
10. [ ] 数据埋点是否覆盖所有关键转化节点？
11. [ ] 验收标准是否每条都可用GWT重写？
12. [ ] 是否标注了本版本明确不做（Out of Scope）的内容？
13. [ ] 是否有关键术语表（防止不同角色理解偏差）？
14. [ ] 风险依赖是否有负责人与具体缓解动作？
15. [ ] 原型是否与需求章节有交叉引用？

**最佳实践附加建议**：
- 使用PRD模板库（避免每次从零构建遗忘章节）
- 评审前邀请测试工程师参与“可测试性预审”
- 对每个易错模块（如边界条件）建立组件级检查规则库
- PRD与交互原型、数据字典三文档同步锁版

以上框架可直接导入Jira/Confluence/PingCode等工具中的PRD模板，也适用于敏捷开发中的“史诗-特性-用户故事”拆分前的高阶需求定义。
