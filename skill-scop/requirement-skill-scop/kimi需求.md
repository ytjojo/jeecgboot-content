互联网/软件产品 PRD 高质量写作指南与检查清单

---

一、文档必备核心模块与子要素

1. 文档元信息与版本控制

子要素	说明	
文档标识	PRD 编号、产品名称、模块名称、关联项目/迭代编号	
版本记录	版本号、修订日期、修订人、修订内容摘要、审批人	
阅读对象	明确目标读者（研发、设计、测试、运营、管理层）	
文档状态	Draft / Review / Approved / Deprecated	
关联文档	产品规划书、竞品分析报告、技术方案、UI 设计稿链接	

2. 项目背景与目标

子要素	说明	
背景陈述	市场机会、用户痛点、业务诉求、数据支撑（现状问题量化）	
项目目标	业务目标（收入、留存、转化、效率提升）+ 产品目标（功能覆盖、体验优化）	
成功指标	北极星指标 + 辅助指标，明确基准值与目标值（如：支付转化率从 3% 提升至 5%）	
范围边界	明确本次迭代包含与不包含的内容，防止范围蔓延	
术语表	专有名词、业务缩写、关键概念定义	

3. 用户画像与场景

子要素	说明	
目标用户画像	用户角色（Persona）、 demographics、行为特征、技术熟练度	
用户分层	新用户/老用户、免费/付费、B 端/C 端、不同权限角色	
场景故事	用户故事（User Story）格式：「作为…，我希望…，以便…」	
场景地图	用户旅程（User Journey）关键触点、情绪曲线、痛点与机会点	
场景优先级	MoSCoW 法则（Must have / Should have / Could have / Won't have）	

4. 功能需求详述

子要素	说明	
功能清单	功能模块 → 功能点 → 子功能三级拆解，唯一编号（如 F-001）	
功能描述	每个功能点的输入、处理逻辑、输出、业务规则、异常分支	
前置条件	触发该功能的前提（登录状态、权限、业务状态）	
后置条件	功能执行后的系统状态变更、数据影响	
权限矩阵	功能 × 角色矩阵，明确谁可见、谁可操作	
状态机/规则引擎	复杂业务的状态流转、规则配置、互斥逻辑	
异常处理	错误码定义、降级策略、超时处理、空状态、极限情况	
第三方集成	接口调用、回调机制、容错逻辑、幂等性要求	

5. 非功能需求（NFR）

子要素	说明	
性能需求	响应时间（TP50/TP90/TP99）、并发量、吞吐量、资源占用阈值	
可用性/可靠性	系统可用性 SLA（如 99.9%）、故障恢复时间 RTO/RPO、降级预案	
安全需求	数据加密（传输/存储）、脱敏规则、权限校验、防刷防爬、合规（GDPR/个保法）	
兼容性	浏览器、操作系统、设备型号、分辨率、APP 版本兼容范围	
扩展性	未来容量预估、模块化设计、配置化能力	
可维护性	日志规范、监控告警、配置管理、热更新能力	
国际化	多语言、多时区、多币种、RTL 布局、本地合规	

6. 业务流程与信息架构

子要素	说明	
业务流程图	跨系统/跨角色的端到端流程（BPMN/泳道图），含决策节点	
系统交互图	时序图（Sequence Diagram）展示系统间调用关系	
信息架构图	站点地图、导航结构、内容分类与层级	
数据流图	数据从产生到消费的全链路流转、存储、加工逻辑	

7. 页面原型与交互说明

子要素	说明	
页面清单	页面名称、页面目的、入口来源、出口去向	
原型标注	低保真/高保真原型链接，元素级标注（按钮、字段、组件）	
交互规则	点击/滑动/长按/拖拽的响应逻辑、动效参数、过渡方式	
数据规则	字段来源、计算逻辑、默认值、格式化规则、校验规则	
边界状态	加载中（Skeleton/Loading）、空数据、网络异常、权限不足、内容溢出	
响应式规则	不同屏幕尺寸下的布局适配、断点定义	
文案规范	按钮文案、提示语、错误信息、空状态文案（语气、字数限制）	

8. 数据埋点需求

子要素	说明	
埋点清单	事件名称、事件 ID、触发时机、上报时机（实时/批量）	
事件属性	公共属性（用户 ID、时间、设备）+ 自定义属性（业务参数）	
页面浏览	PV/UV、停留时长、跳出率、页面来源与去向	
行为事件	点击、曝光、转化漏斗关键步骤、分享、搜索	
业务指标	订单量、GMV、客单价、转化率等可直接计算的业务指标	
埋点验证	测试环境验证方案、数据校验规则、灰度观察期	

9. 验收标准（Acceptance Criteria）

子要素	说明	
功能验收	每个功能点的 Given-When-Then 场景化验收条件	
场景覆盖	正常流程 + 异常流程 + 边界条件 + 并发场景	
数据验收	数据准确性、一致性、完整性、时效性验证标准	
体验验收	视觉还原度、交互流畅度、性能达标、无障碍支持	
兼容性验收	约定设备/浏览器矩阵下的测试通过标准	
UAT 标准	用户验收测试通过条件、参与方、通过准则	

10. 风险与依赖管理

子要素	说明	
风险清单	风险描述、影响程度、发生概率、应对措施、责任人	
外部依赖	第三方接口、法务合规、牌照资质、外部团队排期	
技术依赖	架构改造、中间件升级、数据库迁移、技术债影响	
资源依赖	设计资源、测试资源、运营资源、预算限制	
上线依赖	内容准备、账号配置、运营 SOP、客服培训	

11. 发布与上线计划

子要素	说明	
发布策略	灰度发布比例、AB 实验分组、回滚机制、Feature Toggle	
上线 checklist	配置检查、权限开通、数据初始化、监控告警配置	
运营配合	上线公告、用户引导、帮助文档、客服话术	
回滚预案	触发条件、回滚步骤、数据修复方案、影响范围评估

二、各模块编写规范与质量评判标准

通用质量原则（SMART-TEST 原则）

维度	标准	评判要点	
Specific 具体性	需求描述精确到可执行	无模糊词汇（如「可能」「大概」「优化」），量化指标明确	
Measurable 可度量	目标与验收可量化验证	每个需求有明确的验收条件，可通过测试或数据验证	
Achievable 可实现	技术可行、资源可支撑	已与研发团队预沟通，无已知技术瓶颈	
Relevant 相关性	与业务目标强关联	每个功能点可追溯到项目目标，无镀金需求	
Traceable 可追溯	需求全链路可追踪	需求编号唯一，可关联到设计稿、代码、测试用例、埋点	
Testable 可测试	验收条件可转化为测试用例	测试团队可基于 PRD 直接编写用例，无需额外假设	
Unambiguous 无歧义	仅有一种合理解读	多人独立阅读后对需求的理解一致	
Complete 完整性	覆盖正常/异常/边界	不遗漏分支流程、错误处理、空状态	

分模块编写规范

模块 1：项目背景与目标
- 规范：背景用数据说话（现状痛点量化），目标遵循「指标 = 基准值 + 目标值 + 时间窗口」公式
- 质量标准：研发阅读后能理解「为什么做」和「做到什么程度算成功」

模块 2：用户画像与场景
- 规范：用户故事按「角色-场景-价值」三段式书写；场景覆盖主流程与替代流程
- 质量标准：设计师能据此产出交互方案，测试能识别场景边界

模块 3：功能需求详述
- 规范：采用「功能编号 + 功能名称 + 需求描述 + 业务规则 + 前置/后置条件 + 验收标准」六段式模板
- 质量标准：开发无需口头确认即可开始编码；异常分支覆盖率 ≥ 80%

模块 4：非功能需求
- 规范：性能指标明确场景（如「首页首屏加载时间 ≤ 1.5s，4G 网络下」）；安全需求引用具体合规条款
- 质量标准：运维可据此制定 SLA；安全团队可据此做风险评估

模块 5：业务流程图
- 规范：统一图例标准（开始/结束、流程、判定、数据、子流程）；泳道图按角色/系统划分
- 质量标准：流程图与文字描述无冲突；每个判定节点有 Yes/No 分支

模块 6：页面原型说明
- 规范：原型与需求描述分离但交叉引用（如「详见原型 P3-订单确认页」）；字段说明用表格（字段名、类型、必填、校验、来源）
- 质量标准：视觉设计师可直接基于原型出 UI；前端可基于标注还原页面

模块 7：数据埋点
- 规范：采用事件-属性模型；事件命名遵循「动作_对象_位置」规范（如 `click_submit_order_detail`）
- 质量标准：数据分析师可据此直接配置埋点；每个业务目标至少有一个埋点支撑

模块 8：验收标准
- 规范：使用 Given-When-Then 格式；每条验收标准独立可验证
- 质量标准：测试用例与验收标准一一对应；无「用户体验良好」等主观描述

三、常见遗漏项与低质文档规避要点

🔴 高频遗漏项清单

遗漏类别	具体表现	规避方法	
异常流程缺失	只写正常流程，忽略网络中断、超时、权限变更、并发操作	强制要求每个功能点补充「异常矩阵」（错误类型 × 系统响应 × 用户提示）	
空状态遗漏	未定义无数据、搜索无结果、列表为空的展示逻辑	建立「空状态 checklist」，每个列表/详情页强制检查	
数据一致性	多端数据同步规则、缓存刷新机制、脏数据处理未说明	补充数据状态机图，明确最终一致性策略	
权限边界模糊	未区分「不可见」与「可见但不可操作」，未处理权限动态变更	绘制「角色-功能-数据」三维权限矩阵	
性能基线缺失	只说「要快」，未定义具体指标与测试环境	引用历史基线或竞品基准，明确测试条件	
埋点遗漏	功能上线后无数据可衡量效果	需求评审时同步评审埋点文档，作为上线 gate	
回滚/降级方案	只考虑上线成功场景	强制要求每个迭代写「回滚触发条件与操作步骤」	
文案与多语言	使用「待定」占位，导致开发用临时文案上线	建立文案 review 机制，PRD 中文案必须为最终版或标注 TBD 责任人	
第三方依赖	未评估第三方服务故障时的兜底策略	明确「第三方不可用」时的降级逻辑与用户体验	
运营后向配置	未考虑上线后运营人员如何配置、修改、监控	补充运营后台需求、配置项说明、操作权限	

🔴 低质文档典型特征与规避

低质特征	危害	规避要点	
需求堆砌，无优先级	研发无法识别核心路径，资源分散	强制使用 MoSCoW 标注每个需求点；Must have 不超过 60%	
口语化描述	「做个类似淘宝的功能」导致理解偏差	禁止引用竞品替代需求描述；必须拆解为具体功能点	
原型替代 PRD	只有截图无逻辑说明，无法追溯变更	原型是「附件」，PRD 是「主文档」；变更必须在 PRD 中记录	
技术方案侵入	PRD 中写死技术实现（如「用 Redis 缓存」）	PRD 聚焦「做什么」和「验收标准」，「怎么做」由技术方案决定	
无版本控制	需求频繁变更，各方持有不同版本	使用统一文档工具（Confluence/飞书/石墨），强制留痕变更	
忽视非功能需求	上线后性能崩溃、安全漏洞、无法扩展	将 NFR 作为评审必过项，单独章节不可删减	
验收标准主观化	「体验流畅」「界面美观」无法验证	验收标准必须包含可观测、可量化的指标或明确的行为描述	
忽略数据影响	功能变更导致历史数据不兼容	新增「数据影响分析」章节：存量数据迁移、兼容性、清洗规则	

✅ PRD 评审自检清单（Checklist）

结构完整性
- 包含全部 11 个核心模块（元信息、背景目标、用户场景、功能需求、非功能需求、流程图、原型交互、埋点、验收、风险依赖、发布计划）
- 每个功能点有唯一编号，可追溯到目标与验收标准
- 术语表已定义，全文术语一致

质量达标性
- 所有量化指标符合 SMART 原则
- 每个功能点至少包含 1 条正常流程 + 1 条异常流程的验收标准
- 无「可能」「大概」「优化」等模糊词汇
- 非功能需求章节非空，且包含性能、安全、兼容性至少三类
- 埋点清单覆盖所有业务目标指标

可落地性
- 研发、测试、设计阅读后无需口头确认即可开工
- 原型链接有效，标注覆盖率 ≥ 90%
- 风险清单有明确责任人与应对计划
- 上线 checklist 已准备，回滚方案已确认

四、快速应用模板（一页纸速查）

```
【PRD 标题】产品名_模块名_版本号

1. 背景与目标
   - 为什么做：（数据支撑的痛点）
   - 做到什么程度：（指标 = 基准 → 目标，时间窗口）
   - 范围边界：（In Scope / Out of Scope）

2. 用户与场景
   - 核心用户：（角色 + 特征）
   - 核心场景：（User Story + 优先级）

3. 功能需求（编号：F-001 ~ F-XXX）
   ├─ 功能模块
   │   ├─ 功能点：描述 + 业务规则 + 前置/后置条件
   │   ├─ 页面/原型链接
   │   ├─ 异常处理
   │   └─ 验收标准（Given-When-Then）

4. 非功能需求
   - 性能：（具体指标 + 测试条件）
   - 安全：（合规要求 + 脱敏/加密规则）
   - 兼容：（设备/浏览器矩阵）

5. 数据埋点
   - 事件 ID | 事件名 | 触发时机 | 属性 | 业务目标

6. 风险与依赖
   - 风险 | 影响 | 概率 | 应对 | 责任人
   - 外部依赖 | 状态 | 阻塞影响 | 跟进人

7. 发布计划
   - 灰度策略 / 上线时间 / 回滚条件 / 运营配合

附录：术语表 | 变更记录 | 参考文档
```
