
# SDD 审核 Checklist 系统设计文档评审检查清单（前端 / 后端 / 通用）

| BLOCKER — 必须修复后方可开发 | MAJOR — 里程碑前关闭 | MINOR — 迭代内处理 | □ 未检查   ✓ 通过   ✗ 不通过（需创建 Issue） |

## 一、通用维度（前端 + 后端共用）

### 1.1 完整性 Completeness

| □ | 检查项 | 优先级 | 说明/示例 |
| :--- | :--- | :--- | :--- |
| □ | 文档包含所有必填章节：概述、背景、设计目标、约束、系统架构、接口/数据设计、非功能需求、风险与依赖 | BLOCKER | 缺少任意章节即 Block |
| □ | 每条 RAW 需求（原始需求）都能在 SDD 中找到对应的设计描述（可追溯性矩阵存在） | BLOCKER | 需求覆盖率 = 100% |
| □ | 所有业务主流程（Happy Path）均有对应的流程图或时序图描述 | BLOCKER | 缺失核心流程即 Block |
| □ | 异常/错误路径已列出：网络超时、权限不足、参数非法、第三方服务故障 | MAJOR | 至少覆盖主流程的异常 |
| □ | 边界条件已明确量化：最大值、最小值、空集合、并发场景 | MAJOR | 如：最大分页 size=100 |
| □ | 数据迁移/兼容性方案已描述（如涉及） | MAJOR | 新旧数据如何共存 |
| □ | 文档版本号、作者、最后更新时间已填写 | MINOR | v1.0 基线须有版本号 |

### 1.2 一致性 Consistency

| □ | 检查项 | 优先级 | 说明/示例 |
| :--- | :--- | :--- | :--- |

| □ | SDD 各章节内部自洽，无矛盾描述（如时序图与接口描述一致） | BLOCKER | 常见冲突点：流程图 vs 状态机 |
| □ | 接口命名风格统一（RESTful 风格、snake_case/camelCase 一致） | MAJOR | 如：user_id vs userId 混用 |
| □ | 错误码体系全局统一，无不同模块自定义重复错误码 | MAJOR | 统一错误码注册表 |
| □ | 与上游/下游服务的接口约定版本一致，无独自升级导致不兼容 | MAJOR | 检查所有外部依赖版本 |
| □ | 图表与文字描述同步更新，无过期示意图 | MINOR | 图表日期与文档日期对应 |

### 1.3 可行性 Feasibility

| □ | 检查项 | 优先级 | 说明/示例 |
| :--- | :--- | :--- | :--- |

| □ | 开发工作量估算有依据，不存在明显低估（关键路径已识别） | MAJOR | 对比历史类似功能 |
| □ | 所有外部依赖（第三方服务/SDK/库）已明确版本，SLA 已确认 | MAJOR | 依赖不可用时的降级方案 |
| □ | 已知风险已列出，每个风险有缓解措施和负责人 | MAJOR | 风险矩阵（概率×影响） |
| □ | 方案未引入团队当前技术栈范围外的高风险技术（如有需额外评审） | MINOR | 学习曲线风险 |

### 1.4 可测试性 Testability

| □ | 检查项 | 优先级 | 说明/示例 |

| □ | 安全测试要求已列出（如需渗透测试、OWASP 检查项） | MINOR | 按项目合规要求 |

## 二、后端专项维度

### 2.1 接口规范性 API Specification

| □ | 检查项 | 优先级 | 说明/示例 |
| :--- | :--- | :--- | :--- |
| □ | 所有接口都有 OpenAPI 3.x / Swagger 定义（或等效的接口描述文档） | BLOCKER | 是前后端契约的基础 |
| □ | HTTP 动词使用正确：GET 查询、POST 创建、PUT 全量更新、PATCH 部分更新、DELETE 删除 | BLOCKER | 禁止用 GET 做写操作 |
| □ | API 版本策略已确定（URL 版本 /v1/ 或 Header 版本）并在全部接口中一致执行 | BLOCKER | 无版本策略导致不可维护 |
| □ | 请求参数（Path/Query/Body）已完整定义：字段名、类型、是否必填、默认值、校验规则 | BLOCKER | 如：page 最小值=1 |
| □ | 响应结构统一：成功响应体结构、错误响应体结构（code/message/data）全局一致 | BLOCKER | 避免各接口自定义结构 |
| □ | HTTP 状态码使用规范：200/201/204/400/401/403/404/409/422/500 含义已定义 | MAJOR | 禁止所有接口都返回 200 |
| □ | 分页接口统一：page+size 或 cursor，响应包含 total / has_more / next_cursor | MAJOR | 不同接口分页风格一致 |
| □ | 列表接口支持排序和过滤参数，字段已明确定义 | MAJOR | 如：sort=created_at:desc |
| □ | 幂等性：POST/PATCH 关键接口是否需要幂等键（Idempotency-Key）已明确 | MAJOR | 支付/订单类必须幂等 |
| □ | 文件上传接口说明了文件类型限制、大小限制、存储位置 | MAJOR | 如：最大 10MB，仅 PDF/PNG |
| □ | 接口 deprecated 字段或即将废弃的接口已标注，有过渡方案 | MINOR | 提前告知前端调整 |
| □ | 接口文档包含 cURL 或 HTTP 请求示例（至少主要接口） | MINOR | 降低联调成本 |

### 2.2 数据模型 Data Model

| □ | 检查项 | 优先级 | 说明/示例 |
| :--- | :--- | :--- | :--- |
| □ | ER 图存在，覆盖所有核心实体及其关系（1:1 / 1:N / M:N） | BLOCKER | 缺少主实体定义即 Block |
| □ | 每张表的主键、外键、唯一约束已明确定义 | BLOCKER | 未定义主键的表不可接受 |
| □ | 字段类型、长度、NOT NULL 约束、默认值已完整定义 | BLOCKER | 如：status ENUM 值域已列出 |
| □ | 软删除策略已明确（is_deleted / deleted_at / 归档表），全局一致 | MAJOR | 混用软删除和物理删除会出问题 |
| □ | 时间字段统一：created_at / updated_at / deleted_at 命名和时区策略一致（推荐 UTC） | MAJOR | 时区混用是高频 Bug 来源 |
| □ | 索引策略已说明：高频查询字段的索引已设计，复合索引顺序已分析 | MAJOR | 缺索引导致生产性能问题 |
| □ | M:N 关系通过关联表实现，关联表有明确的字段和索引设计 | MAJOR | 如：user_roles(user_id, role_id) |
| □ | 大字段（TEXT/BLOB/JSON 列）的使用有合理理由，未滥用 JSON 列绕过规范化 | MAJOR | JSON 列难以索引和约束 |
| □ | 数据量增长预估已给出（1年/3年的预估行数），分库分表方案是否需要 | MAJOR | >1亿行需考虑分区/分表 |
| □ | 敏感字段（手机/身份证/密码）已标注加密存储或脱敏方式 | BLOCKER | 合规要求，明文存储不可接受 |
| □ | 历史数据归档策略已描述（如订单数据超 N 年归档到冷存储） | MINOR | 影响长期存储成本 |

### 2.3 非功能需求 NFR

| □ | 检查项 | 优先级 | 说明/示例 |
| :--- | :--- | :--- | :--- |
| □ | 核心接口的性能基线已量化：目标 QPS / TPS、P50/P95/P99 延迟 | BLOCKER | 如：搜索接口 P99 < 300ms |
| □ | 系统可用性 SLA 已定义（如 99.9% = 每月不超过 43 分钟停机） | BLOCKER | 结合业务重要性定级 |
| □ | 认证/授权方案已明确（JWT/OAuth2/Session，RBAC/ABAC 模型） | BLOCKER | 缺少鉴权设计即 Block |
| □ | SQL 注入、XSS、CSRF 等常见安全威胁的防护措施已描述 | BLOCKER | 至少覆盖 OWASP Top 10 |
| □ | 限流策略已定义：全局/用户维度，限流阈值，限流后的响应（429） | MAJOR | 保护服务不被打垮 |
| □ | 熔断/降级策略已设计：哪些依赖可降级，降级后的兜底行为 | MAJOR | 如：评分服务不可用时返回空 |
| □ | 可观测性设计：关键业务操作的结构化日志字段已定义（trace_id 必须） | MAJOR | trace_id 贯穿全链路 |
| □ | 可观测性设计：核心指标（Metrics）已列出，告警阈值已定义 | MAJOR | 如：错误率 > 1% 则告警 |
| □ | 分布式追踪（Tracing）接入方案已说明（如 OpenTelemetry） | MAJOR | 微服务架构必须有 |
| □ | 数据备份和恢复策略已描述：备份频率、RTO（恢复时间目标）、RPO（恢复点目标） | MAJOR | 如：RPO=1h，RTO=4h |
| □ | 水平扩展方案已描述：无状态服务如何扩容，有状态部分如何处理 | MINOR | 避免单点成为瓶颈 |

### 2.4 依赖与部署 Dependencies & Deployment

| □ | 检查项 | 优先级 | 说明/示例 |
| :--- | :--- | :--- | :--- |

## 四、接口 GAP 处理 Checklist（后端已完成时适用）

### 4.1 GAP 识别与分类

| □ | 检查项 | 优先级 | 说明/示例 |
| :--- | :--- | :--- | :--- |
| □ | 前端已完整输出接口消费清单，包含：HTTP 方法、路径、使用的字段、调用场景 | BLOCKER | 清单不完整则比对无效 |
| □ | 比对结果已录入接口对齐矩阵（Interface Alignment Matrix），状态标注清晰 | BLOCKER | 已实现/已设计未实现/完全缺失/字段不符 |
| □ | 每条 GAP 已分类：完全缺失 / SDD 有但未实现 / 字段/行为不符 / 性能不满足 | BLOCKER | 分类决定处理方案 |
| □ | 每条 GAP 已指定 Owner 和处理截止时间 | BLOCKER | 无 Owner 的 GAP 等于无方案 |
| □ | 完全缺失的接口：已判断是需求变更还是设计遗漏，走对应流程 | MAJOR | 需求变更需产品确认 |
| □ | SDD 有但未实现的接口：已纳入后端 Sprint 排期，前端使用 Mock 继续开发 | MAJOR | 明确 Mock 方案（MSW 等） |
| □ | 字段/行为不符的接口：已前后端联审，确定是后端补字段还是前端加适配层 | MAJOR | 优先最小代价方案 |
| □ | 所有 GAP 处理决策已更新到 SDD 文档（包括标注妥协方案） | MAJOR | 文档要反映最终真实状态 |

### 4.2 GAP 预防机制验证

| □ | 检查项 | 优先级 | 说明/示例 |
