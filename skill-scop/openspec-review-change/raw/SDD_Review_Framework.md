# SDD 审核框架

## 优先级说明

| 标识 | 含义 |
|------|------|
| BLOCKER | 必须修复后方可进入开发 |
| MAJOR | 里程碑前关闭，或需有明确处理计划 |
| MINOR | 迭代内处理即可 |

---

## 一、审核维度

### 1.1 通用维度（前后端共用）

#### 完整性 Completeness

| 检查项 | 优先级 | 说明 |
|--------|--------|------|
| 文档包含所有必填章节：概述、背景、设计目标、约束、系统架构、接口/数据设计、非功能需求、风险与依赖 | BLOCKER | 缺少任意章节即 Block |
| 每条 RAW 需求都能在 SDD 中找到对应的设计描述（可追溯性矩阵存在） | BLOCKER | 需求覆盖率须 = 100% |
| 所有业务主流程（Happy Path）均有对应的流程图或时序图 | BLOCKER | 缺失核心流程即 Block |
| 异常/错误路径已列出：网络超时、权限不足、参数非法、第三方服务故障 | MAJOR | 至少覆盖主流程的异常 |
| 边界条件已明确量化：最大值、最小值、空集合、并发场景 | MAJOR | 如：最大分页 size=100 |
| 数据迁移/兼容性方案已描述（如涉及） | MAJOR | 新旧数据如何共存 |
| 文档版本号、作者、最后更新时间已填写 | MINOR | v1.0 基线须有版本号 |

#### 一致性 Consistency

| 检查项 | 优先级 | 说明 |
|--------|--------|------|
| 术语与 PRD/RAW 需求文档保持一致，无自造缩写或歧义词汇 | BLOCKER | 建立术语表附录 |
| SDD 各章节内部自洽，无矛盾描述（时序图与接口描述一致） | BLOCKER | 常见冲突：流程图 vs 状态机 |
| 接口命名风格统一（RESTful 风格、snake_case/camelCase 全局一致） | MAJOR | 如 user_id vs userId 混用 |
| 错误码体系全局统一，无不同模块自定义重复错误码 | MAJOR | 统一错误码注册表 |
| 与上下游服务的接口约定版本一致，无独自升级导致不兼容 | MAJOR | 检查所有外部依赖版本 |
| 图表与文字描述同步更新，无过期示意图 | MINOR | 图表日期与文档日期对应 |

#### 可行性 Feasibility

| 检查项 | 优先级 | 说明 |
|--------|--------|------|
| 技术选型有明确的选择依据（对比分析或团队决策记录） | MAJOR | 如为何选 Redis 而非 Memcached |
| 关键技术点的 PoC 已完成或已有参考实现 | MAJOR | 高风险技术必须有 PoC |
| 开发工作量估算有依据，不存在明显低估（关键路径已识别） | MAJOR | 对比历史类似功能 |
| 所有外部依赖已明确版本，SLA 已确认，不可用时降级方案存在 | MAJOR | 依赖不可用时的降级方案 |
| 已知风险已列出，每个风险有缓解措施和负责人 | MAJOR | 风险矩阵（概率×影响） |
| 方案未引入团队技术栈范围外的高风险技术（如有需额外评审） | MINOR | 学习曲线风险 |

#### 可测试性 Testability

| 检查项 | 优先级 | 说明 |
|--------|--------|------|
| 设计方案可据此直接编写测试用例（测试充分条件存在） | BLOCKER | 若无法测试则设计不完整 |
| 验收标准（AC）已明确量化，无"性能好""响应快"等模糊表述 | BLOCKER | 如：P99 < 200ms |
| 核心模块接口设计支持 Mock/Stub（依赖注入或接口抽象） | MAJOR | 便于单测隔离 |
| 测试环境与生产环境的差异已记录（配置、数据量等） | MAJOR | 差异影响测试有效性 |
| 性能测试场景已描述：目标 QPS、数据规模、测试时长 | MAJOR | 需与 NFR 对应 |
| 安全测试要求已列出（如需渗透测试、OWASP 检查项） | MINOR | 按项目合规要求 |

### 1.2 后端专项维度

#### 接口规范性 API Specification

| 检查项 | 优先级 | 说明 |
|--------|--------|------|
| 所有接口都有 OpenAPI 3.x / Swagger 定义 | BLOCKER | 是前后端契约的基础 |
| HTTP 动词使用正确：GET 查询、POST 创建、PUT 全量更新、PATCH 部分更新、DELETE 删除 | BLOCKER | 禁止用 GET 做写操作 |
| API 版本策略已确定（URL 版本 /v1/ 或 Header 版本）并全部接口一致执行 | BLOCKER | 无版本策略导致不可维护 |
| 请求参数已完整定义：字段名、类型、是否必填、默认值、校验规则 | BLOCKER | 如：page 最小值=1 |
| 响应结构统一：成功响应体结构、错误响应体结构（code/message/data）全局一致 | BLOCKER | 避免各接口自定义结构 |
| HTTP 状态码使用规范：200/201/204/400/401/403/404/409/422/500 含义已定义 | MAJOR | 禁止所有接口都返回 200 |
| 分页接口统一：page+size 或 cursor，响应包含 total / has_more / next_cursor | MAJOR | 不同接口分页风格一致 |
| 幂等性：POST/PATCH 关键接口是否需要幂等键（Idempotency-Key）已明确 | MAJOR | 支付/订单类必须幂等 |
| 文件上传接口说明了文件类型限制、大小限制、存储位置 | MAJOR | 如：最大 10MB，仅 PDF/PNG |
| 接口文档包含 cURL 或 HTTP 请求/响应示例（至少主要接口） | MINOR | 降低联调成本 |

#### 数据模型 Data Model

| 检查项 | 优先级 | 说明 |
|--------|--------|------|
| ER 图存在，覆盖所有核心实体及其关系（1:1 / 1:N / M:N） | BLOCKER | 缺少主实体定义即 Block |
| 每张表的主键、外键、唯一约束已明确定义 | BLOCKER | 未定义主键的表不可接受 |
| 敏感字段（手机/身份证/密码）已标注加密存储或脱敏方式 | BLOCKER | 合规要求，明文存储不可接受 |
| 字段类型、长度、NOT NULL 约束、默认值已完整定义 | BLOCKER | 如：status ENUM 值域已列出 |
| 软删除策略已明确（is_deleted / deleted_at / 归档表），全局一致 | MAJOR | 混用软删除和物理删除会出问题 |
| 时间字段统一：created_at / updated_at 命名和时区策略一致（推荐 UTC） | MAJOR | 时区混用是高频 Bug 来源 |
| 索引策略已说明：高频查询字段的索引已设计，复合索引顺序已分析 | MAJOR | 缺索引导致生产性能问题 |
| 大字段（TEXT/BLOB/JSON 列）的使用有合理理由，未滥用 JSON 列绕过规范化 | MAJOR | JSON 列难以索引和约束 |
| 数据量增长预估已给出（1年/3年预估行数），分库分表方案是否需要 | MAJOR | >1亿行需考虑分区/分表 |
| 历史数据归档策略已描述（超 N 年归档到冷存储） | MINOR | 影响长期存储成本 |

#### 非功能需求 NFR

| 检查项 | 优先级 | 说明 |
|--------|--------|------|
| 核心接口的性能基线已量化：目标 QPS/TPS、P50/P95/P99 延迟 | BLOCKER | 如：搜索接口 P99 < 300ms |
| 系统可用性 SLA 已定义（如 99.9% = 每月不超过 43 分钟停机） | BLOCKER | 结合业务重要性定级 |
| 认证/授权方案已明确（JWT/OAuth2/Session，RBAC/ABAC 模型） | BLOCKER | 缺少鉴权设计即 Block |
| SQL 注入、XSS、CSRF 等常见安全威胁的防护措施已描述 | BLOCKER | 至少覆盖 OWASP Top 10 |
| 限流策略已定义：全局/用户维度，限流阈值，限流后的响应（429） | MAJOR | 保护服务不被打垮 |
| 熔断/降级策略已设计：哪些依赖可降级，降级后的兜底行为 | MAJOR | 如：评分服务不可用时返回空 |
| 可观测性：关键业务操作的结构化日志字段已定义（trace_id 必须贯穿全链路） | MAJOR | trace_id 贯穿全链路 |
| 可观测性：核心 Metrics 已列出，告警阈值已定义 | MAJOR | 如：错误率 > 1% 则告警 |
| 分布式追踪（Tracing）接入方案已说明（如 OpenTelemetry） | MAJOR | 微服务架构必须有 |
| 数据备份和恢复策略：备份频率、RTO、RPO 已描述 | MAJOR | 如：RPO=1h，RTO=4h |
| 水平扩展方案已描述：无状态服务如何扩容，有状态部分如何处理 | MINOR | 避免单点成为瓶颈 |

#### 依赖与部署 Dependencies & Deployment

| 检查项 | 优先级 | 说明 |
|--------|--------|------|
| 所有外部服务依赖已列出：服务名、版本/协议、调用频率、依赖方向 | BLOCKER | 隐性依赖是事故高发区 |
| 环境配置项已外化，无硬编码的 IP/密码/AK/SK | BLOCKER | 合规要求，硬编码不可接受 |
| 部署架构图存在：服务拓扑、负载均衡、数据库、缓存的关系清晰 | MAJOR | 运维和 SRE 必须能看懂 |
| 启动/停止的 Graceful 处理已设计（优雅关闭时如何处理 in-flight 请求） | MAJOR | 滚动发布时避免请求丢失 |
| 数据库 Migration 方案已说明（如 Flyway/Liquibase），向前/向后兼容性分析 | MAJOR | 零停机发布的关键 |
| 回滚方案已设计：代码回滚 + 数据回滚步骤，回滚判断标准 | MAJOR | 缺少回滚方案等于没有方案 |
| 服务的健康检查端点已定义（/health、/ready、/live） | MINOR | K8s 探针需要 |
| CI/CD 流水线中的质量门禁已描述（覆盖率阈值、安全扫描等） | MINOR | 与 DevOps 团队确认 |

### 1.3 前端专项维度

#### 接口消费完整性 API Consumption

| 检查项 | 优先级 | 说明 |
|--------|--------|------|
| 已输出接口消费清单（API Consumption Map）：每个页面/组件依赖哪些接口 | BLOCKER | 是前后端对齐的基础文档 |
| 消费清单与后端 SDD 接口列表已做比对，无 GAP（或 GAP 已有处理决策） | BLOCKER | GAP 无决策即 Block |
| 接口返回的每个字段已映射到对应的 UI 展示位置 | MAJOR | 字段未用或 UI 无来源是设计遗漏 |
| 接口调用时序已明确：哪些接口并行调用，哪些存在顺序依赖（waterfall） | MAJOR | 顺序依赖影响首屏加载 |
| 接口缺失时的 Mock 方案已描述（MSW / json-server / 契约测试） | MAJOR | 保障前端开发不阻塞 |
| 分页/无限滚动的接口调用策略已设计（何时触发、数据合并方式） | MAJOR | 避免重复请求和数据错乱 |
| 接口版本升级时的前端适配策略已说明（如 BFF 层转换） | MINOR | 多版本并存时需要 |

#### 状态与交互设计 State & Interaction

| 检查项 | 优先级 | 说明 |
|--------|--------|------|
| 每个核心页面/组件的三态设计已完成：Loading / Error / Empty | BLOCKER | 缺少任何一态是不完整设计 |
| 全局状态与局部状态的边界已划分（哪些数据放 Store，哪些放 local state） | MAJOR | 混用导致数据一致性问题 |
| 用户操作路径（User Flow）完整覆盖：主流程、取消/返回、跳出再返回 | MAJOR | 测试路径的来源 |
| 表单设计：实时校验 vs 提交时校验策略已定义，错误提示位置已确定 | MAJOR | 统一用户体验 |
| 乐观更新（Optimistic Update）场景已识别，失败时的回滚 UI 已设计 | MAJOR | 如：点赞立即显示，失败后撤销 |
| 并发操作场景已考虑：快速多次点击、多 Tab 同步等（防抖/节流策略） | MAJOR | 按钮防抖/节流策略 |
| 页面间数据传递方式已明确（URL params / router state / store） | MAJOR | 刷新后状态是否保留 |
| 键盘导航和焦点管理已设计（特别是模态框/表单场景） | MINOR | a11y 基本要求 |

#### 性能与可访问性 Performance & A11y

| 检查项 | 优先级 | 说明 |
|--------|--------|------|
| Web 性能目标已量化：LCP < 2.5s / INP < 200ms / CLS < 0.1（Core Web Vitals） | BLOCKER | 与 SEO 和用户体验直接相关 |
| 首屏关键资源已识别，Critical CSS / 关键 JS 的加载策略已设计 | MAJOR | 避免 render-blocking 资源 |
| 代码分割（Code Splitting）策略已确定：路由级 / 组件级 lazy load 的边界 | MAJOR | 减少初始 bundle 体积 |
| 图片/媒体资源优化策略：格式（WebP/AVIF）、懒加载、尺寸响应式 | MAJOR | 图片通常是最大性能开销 |
| 虚拟化列表（Virtual Scroll）场景已识别（>100 条数据的列表） | MAJOR | DOM 节点过多导致卡顿 |
| 缓存策略已设计：接口数据缓存（SWR/React Query）、Service Worker 缓存 | MAJOR | 降低重复请求 |
| 无障碍（a11y）级别已定义（至少达到 WCAG 2.1 AA） | MAJOR | 部分行业有法律合规要求 |
| 颜色对比度满足 WCAG AA 标准（正文 4.5:1，大文字 3:1） | MAJOR | 视觉障碍用户 |
| 多端/断点设计已说明：移动端（< 768px）、平板、桌面的布局策略 | MAJOR | 避免开发阶段遗漏 |
| 国际化（i18n）需求已确认，文本硬编码 vs 翻译 key 的边界已定义 | MINOR | 后期迁移成本极高 |

#### 错误处理与降级 Error Handling & Fallback

| 检查项 | 优先级 | 说明 |
|--------|--------|------|
| 后端 HTTP 错误码与前端 UI 行为的映射表已定义（400/401/403/404/422/500） | BLOCKER | 401 → 跳登录；403 → 提示无权限 |
| 全局错误边界（Error Boundary）已设计，避免组件错误导致整页崩溃 | MAJOR | React ErrorBoundary 或等效实现 |
| 网络请求失败的重试策略已定义（次数、退避算法、哪些接口可重试） | MAJOR | 幂等接口可重试，非幂等不可 |
| 关键功能的降级方案已设计：核心服务不可用时的 fallback UI | MAJOR | 如：推荐系统不可用时显示热门 |
| Token 过期场景的处理已设计：自动刷新 Token 还是跳登录，防止刷新死循环 | MAJOR | 常见坑：刷新接口也 401 死循环 |
| 用户操作冲突提示已设计：并发编辑（乐观锁冲突）的 UI 提示和解决引导 | MAJOR | 防止数据静默丢失 |
| 离线/弱网场景已考虑（如有 PWA 要求）：哪些操作支持离线，数据同步策略 | MINOR | 按产品要求决定是否覆盖 |
| 错误日志上报已设计（Sentry / 自建）：上报字段、采样率、PII 脱敏策略 | MINOR | 前端错误可观测性 |

## 二、审核工作流

```
PRD/RAW 需求冻结
       │
       ▼
 [Step 1] 作者自审（SDD v0.1 初稿）
       │  负责人：SDD 作者（前后端分别）
       │  输出：自查 checklist 完成
       │
       ▼
 [Step 2] 静态分析 & 自动化检查
       │  负责人：CI 自动触发
       │  检查项：OpenAPI 规范、必填章节、需求覆盖率
       │  Gate：无 BLOCKER 级别错误方可继续
       │
       ▼
 [Step 3] 技术 Peer Review（同层横向）
       │  负责人：Tech Lead 指派，≥1 位同侧开发
       │  检查项：技术方案合理性、数据模型、接口一致性、安全/性能隐患
       │  意见标注：BLOCKER / MAJOR / MINOR
       │  Gate：所有 BLOCKER/MAJOR 关闭，评审人 LGTM
       │
       ▼
 [Step 4] 前后端联合接口评审 ← 解决"前端发现后端缺接口"的关键节点
       │  负责人：Tech Lead 主持，前后端核心开发参与
       │  动作：前端输出接口消费清单 → 逐一比对后端 SDD 接口列表
       │        → 识别 GAP → 分类处理（补设计 / 临时 Mock / 调整优先级）
       │  输出：接口对齐矩阵（Interface Alignment Matrix）
       │  Gate：矩阵中所有 GAP 条目均有明确处理决策
       │
       ▼
 [Step 5] 架构 / Tech Lead 终审
       │  负责人：架构师 / Tech Lead
       │  检查项：整体架构一致性、跨团队依赖告知、NFR 合理性、安全合规
       │  Gate：Tech Lead 正式批准（Approved 签名）
       │
       ▼
 [Step 6] 问题追踪与关闭
       │  负责人：PM / Scrum Master 跟踪
       │  BLOCKER：必须关闭后才能进入开发
       │  MAJOR：里程碑前关闭
       │  MINOR：迭代内处理
       │  Gate：BLOCKER=0，MAJOR 有处理计划，文档版本更新至 v1.0
       │
       ▼
 [Step 7] 文档基线化（冻结）
          负责人：文档管理员 / Tech Lead
          动作：标记为 APPROVED 基线；后续变更走 CR 流程；版本与代码 Tag 对齐
          输出：SDD v1.0 APPROVED，归档到知识库
```

### 各步骤负责人与耗时参考

| 步骤 | 负责人 | 建议耗时 | 强制 Gate |
|------|--------|----------|-----------|
| 1. 作者自审 | SDD 作者 | 0.5 天 | 初稿完整性自查通过 |
| 2. 自动化检查 | CI 工具 | < 10 分钟 | 无 BLOCKER 告警 |
| 3. Peer Review | Tech Lead 指派（≥1人） | 1-2 天 | 所有 BLOCKER/MAJOR 关闭，LGTM |
| 4. 前后端联合接口评审 | Tech Lead 主持 | 1-3 小时 | 接口对齐矩阵 GAP 全有决策 |
| 5. 架构终审 | 架构师 / Tech Lead | 0.5-1 天 | Tech Lead Approved |
| 6. 问题追踪关闭 | PM / Scrum Master | 持续跟踪 | BLOCKER=0，MAJOR 有计划 |
| 7. 基线化 | 文档管理员 | < 1 小时 | 归档完成，版本号锁定 |

## 三、验收指标（Definition of Done）

以下为可量化的硬性指标，直接作为 SDD 审核通过的 DoD。

### 3.1 通用指标

| 指标 | 通过标准 | 不通过处理 | 适用 |
|------|----------|-----------|------|
| 需求覆盖率 | = 100%（每条 RAW 需求都有对应设计） | < 100% 即 BLOCKER | 前端 + 后端 |
| BLOCKER 问题数 | = 0（全部关闭） | 任意 1 个未关闭 = 不可进入开发 | 前端 + 后端 |
| MAJOR 问题数 | = 0 或每条都有关闭计划（Owner + 截止时间） | 无计划则不可进入开发 | 前端 + 后端 |
| 评审人数 | 后端 ≥ 2 人 LGTM；前端 ≥ 1 人 LGTM + 1 人后端联审 | 未满足即不进入基线 | 前端 + 后端 |
| 文档版本关联 | SDD 版本号与 git tag / PR 对应 | 未关联为 MINOR | 前端 + 后端 |

### 3.2 后端专项指标

| 指标 | 通过标准 | 不通过处理 |
|------|----------|-----------|
| 接口规范合规率 | ≥ 95%（HTTP 动词/命名/版本/错误码规范） | < 90% 为 BLOCKER；90-95% 为 MAJOR |
| 数据模型完整率 | = 100%（无未定义主实体） | 缺失主实体为 BLOCKER |
| NFR 指标明确率 | 核心接口 100% 有 P99 延迟 / QPS 目标 | 缺失为 MAJOR |
| 安全覆盖率 | OWASP Top 10 全部有对应防护措施描述 | 缺失任意 1 项为 MAJOR |

### 3.3 前端专项指标

| 指标 | 通过标准 | 不通过处理 |
|------|----------|-----------|
| 接口定义完整率 | = 100%（前端消费清单中每条接口都有后端定义或 Mock 方案） | 有 GAP 无方案为 BLOCKER |
| 前端状态覆盖率 | 核心页面 100% 覆盖 Loading / Error / Empty 三态 | 缺失为 MAJOR |
| Core Web Vitals 目标明确率 | LCP / INP / CLS 目标值全部量化 | 缺失为 BLOCKER |
| 错误码映射完整率 | 400/401/403/404/422/500 全部有 UI 处理方案 | 缺失 401/403/500 为 MAJOR |

### 3.4 接口对齐矩阵指标

| 指标 | 通过标准 |
|------|----------|
| GAP 条目有处理决策率 | = 100%（每条 GAP 都有 Owner + 截止时间 + 方案） |
| BLOCKER 级 GAP 关闭率 | = 100% 方可进入开发 |
| Mock 方案覆盖率 | 所有"已设计未实现"和"完全缺失"的接口都有 Mock 实现 |

## 四、后端已完成时的特殊处理（接口 GAP 处理流程）

> 适用场景：后端已开发部分或全部完成，前端开发过程中仍发现接口缺失或不符。

### 4.1 GAP 类型与处理矩阵

| GAP 类型 | 判断标准 | 处理方案 | 优先级 |
|----------|----------|----------|--------|
| **完全缺失** | 前端消费清单有记录，后端 SDD 和代码均无 | ① 判断是需求变更还是设计遗漏 → ② 走 CR 流程 + 排期 → ③ 前端用 Mock 继续开发 | BLOCKER |
| **SDD 有但未实现** | SDD 有接口定义，后端开发进度未到 | ① 确认后端交付时间窗口 → ② 前端 Mock + 契约测试 → ③ 纳入后端 Sprint 排期跟踪 | MAJOR |
| **字段/行为不符** | 接口存在但 response 字段缺失或语义不符 | ① 前后端联审字段差异 → ② 后端补充字段（代价小）或前端加适配层（后端改动风险大） | MAJOR |
| **性能不满足** | 接口存在但不满足前端调用频率或延迟需求 | ① 后端提供 Batch 接口或 WebSocket/SSE 推送 → ② 前端调整交互减少请求频次 → ③ 协商新 SLA | MAJOR |

### 4.2 轻量化变更流程（CR）

后端已完成时，补充接口或修改接口走以下轻量化 CR 流程：

```
① 前端提 Gap Issue
   包含：接口路径、缺失字段、使用场景、影响范围、建议优先级
         ↓
② 后端评估代价
   区分：「补设计文档」（低成本）vs「需要修改实现」（高成本）
         ↓
③ Tech Lead 联合决策
   三选一：补实现 / 前端加适配层 / 推迟到下个迭代
         ↓
④ 更新 SDD
   无论哪种决策，SDD 都要更新反映最终接口契约
   包括标注哪些是妥协方案、为什么这样决策
         ↓
⑤ 补充契约测试
   用 Pact / OpenAPI Schema Validation 锁定接口契约
   防止后续后端迭代静默破坏前端依赖
         ↓
⑥ 复盘记录
   GAP 根因写入项目复盘
   改进下次 SDD 评审的前后端联审节点
```

### 4.3 预防机制（从源头减少 GAP）

| 机制 | 说明 | 实施时机 |
|------|------|----------|
| 前端输出接口消费清单 | 列出每个页面/组件依赖的接口，在 SDD 评审阶段（而非开发阶段）与后端比对 | SDD v0.1 评审前 |
| API First 工作流 | 后端先出 OpenAPI Spec → 生成 Mock → 前后端并行开发，消灭联调阶段的惊喜 | SDD 设计阶段 |
| Sprint 接口对齐会 | 每个 Sprint 有一次 15 分钟前后端接口对齐，消灭微小偏差，不等到集成测试才发现 | 每 Sprint 启动时 |
| 契约测试 CI 门禁 | Pact 或 OpenAPI Schema Validation 接入 CI，后端变更自动检测是否 breaking 前端契约 | 持续集成中 |

### 4.4 GAP 根因分类（用于复盘）

| 根因类型 | 描述 | 改进方向 |
|----------|------|----------|
| 设计遗漏 | SDD 评审阶段前端未输出消费清单，设计时遗漏了某些接口 | 强制要求前端在评审前输出消费清单 |
| 需求变更 | 后端完成后需求发生变化，产生新的接口需求 | 需求变更必须触发接口影响评估 |
| 沟通缺失 | 后端实现时裁剪了部分接口但未通知前端 | 接口变更必须同步前端，纳入 CR 流程 |
| 理解偏差 | 前后端对同一个接口的字段语义理解不同 | 接口文档必须包含字段语义说明和示例值 |
