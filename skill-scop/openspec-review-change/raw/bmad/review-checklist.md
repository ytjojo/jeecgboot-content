# SDD 审核清单

## 通用审核（前后端共享）

### L1 结构完整性

| # | 检查项 | 严重级别 | 说明 |
|---|--------|---------|------|
| 1.1 | SPEC.md 包含完整的五字段内核（Why/Capabilities/Constraints/Non-goals/Success Signal） | BLOCKER | 任一字段缺失即阻塞 |
| 1.2 | 每个 Capability 有 intent + success | BLOCKER | Spec Law Rule 1 |
| 1.3 | Capability ID 稳定且唯一 | BLOCKER | Spec Law Rule 6 |
| 1.4 | Non-goals ≥ 1 条 | BLOCKER | Spec Law Rule 4 |
| 1.5 | Companion 文件按内容类型命名 | MEDIUM | 如 glossary.md、api-contracts.md |
| 1.6 | 图表在 Companion 中，SPEC.md 仅散文 | MEDIUM | Spec Law 隐含要求 |
| 1.7 | Frontmatter 的 companions/sources 正确声明 | HIGH | 下游消费依赖 |
| 1.8 | .decision-log.md 存在 | MEDIUM | 决策可追溯 |

### L2 内容质量

| # | 检查项 | 严重级别 | 说明 |
|---|--------|---------|------|
| 2.1 | Intent 描述 WHAT 而非 HOW | HIGH | 无实现处方 |
| 2.2 | Constraint 真正约束设计决策 | HIGH | 删除后设计不应改变则为装饰 |
| 2.3 | Success signal 可测试/可演示 | HIGH | 不接受"用户喜欢"类描述 |
| 2.4 | 精简散文，无装饰性文字 | LOW | Spec Law Rule 8 |
| 2.5 | 无歧义术语，同一概念同一用词 | MEDIUM | 一致性 |
| 2.6 | 无模糊量词（"尽快"、"适当"、"足够"） | HIGH | 不可操作 |

## 前端审核专项

### F1 页面-接口映射

| # | 检查项 | 严重级别 | 说明 |
|---|--------|---------|------|
| F1.1 | 每个页面/组件有对应的 API 需求列表 | BLOCKER | 无 API 支撑的页面 = 开发阻塞 |
| F1.2 | 每个 API 需求有对应的后端 API 声明 | BLOCKER | 悬空需求 = 契约不完整 |
| F1.3 | API 返回数据结构满足前端渲染需求 | HIGH | 字段缺失、类型不匹配 |
| F1.4 | 分页/排序/筛选参数在 API 契约中明确 | MEDIUM | 前端列表页必需 |

### F2 错误处理覆盖

| # | 检查项 | 严重级别 | 说明 |
|---|--------|---------|------|
| F2.1 | 每个 API 声明了错误码列表 | HIGH | 前端需据此实现错误处理 |
| F2.2 | 401 → 跳转登录页的策略明确 | HIGH | 认证失败处理 |
| F2.3 | 403 → 无权限提示的策略明确 | HIGH | 授权失败处理 |
| F2.4 | 404 → 空状态展示的策略明确 | MEDIUM | 资源不存在处理 |
| F2.5 | 500 → 重试/错误页的策略明确 | MEDIUM | 服务端错误处理 |
| F2.6 | 网络超时的前端降级策略明确 | MEDIUM | 离线/弱网场景 |

### F3 前端并行开发支持

| # | 检查项 | 严重级别 | 说明 |
|---|--------|---------|------|
| F3.1 | 每个 API 标注就绪状态（已就绪/待开发/需重构） | HIGH | 前端据此制定 mock 策略 |
| F3.2 | Mock 数据结构与最终 API 返回结构一致 | HIGH | 避免联调时大面积返工 |
| F3.3 | 前端职责范围内的逻辑在 SDD 中明确 | HIGH | 避免前后端重复实现 |

### F4 前端命名敏感度

| # | 检查项 | 严重级别 | 说明 |
|---|--------|---------|------|
| F4.1 | API 路径语义清晰，前端可直接映射为代码变量名 | HIGH | 如 `/users/{id}/block` 而非 `/blackList` |
| F4.2 | API 返回字段名符合前端命名惯例（camelCase） | MEDIUM | 避免前端做无意义转换 |
| F4.3 | 枚举值在 SDD 中有明确的前端展示映射 | MEDIUM | 如 status: 1 → "启用" |

## 后端审核专项

### B1 API 命名规范

| # | 检查项 | 严重级别 | 说明 |
|---|--------|---------|------|
| B1.1 | 资源名用复数名词 | HIGH | `/users` 而非 `/user` |
| B1.2 | 动作用 HTTP method 表达 | HIGH | `DELETE /users/{id}/block` 而非 `/removeBlackList` |
| B1.3 | 路径中无技术实现术语 | BLOCKER | 禁止 `blackList`、`cache`、`queue` 等 |
| B1.4 | 路径层级反映资源关系 | HIGH | `/users/{id}/blocks` 而非 `/userBlock` |
| B1.5 | 无 RPC 风格路径 | HIGH | 禁止 `/getUserList`、`/submitOrder` |
| B1.6 | 路径全小写，单词用连字符分隔 | MEDIUM | `/user-profiles` 而非 `/userProfiles` |

### B2 存量 API 兼容

| # | 检查项 | 严重级别 | 说明 |
|---|--------|---------|------|
| B2.1 | 已有 API 清单已盘点 | BLOCKER | 无盘点则无法审核兼容性 |
| B2.2 | 已有 API 在 SDD 中标记为"已就绪" | HIGH | 约束后续设计 |
| B2.3 | 新增 API 不与已有 API 路径冲突 | BLOCKER | 路径冲突 = 运行时错误 |
| B2.4 | 需重构的 API 有明确的迁移方案 | HIGH | 避免前后端同时断裂 |
| B2.5 | 已有 API 的命名违规单独记录 | MEDIUM | 不阻塞本轮，但需技术债务记录 |

### B3 接口粒度

| # | 检查项 | 严重级别 | 说明 |
|---|--------|---------|------|
| B3.1 | 单个 API 不超过 3 个主要返回数据块 | MEDIUM | 过粗 = 前端不需要的数据也被传输 |
| B3.2 | 单个页面调用 API 数不超过 8 个 | MEDIUM | 过碎 = 前端性能和维护成本 |
| B3.3 | 列表 API 支持分页 | HIGH | 数据量大时必需 |
| B3.4 | 批量操作 API 有并发控制说明 | HIGH | 避免数据不一致 |

### B4 数据契约

| # | 检查项 | 严重级别 | 说明 |
|---|--------|---------|------|
| B4.1 | 每个 API 的请求/响应字段有类型定义 | HIGH | 无类型 = 联调时的猜谜游戏 |
| B4.2 | 字段名用 camelCase | MEDIUM | Java 后端惯例 |
| B4.3 | 必填/可选字段标注明确 | HIGH | 避免 NPE 和前端空指针 |
| B4.4 | 日期时间格式统一（ISO 8601） | MEDIUM | 避免格式解析错误 |
| B4.5 | 枚举值有完整列表 | HIGH | 前端需据此渲染 |

## 跨端一致性审核

### C1 前后端对齐

| # | 检查项 | 严重级别 | 说明 |
|---|--------|---------|------|
| C1.1 | 前端需求的每个 API 在后端有声明 | BLOCKER | 契约双向完整 |
| C1.2 | 后端声明的每个 API 在前端有消费方 | HIGH | 孤立 API = 可能无用 |
| C1.3 | 请求/响应字段名前后端一致 | HIGH | 前端 camelCase，后端也 camelCase |
| C1.4 | 错误码前后端含义一致 | HIGH | 避免前后端对同一错误码理解不同 |
| C1.5 | 业务逻辑归属无歧义 | HIGH | 每个逻辑点仅一个端负责 |

### C2 术语一致性

| # | 检查项 | 严重级别 | 说明 |
|---|--------|---------|------|
| C2.1 | 术语表存在且覆盖核心业务概念 | MEDIUM | glossary.md |
| C2.2 | 同一概念前后端用同一术语 | HIGH | 避免"用户"vs"会员"vs"账户"混用 |
| C2.3 | 枚举值前后端定义一致 | HIGH | 状态码、类型码等 |

## 依赖分析清单（Step 6）

### D1 前端依赖

| # | 检查项 | 严重级别 | 说明 |
|---|--------|---------|------|
| D1.1 | SDD 中引用的前端公共组件是否已存在 | HIGH | 缺失 → 启动 subagent 补充组件 |
| D1.2 | SDD 中引用的前端工具函数是否已存在 | HIGH | 缺失 → 启动 subagent 补充工具 |
| D1.3 | SDD 中引用的前端状态管理模块是否已存在 | HIGH | 缺失 → 启动 subagent 补充模块 |
| D1.4 | SDD 中引用的前端路由配置是否已存在 | MEDIUM | 缺失 → 启动 subagent 补充路由 |

### D2 后端依赖

| # | 检查项 | 严重级别 | 说明 |
|---|--------|---------|------|
| D2.1 | SDD 中每个 API 对应的后端 Controller 是否已存在 | BLOCKER | 缺失 → 启动 subagent 创建 Controller |
| D2.2 | SDD 中每个 API 对应的后端 Service 是否已存在 | HIGH | 缺失 → 启动 subagent 创建 Service |
| D2.3 | SDD 中引用的数据库表是否已存在 | BLOCKER | 缺失 → 启动 subagent 创建表结构 |
| D2.4 | SDD 中引用的枚举/常量是否已在后端定义 | MEDIUM | 缺失 → 启动 subagent 补充定义 |
| D2.5 | SDD 中依赖的第三方服务是否可调用 | HIGH | 缺失 → 记录为外部依赖风险 |

### D3 依赖修复优先级

| 优先级 | 依赖类型 | 说明 |
|--------|---------|------|
| P0 | 后端 API 未实现 | 前端无法开发，必须先补齐 |
| P1 | 数据库表未创建 | 后端 API 无法实现 |
| P2 | 前端公共组件缺失 | 前端可先 mock，但需尽快补充 |
| P3 | 枚举/常量未定义 | 可临时硬编码，后续统一补充 |

## 审核结果与 subagent 启动提示

审核完成后，根据发现类型生成对应的 subagent 启动提示：

| 发现类型 | 提示内容 | subagent 职责 |
|---------|---------|--------------|
| 规范文档问题 | "建议启动 subagent 修复规范文档" | 修改 SPEC.md / Companions |
| 后端 API 缺失 | "建议启动 subagent 创建后端接口" | 创建 Controller/Service/DAO |
| 后端命名违规 | "建议启动 subagent 修复后端 API 命名" | 重命名 API 路径 |
| 前端模块缺失 | "建议启动 subagent 补充前端模块" | 创建组件/工具/状态管理 |
| 数据库表缺失 | "建议启动 subagent 创建数据库表" | 编写 DDL 迁移脚本 |
