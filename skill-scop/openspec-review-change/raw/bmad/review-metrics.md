# SDD 审核指标与验收标准

## 门禁指标（BLOCKER 级）

进入开发阶段前必须全部满足：

| 指标 | 计算方式 | 阈值 | 说明 |
|------|---------|------|------|
| Spec Law 合规率 | 通过规则数 / 8 × 100% | = 100% | 8 条规则全部通过 |
| Capability 完整度 | 有 intent+success 的 CAP 数 / 总 CAP 数 × 100% | = 100% | 零缺失 |
| 悬空需求数 | 前端需要但后端未声明的 API 数 | = 0 | 契约双向完整 |
| 命名违规（CRITICAL） | 路径中含技术实现术语的 API 数 | = 0 | 如 blackList、cache、queue |
| 已有 API 冲突数 | 新增 API 与已有 API 路径冲突数 | = 0 | 运行时错误 |
| BLOCKER 发现总数 | 所有审核维度的 BLOCKER 发现数 | = 0 | 发现则阻塞 |

## 质量指标（HIGH 级）

Sprint 内必须修复：

| 指标 | 计算方式 | 阈值 | 说明 |
|------|---------|------|------|
| HIGH 发现数 | 所有审核维度的 HIGH 发现数 | ≤ 2 | 超出需升级审查 |
| 错误码覆盖率 | 有错误码声明的 API 数 / 总 API 数 × 100% | = 100% | 前端错误处理必需 |
| API 就绪状态标注率 | 有就绪状态的 API 数 / 总 API 数 × 100% | = 100% | 并行开发必需 |
| 前后端字段对齐率 | 字段名一致的 API 数 / 总 API 数 × 100% | = 100% | 避免联调试错 |
| 业务逻辑归属清晰率 | 归属无歧义的逻辑点数 / 总逻辑点数 × 100% | = 100% | 无前后端重复实现 |

## 健康指标（MEDIUM 级）

记录为技术债务，可在实现阶段迭代修复：

| 指标 | 计算方式 | 阈值 | 说明 |
|------|---------|------|------|
| MEDIUM 发现数 | 所有审核维度的 MEDIUM 发现数 | ≤ 5 | 技术债务 |
| 命名违规（一般） | 资源名非复数、路径含大写等 | ≤ 3 | 非阻塞但需记录 |
| 接口粒度偏差 | 过粗或过细的 API 数 | ≤ 2 | 后续迭代优化 |
| 术语不一致数 | 前后端对同一概念用不同术语的次数 | = 0 | 零容忍但归为 MEDIUM |
| 日期格式统一率 | 使用 ISO 8601 的 API 数 / 总 API 数 × 100% | ≥ 90% | 渐进统一 |

## 改进指标（LOW 级）

改进建议，不阻塞流程：

| 指标 | 计算方式 | 阈值 | 说明 |
|------|---------|------|------|
| 精简度 | 每 Capability 平均字数 | ≤ 50 字 | Spec Law Rule 8 |
| Companion 文件合理性 | Companion 数 / 总内容量 | 1-5 个 | 过多说明内核过载 |
| 可读性 | 文档结构清晰度评分 | ≥ 7/10 | 主观评估 |

## 已有 API 命名审计指标

针对后端已部分完成的场景，单独追踪：

| 指标 | 计算方式 | 阈值 | 说明 |
|------|---------|------|------|
| 命名违规总数 | 已有 API 中命名不规范的数量 | 记录 | 不阻塞本轮 |
| CRITICAL 命名违规 | 含歧视性/误导性术语的 API 数 | = 0 | 如 blackList，必须迁移 |
| 迁移方案覆盖率 | 有迁移方案的命名违规数 / 总违规数 × 100% | = 100% | CRITICAL 级必须 |

## 审核效率指标

| 指标 | 计算方式 | 目标 | 说明 |
|------|---------|------|------|
| 审核周期 | 从启动到通过的天数 | ≤ 2 天 | 避免审核阻塞开发 |
| 首次通过率 | 一次通过的审核数 / 总审核数 × 100% | ≥ 60% | 衡量 SDD 生成质量 |
| 复审轮数 | BLOCKER 修复后复审次数 | ≤ 2 | 修复质量 |
| 发现密度 | 总发现数 / SDD 字数 × 1000 | ≥ 2.0/千字 | 衡量审核深度 |

## 依赖满足指标

| 指标 | 计算方式 | 阈值 | 说明 |
|------|---------|------|------|
| 后端 API 满足率 | 已实现的 API 数 / SDD 声明的 API 数 × 100% | = 100% | 缺失 API = 启动 subagent |
| 数据库表满足率 | 已存在的表数 / SDD 引用的表数 × 100% | = 100% | 缺失表 = 启动 subagent |
| 前端组件满足率 | 已存在的组件数 / SDD 引用的组件数 × 100% | ≥ 90% | 缺失可 mock，但需补充 |
| 依赖阻塞项数 | P0 级依赖缺失数 | = 0 | P0 阻塞开发启动 |

## 门禁判定规则

```
Step 1: 规范审核
IF BLOCKER 发现数 > 0 THEN
  判定: REJECTED
  动作: 提示用户启动 subagent 修复规范文档 → 修复后重新审核
ELSE IF HIGH 发现数 > 2 THEN
  判定: CONDITIONAL
  动作: 提示用户启动 subagent 修复规范文档 → 修复后重新审核
ELSE
  进入 Step 2
END IF

Step 2: 依赖检查
IF 依赖阻塞项数 > 0 THEN
  判定: NEEDS_DEPENDENCIES
  动作: 提示用户启动 subagent 完善依赖模块 → 完成后重新审核
ELSE
  判定: APPROVED — 可进入开发阶段
END IF

完整判定:
  规范审核通过 (BLOCKER=0, HIGH≤2) AND 依赖全部满足 → APPROVED
  否则 → 需要迭代修复后重新审核
```

## subagent 启动提示模板

审核报告（VOTE.md）中必须包含以下提示：

```
## 修复建议

### 需要修复的规范文档问题（共 N 项）
- [BLOCKER] 问题描述 → 建议启动 subagent 修复规范文档
- [HIGH] 问题描述 → 建议启动 subagent 修复规范文档

### 需要完善的依赖模块（共 N 项）
- [P0] 后端 API 缺失: /xxx/yyy → 建议启动 subagent 创建后端接口
- [P1] 数据库表缺失: table_name → 建议启动 subagent 创建表结构
- [P2] 前端组件缺失: ComponentName → 建议启动 subagent 补充组件

### 需要修复的后端代码（共 N 项）
- [CRITICAL] API 命名违规: /blackList → 建议启动 subagent 修复后端 API 命名

所有问题修复后，请重新执行审核以确认"可开发"状态。
```
