# SDD 审核工作流

## 核心原则

**审核阶段只发现问题，不直接修改。** 所有修复通过用户启动 subagent 执行。审核是独立的质量关卡，与修复解耦。

## 审核阶段与开发阶段的边界

```
审核阶段（本文档定义）              开发阶段（本文档不涉及）
┌─────────────────────────┐      ┌─────────────────────────┐
│ 审核 → 确认 → 修复 → 再审核 │ ──→ │ 编码 → 测试 → 集成       │
└─────────────────────────┘      └─────────────────────────┘
         ↑                                    │
         │ SDD 判定"可开发"才可进入              │
         └────────────────────────────────────┘
```

## 审核角色

| 角色 | 职责 | 人数 |
|------|------|------|
| 审核 Agent | 执行全部审核维度，汇总发现，提示用户启动 subagent 修复 | 1（AI agent） |
| 用户 | 确认审核结果，决定是否启动 subagent 修复 | 1 |

> 注：在 AI agent 驱动的审核中，审核 Agent 同时承担前端/后端/架构审核员的角色，按清单逐项执行。

## 审核主流程

```
┌──────────────────────────────────────────────────────────────────┐
│                     SDD 审核主流程                                 │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Step 1: 收集                                                     │
│  ├─ 读取 SDD 文档（SPEC.md + Companions）                          │
│  ├─ 读取已有 API 清单（代码扫描）                                    │
│  ├─ 读取上游 PRD/GDD（如有）                                       │
│  └─ 输出: 审核输入确认                                              │
│                                                                  │
│  Step 2: 结构预审                                                  │
│  ├─ L1 结构完整性检查                                              │
│  ├─ L2 内容质量检查                                                │
│  ├─ BLOCKER → 记录，继续审核其余维度（不中断）                        │
│  └─ 输出: 结构审查报告                                              │
│                                                                  │
│  Step 3: 前端审核                                                  │
│  ├─ F1 页面-接口映射                                               │
│  ├─ F2 错误处理覆盖                                                │
│  ├─ F3 并行开发支持                                                │
│  ├─ F4 命名敏感度                                                  │
│  └─ 输出: 前端审核报告                                              │
│                                                                  │
│  Step 4: 后端审核                                                  │
│  ├─ B1 API 命名规范                                               │
│  ├─ B2 存量 API 兼容                                               │
│  ├─ B3 接口粒度                                                    │
│  ├─ B4 数据契约                                                    │
│  └─ 输出: 后端审核报告                                              │
│                                                                  │
│  Step 5: 跨端对齐                                                  │
│  ├─ C1 前后端对齐                                                  │
│  ├─ C2 术语一致性                                                  │
│  └─ 输出: 跨端对齐报告                                              │
│                                                                  │
│  Step 6: 依赖分析                                                  │
│  ├─ 识别 SDD 依赖的外部模块（前端其他模块、后端未实现接口）             │
│  ├─ 每个依赖项标注：归属模块、缺失内容、修复建议                       │
│  └─ 输出: 依赖清单                                                 │
│                                                                  │
│  Step 7: 汇总与判定                                                │
│  ├─ 合并所有审核报告                                               │
│  ├─ 按严重级别分类统计                                              │
│  ├─ 生成 VOTE.md                                                  │
│  └─ 输出: 审核报告 + 判定结果                                       │
│                                                                  │
│  Step 8: 提示用户                                                  │
│  ├─ 展示审核结果摘要                                               │
│  ├─ 有 BLOCKER? → 提示启动 subagent 修复规范文档                    │
│  ├─ 有依赖缺失? → 提示启动 subagent 完善依赖模块                    │
│  └─ 全部通过? → 判定"可开发"                                       │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 审核判定与用户动作

### 判定结果

| 判定 | 条件 | 用户动作 |
|------|------|---------|
| **REJECTED** | BLOCKER > 0 | 启动 subagent 修复规范文档，修复后重新审核 |
| **CONDITIONAL** | BLOCKER = 0 且 HIGH > 2 | 启动 subagent 修复规范文档，修复后重新审核 |
| **NEEDS_DEPENDENCIES** | 存在未满足的依赖项 | 启动 subagent 完善依赖模块，完成后重新审核 |
| **APPROVED** | BLOCKER = 0 且 HIGH ≤ 2 且依赖全部满足 | 进入开发阶段 |

## 三类修复与 subagent 启动提示

### 类型 1: 规范文档修复

**触发条件:** SDD 内容问题（结构不完整、Capability 缺失、命名违规、职责边界模糊等）

**提示模板:**
```
审核发现 [N] 个问题需要修复规范文档：
- BLOCKER: [列表]
- HIGH: [列表]

建议启动 subagent 修复规范文档。修复完成后重新审核。
```

**subagent 职责:** 修改 SPEC.md 和 Companion 文件，不修改代码

### 类型 2: 依赖模块完善

**触发条件:** SDD 依赖的功能模块缺失（前端其他模块未完成、后端接口未实现）

**提示模板:**
```
SDD 依赖以下模块，当前缺失：
- [模块名]: [缺失内容描述] — 建议启动 subagent 完善
- [模块名]: [缺失内容描述] — 建议启动 subagent 完善

依赖全部满足后重新审核。
```

**subagent 职责:** 在对应模块中补充缺失的功能代码或接口定义

### 类型 3: 后端代码修复

**触发条件:** 后端已有代码存在问题（命名违规 API、缺失接口等）

**提示模板:**
```
后端代码存在以下问题：
- [API 路径]: [问题描述] — 建议启动 subagent 修复后端代码
- [API 路径]: [问题描述] — 建议启动 subagent 修复后端代码

修复完成后重新审核。
```

**subagent 职责:** 修改后端代码，重命名 API、补充缺失接口等

## 审核模式

### 模式 A: 前后端均未开发

标准流程，Step 3-4 并行执行。SDD 是全新契约，无存量约束。
- 依赖分析主要检查：是否有依赖其他模块的设计意图

### 模式 B: 后端已部分完成

额外步骤：
1. **B2.1 前置** — 先盘点已有 API 清单，标记就绪状态
2. **B2.3 强制** — 新增 API 必须与已有 API 路径不冲突
3. **F3.1 强制** — 每个 API 必须标注就绪状态
4. **命名审计** — 已有 API 命名违规记录，审核完成后提示用户启动 subagent 修复
5. **依赖分析重点** — 前端发现的后端缺失接口，归入"依赖模块完善"

### 模式 C: 后端已全部完成

额外步骤：
1. **反向审核** — 从后端已有 API 出发，审核前端需求是否被完整覆盖
2. **接口缺失 → 依赖模块完善** — 前端需要但后端没有的 API，提示用户启动 subagent 补齐
3. **命名审计** — CRITICAL 级命名违规，提示用户启动 subagent 修复后端代码

## 审核产出物

| 产出物 | 内容 | 时机 |
|--------|------|------|
| 结构审查报告 | L1/L2 检查结果 | Step 2 |
| 前端审核报告 | F1-F4 发现列表 | Step 3 |
| 后端审核报告 | B1-B4 发现列表 | Step 4 |
| 跨端对齐报告 | C1-C2 发现列表 | Step 5 |
| 依赖清单 | 缺失的外部模块和接口 | Step 6 |
| VOTE.md | 最终审核判定 + 修复建议 + subagent 启动提示 | Step 7 |
| 技术债务清单 | 非阻塞问题记录 | Step 7 |

## "可开发"判定标准

SDD 同时满足以下全部条件时，判定为"可开发"：

1. **结构完整** — L1/L2 零 BLOCKER
2. **契约完整** — 前端零悬空需求，后端零路径冲突
3. **命名合规** — 零 CRITICAL 级命名违规
4. **职责清晰** — 零前后端逻辑归属歧义
5. **依赖满足** — 依赖清单中所有项已标记为"已解决"
6. **错误覆盖** — 每个 API 有错误码声明
7. **审核通过** — BLOCKER = 0 且 HIGH ≤ 2
