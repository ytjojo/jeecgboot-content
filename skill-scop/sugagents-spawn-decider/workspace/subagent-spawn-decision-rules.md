# Subagent 启动决策规则

> **范围约束**：本文档仅关注「何时启动 subagent」的判断规则，不涉及启动后的具体执行契约与验收标准等内容。

---

## Subagent 的本质定位

Subagent 解决的问题：

1. 上下文隔离
    避免主 agent 被大量无关探索、日志、代码细节污染。
2. 并行探索
    让多个 agent 从不同角度同时分析同一个问题。
3. 职责专门化
    让不同 agent 只关注安全、性能、测试、架构、兼容性等单一维度。
4. 文件级隔离实现
    让多个实现 agent 分别修改互不重叠的文件或模块。
5. 超大任务拆分
    当单个 agent 的上下文窗口不足以覆盖完整代码库、需求和修改计划时，subagent 是必要手段。

## 前提假设与平台能力依赖

本规则框架建立在以下平台能力假设之上，使用前请确认目标 agent 平台是否满足。

> 表格中引用的 Q1-Q6 代号对应第四节决策树中的判断问题，具体定义见决策树各节点。

| # | 假设 | 依赖的规则 | 失效影响 |
|---|------|-----------|---------|
| 1 | **上下文隔离**：subagent 上下文与主 agent 真正隔离，互不污染 | 本质定位第1点、决策树 Q2（上下文窗口是否足够）、Q5（上下文是否可完整自包含） | 「上下文减负」价值降低，拆分意义大打折扣 |
| 2 | **文件传递**：subagent 可将产出物（文件、报告）回传给主 agent 或其他 subagent | 决策树 Q4（执行顺序依赖）、串行 subagent 模式、阶段门流水线 | 串行 subagent 模式完全失效 |
| 3 | **并行执行**：平台支持多个 subagent 同时运行，无互斥锁或排队限制 | 决策树 Q6（并行收益临界点）、并行 subagent 模式 | 并行收益归零，所有 subagent 退化为串行 |
| 4 | **文件写入**：subagent 可执行文件修改操作，且修改结果可被主 agent 获取 | 本质定位第4点、并行写场景 | 所有实现型 subagent 不可用，仅保留只读模式 |
| 5 | **Spawn 开销可控**：启动 subagent 的延迟和 token 开销显著低于单 agent 串行完成全部工作 | 决策树 Q6 的收益临界点 | 「5 个单元 / 15 分钟」等阈值需重新校准，拆分可能得不偿失 |
| 6 | **质量不劣化**：subagent 基于部分上下文能产出质量不低于全量上下文的判断和修改。此假设在决策树 Q5（完整自包含上下文）通过后成立——若 Q5 未通过，此假设失效 | 决策树 Q5、并行审查型 subagent、并行实现型 subagent | 并行审查型 subagent 的发现可能不可靠，并行实现型 subagent 的代码可能需大量返工 |

> 若平台不支持上述任一能力，相应的 subagent 模式（串行/并行写/实现型）不可用，仅保留「并行只读」模式。假设 #6（质量不劣化）失效时，只读型 subagent 仍可用（仅探索/审查，不承担修改职责），但实现型 subagent 应禁用。

## 一、拆分决策的概念模型

```
拆分价值 = (独立性 × 并行度 × 上下文减负) − (同步成本 × 合并成本 × 冲突风险)
```

- **拆分价值 > 0** → 可以启动 subagent
- **拆分价值 ≤ 0** → 单 agent 完成

> **重要说明**：本公式是概念模型而非计算工具。六个变量（独立性、并行度、上下文减负、同步成本、合并成本、冲突风险）均为定性维度，无统一量纲。公式的作用是帮助思考拆分决策时需要权衡的因素，而非输出精确数值。具体的操作化判断请使用第四节决策树。
>
> **与决策树的关系**：决策树（第四节）将本模型中的六个维度转化为可执行的二元或阈值判断问题——Q1 检验规划完整性（独立性前提），Q2 检验上下文窗口（上下文减负），Q3-Q4 检验依赖关系（同步成本/合并成本），Q5 检验信息自足性（同步成本），Q6 检验规模临界点（并行度收益）。

## 二、必须启动 Subagent 的场景（硬条件）

以下任一条件成立时，**必须**启动 subagent：

| # | 条件 | 说明 |
|---|------|------|
| 1 | **单 agent 上下文窗口不足** | 完整代码库 + 需求 + 修改计划超出单 agent 承载上限 |
| 2 | **存在多个互不重叠的模块需并行修改** | 各模块文件范围无交集，无共享状态 |
| 3 | **需要并行只读审查** | 安全/性能/测试/架构等多维度同时审查，只读无竞争 |
| 4 | **需要竞争性分析** | bug 根因不明，需多角度并行提出假设 |

---

## 三、绝对不要启动 Subagent 的场景（红线）

以下任一条件成立时，**禁止**启动 subagent。红线优先于第二节的硬条件——若同一场景同时触发硬条件和红线，以红线为准。

> **优先级说明**：红线的设计意图是防止「为拆而拆」和「拆分带来的损害超过收益」。即使某个场景满足硬条件（如跨模块），若同时命中红线（如同文件修改），仍应禁止拆分。

| # | 条件 | 反例 |
|---|------|------|
| 1 | **同一文件被多个 agent 修改** | 3 个 agent 同时改 `UserService.java` |
| 2 | **强串行依赖** | A 的输出是 B 的输入，B 无法在 A 完成前开始任何工作 |
| 3 | **任务无法事先规划完整** | 探索性 debug、需要"看情况决定下一步"的诊断任务 |
| 4 | **拆分后需要频繁实时同步** | 各 agent 之间需要不断沟通中间状态 |
| 5 | **只为「看起来更工程化」而拆** | 形式大于实质 |

## 四、决策树（按顺序判断）

```
Q1: 任务能否在开始前被完整规划？
  ├── 否 → 不拆分，使用单 agent（需要探索性推理，无法预分配工作单元）
  └── 是 → 继续

Q2: 单 agent 上下文窗口是否足够？
  ├── 否 → 必须拆分为 subagent（硬条件，不拆就跑不完）
  └── 是 → 继续

Q3: 子任务之间是否需要共享中间状态（非文件传递）？
  ├── 是 → 不拆分，使用单 agent（只有文件传递才能解耦）
  └── 否 → 继续

    > **Q3 vs Q4 边界说明**：Q3 关注「是否需要在 agent 之间传递无法写成文件的信息」（如推理过程、隐式知识、设计意图），Q4 关注「产出物（文件）的上下游依赖顺序」。
    > 若下游 subagent 依赖上游的推理过程（而非仅消费其输出文件），应视为 Q3「是」——回归单 agent。仅当下游仅需上游的输出文件即可独立工作时，才进入 Q4 判断。

Q4: 子任务是否有明确的执行顺序依赖？
  ├── 是（有阶段门，输出是文件） → 启动 subagent：串行模式
  └── 否 → 继续

Q5: 拆分后每个 subagent 的上下文是否可以完整自包含？
  ├── 否 → 不拆分（公共背景复制成本超过并行收益）
  └── 是 → 继续

    「完整自包含上下文」的操作性判断标准（含事前检查清单与事后验证项）：

    事前检查清单（拆分前逐项确认）：
    - □ subagent 的 prompt 是否包含任务所需的全部文件内容（或可直接访问的文件路径）？
    - □ subagent 的 prompt 是否包含了所有相关的设计决策、约束条件和禁止修改项？
    - □ subagent 的 prompt 是否包含了明确的验收标准（知道「做到什么程度算完成」）？
    - □ subagent 是否已获得所有必要的设计背景和历史决策说明，使其在执行中不会因「为什么这样设计」「这段代码的历史背景」的疑问而阻塞？
    - □ 如果 subagent 执行中需要更多上下文，是否有明确的降级渠道（回退到主 agent / 读取指定文件）？

    事后验证（执行完成后复盘）：
    - 每个 subagent 是否能仅凭接收到的输入（任务描述 + 文件/数据）独立完成工作，无需额外查询或回溯主 agent 状态？
    - 换一个不熟悉原始任务背景的 agent，仅凭分配给它的输入材料，能否正确理解任务目标并执行？
    - 如果任一 subagent 的输出中包含「需要先了解 X」「需要查看 Y 的上下文」等回溯请求，则说明上下文未自包含

Q6: 并行单元数量 > 收益临界点？
  ├── < 5 个单元 且 每个 < 15 分钟工作量 → 不拆分（协调成本超过并行收益）
  └── ≥ 5 个单元 或 每个 ≥ 15 分钟工作量 → 启动 subagent：并行模式

    > **阈值说明**：「5 个单元」「15 分钟」为初始参考值，需根据实际平台 spawn 开销和团队经验校准。若平台 spawn 延迟较高（如每次启动 > 30 秒），应相应上调单元数或工作量阈值。补充规则：若总工作量 > 60 分钟且单元数 ≥ 2，即使不满足右侧条件也应纳入拆分考量（大任务即使单元数少，并行仍有收益）。
```

## 五、场景速查表

### ✅ 应该启动 Subagent

| 场景 | 模式 | 触发条件 |
|------|------|----------|
| 多维度代码审查 | 并行只读 | 需安全/性能/测试/架构分工审查。注意：各维度并非完全独立——安全分析可能依赖性能上下文，架构审查的结论可能有安全含义。建议汇总阶段对跨维度发现进行交叉验证 |
| 独立模块实现 | 并行写 | 模块边界清晰，文件范围不重叠（如 `auth/`、`payment/`、`order/`） |
| 复杂 bug 定位 | 并行只读 | 根因不明，需从变更历史/日志/数据流/依赖多角度分析 |
| 大代码库探索 | 并行只读 | 需扫多个子系统，输出小、输入大 |
| 多方案比较 | 并行只读 | 架构方案 A/B/C 独立探索后汇总 |
| 阶段门流水线 | 串行 | 需求分析 → 接口设计 → 实现 → 测试，每阶段产出具体文件 |
| 大规模测试补齐 | 并行写 | 为 N 个独立模块分别生成测试 |
| Review 后按文件并行修复 | 并行写 | Review 已完成，问题按文件隔离，无同文件冲突 |

### 🚫 不应该启动 Subagent

| 场景 | 原因 |
|------|------|
| 小 bug 修复 | 不值得拆 |
| 同一模块内聚修改 | 修改集中在同一文件或同一类 |
| Review → Impl 的低上下文流水线 | Review 只输出 JSON 摘要，Impl 需要重新读代码推理 |
| 配置文件修改 | 多处修改同一 `pom.xml`/`build.gradle`/`application.yml` |
| 数据库 schema 变更 | 多人改同一 schema 必定冲突 |
| 探索性调试 | 需要"看一步走一步"，无法预分配 |

## 六、Review → Impl 模式的特殊判断

这是最容易误用的模式，单独列出判断规则。

### 可以 Review → Impl 拆分的条件（ALL 必须满足）

1. Review agent **只读**
2. Impl agent 拥有**完整自包含的上下文**（不只是 JSON 摘要）
3. Impl 可以按**文件或模块并行**
4. Review 输出**可执行**（包含函数锚点、调用链、禁止修改项、验收标准）
5. 单 agent 上下文**不足**

### 禁止 Review → Impl 拆分的条件（任一成立即禁止）

1. 中小规模任务（< 10 文件）
2. 同一批文件既要 review 又要修改
3. Review 输出只是 `{file, line, desc}` 精简格式
4. 多个 Impl agent 会修改同一文件
5. 修复方案高度依赖 reviewer 的完整上下文理解

### 判断口诀

> Review → Impl 能否拆分的核心判断标准，与决策树 Q5「完整自包含上下文」的检验逻辑一致：如果一个不熟悉这段代码的工程师，仅凭 Review 报告就能正确、安全地完成修改 → 可以拆；如果他还必须重新读大量代码才能知道怎么改 → 不要拆。具体的操作性判断标准见决策树 Q5 下方的三个检验问题。

## 七、启动前快速检查清单

启动任何 subagent 前，逐项确认：

```
□ Q1: 这个任务是否真的需要 subagent？（不是为拆而拆）
□ Q2: 任务是否可以明确拆分？（每个 subagent 的边界清晰）
□ Q3: 每个 subagent 的输入是否完整自包含？（对照 Q5 事前检查清单逐项确认）
□ Q4: 每个 subagent 的输出是否可验证？（非决策树判断项，为拆分后质量门检查）
□ Q5: 是否存在多个 agent 修改同一文件？（是 → 不拆）
□ Q6: 是否存在强串行依赖？（是 → 不拆并行）
□ Q7: 是否会因为拆分造成上下文损耗？
□ Q8: 主 agent 是否保留最终决策权？
```

### 快速否决规则

以下任一条件成立，**优先不拆**：

- 任务很小（< 3 文件，< 30 行修改）
  > 注意：30 行阈值为启发式参考值，核心判断标准是「协调成本是否超过修改本身的工作量」。高风险修改（数据迁移、schema 变更、安全修补）即使 < 30 行也应单独评估是否需拆分。
- 修改集中在同一文件（与 Section 三 条件 #1「同一文件被多个 agent 修改」对应）
- 修复依赖完整上下文
- agent 之间必须严格串行（与 Section 三 条件 #2「强串行依赖」对应）
- review 输出不足以直接指导修改
- 只是为了「看起来更工程化」而拆分（与 Section 三 条件 #5 对应）

## 八、拆分模式选择速查

| 拆分维度 | 适用场景 | 示例 |
|----------|----------|------|
| **按资源拆分** | 文件/目录隔离 | `frontend/`、`backend/`、`docs/` |
| **按领域拆分** | 业务模块独立 | 认证、支付、订单、消息 |
| **按维度拆分** | 审查维度独立 | 性能、安全、可维护性、可测试性。注意：维度间存在交叉关注点，汇总时需交叉验证 |
| **按方案拆分** | 多方案探索 | 方案 A、方案 B、方案 C 独立探索 |
| **按阶段拆分** | 阶段门依赖 | 需求分析 → 接口设计 → 实现 → 测试 |

## 九、核心原则（总结）

1. **并行 review → 可以拆**
2. **串行修复 → 慎重拆**
3. **同文件修改 → 不要拆**
4. **跨模块独立实现 → 可以拆**
5. **中小任务 → 单 agent 通常更好**
6. **只读 subagent 最安全**
7. **实现型 subagent 必须文件级隔离**
8. **上下文隔离 > 并行执行 > 能力分工**（这是 subagent 真正的价值排序）
9. **subagent 的目标不是增加 agent 数量，而是降低任务复杂度**
10. **如果拆分后上下文损耗、冲突风险和协调成本高于收益，就不应该拆**
11. **阈值需根据实际使用反馈校准** — 30 行、10 文件、5 单元、15 分钟等阈值是初始参考值，应在实际使用中记录 spawn 决策的结果（成功/失败/收益），定期回溯调整。有效性评估标准见第十节。

---

## 十、Spawn 决策有效性评估标准

用于原则 #11 中「记录 spawn 决策的结果（成功/失败/收益）」的操作性定义。

### 成功标准（全部满足即为成功）

- 所有 subagent 在预期时间内完成各自子任务，未超时或崩溃
- 主 agent 汇总 subagent 产出时无需额外补充上下文或重新推理
- 拆分总耗时（含 spawn 开销 + 主 agent 汇总时间）≤ 单 agent 预估耗时
- 各 subagent 产出之间无冲突，或冲突在汇总阶段无需额外返工即解决

### 失败标准（任一成立即为失败）

- 任一 subagent 产出不可用（错误、不完整、偏离任务目标），需主 agent 或其他 subagent 重新完成
- 合并阶段出现文件级冲突（同一文件被多个实现型 subagent 修改）需要人工介入
- 拆分总耗时 > 单 agent 预估耗时（即拆分未能产生并行收益）
- 汇总阶段发现 subagent 产出之间逻辑矛盾，需要重新调度

### 收益度量

- **时间收益**: 单 agent 预估耗时 − 拆分总耗时（正值表示有收益）
- **质量收益**: 多维度审查是否发现了单 agent 遗漏的问题（只读型拆分）
- **上下文收益**: 主 agent 上下文占用率降低幅度（通过拆分将探索/实现细节移出主上下文）

> **使用方式**：每次 spawn 决策执行完毕后，按上述标准标记成功/失败并记录关键度量。每 10 次决策或每个迭代周期回溯分析，据此校准决策树阈值和红线条件。
