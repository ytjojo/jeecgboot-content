# Subagent 启动决策规则

> **范围约束**：本文档仅关注「何时启动 subagent」的判断规则，不涉及启动后的具体执行契约与验收标准等内容。

---

## 一、核心决策公式

```
拆分价值 = (独立性 × 并行度 × 上下文减负) − (同步成本 × 合并成本 × 冲突风险)
```

- **拆分价值 > 0** → 可以启动 subagent
- **拆分价值 ≤ 0** → 单 agent 完成

---

## 二、必须启动 Subagent 的场景（硬条件）

以下任一条件成立时，**必须**启动 subagent：

| # | 条件 | 说明 |
|---|------|------|
| 1 | **单 agent 上下文窗口不足** | 完整代码库 + 需求 + 修改计划超出单 agent 承载上限 |
| 2 | **存在多个互不重叠的模块需并行修改** | 各模块文件范围无交集，无共享状态 |
| 3 | **需要并行只读审查** | 安全/性能/测试/架构等多维度同时审查，只读无竞争 |
| 4 | **需要竞争性分析** | bug 根因不明，需多角度并行提出假设 |

---

## 三、绝对不要启动 Subagent 的场景（红线）

以下任一条件成立时，**禁止**启动 subagent：

| # | 条件 | 反例 |
|---|------|------|
| 1 | **同一文件被多个 agent 修改** | 3 个 agent 同时改 `UserService.java` |
| 2 | **强串行依赖** | A 的输出是 B 的输入，B 无法在 A 完成前开始任何工作 |
| 3 | **修改量 < 30 行** | 小修小补拆 agent 是浪费 |
| 4 | **任务无法事先规划完整** | 探索性 debug、需要"看情况决定下一步"的诊断任务 |
| 5 | **拆分后需要频繁实时同步** | 各 agent 之间需要不断沟通中间状态 |
| 6 | **只为「看起来更工程化」而拆** | 形式大于实质 |

---

## 四、决策树（按顺序判断）

```
Q1: 任务能否在开始前被完整规划？
  ├── 否 → 🚫 单 agent（需要探索性推理，无法预分配工作单元）
  └── 是 → 继续

Q2: 单 agent 上下文窗口是否足够？
  ├── 否 → ✅ 必须拆分（这是硬条件，不拆就跑不完）
  └── 是 → 继续

Q3: 子任务之间是否需要共享中间状态（非文件传递）？
  ├── 是 → 🚫 单 agent（只有文件传递才能解耦）
  └── 否 → 继续

Q4: 子任务是否有明确的执行顺序依赖？
  ├── 是（有阶段门，输出是文件） → ✅ 串行 subagent
  └── 否 → 继续

Q5: 拆分后每个 subagent 的上下文是否可以完整自包含？
  ├── 否 → 🚫 不拆（公共背景复制成本超过并行收益）
  └── 是 → 继续

Q6: 并行单元数量 > 收益临界点？
  ├── < 5 个单元 且 每个 < 5 分钟工作量 → 🚫 不拆
  └── ≥ 5 个单元 或 每个 > 15 分钟工作量 → ✅ 并行 subagent
```

## 五、场景速查表

### ✅ 应该启动 Subagent

| 场景 | 模式 | 触发条件 |
|------|------|----------|
| 多维度代码审查 | 并行只读 | 需安全/性能/测试/架构分工审查 |
| 独立模块实现 | 并行写 | 模块边界清晰，文件范围不重叠（如 `auth/`、`payment/`、`order/`） |
| 复杂 bug 定位 | 并行只读 | 根因不明，需从变更历史/日志/数据流/依赖多角度分析 |
| 大代码库探索 | 并行只读 | 需扫多个子系统，输出小、输入大 |
| 多方案比较 | 并行只读 | 架构方案 A/B/C 独立探索后汇总 |
| 阶段门流水线 | 串行 | 需求分析 → 接口设计 → 实现 → 测试，每阶段产出具体文件 |
| 大规模测试补齐 | 并行写 | 为 N 个独立模块分别生成测试 |
| Review 后按文件并行修复 | 并行写 | Review 已完成，问题按文件隔离，无同文件冲突 |

### 🚫 不应该启动 Subagent

| 场景 | 原因 |
|------|------|
| 小 bug 修复 | 不值得拆 |
| 同一模块内聚修改 | 修改集中在同一文件或同一类 |
| Review → Impl 的低上下文流水线 | Review 只输出 JSON 摘要，Impl 需要重新读代码推理 |
| 配置文件修改 | 多处修改同一 `pom.xml`/`build.gradle`/`application.yml` |
| 数据库 schema 变更 | 多人改同一 schema 必定冲突 |
| 探索性调试 | 需要"看一步走一步"，无法预分配 |

## 六、Review → Impl 模式的特殊判断

这是最容易误用的模式，单独列出判断规则。

### 可以 Review → Impl 拆分的条件（ALL 必须满足）

1. Review agent **只读**
2. Impl agent 拥有**完整自包含的上下文**（不只是 JSON 摘要）
3. Impl 可以按**文件或模块并行**
4. Review 输出**可执行**（包含函数锚点、调用链、禁止修改项、验收标准）
5. 单 agent 上下文**不足**

### 禁止 Review → Impl 拆分的条件（任一成立即禁止）

1. 中小规模任务（< 10 文件）
2. 同一批文件既要 review 又要修改
3. Review 输出只是 `{file, line, desc}` 精简格式
4. 多个 Impl agent 会修改同一文件
5. 修复方案高度依赖 reviewer 的完整上下文理解

### 判断口诀

> 如果一个不熟悉这段代码的工程师，仅凭 Review 报告就能正确、安全地完成修改 → 可以拆。
> 如果他还必须重新读大量代码才能知道怎么改 → 不要拆。

## 七、启动前快速检查清单

启动任何 subagent 前，逐项确认：

```
□ Q1: 这个任务是否真的需要 subagent？（不是为拆而拆）
□ Q2: 单 agent 是否已经足够？
□ Q3: 任务是否可以明确拆分？（每个 subagent 的边界清晰）
□ Q4: 每个 subagent 的输入是否完整自包含？
□ Q5: 每个 subagent 的输出是否可验证？
□ Q6: 是否存在多个 agent 修改同一文件？（是 → 不拆）
□ Q7: 是否存在强串行依赖？（是 → 不拆并行）
□ Q8: 是否会因为拆分造成上下文损耗？
□ Q9: 主 agent 是否保留最终决策权？
```

### 快速否决规则

以下任一条件成立，**优先不拆**：

- 任务很小（< 3 文件，< 30 行修改）
- 修改集中在同一文件
- 修复依赖完整上下文
- agent 之间必须严格串行
- review 输出不足以直接指导修改
- 只是为了「看起来更工程化」而拆分

## 八、拆分模式选择速查

| 拆分维度 | 适用场景 | 示例 |
|----------|----------|------|
| **按资源拆分** | 文件/目录隔离 | `frontend/`、`backend/`、`docs/` |
| **按领域拆分** | 业务模块独立 | 认证、支付、订单、消息 |
| **按维度拆分** | 审查维度独立 | 性能、安全、可维护性、可测试性 |
| **按方案拆分** | 多方案探索 | 方案 A、方案 B、方案 C 独立探索 |
| **按阶段拆分** | 阶段门依赖 | 分析 → 设计 → 实现 → 验证 |

---

## 九、核心原则（总结）

1. **并行 review → 可以拆**
2. **串行修复 → 慎重拆**
3. **同文件修改 → 不要拆**
4. **跨模块独立实现 → 可以拆**
5. **中小任务 → 单 agent 通常更好**
6. **只读 subagent 最安全**
7. **实现型 subagent 必须文件级隔离**
8. **上下文隔离 > 并行执行 > 能力分工**（这是 subagent 真正的价值排序）
9. **subagent 的目标不是增加 agent 数量，而是降低任务复杂度**
10. **如果拆分后上下文损耗、冲突风险和协调成本高于收益，就不应该拆**
