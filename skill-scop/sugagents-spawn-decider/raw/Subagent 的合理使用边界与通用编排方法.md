Claude Code / Codex 中 Subagent 的合理使用边界与通用编排方法

⸻

1. Subagent 的本质定位

1.1 Subagent 不是“小号 Agent”

很多团队对 Subagent 的理解存在偏差：

错误理解：

主 Agent 做不完 → 多开几个 Agent 一起做

正确理解：

Subagent 是一种任务隔离机制（Task Isolation Mechanism）

其核心目标不是增加 Agent 数量，而是：

* 降低上下文复杂度
* 降低任务耦合度
* 提高并行度
* 提高结果可验证性
* 降低主 Agent 的认知负载

⸻

1.2 Subagent 的本质能力

本质上：

主 Agent
    ↓
任务分解
    ↓
多个独立上下文
    ↓
结果汇总

Subagent 最重要的价值：

上下文隔离
＞并行执行
＞能力分工

很多时候即使不并行，仅仅因为上下文隔离，就已经值得拆分。

⸻

1.3 Subagent 解决的问题

问题1：上下文污染

例如：

需求分析
架构设计
数据库设计
前端实现
测试设计

全部放一个上下文：

Context = 巨大混合体

Agent 很容易：

* 忘记前文
* 关注点漂移
* 错误引用信息

⸻

问题2：角色冲突

例如：

设计者
实现者
验证者

放在同一个 Agent：

自己设计
自己实现
自己验证

容易形成确认偏差。

⸻

问题3：任务复杂度爆炸

任务规模增长时：

复杂度 ≠ 线性增长
复杂度 ≈ n²

此时需要拆分上下文。

⸻

2. 什么任务适合拆成 Subagent

2.1 高独立性任务

标准：

一个任务完成
不依赖另一个任务实时结果

例如：

模块A分析
模块B分析
模块C分析

适合。

⸻

2.2 文件级隔离任务

例如：

frontend/
backend/
docs/

每个 Subagent 负责不同目录。

适合。

⸻

2.3 大规模扫描任务

例如：

代码审计
技术债统计
依赖分析
日志分析

特点：

输入大
输出小

非常适合 Subagent。

⸻

2.4 多方案探索任务

例如：

架构方案A
架构方案B
架构方案C

多个 Subagent 独立探索。

最后汇总。

⸻

2.5 多维度评审任务

例如：

性能
安全
可维护性
可测试性
扩展性

天然适合拆分。

⸻

3. 什么任务不适合拆成 Subagent

⸻

3.1 强串行依赖任务

例如：

Step1 输出
必须成为 Step2 输入

链路：

A → B → C → D

拆分收益极低。

⸻

3.2 高频交互任务

例如：

每2分钟需要同步一次状态

此时：

协调成本 > 执行成本

不适合。

⸻

3.3 小任务

例如：

修改一个函数
修一个Bug
改一个文案

不值得拆。

⸻

3.4 同一文件多人修改

例如：

5个 Subagent
同时修改 UserService.java

属于灾难模式。

⸻

3.5 无法定义边界的任务

如果无法回答：

负责什么
不负责什么
何时完成

不要拆。

⸻

4. 如何判断拆分收益是否大于协调成本

四维判断模型

维度1：任务规模

小 = 不拆
大 = 倾向拆

⸻

维度2：任务独立性

独立度高 → 拆
独立度低 → 不拆

⸻

维度3：上下文体积

Context > 单Agent舒适区

倾向拆。

⸻

维度4：结果可合并性

如果最终结果：

容易汇总

适合拆。

如果：

最终需要大量人工整合

不适合拆。

⸻

快速决策公式

拆分价值
=
(独立性 × 并行度 × Context减负)
-
(同步成本 × 合并成本 × 冲突风险)

若结果大于零：

可以拆

统一格式：

建议:

⸻

权限（Permission）

建议四级模型：

Level 0

只读

⸻

Level 1

允许生成文件

⸻

Level 2

允许修改指定文件

⸻

Level 3

允许提交结果

⸻

必须可验证。

- 问题列表

- 修复建议

7. 如何避免常见问题

⸻

避免上下文损耗

不要传：

请参考前面讨论

应该传：

明确摘要

例如：

背景:
目标:
约束:
现状:

⸻

避免任务漂移

增加：

不在职责范围内的问题
记录即可
不要处理

⸻

避免越权执行

明确：

允许:
禁止:

例如：

允许:
  修改 docs/
禁止:
  修改 src/

⸻

避免重复工作

建立：

Ownership

例如：

A:
  frontend/
B:
  backend/
C:
  docs/

唯一负责人。

⸻

避免结果冲突

要求：

一个资源
一个Owner

不要：

多Agent同时改同文件

最终原则

Subagent 编排的核心不是：

如何使用更多 Agent

而是：

如何让每个 Agent
拥有最小上下文、
明确边界、
唯一职责、
可验证输出、
可组合结果。

当一个任务无法清晰定义：

输入
边界
权限
输出
完成标准

时，不应该创建 Subagent；

当这些要素能够被严格定义时，Subagent 才会真正带来规模化收益，而不是制造新的复杂度。
