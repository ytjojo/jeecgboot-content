## 二、CSO 审核

### 2.1 description 字段

```
当前内容:
"以证据为先的已完成变更回顾。对技术实现、需求实现、代码实现的完整流程进行复盘，
生成结构化回顾报告。openspec change 优先，也支持通用开发任务。
触发：/apply-retrospective、'复盘'、'retrospective'、openspec apply 完成后自动调用。"
```

**判定：WARN — 部分违反 CSO 规则**

| 检查项 | 结果 | 说明 |
|--------|------|------|
| 以 "Use when..." 开头 | ❌ | 以中文「以证据为先的…」开头 |
| 仅描述触发条件 | ⚠️ | 「生成结构化回顾报告」是工作流摘要 |
| 第三人称 | ✅ | 无第一人称 |
| 触发条件具体 | ✅ | 列出了具体触发词和场景 |

**问题分析**：description 中「生成结构化回顾报告」描述了技能**做什么**而非**何时使用**。根据 writing-skills 的 CSO 规则，这可能导致 AI 跳过完整技能内容，仅按 description 摘要执行。

**建议修改**：
```yaml
description: "Use when a completed change needs retrospective review — analyzing process quality across requirements, technical decisions, and implementation. Trigger on: /apply-retrospective, '复盘', 'retrospective', '回顾变更', or after openspec apply phase completes. Supports openspec changes (full mode) and general development tasks (degraded mode)."
```

## 十、审核结论

`apply-retrospective` 是一个**高质量、结构完善**的技能。它的核心优势在于：

1. **防理性化设计出色** — Red Flags + 借口对照表 + 禁止理由清单三层防护，是 writing-skills 方法论的最佳实践
2. **边界意识极强** — 越界红线表具体可操作，与 code-review / openspec-review 的边界清晰
3. **双模式设计合理** — openspec 完整模式 + 通用降级模式，覆盖不同成熟度的使用场景
4. **长期学习机制** — Promote candidates + Carry-forward 将单次复盘转化为持续改进

**主要改进点**仅 CSO description 字段需修正，其余为锦上添花。**建议修复 description 后即可正式使用。**
