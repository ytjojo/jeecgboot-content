SubAgent 约束框架终版评审与改进报告

文档定位

本文档评审对象为《SubAgent 约束系统性方案》。

评审目标不是构建 Agent OS、企业级 Agent 治理平台或多 Agent 基础设施，而是：

在 Claude Code、Codex、Cursor Agent、Gemini CLI 等成熟 Agent 系统中，作为使用者，如何定义 SubAgent 规范，使其执行更稳定、更可控、更符合目标，同时不过度增加主 Agent 的管理成本。

因此评审标准采用：

* 使用者视角
* 实战效率优先
* 最小必要约束
* 避免过度工程化

⸻

总体结论

当前文档整体方向正确。

其核心价值在于：

限制 Agent 的搜索空间
而不是控制 Agent 的全部行为

对于 Claude Code、Codex 等成熟系统而言：

Agent 本身已经具备：

* Context 管理
* Tool 调用
* 文件修改记录
* Git 回滚能力
* Token 管理
* 执行日志
* 多 Agent 调度

因此：

SubAgent 框架的职责
≠ 重新实现 Agent 平台
SubAgent 框架的职责
= 约束任务边界

这一定位是正确的。

⸻

当前框架的优点

1. 分层结构合理

当前五层模型：

权限边界
↓
作用域边界
↓
任务契约
↓
输出契约
↓
协调规则

符合实际 Agent 工作流程。

其中：

* 前三层定义 SubAgent 边界
* 后两层定义 Main Agent 管理方式

职责清晰。

⸻

2. 强调 Scope 优于 Prompt

文档最有价值的部分是：

路径白名单
作用域边界

实际项目中：

Agent 失控最常见原因并不是：

任务目标不清晰

而是：

为了修复 A
顺手修改 B
然后影响 C

因此：

Scope > Prompt

这一设计应当保留。

⸻

3. 强调升级机制

当前定义：

完成
完成但有疑虑
受阻
需要信息

符合 Agent 实际工作状态。

相比简单的：

成功
失败

更具有可操作性。

⸻

4. 强调主 Agent 最终裁决权

文档明确：

质量裁决
规则修改
跨任务冲突
外部影响操作

保留在 Main Agent。

这是正确设计。

否则容易出现：

SubAgent 自行修改架构
SubAgent 自行修改需求
SubAgent 自行决定发布

最终导致系统漂移。

⸻

评审意见采纳情况

应采纳的问题

一、升级机制与 STOP 条件关系不清

当前存在：

升级机制
STOP 条件

两套规则。

但缺少关系说明。

建议明确：

STOP 是停止执行条件
升级机制是停止后的汇报状态

例如：

触发 STOP
↓
选择状态
完成
完成但有疑虑
受阻
需要信息

这样逻辑完整。

⸻

二、缺少 SubAgent 协作模式

当前框架默认：

SubAgent 独立完成任务

适用于：

80%
单任务

但复杂项目经常出现：

Agent A
↓
Agent B
↓
Agent C

因此建议增加：

协作模式

串行协作

A 输出
↓
Main Agent Review
↓
B 输入

并行协作

Main Agent
├─ Agent A
├─ Agent B
└─ Agent C

冲突原则

多个 Agent
不得修改同一文件

如需修改：

由 Main Agent 合并

⸻

三、缺少重试策略

当前只有：

受阻
重新派发

建议补充：

受阻
↓
分析原因
↓
调整任务
↓
重新派发

并明确：

不要原样重派发

否则容易进入循环失败。

⸻

四、缺少实际案例

建议增加：

* 编程案例
* 文档案例
* 调研案例

帮助使用者理解如何落地。

⸻

不建议采纳的意见

一、删除路径白名单

部分评审认为：

默认允许整个项目

即可。

不建议采纳。

原因：

路径白名单是整个框架最核心的约束。

如果允许：

整个仓库

则：

作用域边界
基本失效

⸻

二、采用探索-确认-执行模式

部分评审建议：

先探索
再申请权限
再执行

不建议作为默认模式。

原因：

对于：

90%
普通任务

会增加大量往返成本。

应作为特殊模式存在：

任务范围未知时使用

而不是默认工作流。

⸻

三、合并五层结构

部分评审建议：

5层
↓
3层

不建议采纳。

原因：

当前五层虽然略显冗长。

但职责明确：

权限 ≠ 范围
契约 ≠ 输出
协调 ≠ 权限

合并后会导致概念混淆。

⸻

四、削弱约束提高灵活性

部分观点认为：

现代 LLM 足够可靠

因此不需要严格约束。

不建议采纳。

实际生产经验表明：

Agent 常见失败模式包括：

任务漂移
范围扩张
过度重构
自行假设
顺手优化

约束的目的不是限制能力。

而是：

防止已知失败模式

如果目标是成熟 Agent 系统。

建议将框架进一步聚焦。

最终结论

这份框架最适合作为：

Claude Code
Codex
Cursor Agent
Gemini CLI

等成熟 Agent 系统中的 SubAgent 约束规范。

其核心思想应保持不变：

限制搜索空间
而非控制全部行为
约束危险操作
而非约束所有操作
保持 Main Agent 裁决权
而非追求 SubAgent 自治

在此基础上补充：

协作机制
冲突规则
重试机制
STOP 与状态关系

即可形成一套兼顾：

* 可执行性
* 低管理成本
* 高稳定性
* 高适配性

的 SubAgent 约束体系。
