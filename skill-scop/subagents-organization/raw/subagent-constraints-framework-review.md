# subagent 约束框架审查报告

审查对象：`subagent-constraints-framework.md`
审查时间：2026-06-21
审查模式：compound-engineering 四角色审查

---

## 一、内部一致性审查 (ce-coherence-reviewer)

### 审查维度
- 文档内部概念是否一致
- 层级之间的关系是否清晰
- 是否存在自相矛盾

### 发现的问题

#### 问题 1：第 1 层与第 5 层的边界模糊
**严重程度**：中

**问题描述**：
- 第 1 层（权限边界）定义了"禁止操作"，包括"外部影响：git push"
- 第 5 层（协调规则）定义了"绝不委托：外部影响操作 push、merge、发布、通知"
- 两处内容高度重叠，职责边界不清晰

**建议**：
- 第 1 层聚焦于"subagent 不能做什么"（行为约束）
- 第 5 层聚焦于"主 agent 保留什么权力"（决策权分配）
- 删除第 5 层中与第 1 层重复的内容，或明确说明第 5 层是"为什么这些操作不能委托"的理由

#### 问题 2：read-only subagent 约束位置不当
**严重程度**：低

**问题描述**：
- read-only subagent 的输出边界定义在第 1 层（权限边界）
- 但"输出边界"本质上是输出契约，应该属于第 4 层
- 当前位置容易造成混淆：这是"禁止操作"还是"输出格式"？

**建议**：
- 将 read-only subagent 约束移到第 4 层，作为"特殊输出格式"的子节
- 或者在第 1 层只保留"禁止修改文件"，将"输出边界"移到第 4 层

#### 问题 3：升级机制与 STOP 触发条件的关系不清
**严重程度**：中

**问题描述**：
- 第 3 层定义了"升级机制"（4 种情况）和"STOP 触发条件"（4 种情况）
- 两者的关系不明确：是互斥的？还是并行的？
- "完成但有疑虑"属于升级机制，但是否也触发 STOP？
- "反复失败"触发 STOP，但应该报告什么状态？

**建议**：
- 明确说明：升级机制是"状态分类"，STOP 触发条件是"何时停止"
- 或者合并为一个表格：情况 → 行动 → 报告状态

#### 问题 4：派发模板与检查清单的对应关系
**严重程度**：低

**问题描述**：
- 派发模板中有"升级规则"部分，列出了 3 种情况
- 但第 3 层的升级机制有 4 种情况（缺少"完成，符合预期"）
- 检查清单中没有验证"升级规则是否完整"

**建议**：
- 派发模板的"升级规则"应该引用第 3 层的完整定义，而非重复列举
- 或者在模板中列出所有 4 种情况

### 一致性评分：7/10

**优点**：
- 5 层结构清晰，层层递进
- 每层都有明确的核心原则

**不足**：
- 部分概念在不同层级重复出现
- 升级机制与 STOP 条件的关系需要澄清

## 二、完整性与范围审查 (ce-scope-guardian-reviewer)

### 审查维度
- 是否遗漏关键内容
- 范围是否合理
- 是否有冗余

### 发现的问题

#### 问题 1：缺少错误处理和重试机制
**严重程度**：高

**问题描述**：
- 文档只定义了"何时停止"，但没有定义"停止后怎么办"
- 派发后跟踪中提到了"重派发"，但没有定义重试策略
- 缺少：最大重试次数、重试间隔、重试时的上下文调整

**建议**：
- 在第 3 层增加"重试策略"子节
- 或者在派发检查清单中增加"重试策略是否明确"

#### 问题 2：缺少 subagent 之间的协作机制
**严重程度**：中

**问题描述**：
- 文档假设 subagent 是独立的，没有定义 subagent 之间如何协作
- 如果 subagent A 的输出是 subagent B 的输入，如何传递？
- 如果 subagent A 和 B 需要共享中间结果，如何处理？

**建议**：
- 增加"协作模式"子节，定义：
  - 串行协作：A 的输出写入文件，B 读取
  - 并行协作：通过主 agent 中转
  - 共享状态：通过 ledger 或共享文件

#### 问题 3：缺少性能优化建议
**严重程度**：低

**问题描述**：
- 文档没有定义如何选择 subagent 的数量
- 没有定义何时使用并行 vs 串行
- 没有定义如何平衡 subagent 的粒度（太粗 vs 太细）

**建议**：
- 增加"性能优化"子节，定义：
  - 任务分解原则：独立性 > 依赖关系
  - 并行度建议：3-5 个 subagent 为宜
  - 粒度平衡：单个 subagent 工作量 5-15 分钟

#### 问题 4：缺少实际案例
**严重程度**：中

**问题描述**：
- 文档提供了通用模板，但没有实际案例
- 读者难以理解如何将框架应用到实际场景
- 缺少：编程场景、写作场景、研究场景的完整示例

**建议**：
- 增加"实际案例"章节，提供 2-3 个完整示例
- 或者在 `111.md` 中补充案例，并在此文档中引用

### 完整性评分：6/10

**优点**：
- 5 层结构覆盖了核心约束维度
- 提供了通用模板和检查清单

**不足**：
- 缺少错误处理和重试机制
- 缺少 subagent 协作机制
- 缺少实际案例

## 三、实际可行性审查 (ce-feasibility-reviewer)

### 审查维度
- 规则是否可执行
- 是否有过度理想化的假设
- 实际使用中的障碍

### 发现的问题

#### 问题 1："路径白名单"的执行成本
**严重程度**：高

**问题描述**：
- 要求主 agent 为每个 subagent 指定"路径白名单"
- 但主 agent 可能不知道具体需要哪些文件
- 实际使用中，主 agent 可能：
  - 给出过宽的范围（失去约束意义）
  - 给出过窄的范围（subagent 无法完成任务）
  - 需要多次迭代才能确定正确范围

**建议**：
- 允许"默认范围 + 例外"模式：
  - 默认：整个项目目录
  - 例外：明确禁止的目录（如 `node_modules`、`.git`）
- 或者提供"范围推断"机制：
  - 主 agent 给出任务描述
  - subagent 报告需要的文件范围
  - 主 agent 确认后执行

#### 问题 2："禁止操作清单"的维护成本
**严重程度**：中

**问题描述**：
- 要求"按领域实例化"禁止操作清单
- 但"领域"如何定义？编程、写作、研究是领域吗？
- 如果一个任务跨越多个领域（如"编写技术文档"），如何处理？
- 每次派发都要重新列举禁止操作，增加主 agent 负担

**建议**：
- 提供"领域模板库"：预定义的编程、写作、研究等场景的禁止操作
- 主 agent 只需选择领域模板，而非从零列举
- 或者采用"默认禁止 + 例外"模式

#### 问题 3："report_file"的 I/O 开销
**严重程度**：中

**问题描述**：
- 要求"详细报告写入 REPORT_FILE"
- 但频繁的文件 I/O 会增加延迟
- 对于简单任务（如"检查文件是否存在"），写文件是过度设计

**建议**：
- 已经改进：增加了"简单任务可以直接返回文本"的灵活性
- 但需要更明确的判断标准：
  - 报告长度 > 50 行 → 写文件
  - 报告长度 ≤ 50 行 → 直接返回

#### 问题 4："ledger"的实现细节缺失
**严重程度**：低

**问题描述**：
- 派发后跟踪提到"写入 ledger（防止上下文压缩后重复派发）"
- 但 ledger 是什么？如何写入？如何查询？
- 缺少实现细节

**建议**：
- 明确 ledger 的定义：
  - 可以是内存中的列表
  - 可以是文件（如 `.subagent-ledger.json`）
  - 可以是数据库
- 提供 ledger 的 schema 和 API

### 可行性评分：6/10

**优点**：
- 增加了灵活性（如路径可以用相对路径）
- 删除了过度量化的约束（如"≤15行"）

**不足**：
- 路径白名单的执行成本高
- 禁止操作清单的维护成本高
- ledger 的实现细节缺失

## 四、对抗性审查 (ce-adversarial-document-reviewer)

### 审查维度
- 挑战文档的前提假设
- 找出隐藏的依赖
- 质疑设计的合理性

### 发现的问题

#### 假设 1：subagent 需要被严格约束
**挑战**：
- 文档假设 subagent 会尽可能越界，所以需要 5 层约束
- 但实际上，现代 LLM 在明确任务目标后，行为是可预测的
- 过度约束会降低 subagent 的灵活性和效率

**替代方案**：
- 采用"原则 + 反例"模式，而非"规则 + 例外"
- 只约束真正危险的操作（如 git push、删除文件）
- 其他操作依赖 subagent 的判断力

**当前文档的改进**：
- 已经采用了"原则指导"的设计原则
- 但实际规则仍然过于具体（如路径白名单、禁止操作清单）

#### 假设 2：主 agent 能够准确定义任务范围
**挑战**：
- 文档假设主 agent 知道"需要哪些文件"、"禁止哪些操作"
- 但实际上，主 agent 可能：
  - 对项目结构不了解
  - 对任务的复杂性估计不足
  - 需要多次迭代才能确定正确范围

**替代方案**：
- 采用"探索-确认-执行"模式：
  - subagent 先探索，报告需要的范围和权限
  - 主 agent 确认后，subagent 再执行
- 或者提供"范围自适应"机制：
  - subagent 发现需要超出范围时，报告并等待指示
  - 而非直接禁止

**当前文档的改进**：
- 已经增加了"超出范围发现相关材料？→ 报告位置，等待指示"
- 但仍然要求主 agent 预先定义范围

#### 假设 3：5 层结构是必要的
**挑战**：
- 文档假设需要 5 层约束（权限、范围、契约、输出、协调）
- 但实际上，部分层级可以合并：
  - 第 1 层（权限）和第 5 层（协调）可以合并为"边界"
  - 第 3 层（任务契约）和第 4 层（输出契约）可以合并为"执行契约"
- 5 层结构增加了认知负担

**替代方案**：
- 采用 3 层结构：
  - 第 1 层：边界（不能做什么 + 主 agent 保留什么）
  - 第 2 层：范围（能在哪里做）
  - 第 3 层：契约（做什么 + 怎么汇报）

**当前文档的改进**：
- 5 层结构确实增加了认知负担
- 但每层都有明确的职责，合并可能导致混乱

#### 假设 4：subagent 是独立的
**挑战**：
- 文档假设 subagent 是独立的，没有定义协作机制
- 但实际上，复杂任务需要 subagent 之间的协作
- 缺少协作机制会导致：
  - 重复工作（多个 subagent 读取同一文件）
  - 冲突（多个 subagent 修改同一文件）
  - 信息孤岛（subagent A 的发现无法传递给 subagent B）

**替代方案**：
- 增加"协作模式"子节，定义：
  - 串行协作：A 的输出写入文件，B 读取
  - 并行协作：通过主 agent 中转
  - 共享状态：通过 ledger 或共享文件

**当前文档的改进**：
- 文档假设 subagent 是独立的
- 但实际使用中，协作是不可避免的

### 前提假设评分：5/10

**优点**：
- 增加了灵活性，减少了过度约束
- 采用了"原则指导"的设计理念

**不足**：
- 仍然假设主 agent 能够准确定义任务范围
- 缺少 subagent 协作机制
- 5 层结构增加了认知负担

## 五、综合评估

### 总体评分：6/10

### 优点
1. **结构清晰**：5 层结构层层递进，逻辑清晰
2. **原则导向**：每层都有明确的核心原则，避免过度约束
3. **灵活性**：增加了多处灵活性说明（如路径可以用相对路径、简单任务可以直接返回）
4. **实用性**：提供了通用模板和检查清单，便于实际应用

### 不足
1. **内部一致性**：部分概念在不同层级重复出现，关系不清
2. **完整性**：缺少错误处理、重试机制、协作机制
3. **可行性**：路径白名单和禁止操作清单的执行成本高
4. **前提假设**：假设主 agent 能够准确定义任务范围，假设 subagent 是独立的

### 改进建议

#### 短期改进（立即可做）
1. 澄清第 1 层与第 5 层的关系，删除重复内容
2. 将 read-only subagent 约束移到第 4 层
3. 明确升级机制与 STOP 触发条件的关系
4. 增加 2-3 个实际案例

#### 中期改进（需要设计）
1. 增加错误处理和重试机制
2. 增加 subagent 协作机制
3. 提供"领域模板库"，降低禁止操作清单的维护成本
4. 明确 ledger 的实现细节

#### 长期改进（需要重构）
1. 考虑合并部分层级，减少认知负担
2. 采用"探索-确认-执行"模式，降低主 agent 的负担
3. 提供"范围自适应"机制，让 subagent 参与范围定义

---

## 六、审查结论

这份文档是一个良好的起点，提供了清晰的 5 层约束框架。但在实际使用中，存在以下问题：

1. **执行成本高**：主 agent 需要为每个 subagent 定义路径白名单和禁止操作清单
2. **灵活性不足**：假设主 agent 能够准确定义任务范围，但实际情况可能更复杂
3. **缺少协作机制**：假设 subagent 是独立的，但复杂任务需要协作

**建议**：
- 短期：澄清内部一致性问题，增加实际案例
- 中期：增加错误处理、重试机制、协作机制
- 长期：考虑重构，减少层级，增加灵活性

**最终评价**：这份文档是一个"可用但不完美"的框架，需要在实际使用中不断迭代和优化。
