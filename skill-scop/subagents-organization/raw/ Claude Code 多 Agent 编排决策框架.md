# Claude Code 多 Agent 编排决策框架

## 先建立分析基础：Claude Code 的实际约束

在讨论"何时拆分"之前，必须理解几个硬约束：

**上下文隔离是单向的**：子 agent 启动时继承 main agent 注入的内容，但子 agent 运行期间产生的上下文对其他子 agent 完全不可见，最终只有 Task tool 的返回值（文本摘要）回传给 main agent。这意味着**跨 agent 的信息传递代价很高**——你必须把信息序列化成文本，通过 main agent 中转。

**并行有实际上限**：Claude Code 并行 subagent 的实际稳定并发大约在 **10 个以内**，超过后调度本身的 overhead 和错误率都会上升。

**Token 成本的真实结构**：每个子 agent 都需要重新加载系统 prompt、CLAUDE.md、以及你注入的上下文。如果你有一个 5000 token 的 CLAUDE.md，启动 10 个子 agent 的固定成本就是 50,000 token，任务还没开始做。

**Task tool 的返回是有损压缩**：子 agent 完成后，main agent 拿到的是结果摘要，不是完整的执行轨迹。如果后续 agent 需要前面 agent 的中间细节，这个细节必须在任务定义阶段就要求写入文件系统，而不能依赖"我看看前一个 agent 做了什么"。

## 1. 单个 Main Agent 就够了的情况

**判断标准**：以下条件同时满足时，不要拆分。

- 任务的逻辑是**线性推理链**，每一步依赖上一步的完整中间状态
- 总代码涉及范围 < 20 个文件，或上下文窗口用量预估 < 60%
- 任务中有大量**隐性决策**（比如"根据看到的情况决定下一步"），无法在开始前规划完整

**典型场景**：
```
"帮我 debug 这个 Spring Boot 启动失败的问题"
```
这个任务的核心是诊断——看报错 → 找代码 → 推断原因 → 验证假设 → 修复。每一步都依赖上一步的发现，根本无法提前切分给子 agent，强行拆分只会让 main agent 在中间不断写"诊断报告文件"再让下一个 agent 读，效率比单 agent 差一个数量级。

**反直觉的情况**：一个要修改 50 个文件的重构任务，如果修改逻辑是统一的（比如统一替换一个 API 调用），单 agent 依然可能更合适——因为一致性检查比并行收益更重要。

---

## 2. 串行子 Agent 的适用场景

**核心判断**：任务有**明确的阶段门（Stage Gate）**，后一阶段的输入依赖前一阶段的**完整输出**，但阶段内部的上下文不需要透传。

用公式表达：**阶段 N+1 的输入 = f(阶段 N 的输出文件)** 而不是 f(阶段 N 的完整上下文)

**典型场景**：

```
需求分析 Agent → 接口设计 Agent → 实现 Agent → 测试 Agent
```

每个阶段产出一个文档或代码文件，下一个 agent 读文件开始工作。Main agent 在这里充当的是**流水线调度器**：

```
Main Agent 职责（串行模式）：
1. 阶段启动前：将上一阶段的输出路径和当前阶段目标注入 Task prompt
2. 阶段运行中：等待，不干预（除非 Task 返回失败）
3. 阶段完成后：验证输出文件是否存在且符合格式约定
4. 错误处理：判断是重试当前阶段还是回滚到上一阶段重做
```

**关键实践**：阶段间的"契约"必须是**文件系统上的具体产物**，而不是"理解"或"状态"。比如：

- ✅ 需求 Agent 输出 `requirements.md`，设计 Agent 的 prompt 明确写 "读取 requirements.md"
- ❌ 需求 Agent 完成后，main agent 把需求"总结"传给设计 Agent——这是有损的，设计 Agent 可能丢失细节

**串行的成本划算点**：当单个阶段的上下文需求 > 30K token，串行比单 agent 更省钱，因为每个子 agent 只加载自己阶段的上下文，而不是累积所有阶段的历史。

## 3. 并行子 Agent 的适用场景

**核心判断**：任务可以分解为 **N 个相互独立的工作单元**，满足：

- 单元之间**无读写竞争**（不能同时修改同一个文件）
- 单元的成功/失败**相互独立**（一个失败不 block 其他）
- 每个单元的上下文是**可以完整自包含描述**的

**典型场景**：

```
"为这 8 个模块分别生成单元测试"
"把这 12 个 API 端点的文档补全"
"对这 6 个微服务分别做依赖升级"
```

**Main Agent 在并行模式的实际职责**（这是最容易做错的地方）：

```
阶段一：任务分解（最关键，必须在启动任何子 agent 前完成）
  - 枚举所有工作单元
  - 为每个单元定义：输入边界、输出路径、成功标准
  - 检查单元间是否真的无竞争（特别是公共文件、shared config）

阶段二：上下文包（Context Bundle）构建
  - 每个子 agent 的 prompt 必须是完全自包含的
  - 不能依赖"子 agent 自己去发现"
  - 将公共约定（代码风格、命名规范）以文本形式复制进每个 prompt
  
阶段三：并发启动（实际用多个 Task tool calls）
  - 10 个以内可以一次全部启动
  - 超过 10 个考虑分批

阶段四：结果聚合（最容易被低估的复杂度）
  - 收集所有子 agent 的返回状态
  - 对失败的单元决策：重试/跳过/中止整批
  - 执行跨单元的一致性检查（比如 8 个测试文件的 import 风格是否统一）
  - 如果有必要，启动"整合子 agent"做最终收尾
```

**并行失效的隐藏原因**：

并行 agent 同时读取同一个大文件（比如 `package.json`）没有问题，但如果它们都要**写**公共文件，就会产生竞争。典型陷阱：
- 多个 agent 同时修改 `pom.xml` 或 `build.gradle`
- 多个 agent 同时向同一个 `index.ts` 追加 export
- 多个 agent 同时更新 `README.md`

解决方案是让并行 agent 只写自己独立的文件，main agent 在最后做合并。

## 4. 拆分粒度的实用判断标准

这是整个框架最核心的部分。用一个决策树：

```
问题1：任务能否在开始前被完整规划？
  否 → 单 agent（需要探索性推理）
  是 → 继续

问题2：各子任务间是否需要共享中间状态？
  是（需要实时共享）→ 单 agent
  否（只通过文件传递）→ 继续

问题3：子任务是否有明确的执行顺序依赖？
  是 → 串行子 agent
  否 → 继续

问题4：拆分后每个子 agent 的上下文是否可以完整自包含？
  否（需要大量公共背景）→ 不拆，公共背景的复制成本会超过并行收益
  是 → 并行子 agent

问题5：并行单元数量 > 收益临界点？
  每个单元 < 5 分钟工作量，且 < 5 个单元 → 不值得拆（编排开销不合算）
  每个单元 > 15 分钟工作量，或 > 5 个单元 → 值得并行
```

**净收益为负的典型拆分**（反面教材）：

```
❌ 错误拆分："帮我重构 UserService，把它拆给 3 个 agent 分别处理"
   - 问题：UserService 的方法之间有调用关系，任何一个 agent 的修改都影响其他
   - 实际结果：3 个 agent 产出 3 个互相冲突的版本，main agent 花大量时间合并冲突

✅ 正确使用：
   - 并行："为 UserService、OrderService、PaymentService 分别生成测试"（三个服务独立）
   - 串行："先分析 UserService 的现有接口，再设计新接口，再实现迁移"（阶段依赖）
```

## 5. 一个综合场景：PRD → 代码的完整编排

结合你之前做的需求工程框架，一个实际的多 agent pipeline 示例：

```
Main Agent 全程职责：
│
├── [阶段 0：Main Agent 自己执行] 
│   读取 PRD，执行任务分解，生成 task-manifest.json
│   （这步不能交给子 agent，因为需要 main 的完整推理能力来做全局规划）
│
├── [串行 Agent 1] 需求分析 → requirements.md
│   注入：PRD 文件路径 + 分析模板
│
├── [Main Agent 执行阶段门检查]
│   验证 requirements.md 格式，提取模块列表
│
├── [串行 Agent 2] 接口设计 → api-spec.yaml  
│   注入：requirements.md + 接口设计规范
│
├── [Main Agent 分解并行任务]
│   从 api-spec.yaml 提取 N 个独立模块
│
├── [并行 Agent 3-1 ~ 3-N] 各模块实现
│   每个注入：自己模块的接口定义 + 公共代码规范 + 相关依赖说明
│
├── [Main Agent 聚合 + 一致性检查]
│   检查各模块输出，执行集成验证
│
└── [串行 Agent 4] 测试生成（基于已稳定的实现）
```

**这个结构的关键设计决策**：
1. 全局规划永远由 main agent 做，不外包
2. 有依赖的阶段串行，无依赖的实现并行
3. 每个阶段门都有 main agent 的显式验证，不是盲目传递
4. 并行 agent 的产出是独立文件，main agent 负责最终集成

## 最后：何时放弃编排，回归简单

多 agent 编排有一个容易忽视的隐性成本：**调试复杂度**。当出错时，你需要追踪哪个 agent 在哪个步骤出了什么问题，这个排查成本随 agent 数量非线性增长。

**实用原则**：如果你无法用一段清晰的文字描述每个子 agent 的输入/输出边界，那这个拆分还没有成熟到可以实施——先在单 agent 下跑通逻辑，再提取可并行的部分。
