## 二、四个维度一致的发现（高置信度交叉验证）

以下发现被 3-4 个 reviewer 独立识别，属于确凿问题：

### 🔴 严重问题（4/4 reviewer 一致）

**1. 缺少 STOP/升级机制**

文档在"行为约束"中提到"边界：遇到不确定时的处理方式"，但三个模板均未定义 subagent 遇到困难时的标准停止和报告路径。对照文档第 3 层有完整的 4 状态升级机制（DONE / DONE_WITH_CONCERNS / BLOCKED / NEEDS_CONTEXT），本方案完全没有。

**2. 并行调度冲突解决缺位**

模板 B 仅说"检查修改是否冲突"，没有：
- 冲突检测方法（diff？文件锁？）
- 冲突解决策略（谁优先？回退？人工介入？）
- 部分失败回退机制
- 并发数上限控制
- subagent 间依赖声明

**3. 缺少派发后生命周期**

方案只有"启动前"和"完成后"两张检查清单，缺少：
- 收到结果后的 review 流程
- review 不通过时的返工机制
- 上下文压缩后防止重复派发的 ledger 记录
- 对照文档的"派发后跟踪"清单完全覆盖了这些

### 🟡 重要问题（3/4 reviewer 一致）

**4. Hermes Agent 代码示例是伪代码**

`delegate_task()` API 和 `toolsets` 参数在此仓库中不存在。文档未标注"此为示意代码"，读者会误以为可复制执行。4 个 reviewer 中有 3 个建议删除或替换。

**5. 与对照文档的约束分类体系不兼容**

编排文档用 4 维并列模型，对照文档用 5 层递进模型。两套体系描述同一问题但互不引用，导致：
- 读者面临"用哪个"的选择困惑
- 维护者需同步更新两份文档
- 编排文档缺少对照文档中最关键的层（第 3 层任务契约、第 5 层协调规则）

**6. 模板 C（TDD）分类错误**

TDD 是质量策略，不是编排策略。将其与"单任务""并行调度"并列，混淆了"如何组织工作"与"如何保证质量"两个正交维度。

**7. 验证清单衡量合规性而非正确性**

清单可以全部打勾而 subagent 产出架构错误的代码。关键的检查项缺失：
- subagent 是否读取了项目规则文件？
- subagent 是否修改了白名单之外的"顺手"文件？
- subagent 的输出是否写入了指定的 REPORT_FILE？

### 🟢 次要问题（1-2 reviewer 识别）

**8. 双重否定歧义（coherence reviewer）**

模板 C 阶段 1："禁止：不实现任何业务逻辑" = 必须实现业务逻辑（与意图相反）
模板 C 阶段 3："禁止：不改变外部行为" = 必须改变外部行为（与意图相反）

**9. 预启动检查"是否与其他 subagent 有冲突"逻辑位置错误**

在 subagent 启动前无法判断冲突——subagent 尚未执行。该检查项属于"集成验证"阶段。

**10. 上下文最小化原则缺少操作指导**

"只传递完成任务必需的信息"没有判断标准——什么是"必需"？没有像对照文档那样将 BRIEF_FILE 作为硬要求。

## 三、与对照文档的差距分析

`subagent-constraints-framework.md` 在以下维度显著优于被审阅文档：

| 维度 | 被审阅文档 | 对照文档 |
|------|-----------|----------|
| 架构 | 4 维并列 | 5 层递进（权限→范围→契约→输出→协调） |
| 哲学 | 控制导向（"禁止"优先） | 信任导向（"信任判断力，保留最终裁决权"） |
| 灵活性 | 刚性约束 | Escape hatches：STOP 条件、升级机制 |
| 领域分离 | 框架与质量策略混在一起 | 框架（层 1/2/4/5）与领域实例化（层 3）明确分离 |
| 平台无关 | 含 Hermes 特定代码 | 完全平台无关 |
| 输出模型 | 强制结构化返回 | 简单任务可直接返回文本 |
| 可追溯 | 无 | Ledger 追踪（防止上下文压缩后重复派发） |
| 设计哲学 | "用规则束缚" | "用原则指导，而非用规则束缚" |

## 四、被审阅文档的独特价值

不是全盘否定。这份文档有对照文档缺少的优点：

1. **正反例对比**（第三节"常见错误与规避"）—— 对照文档高度抽象，缺少可操作的坏/好对比
2. **按工作模式的模板分类**（单任务 / 并行 / TDD）—— 对照文档只有一个通用模板
3. **更易上手的结构**（4 维 + 3 模板 + 检查清单）—— 对新手更友好
4. **并行调度显式模式**（模板 B）—— 对照文档没有展开并行场景

---

## 五、修复建议（按优先级排序）

### 必须修复（阻塞级）

1. **合并 STOP/升级机制**：从对照文档第 3 层复制 4 状态升级机制到所有模板
2. **补充派发后生命周期**：从对照文档复制"派发后跟踪"清单（DONE → review, BLOCKED → 诊断, etc.）
3. **模板 B 补充冲突解决**：增加并发上限、文件锁/merge 策略、部分失败回退
4. **删除或标注第四节代码示例**：标注为"伪代码示意"，或替换为真实 Claude Code Task 工具调用
5. **修复双重否定歧义**（模板 C 阶段 1 和阶段 3 的"禁止"行）
6. **所有模板增加项目规则读取指令**：`开始前必读: jeecg-boot/AGENTS.md` 等

### 建议修复（重要级）

7. **声明与对照文档的关系**：明确本方案是对照文档 5 层模型的"模板实例化"视图
8. **统一约束分类**：采用 5 层模型，或在文档开头声明 4 维模型的定位
9. **模板 C 降级为模板 A 的参数**：`quality_mode: "tdd"` 而非独立模板
10. **合并两份验证清单**：消除与对照文档派发检查清单的重叠
11. **修复预启动检查项 #6 的逻辑位置**：移入"集成验证"阶段

### 可选优化

12. **移除或精简第三节常见错误**：信息已在模板中编码，属于教学性冗余
13. **上下文隔离改为有条件的默认**：默认传递关键上下文，按需裁剪
14. **补充 subagent 角色类型**（analyst / implementer / reviewer），每种有预设约束配置

---

## 六、最终建议

**不要独立修订此文档。**

推荐路径：

1. 以 `subagent-constraints-framework.md` 的 5 层模型为唯一框架基础
2. 将被审阅文档中独有的有价值内容（正反例对比、并行调度模式、工作模式分类）提取并整合到对照文档
3. 将被审阅文档重新定位为对照文档的"快速入门指南"或"模板实例化手册"
4. 删除两份文档之间的重复内容

**核心理由**：对照文档的架构更清晰（5 层递进）、设计更灵活（escape hatches）、哲学更健康（信任+赋能而非控制）。两份竞争框架维护成本高，且会让使用者困惑。
