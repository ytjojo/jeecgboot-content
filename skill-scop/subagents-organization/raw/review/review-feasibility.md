# Subagent 编排系统性方案 — 可行性评估报告

评估日期：2026-06-21
评估对象：`skill-scop/subagents-organization/raw/Subagent 编排系统性方案.md`
参考基线：仓库现有 `AGENTS.md` subagent 规则、同伴文档 `subagent-constraints-framework.md`、`.claude/skills/` 下现有技能文件

---

## 1. 现实使用中最先出问题的地方

### 1.1 缺少项目约定注入机制（第一断裂点）

三个模板均未告诉 subagent 在动手前必须阅读项目规则文件。AGENTS.md 明确规定：

> "修改后端代码前，先阅读 `jeecg-boot/AGENTS.md`"
> "修改前端代码前，先阅读 `jeecgboot-vue3/AGENTS.md`"

但模板 A/B/C 中没有一行指令引导 subagent 执行此路由规则。结果是：subagent 会在不了解 `springboot-coding-conventions.md`、`api-guidelines.md` 等文件的情况下直接写代码，代码大概率违反项目规范。这不是理论问题——已在 AGENTS.md 规则中被验证为必须处理的场景。

### 1.2 Hermes Agent 代码是伪代码，不是可执行示例（第二断裂点）

```typescript
delegate_task({
  goal: "修复 src/auth/login.test.ts 中的 3 个失败测试",
  toolsets: ["terminal", "file"]
})
```

`delegate_task()` 在此仓库中不存在。仓库实际使用的是 Claude Code 的 Task 工具或 `/superpowers:subagent-driven-development` 编排入口。`toolsets` 参数在 Claude Code SDK 和本仓库的工具链中无对应概念。文档未标注"此为示意代码"，实现者拿到这段代码无法执行，需要自行推断映射关系。

### 1.3 模板 B 缺少冲突解决机制（第三断裂点）

模板 B 唯一涉及集成的部分是：

> "检查修改是否冲突"

没有说明如何检查（diff？文件锁？merge strategy？），没有冲突后处理流程（谁优先？回退谁的？人工介入？）。两个并行 subagent 修改同一文件时，后完成的会静默覆盖先完成的修改。现实场景中（如两个 subagent 各自修同一模块的不同 bug），文件重叠概率不低。

### 1.4 缺少 STOP/升级机制

同伴文档 `subagent-constraints-framework.md` 第 3 层有明确的升级机制：

| 情况 | 行动 |
|------|------|
| 完成，但有疑虑 | 报告完成，同时说明疑虑 |
| 无法继续（遇到阻碍） | 报告卡点，等待指示 |
| 缺少信息/决策 | 说明需要什么，等待补充 |

但本方案的三个模板中完全没有类似指令。subagent 遇到"目标文件不存在"、"测试本来就通过"、"需要做未授权的架构决策"等情况时，没有明确的停止和报告路径，会自行猜测并继续执行，造成难以追踪的错误。

## 2. 缺失的边界情况

### 2.1 影子路径追踪

按 happy / nil / empty / error 四条路径逐一检查：

| 路径 | 场景 | 模板是否覆盖 |
|------|------|------------|
| **happy** | 文件存在，错误可定位，修复成功 | 覆盖（模板 A/B/C） |
| **nil** | 目标文件不存在 | **未覆盖** — subagent 会拿到文件路径但找不到文件，无指令如何处理 |
| **empty** | 测试文件存在但为空、错误日志为空 | **未覆盖** — "根据错误日志修复"的前提是日志有内容 |
| **error** | 修复后测试仍失败（方案错误）、依赖的模块不编译 | **未覆盖** — 模板只说"验证标准：所有测试通过"，没说通不过怎么办 |

### 2.2 并行派发部分失败

模板 B 假设"启动所有 subagent → 全部完成 → 汇总"。现实是：30 个 subagent 中至少一个失败的概率随数量急剧上升。文档没有定义：部分成功时是否回滚？是回滚全部还是保留成功的？失败的 subagent 是否重新派发？

### 2.3 Token 预算耗尽

项目 RULES.md 设定单会话上限 30,000 Token。30 个 subagent 各自的上下文 + 返回结果 + 主 agent 的编排成本，很可能超出预算。模板 B 没有为每个 subagent 设定 token 预算限制。

### 2.4 subagent 超时/卡住

没有超时处理。如果 subagent 陷入无限循环或等待用户输入（在某些模式下），编排层没有中断机制。

### 2.5 跨 subagent 依赖

模板 B 假设子任务是"独立"的。但在修复 bug 场景中，Subagent 1 的修复可能改变 Subagent 2 依赖的函数签名。没有依赖声明和排序机制。

## 3. 模板是否可直接使用

### 模板 A（单任务）：基本可用，需补充

- 结构清晰，目标/上下文/约束/输出四段式合理
- **缺失**：项目约定注入指令、STOP 条件、文件不存在时的处理
- **结论**：作为基础框架可用，但不能原样复制到生产派发

### 模板 B（并行派发）：不可直接使用

- 缺少：冲突检测算法、文件锁、部分失败回退、依赖声明、token 预算
- "检查修改是否冲突"和"运行完整测试套件"过于模糊
- **结论**：仅适用于 <5 个完全无文件重叠的独立任务

### 模板 C（TDD 驱动）：概念正确但需适配

- TDD 三阶段（红→绿→重构）结构正确
- **缺失**：未要求读取项目测试规范文件（`docs/agent-context/springboot-testing-conventions.md`），未指定运行哪些测试（本模块？全量？），未说明测试框架和断言风格需与项目一致
- 同伴文档要求"需求是否已写入 BRIEF_FILE？不在 prompt 中粘贴全文"，此模板未遵循

## 4. Hermes Agent 代码示例的价值评估

**结论：制造混淆 > 增加价值。**

理由：

1. **非真实 API**：`delegate_task()` 和 `toolsets` 在此仓库不存在。实现者会困惑"这是什么东西？我应该用 Task 工具还是这个？"
2. **无免责声明**：文档未注明"此为伪代码，仅供示意"
3. **与同伴文档风格不一致**：`subagent-constraints-framework.md` 全部用 YAML/Markdown 模板，没有一行代码——这种风格更可维护
4. **误导性强**：看起来像"复制粘贴就能用"的代码，实际完全不是

**建议**：删除此章节，或将示例改为实际可用的 Claude Code Task 工具调用格式，并标注版本和环境依赖。

## 5. 扩容性：从 3 到 30 个 subagent

| 维度 | 3 subagent | 30 subagent | 方案覆盖 |
|------|-----------|-------------|---------|
| 派发复杂度 | 可手动 | 需要编排引擎 | **未覆盖** |
| 并行度控制 | 不限 | 需要限制并发数 | **未覆盖** |
| 冲突检测 | 可手动 diff (3 pair) | O(n^2) = 435 pair | **未覆盖** |
| Token 预算 | 可控 | 大概率超限 | **未覆盖** |
| 失败概率 | 低 | >60% 至少一个失败 | **未覆盖** |
| 汇总报告 | 手动可做 | 需自动聚合 | **未覆盖** |
| subagent 间依赖 | 可声明 | 需要 DAG | **未覆盖** |

方案本质是"手工作坊级"的编排思路——适合 3-5 个 subagent 的场景，对 30 个不具备可操作性。需要引入 wave-based 编排（如同伴文档提到的"派发后跟踪"和"ledger"概念）或 DAG 依赖图。

## 6. 验证清单能否捕获常见失败模式

### 派发前清单分析：

- "目标是否具体、可衡量？" — **有效**，能拦截"修好它"这类模糊指令
- "上下文是否最小化且充分？" — **部分有效**，但"充分"的判断标准不明确。"最小化"和"充分"之间的张力没有解决指南
- "约束是否明确？" — **部分有效**，但不检查"是否告诉了 subagent 去哪里读项目规则"
- "输出格式是否清晰？" — **有效**
- "验证标准是否可执行？" — **有效**
- "是否与其他 subagent 有冲突？" — **意图好但无操作手段**，只是提醒主 agent "考虑一下冲突"，没有提供冲突检测方法

### 派发后清单分析：

- "返回结果是否符合预期格式？" — **有效**
- "修改是否在允许范围内？" — **弱有效**，但"如何验证"未说明（检查 diff 中所有文件路径？手工比对？）
- "测试是否全部通过？" — **有效**
- "是否引入新的问题？" — **太模糊**，应具体化为："lint 是否通过？类型检查是否通过？相关模块测试是否通过？"

### 清单遗漏的常见失败模式：

1. **subagent 未读取项目规则文件就开始修改** — 未检查（本仓库最可能出现的失败模式）
2. **subagent 修改了不在白名单的"顺手"文件** — 未检查
3. **subagent 的输出未写入指定报告文件** — 未检查（同伴框架要求写入 REPORT_FILE）
4. **subagent 卡住/超时** — 未检查
5. **并行 subagent 的文件修改互相覆盖** — 未检查（清单说"是否冲突"但无检查手段）

## 7. 与同伴文档的关系

`subagent-constraints-framework.md` 是本方案的升级版和补充版：
- 同伴文档有 5 层约束体系，本方案只有 4 维（缺少协调层）
- 同伴文档有 STOP 触发条件和升级机制，本方案缺失
- 同伴文档有派发后跟踪流程（dispatch → review → fix loop），本方案缺失
- 同伴文档有"设计原则"指导（"用原则指导，而非用规则束缚"），本方案是规则集合

**建议**：两文档存在大量重叠。要么明确分工（本方案聚焦模板，同伴文档聚焦约束框架），要么合并。当前状态是两份文档描述同一体系但细节不一致，维护者必须同时更新两份。

## 8. 总结

### 总体实用性评分：5/10

**能用的部分**：
- 四维约束概念（范围/上下文/行为/输出）是合理的分类框架
- 模板 A 的结构作为单任务派发模板基本可用（需补充）
- 四个反例（范围过大、上下文不足、缺少约束、输出模糊）是真实且常见的错误

**关键风险（按严重程度排序）**：

1. **subagent 不读取项目规则文件就写代码** — 无模板或清单处理此问题，违反 AGENTS.md 路由规则
2. **Hermes Agent 代码示例是伪代码** — 误导实现者以为可执行，实际需要完全重写映射到真实 API
3. **并行派发无冲突解决和部分失败回退** — 超过 3 个 subagent 即不可靠
4. **缺少 STOP/升级机制** — subagent 遇到意外时没有停止路径，会自行猜测
5. **验证清单遗漏了"是否遵守项目约定"和"是否读取项目规则"** — 本仓库最关键的检查点
6. **与 `subagent-constraints-framework.md` 重叠且不一致** — 缺少协调层(第 5 层)、升级机制(第 3 层 STOP 条件)、派发跟踪(ledger)

### 修复路径建议：

1. 所有模板增加"工作前必读"段，指向项目规则文件（AGENTS.md 路由规则）
2. 删除或标注 Hermes Agent 代码为"伪代码示意"，或替换为真实 Task 工具调用
3. 模板 B 增加：并发上限、文件锁/merge 策略、部分失败回退、依赖声明
4. 合并同伴文档的 STOP 触发条件和升级机制到模板
5. 派发前清单增加："[ ] subagent 是否收到了读取项目规则文件的指令？"
6. 明确与 `subagent-constraints-framework.md` 的分工或考虑合并
