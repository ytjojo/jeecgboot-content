# 一致性审阅报告：Subagent 编排系统性方案

## 结构性发现

### 1. 核心约束定义的失败处理从未出现在模板中

**严重级别**: 中

**出处**: 第三节"1.3 行为约束"要求"失败处理：出错时的回退策略"。

**矛盾点**: 三个模板（A、B、C）均未包含任何错误回退策略字段。模板 A 仅有"边界：[遇到不确定时的处理方式]"，不等于"出错时的回退策略"。这是一个在框架层声明了但模板层完全未实现的要求——读者无法知道在实际调用时应当在哪里、以何种方式表达回退策略。

---

### 2. 模板 B 缺少"预期输出"规格，与 1.4 节约束脱节

**严重级别**: 中

**出处**: 第三节"1.4 输出约束"要求"验证标准：明确成功条件"和"可追溯性：要求列出修改的文件和原因"。

**矛盾点**: 模板 B（并行多 Subagent）仅列出"集成验证"步骤（检查冲突、运行测试、汇总结果），但没有"预期输出"字段。与模板 A 和模板 C 的"预期输出"结构不一致，且缺少对单个 subagent 的明确验证标准。读者无法知道每个并行 subagent 应以何种格式返回结果。

---

### 3. 预启动检查项"是否与其他 subagent 有冲突"逻辑位置错误

**严重级别**: 中

**出处**: 第五节"验证清单"之"启动 subagent 前检查"列表第 6 项。

**矛盾点**: "是否与其他 subagent 有冲突"在 subagent 启动前无法判断——subagent 尚未执行，冲突（如同一文件的并发修改）只能在运行后或集成时发现。该检查项实质上属于"完成后检查"或"集成验证"阶段。同时，模板 B 的"任务分解"要求"每个 subagent 负责一个独立领域"，若确实独立，冲突本不应出现。该检查项的存在暗示"独立领域"的划分未必可靠，但文档未就此做出任何说明。

---

### 4. 模板 C 的"禁止"表述存在双重否定歧义

**严重级别**: 高

**出处**: 模板 C 阶段 1 和阶段 3 的约束行。

**原文**:
- 阶段 1：`禁止：不实现任何业务逻辑`
- 阶段 3：`禁止：不改变外部行为`

**分析**: 在标准中文中，"禁止 [X]" = X 行为被禁止。因此：
- "禁止：不实现任何业务逻辑" = 不允许*不实现*业务逻辑 = 必须实现业务逻辑
- "禁止：不改变外部行为" = 不允许*不改变*外部行为 = 必须改变外部行为

但在阶段 1（测试先行）中，意图是不允许实现业务逻辑（只写测试）。在阶段 3（重构）中，意图是不允许改变外部行为。当前表述恰好与意图相反。

**可能的善意解读**: 作者可能意图将"禁止："读作"禁止事项："然后以"不XXX"作为一条规则说明。但这种解读依赖读者对作者意图的猜度，而非语法规则。两名严谨读者会得出相反的结论。

---

### 5. 模板之间对"边界条件"约束的实现不一致

**严重级别**: 低

**出处**: 1.3 行为约束要求"边界条件：遇到不确定时的处理方式"。

**矛盾点**: 模板 A 实现了该字段（"边界：[遇到不确定时的处理方式]"），但模板 B 和模板 C 均未包含。作为一个跨模板的共同约束不应当只在部分模板中出现。
