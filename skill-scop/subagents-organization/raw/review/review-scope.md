## 一、范围覆盖：缺失的关键主题

### 1.1 缺少升级/停止（STOP）机制

方案在「行为约束」中提及「边界：遇到不确定时的处理方式」，但**从未定义 subagent 遇到不确定时应该做什么**。companion 文档 `subagent-constraints-framework.md` 第 3 层有明确的 STOP 触发条件：缺少信息、需要决策、任务本身有问题、反复失败。编排文档的 3 个模板都没有给 subagent 一个「我做不到，需要主 agent 介入」的协议。

### 1.2 并行 subagent 协调过于肤浅

「模板 B：并行多 Subagent」提到「检查修改是否冲突」但没有说明**如何**检测或解决冲突。两个 subagent 修改同一文件怎么办？一个 subagent 的输出是另一个的输入怎么办？这些是并行编排的核心难点，方案只提了一句就略过。

### 1.3 缺少派发后生命周期

方案描述了如何启动 subagent，但缺少完成后的处理流程：
- 收到结果后如何 review？
- review 不通过怎么办（返工还是重新派发）？
- 如何防止上下文压缩后重复派发相同任务（ledger 机制）？
- 这几点 companion 文档的「派发后跟踪」清单已覆盖，编排文档完全没有。

### 1.4 缺少 read-only subagent 的区分

方案将所有 subagent 一视同仁，没有区分只读 subagent（Explore、验证）和读写 subagent（实现）。companion 文档第 1 层有专门的 read-only 输出边界规则。这对防止只读 subagent 越界输出代码实现至关重要。

### 1.5 缺少失败恢复策略

「行为约束」列出「失败处理：出错时的回退策略」作为一个约束类别，但全文没有任何地方描述实际的失败恢复策略，仅停留于分类标签。

## 二、过度工程：不必要的复杂度

### 2.1 TypeScript API 示例是推测性设计

第四节「实际调用示例（Hermes Agent）」提供了具体的 `delegate_task()` TypeScript API 调用示例。这属于**框架先于需求（framework-ahead-of-need）**——没有证据表明这个 API 存在。编排方案作为一个约束框架文档，不需要规定具体的 API 形状。这是实现细节，不应出现在约束方案中。

### 2.2 与 companion 文档的约束维度重叠

编排文档的 4 个约束维度与 companion 文档的 5 层模型存在显著重叠，形成了**两套并行的分类体系**：

| 编排文档（4 维） | companion 文档（5 层） | 关系 |
|---|---|---|
| 范围约束 | 第 2 层：作用域边界 | 实质等同 |
| 行为约束（部分） | 第 1 层：权限边界 | 实质等同 |
| 输出约束 | 第 4 层：输出契约 | 实质等同 |
| 上下文约束 | 无独立层（分散于各层） | 编排文档独有 |
| （无） | 第 3 层：任务契约 | companion 独有 |
| （无） | 第 5 层：协调规则 | companion 独有 |

维护两套分类体系给使用者造成认知负担——接到任务时该参照哪个分类？

### 2.3 验证清单与 companion 文档的派发检查清单重叠

第五节「启动 subagent 前检查」的 6 项与 companion 文档的「派发前检查」7 项存在大量重叠（目标明确性、路径白名单、冲突检查）。维护两份功能相似的清单是浪费。

## 三、工程不足：过于肤浅的方面

### 3.1 上下文最小化的原则缺少操作指导

「上下文约束 - 最小化原则」声明「只传递完成任务必需的信息」，但没有任何判断标准——什么是「必需」？多即是少还是少即是多少？如何评估 subagent 需要多少上下文？companion 文档至少把 BRIEF_FILE 作为硬要求，但编排文档只陈述了目标。

### 3.2 TDD 模板仅覆盖 happy path

「模板 C：TDD 驱动的 Subagent」描述了三阶段流程（测试 → 实现 → 重构），但只覆盖理想情况。当实现揭示设计缺陷时怎么办？当 subagent 发现需求本身有问题时怎么办？当测试编写阶段就遇到歧义时怎么办？这些真实场景在 TDD 中极其常见，模板没有给出应对策略。

---

## 四、边界清晰度

### 4.1 编排文档与约束文档的边界模糊

两份文档的目标受众和使用场景没有明确区分。编排文档标题为「编排系统性方案」，侧重**主 agent 如何组织和启动 subagent**；companion 文档标题为「约束系统性方案」，侧重**subagent 的边界定义**。但这只是主观解读——文档内部没有声明自己的定位和与另一个文档的关系。

### 4.2 约束维度内部边界模糊

「行为约束」同时包含禁止操作、边界条件和失败处理，这三个子项的粒度差异很大——禁止操作是硬规则，边界条件是判断准则，失败处理是协议——放在同一个维度下造成混淆。

## 五、与 companion 文档的对比

### 编排文档新增的价值（companion 文档所无）

- **具体的正反例对比**（第三节「常见错误与规避」）——companion 文档高度抽象，缺少可操作的例子
- **按工作模式的模板分类**（单任务 / 并行 / TDD）——companion 文档只有一个通用派发模板
- **TDD 模板**——companion 文档将其视为「领域实例化」，没有展开
- **更易上手的结构**——4 维度 + 3 模板 + 检查清单，对新手更友好

### 编排文档的不足（companion 文档已有）

- 无升级/STOP 机制
- 无派发后生命周期（review → 返工/重派发 → ledger）
- 无主 agent 保留权力的明确声明（第 5 层：协调规则）
- 无 read-only subagent 的专门处理
- 无设计原则章节（「为什么这样设计」）
- 两套约束分类体系并存，未说明关系

## 六、总结

### 关键差距（按重要性排序）

1. **无升级/STOP 协议**：subagent 遇到困难时没有通知主 agent 的标准方式
2. **并行协调浅尝辄止**：冲突检测和依赖管理是并行编排的核心难点，文档只提了一句
3. **无派发后生命周期**：缺少 review、返工、ledger 记录等完整闭环
4. **推测性 API 示例**：第四节的实际调用示例是未经验证的 API 设计，应移除或标注为示例
5. **与 companion 文档分类体系未对齐**：维护 4 维和 5 层两套并行分类，增加认知负担

### 简化机会

1. **移除第四节 API 调用示例**（推测性设计），或将其融入第三节的错误示例
2. **统一约束分类体系**：要么采用 companion 文档的 5 层模型（更完整），要么在编排文档中显式声明 4 维模型是 5 层模型的简化视图
3. **合并两份验证清单**：编排文档的检查清单与 companion 文档的派发检查清单存在大量重复，应合并为一份
4. **补充设计原则**：将 4 个约束维度与 companion 文档的 5 层模型的关系说清楚，消除读者的困惑
