# 对抗性审阅：Subagent 编排系统性方案

## 审阅范围与深度

- **被审阅文档**：`Subagent 编排系统性方案.md`（~2100 字，4 个主要章节，10+ 个模板/约束/清单）
- **对照文档**：`subagent-constraints-framework.md`（同目录，作为对照基准）
- **风险信号**：文档定义了一套 agent 编排框架，属于新建架构抽象。涉及上下文隔离、并行调度、TDD 流程——每一项都有系统性影响。
- **审阅深度**：Deep。对所有 5 个维度进行多轮压力测试。

---

## 1. 前提挑战（Premise Challenging）

### 1.1 问题-方案错配

文档将问题框定为"subagent 需要约束，否则会失控"。其解决方案围绕四个约束维度展开——范围、上下文、行为、输出。但它在解决的*真正问题*可能完全不同。

**文档声称的问题**：subagent 可能修改不该改的文件、做范围之外的事、产生不可追溯的输出。

**实际观察到的项目问题**（来自 AGENTS.md 中的硬约束）：项目中已有的 subagent 保护措施（禁止 openspec archive、git push、修改 AGENTS.md）针对的是*特定危险操作*，而非通用越界行为。这些保护比文档提出的通用约束更精确、更有效。

**错配**：如果真正的失败模式是"subagent 做了主 agent 没授权的事"（如合并到 master），那么通用范围约束是低效的——只需禁止那 5-6 个具体操作即可。如果真正的失败模式是"subagent 产出质量低"，那么约束范围不解决质量问题——需要的是更好的任务描述和审阅流程。

**置信度**：75。文档从未明确说明它要解决的失败模式是什么，也未提供 subagent 不当行为的具体案例来支撑其约束设计。

### 1.2 成功标准怀疑

文档第五节验证清单假设：如果 subagent 满足所有约束条件并返回结构化输出，任务就成功了。但这个清单可以全部打勾，而 subagent 产出的代码可能：
- 在技术上正确但架构上错误
- 与并发 subagent 的修改冲突
- 解决了症状而非根因
- 引入技术债务

验证清单衡量的是*合规性*，而非*正确性*。这两者在复杂软件任务中常常背离。

### 1.3 框架效应

整个文档以安全/控制框架构建 subagent 关系——subagent 是潜在的危险行为者，需要被约束。这种框架自然导向过度约束的方案。

对照文档（`subagent-constraints-framework.md`）使用了更有效的框架："信任 subagent 的判断力，同时保持主 agent 的最终裁决权"。它不是从"如何防止 subagent 做坏事"出发，而是从"如何让 subagent 在清晰边界内有效工作"出发。这种框架差异解释了为什么两份文档的解决方案如此不同——一份是控制导向，一份是赋能导向。

## 2. 假设浮现（Assumption Surfacing）

### 2.1 环境假设：通用 AI Agent 平台

第四节包含 Hermes-agent 特定的代码：

```typescript
delegate_task({
  goal: "修复...",
  toolsets: ["terminal", "file"]
})
```

文档隐含假设读者使用 Hermes agent 框架。`delegate_task()` API、`toolsets` 参数——这些是平台特定的。框架声称是"系统性方案"，但实现细节泄露了平台绑定。如果在 Claude Code、LangChain agents、或其他平台上实施，读者需要自行翻译这些概念。

**后果**：读者可能将 API 设计误认为框架要求，而非实施细节。

### 2.2 用户行为假设：主 agent 具有完美任务分解能力

整个框架——尤其是并行调度模板——假设主 agent 能够：
1. 正确识别根因
2. 将任务分解为完全独立的子任务
3. 为每个子任务精确指定文件范围
4. 预测所有需要的上下文

**现实**：如果主 agent 能做到以上所有，它就不需要 subagent——它可以直接完成任务。subagent 的价值恰恰在于处理主 agent *不完全了解*的工作。但框架要求主 agent 在派发时就已经完全了解任务边界。

**后果**：主 agent 的分解错误会被放大——subagent 在错误范围内高效执行错误的任务。

### 2.3 规模假设：任务适合单会话完成

"时间范围：单次任务，不跨会话"（第一节，约束 1.3）假设所有可委派的任务都能在一个 agent 会话内完成。

**现实**：复杂实现任务（如"为整个模块添加审计日志"）可能需要跨越多轮迭代、审阅、和修订。强制单会话边界要么导致不完整的工作，要么导致 subagent 赶工而牺牲质量。

**后果**：大型任务无法委派，或委派后产出不完整。

### 2.4 上下文假设：会话历史是噪音而非信号

"无继承：subagent 不继承主 agent 的会话历史"（第一节，约束 2.2）假设主 agent 的会话历史对 subagent 没有价值。

**现实**：在软件任务中，"我们如何到达当前状态"常常是关键信息：
- 主 agent 可能已经尝试并放弃了某些方法
- 项目约束可能在会话中被发现和讨论
- 关于代码库的隐性知识在对话中积累

剥夺 subagent 的这些信息意味着它可能重复失败的尝试，或违反在会话中建立的约定。

**后果**：上下文隔离节省了 prompt token，但浪费了认知 token——subagent 重新发现已知信息。

### 2.5 认知假设：所有 subagent 类型需要相同约束

框架对所有 subagent 统一应用四维约束——无论它是做研究、实现、审阅还是修复。

**现实**：不同类型的工作需要不同的自主权：
- 研究型 subagent 需要*更宽*的探索范围
- 实现型 subagent 需要精确的文件范围
- 审阅型 subagent 已经自然受限（只读）
- 修复型 subagent 需要理解更广的上下文来诊断根因

统一约束对某些任务类型过度限制，对另一些则限制不足。

## 3. 决策压力测试（Decision Stress-Testing）

### 3.1 决策：上下文隔离作为默认策略

**证伪条件**：如果能证明 subagent 在拥有更多上下文（包括会话历史）时产出更高质量的结果，那么上下文隔离作为默认策略就是错误的。

**证据质量**：文档中没有提供任何证据证明上下文隔离能提升 subagent 质量。它被作为"最小化原则"抛出，但没有与"充分性原则"（提供足够上下文以确保正确性）平衡。

**逆转成本**：高。上下文隔离是整个框架的基础。如果它是错的，所有派发模板都需要重新设计以包含足够的上下文传递。

**负载**：上下文隔离是框架中其他决策的依赖项——结构化输出要求、TDD 模板、验证清单——它们都假设 subagent 只拥有有限的信息。

**置信度**：75。上下文隔离在某些场景下确实有价值（避免 token 浪费），但作为无条件的默认策略是过度简化。

### 3.2 决策：TDD 作为独立模板（模板 C）

**证伪条件**：如果存在常见的委派场景（如简单 bug 修复、配置更改、文档更新）中 TDD 增加开销而无益，那么 TDD 作为独立核心模板就是过度规定。

**证据**：AGENTS.md 中的 DoD 确实要求 TDD 用于代码实现。但这不意味着 TDD 模板应该在 subagent 编排框架中占据与"单任务"和"并行调度"平级的地位。TDD 是一种*质量策略*，而非一种*编排策略*。

**逆转成本**：低。模板 C 可以降级为模板 A 的一个参数（`quality_mode: "tdd"`）。

**决策-范围错配**：将一个具体的质量流程提升到与编排模式平级，混淆了"如何组织工作"与"如何保证质量"两个不同维度。

### 3.3 决策：并行调度中的冲突检测但无冲突解决

模板 B 规定集成验证："等待所有 subagent 完成 → 检查修改是否冲突 → 运行完整测试套件 → 汇总结果"。

**证伪条件**：如果并行 subagent 的修改冲突率高于预期，且"检查冲突"步骤没有配套的解决机制，那么并行调度在实际中会产生净负收益。

**问题**：框架检测冲突但未定义谁来解决。主 agent 既没有参与修改，又需要裁决冲突——这需要它理解两个 subagent 的修改意图，而在上下文隔离下它缺乏这个理解。

**后果**：并行调度在理论上节省时间，但在实践中可能因为冲突解决开销而比串行执行更慢。

**置信度**：75。这是框架中一个真实但未被承认的差距。

### 3.4 决策：禁止操作作为通用列表

第一节约束 3.1 要求"明确列出不可做的事情"。

**证伪条件**：随着项目演进，新工具、新 API、新风险出现。维护一个全面的禁止清单需要持续更新。如果清单变得过长或不完整，subagent 可能通过清单未覆盖的路径越界。

**对比**：对照文档采用了更优越的策略："列出禁止事项，而非列举允许事项。只约束真正危险的操作"——并进一步将禁止项分为通用（越界操作）+ 按领域实例化。这更具可扩展性。

**置信度**：50。文档的意图正确，但实现比对照文档薄弱。

## 4. 简化压力（Simplification Pressure）

### 4.1 抽象审计

| 抽象 | 消费者数量 | 评估 |
|------|-----------|------|
| 四维约束模型 | 被 3 个模板引用 | 合理——但对照文档的 5 层模型更清晰 |
| 模板 A（单任务） | 核心用例 | 必要 |
| 模板 B（并行调度） | 特定场景 | 可以参数化到模板 A 中（`parallel: true`） |
| 模板 C（TDD） | 特定质量策略 | 应是模板 A 的一个参数，不是独立模板 |
| 常见错误（第三节） | 教学辅助 | 信息已在模板中编码，属于冗余 |
| 代码示例（第四节） | 平台绑定 | 应与通用框架分离 |
| 验证清单（第五节） | 派发检查 | 必要，但可与对照文档的检查清单合并 |

### 4.2 减法测试

**如果移除模板 B（并行调度）**：失去并行调度的显式指导，但主 agent 仍可通过多次调用模板 A 实现并行调度。模板 B 的价值仅在于显式展示并行模式——价值适中。

**如果移除模板 C（TDD）**：TDD 要求仍然存在于 AGENTS.md 的 DoD 中。移除不会降低质量要求，但会减少框架中的冗余。

**如果移除第三节（常见错误）**：信息冗余——模板本身已经体现了"好"的写法。移除后信息不丢失。

**如果移除第四节（代码示例）**：框架变得平台无关，更具通用性。损失：失去对 Hermes 用户的即用参考。

### 4.3 复杂度预算

文档总复杂度：4 个约束维度 + 3 个模板 + 4 个反模式 + 代码示例 + 2 个验证清单 = ~15 个独立概念。

对照文档：5 个层次（每个 2-4 个子概念）+ 1 个通用模板 + 1 个检查清单 = ~12 个独立概念。但对照文档的概念之间具有层级递进关系，而主文档的概念是并列的。

**判断**：主文档用更高的概念数量覆盖了与对照文档大致相同甚至更少的内容。对照文档用更少的概念说出了更多东西。

## 5. 替代方案盲点（Alternative Blindness）

### 5.1 被忽略的替代方案

**替代方案 A：能力声明模型**
- 定义 subagent 的*能力*（可读哪些文件、可写哪些文件、可用的工具），而非禁止事项
- 优势：不需要预判所有可能的错误行为。任何不在能力范围内的操作自然不可行
- 文档中未提及

**替代方案 B：信任分级模型**
- 按任务风险等级授予不同自主权
  - Tier 1 只读（研究/分析）：宽范围，最小约束
  - Tier 2 限定写入：精确文件范围，标准约束
  - Tier 3 自由写入：目录级范围，事后审阅
- 对照文档暗示了这种思路（"按领域实例化"），主文档没有
- 文档中未提及

**替代方案 C：结果导向委派**
- 只指定期望结果和验收标准，不指定操作范围
- 这正是 AGENTS.md 规则四（"目标驱动执行"）倡导的原则，但在 subagent 框架中被反转
- 文档中未提及

**替代方案 D：审阅门控执行**
- Subagent 自由执行 → 产出 → 主 agent 审阅 → 通过则接受，不通过则拒绝并重派
- 利用审阅步骤（已在 DoD 中）作为主要的约束机制，而非前置约束
- 文档中未提及

### 5.2 构建 vs 使用

对照文档（`subagent-constraints-framework.md`）已经存在于同一目录中，且提供了：
- 更清晰的架构（5 层递进）
- 更灵活的设计（escape hatches、STOP 条件）
- 更通用的模板（领域无关）
- 更实用的派发检查清单

主文档似乎没有参考或复用对照文档的任何内容。这是重复建设。

### 5.3 什么都不做的基线

如果此文档不被采纳，项目仍有：
- AGENTS.md 中的 subagent 行为边界（硬约束）
- AGENTS.md 中的 subagent 作用域约束
- AGENTS.md 中的 DoD 流程
- 对照文档中的 5 层约束框架

什么都不做的成本：缺少一个统一的"如何写 subagent prompt"的模板。这是一个真实的但较小的缺口——对照文档的"通用派发模板"已经填补了大部分。

## 6. 对照文档分析：subagent-constraints-framework.md 更优秀之处

对照文档在以下维度显著优于主文档：

| 维度 | 主文档 | 对照文档 |
|------|--------|----------|
| **架构** | 4 个并列维度，无层级 | 5 层递进模型：权限 → 范围 → 契约 → 输出 → 协调 |
| **哲学** | 控制导向（"禁止"优先） | 信任与赋能（"信任 subagent 的判断力"） |
| **灵活性** | 刚性约束 | Escape hatches：STOP 条件、升级机制 |
| **领域分离** | 框架与领域混在一起（TDD 在编排框架中） | 明确区分框架（层 1/2/4/5）与领域实例化（层 3） |
| **输出模型** | 强制结构化返回 | "简单任务可以直接返回文本，不需要强制写文件" |
| **操作可追溯** | 无 | Ledger 追踪（"写入 ledger，防止上下文压缩后重复派发"） |
| **禁止操作设计** | 需要"明确列出" | "列出禁止事项，而非列举允许事项。只约束真正危险的操作" |
| **平台无关性** | 含 Hermes 特定代码 | 完全平台无关 |
| **搜索范围** | 刚性边界 | "如果需要跨目录搜索，主 agent 应显式扩大范围" |

对照文档的核心洞察：约束应该*保护真正危险的操作*，而非*限制所有可能的自主行为*。这个洞察在主文档中完全缺失。

## 7. 从零重新设计

如果从零设计这个框架，我会：

1. **采用对照文档的 5 层模型作为基础架构**——它已经证明了其优越性。

2. **定义 subagent 角色类型而非统一约束**：
   - 分析师（只读）：宽搜索范围，最小约束
   - 实施者（限定写入）：精确文件范围 + 禁止操作清单
   - 审阅者（只读 + 评估）：无文件修改能力
   每种角色有预设的约束配置，主 agent 选择角色而非手动指定每个约束。

3. **单一参数化派发模板**：
   ```
   派发参数：
   - 角色：analyst | implementer | reviewer
   - 目标：一句话描述
   - 需求：BRIEF_FILE 路径
   - 范围：允许的文件/目录
   - 质量要求：tdd | review-gated | coverage-targeted | none
   - 并行：true | false
   ```
   一个模板替代三个，参数区分行为而非模板区分。

4. **取消通用上下文隔离**：改为"默认传递关键上下文 + 按需裁剪"模型。除非 token 预算紧张，否则传递会话摘要而非完全隔离。

5. **添加集成冲突解决协议**：不仅仅是"检查冲突"，而是定义主 agent 如何裁决跨 subagent 冲突——基于 diff 比较、意图推断、或回退到串行重新执行。

6. **取消单会话限制**：允许长任务 subagent 通过中间检查点报告进度，而非强制单次完成。

7. **与 AGENTS.md 的硬约束整合**：直接引用 AGENTS.md 中的禁止操作清单，而非在框架中重复定义。

| A2 | 信任分级委派（按任务风险给予不同自主权） | 匹配实际需求多样性 |
| A3 | 审阅门控（少前置约束，多后置审阅） | 与 AGENTS.md DoD 流程更一致 |
| A4 | 直接采用对照文档作为唯一框架 | 消除重复，提升质量 |

## 总体裁决

这份文档试图解决一个真实问题（subagent 编排），但它在三个关键维度上走错了方向：

1. **哲学上**：选择了"控制"而非"赋能"，导致框架对 subagent 的不信任内嵌到每个约束中。
2. **架构上**：采用并列维度而非层级递进，且缺少对照文档中已实现的 escape hatches 和信任模型。
3. **范围上**：框架、质量策略、平台代码混在一起，缺乏清晰的关注点分离。

**建议**：不要修订此文档——而是以对照文档（`subagent-constraints-framework.md`）为起点，将其 5 层模型作为唯一框架。对照文档的架构更清晰、设计更灵活、哲学更健康。将主文档中独有的有价值部分（并行调度模式、验证清单）提取并整合到对照文档中，而非维护两份竞争的框架文档。

对照文档中有一句话本身就是对此文档的最佳判词：**"用原则指导，而非用规则束缚。"**
