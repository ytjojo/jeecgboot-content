# subagent 约束系统性方案(subagent-constraints-framework)

约束体系分为 **5 层**，从硬性边界到执行契约层层递进。

---

## 第 1 层：权限边界 — "你不能做什么"

核心原则：**列出禁止事项，而非列举允许事项**。只约束真正危险的操作，不要试图覆盖所有边界情况。

### 通用禁止项

```
禁止操作：
- 外部影响：git push、调用外部 API、发送消息/邮件
- 规则修改：修改项目配置文件、规则文件、构建脚本
- 越权操作：执行需要人类审批的操作（如合并到主分支、发布、付费操作）
```

### 按领域实例化

```
编程场景：
  禁止：git push, 合并 master, 修改 AGENTS.md, 删除 openspec changes

写作场景：
  禁止：修改原文的事实信息, 删除未标记为冗余的段落

研究场景：
  禁止：跳过指定信息源
```

### read-only subagent 输出边界

优先描述问题，代码示例仅在必要时使用。

- ✅ 允许：描述问题（路径+行号）、指出差异、标注严重级别、方向性建议
- ⚠️ 谨慎：完整代码实现（仅在代码片段比文字描述更清晰时使用）

---

## 第 2 层：作用域边界 — "你只能在哪儿操作"

核心原则：**明确操作范围，但不限制合理的探索**。

| 规则 | 含义 |
|------|------|
| **路径白名单** | 主 agent 指定允许操作的文件/目录清单，subagent 不得超出 |
| **搜索边界** | 使用有界 glob，避免无意义的跨目录扫描 |

### 通用模板

```
操作范围：
- 允许操作的文件/目录：[路径清单，相对或绝对均可]
- 允许搜索的范围：[有界 glob，如 src/docs/**/*.md]
- 超出范围发现相关材料？→ 报告位置，等待指示
```

**灵活性**：如果任务需要跨目录搜索（如"找到所有引用 X 的地方"），主 agent 应显式扩大范围，而非让 subagent 自行突破边界。

---

## 第 3 层：任务契约 — "做什么 & 何时停止"

核心原则：**明确目标和完成标准，让 subagent 自主判断何时停止**。

### 前置澄清

开始前确认：
- 输出物是什么？（文件/报告/分析/修改）
- 验收标准是什么？（什么叫"完成"）

### STOP 触发条件

STOP 是**停止执行的条件**。触发以下任一条件时，subagent 必须立即停止执行，不得继续：

- 需要做主 agent 未授权的决策
- 遇到任务描述中没有覆盖的情况，存在多种合理选择
- 发现任务本身可能有问题
- 反复失败（同一操作多次失败时停止，而非硬编码"2次"）

### 升级机制（停止后的状态汇报）

触发 STOP 后，subagent 根据实际情况选择以下状态向主 agent 汇报：

| 情况 | 行动 |
|------|------|
| 完成，符合预期 | 报告完成，进入 review |
| 完成，但有疑虑 | 报告完成，同时说明疑虑 |
| 无法继续（遇到阻碍） | 报告卡点，等待指示 |
| 缺少信息/决策 | 说明需要什么，等待补充 |

**STOP 与升级机制的关系**：

```
触发 STOP 条件
    ↓
停止执行
    ↓
选择汇报状态
    完成 / 完成但有疑虑 / 受阻 / 需要信息
```

STOP 定义"何时必须停"，升级机制定义"停后如何汇报"。两者配合形成完整的执行边界。

### 重试原则

收到"受阻"状态后，主 agent 的处理规则：

- **禁止原样重派发**：不分析原因直接重试，大概率进入循环失败
- 正确流程：**诊断原因 → 调整任务（描述/范围/输入）→ 重新派发**
- 同一任务调整后仍失败 2 次以上：主 agent 应重新评估任务是否可进一步分解，或是否需要人工介入

---

## 第 4 层：输出契约 — "怎么汇报"

核心原则：**简洁明了，便于主 agent 快速理解**。

### 通用模板

```
报告格式：
1. 将完整报告写入 [REPORT_FILE]（如果内容较长）
2. 向主 agent 返回摘要：
   - 状态：完成/完成但有疑虑/受阻/需要信息
   - 产出物：[文件/提交/修改的清单]
   - 验证结果：[如果任务要求验证]
   - 疑虑（如有）
```

**灵活性**：简单任务可以直接返回文本，不需要强制写文件。内容复杂度决定输出形式。

---

## 第 5 层：协调规则 — "主 agent 保留什么权力"

| 主 agent 保留 | 绝不委托 |
|-------------|---------|
| 最终质量裁决 | 是否通过 |
| 跨任务冲突裁决 | "A 任务和 B 任务改了同一个文件——谁对？" |
| 规则/计划修改 | 修改 AGENTS.md、变更计划 scope |
| 外部影响操作 | push、merge、发布、通知 |

### 冲突规则

- **一个文件，同一时刻，只允许一个 Agent 修改**
- 如需多个 Agent 修改同一文件：由主 agent 合并变更，subagent 不得自行合并
- 派发并行任务时，主 agent 负责确保各 subagent 的操作范围无文件交集
- 若任务间存在共享文件依赖 → 改用串行模式（见协作模式）

---

## SubAgent 协作模式

### 独立模式（默认）

适用于 **80%** 的任务。SubAgent 独立完成，不依赖其他 subagent 的中间结果。

```
Main Agent → Agent → 完成 → Review
```

### 串行协作

```
Agent A 完成
    ↓
Main Agent Review（验证 A 的输出）
    ↓
Agent B 以 A 的输出为输入
```

**适用场景**：A 分析 → B 实现、A 设计 → B 编码、A 调研 → B 写报告。

**关键约束**：必须经 Main Agent 中转验证后才交给下一个 Agent，不允许 Agent 之间直接传递。

### 并行协作

```
Main Agent
├─ Agent A（独立任务，文件集 X）
├─ Agent B（独立任务，文件集 Y）
└─ Agent C（独立任务，文件集 Z）
```

**适用场景**：多个独立模块同时开发、多维度同时分析、多文件并行 review。

**派发前提**：各任务操作的文件范围无交集，否则改用串行模式。

## 通用派发模板（领域无关）

```
---
agent: [类型]
description: "[一句话描述任务]"
prompt: |
  你的任务：[一句话]

  ## 需求
  先读取：[BRIEF_FILE — 需求详情]
  这是你唯一的需求来源。

  ## 上下文
  [这个任务在整个工作中的位置，与其他任务的依赖关系]

  ## 协作模式
  [独立模式 / 串行（上游 Agent：[名称]，输出：[文件]）/ 并行（兄弟 Agent：[列表]）]

  ## 操作范围
  允许操作：[路径白名单]
  禁止超出此范围。

  ## 禁止操作
  [从第 1 层按需列出]

  ## STOP 与升级规则
  遇到 STOP 条件 → 立即停止
  停止后 → 选择状态汇报：完成/完成但有疑虑/受阻/需要信息

  ## 输出
  详细报告写入 [REPORT_FILE]，然后返回摘要：
  - 状态
  - 产出物清单
  - 验证结果
  - 疑虑（如有）
```

## 派发检查清单

主 agent 每次派发 subagent 前确认核心项：

```
派发前检查：
□ 1. 任务是否可独立完成？（不依赖其他 subagent 的中间结果）
□ 2. 需求是否已写入 BRIEF_FILE？（不在 prompt 中粘贴全文）
□ 3. 路径白名单是否明确？
□ 4. 禁止操作清单是否按领域实例化列出？
□ 5. 是否没有把历史任务的冗长上下文粘贴进去？
□ 6. 并行派发时，各 Agent 文件操作范围是否无交集？
```

**如果是实现类 subagent（编程等）额外检查**：

```
□ 7. 是否没有替 reviewer 预先判定问题级别？
□ 8. 全局约束是否从 plan/spec 逐字复制？
```

**派发后跟踪**：

```
□ 收到完成 → 进入 review
□ 收到完成但有疑虑 → 先读疑虑，决定是否返工
□ 收到受阻 → 诊断原因 → 调整任务 → 重派发（禁止原样重派发）
□ 收到需要信息 → 补充 → 重派发
□ 同一任务调整 2 次仍受阻 → 评估是否需要分解或人工介入
□ review 通过 → 写入 ledger（防止上下文压缩后重复派发）
□ review 不通过 → 派发 fix subagent → 重新 review
```

## 实际案例

### 编程案例：实现用户导出功能

```
派发模板：
  任务：实现用户列表 CSV 导出功能

  操作范围：
  - 允许操作：src/services/user-export.ts, src/controllers/user.ts, tests/user-export/
  - 禁止操作：src/services/user-auth.ts（认证逻辑，不相关）

  禁止：修改数据库 schema、修改 API 路由前缀

  STOP 条件：
  - 发现导出格式需要产品确认 → 停止，报告"需要信息"
  - CSV 库版本不兼容 → 停止，报告"受阻"

  协作模式：独立模式
```

### 文档案例：更新 API 文档

```
派发模板：
  任务：为 /api/v1/content/ 下的接口更新文档

  操作范围：
  - 允许操作：docs/api/content/*.md
  - 禁止操作：docs/api/sys/（系统 API 文档，不相关）

  禁止：修改接口事实描述（参数名、返回值结构），只能补充说明文字和示例

  STOP 条件：
  - 发现文档与代码不一致 → 停止，报告差异位置，等待确认后再修改

  协作模式：独立模式
```

### 调研案例：评估日志框架迁移方案

```
派发模板：
  任务：调研从 Log4j 迁移到 Logback 的可行性和影响范围

  操作范围：
  - 允许搜索：src/**/*.java（找 Log4j 引用）, pom.xml, build.gradle
  - 允许操作：无（只读调研，输出报告）

  禁止：修改任何代码或配置文件

  STOP 条件：
  - 发现 Log4j 使用量超出预期 → 停止，报告初步统计，等待是否继续

  协作模式：独立模式（纯调研，无文件修改）
```

## 总结：五层约束的本质

```
第 1 层 → 权限：不能做什么（领域相关，实例化）
第 2 层 → 范围：能在哪里做（通用规则）
第 3 层 → 契约：做什么 + 何时停（通用，STOP 条件 + 状态汇报 + 重试原则）
第 4 层 → 输出：怎么汇报（通用，简洁优先）
第 5 层 → 协调：谁有最终决定权 + 冲突规则（通用，主 agent 保留）
```

前三层回答"subagent 的边界"，后两层回答"主 agent 怎么管 subagent"。协作模式（独立/串行/并行）是这五层约束之上的**编排维度**——定义 subagent 之间的协作拓扑。编程场景的 TDD、reviewer、coverage、commit 规范等等，都是第 3 层（任务契约）的**领域实例化**，不是框架本身。

**设计原则**：
- 约束真正危险的操作，不试图覆盖所有边界情况
- 给 subagent 留 escape hatch，而非死板的"禁止"
- 用原则指导，而非用规则束缚
- 信任 subagent 的判断力，同时保持主 agent 的最终裁决权
