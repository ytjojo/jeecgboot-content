# subagent 约束系统性方案

约束体系分为 **5 层**，从硬性边界到执行契约层层递进。

---

## 第 1 层：权限边界 — "你不能做什么"

这是唯一与领域绑定的层。核心原则：**列出禁止事项，而非列举允许事项**。

### 通用模板

```
禁止操作：
- [写操作类]：删除、移动、重命名文件/目录，除非任务明确要求
- [外部影响类]：git push、调用外部 API、发送消息/邮件
- [规则修改类]：修改项目配置文件、规则文件、构建脚本
- [越权类]：执行需要人类审批的操作（如合并到主分支、发布、付费操作）
```

### 按领域实例化

```
编程场景：
  禁止：git push, 合并 master, 修改 AGENTS.md, 删除 openspec changes

写作场景：
  禁止：修改原文的事实信息, 删除未标记为冗余的段落,
        添加未经核实的引用

研究场景：
  禁止：编造数据来源, 跳过指定信息源, 输出确定性结论当证据不足时
```

### read-only subagent 输出边界

- ✅ 允许：描述问题（路径+行号）、指出差异、标注严重级别、方向性建议
- ❌ 禁止：输出可落地代码实现（diff、完整组件代码）

## 第 2 层：作用域边界 — "你只能在哪儿操作"

通用规则只有两条：

| 规则 | 含义 |
|------|------|
| **路径白名单** | 主 agent 指定允许操作的文件/目录清单，subagent 不得超出 |
| **搜索边界** | 绝对路径、有界 glob、禁止 `find /` / `find ~/` / `find ../` |

### 通用模板

```
操作范围：
- 允许操作的文件/目录：[绝对路径清单]
- 允许搜索的范围：[有界 glob，如 /project/src/docs/**/*.md]
- 超出范围发现相关材料？→ 报告位置，不自行扩展
```

---

## 第 3 层：任务契约 — "做什么 & 何时停止"

**这是最核心的通用层**。四条规则适用所有 subagent：

### 前置澄清

```
开始前必须确认：
- 输出物是什么？（文件/报告/分析/修改）
- 验收标准是什么？（什么叫"完成"）
- 有任何模糊点？（不问 = 猜 = 出错）
```

### 升级机制（4 种状态，领域无关）

| 状态 | 何时使用 | 主 agent 响应 |
|------|---------|-------------|
| `DONE` | 完成，自信符合要求 | 进入 review |
| `DONE_WITH_CONCERNS` | 完成但有疑虑 | 先读疑虑，决定是否返工 |
| `BLOCKED` | 无法继续 | 诊断阻碍 → 调整 → 重派发 |
| `NEEDS_CONTEXT` | 缺信息/决策 | 补充后重派发 |

### STOP 触发条件（通用）

- 需要做主 agent 未授权的决策
- 遇到任务描述中没有覆盖的情况，存在多种合理选择
- 发现任务本身可能有问题
- 连续失败两次同样的操作

---

## 第 4 层：输出契约 — "怎么汇报"

**格式分离原则**：详细内容写入文件，返回摘要（≤15 行）。

### 通用模板

```
报告格式：
1. 将完整报告写入 [REPORT_FILE]
2. 向主 agent 只返回：
   - Status: DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
   - 产出物：[文件/提交/修改的清单]
   - 验证结果：[如果任务要求验证]
   - 疑虑（如有）
```

### read-only subagent 额外约束

```
只读 subagent → 只描述问题，不输出实现。

✅ 允许：路径+行号、差异描述、严重级别、方向建议
❌ 禁止：完整实现代码、可直接应用的 diff、可替换代码块
```

---

## 第 5 层：协调规则 — "主 agent 保留什么权力"

| 主 agent 保留 | 绝不委托 |
|-------------|---------|
| 最终质量裁决 | 是否通过 |
| 跨任务冲突裁决 | "A 任务和 B 任务改了同一个文件——谁对？" |
| 规则/计划修改 | 修改 AGENTS.md、变更计划 scope |
| 外部影响操作 | push、merge、发布、通知 |

---

## 通用派发模板（领域无关）

```
---
agent: [类型]
description: "[一句话描述任务]"
model: [按任务复杂度选择]
prompt: |
  你的任务：[一句话]

  ## 需求
  先读取：[BRIEF_FILE — 需求详情]
  这是你唯一的需求来源。

  ## 上下文
  [这个任务在整个工作中的位置，与其他任务的依赖关系]

  ## 操作范围
  允许操作：[绝对路径白名单]
  禁止超出此范围。

  ## 禁止操作
  [从第 1 层按需列出]

  ## 开始前
  对需求、验收标准、方法有任何不清楚？**现在问**。不要猜测。

  ## 升级规则
  做不了 → BLOCKED（描述卡点）
  缺信息 → NEEDS_CONTEXT（描述需要什么）
  完成了但有疑虑 → DONE_WITH_CONCERNS（说明疑虑）

  ## 输出
  详细报告写入 [REPORT_FILE]，然后返回（≤15行）：
  - Status
  - 产出物清单
  - 验证结果
  - 疑虑（如有）
```

## 派发检查清单

主 agent 每次派发 subagent 前逐项确认：

```
派发前检查：
□ 1. 任务是否可独立完成？（不依赖其他 subagent 的中间结果）
□ 2. 需求是否已写入 BRIEF_FILE？（不在 prompt 中粘贴全文）
□ 3. 路径白名单是否使用绝对路径？
□ 4. 禁止操作清单是否按领域实例化列出？
□ 5. REPORT_FILE 路径是否已指定？
□ 6. model 是否显式指定？（省略 = 继承最贵模型）
□ 7. 是否没有把历史任务的冗长上下文粘贴进去？
```

**如果是实现类 subagent（编程等）额外检查**：

```
□ 8. 是否没有替 reviewer 预先判定问题级别？
    （禁止："treat it as Minor"、"不用检查 X"、"plan 已决定所以不算问题"）
□ 9. 全局约束是否从 plan/spec 逐字复制？
□ 10. diff 文件是否通过脚本生成，而非粘贴到 prompt？
```

**派发后跟踪**：

```
□ 收到 DONE → 进入 review
□ 收到 DONE_WITH_CONCERNS → 先读疑虑，决定是否返工
□ 收到 BLOCKED → 诊断 → 调整 → 重派发
□ 收到 NEEDS_CONTEXT → 补充 → 重派发
□ review 通过 → 写入 ledger（防止上下文压缩后重复派发）
□ review 不通过 → 派发 fix subagent → 重新 review
```

---

## 总结：五层约束的本质

```
第 1 层 → 权限：不能做什么（领域相关，实例化）
第 2 层 → 范围：能在哪里做（通用规则）
第 3 层 → 契约：做什么 + 何时停（通用，4种状态）
第 4 层 → 输出：怎么汇报（通用，文件/摘要分离）
第 5 层 → 协调：谁有最终决定权（通用，主 agent 保留）
```

前三层回答"subagent 的边界"，后两层回答"主 agent 怎么管 subagent"。编程场景的 TDD、reviewer、coverage、commit 规范等等，都是第 3 层（任务契约）的**领域实例化**，不是框架本身。
