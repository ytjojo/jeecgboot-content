Subagent 编排系统性方案
    
    一、核心约束维度
    
    1. 范围约束 (Scope Isolation)
    
    - 文件范围：只允许修改指定文件/目录
    - 功能范围：只解决指定问题，不做额外重构
    - 时间范围：单次任务，不跨会话
    
    
    2. 上下文约束 (Context Isolation)
    
    - 最小化原则：只传递完成任务必需的信息
    - 无继承：subagent 不继承主 agent 的会话历史
    - 显式注入：所有背景信息通过 context 参数明确提供
    
    
    3. 行为约束 (Behavioral Constraints)
    
    - 禁止操作：明确列出不可做的事情
    - 边界条件：遇到不确定时的处理方式
    - 失败处理：出错时的回退策略
    
    
    4. 输出约束 (Output Format)
    
    - 结构化返回：要求具体格式的总结
    - 验证标准：明确成功条件
    - 可追溯性：要求列出修改的文件和原因
    
    
    二、主 Agent 启动 Subagent 提示词模板
    
    模板 A：单任务 Subagent（focused task）
    
    
    目标：[具体、可衡量的目标]
    
    上下文：
    - 背景：[为什么需要这个任务]
    - 相关文件：[文件路径列表]
    - 当前状态：[问题描述或需求说明]
    
    约束：
    - 只修改：[明确允许修改的文件/目录]
    - 禁止：[明确不允许的操作]
    - 边界：[遇到不确定时的处理方式]
    
    预期输出：
    - 返回格式：[结构化总结的要求]
    - 验证标准：[如何确认任务成功]
    - 必须包含：[修改清单、测试结果等]
    
    示例：
    ────────────────────────────────────────
    目标：修复 src/auth/login.test.ts 中的 3 个失败测试
    
    上下文：
    - 背景：CI 报告显示登录模块测试失败
    - 相关文件：src/auth/login.test.ts, src/auth/login.ts
    - 当前状态：测试 "should validate token expiry" 等 3 个用例失败，
      错误信息显示时间戳比较问题
    
    约束：
    - 只修改：src/auth/login.test.ts 和 src/auth/login.ts
    - 禁止：不修改其他测试文件，不重构无关代码
    - 边界：如果是环境配置问题，返回错误信息而非强行修复
    
    预期输出：
    - 返回：根因分析 + 修复方案 + 修改的文件列表
    - 验证：所有 3 个测试通过
    - 必须包含：每个测试的修复说明
    ────────────────────────────────────────
    
    
    模板 B：并行多 Subagent（parallel dispatch）
    
    
    任务分解：
    将大问题拆分为 N 个独立子任务，每个 subagent 负责一个独立领域
    
    Subagent 1：
      目标：[子任务 1 的具体目标]
      范围：[文件/模块 1]
      约束：[特定限制]
      
    Subagent 2：
      目标：[子任务 2 的具体目标]
      范围：[文件/模块 2]
      约束：[特定限制]
    
    [并行启动所有 subagent]
    
    集成验证：
    - 等待所有 subagent 完成
    - 检查修改是否冲突
    - 运行完整测试套件
    - 汇总结果
    
    
    模板 C：TDD 驱动的 Subagent（test-first）
    
    
    目标：实现 [功能名称]，遵循 TDD 流程
    
    阶段 1 - 测试先行：
    - 根据需求编写失败测试
    - 测试必须覆盖：[核心场景列表]
    - 禁止：不实现任何业务逻辑
    
    阶段 2 - 最小实现：
    - 只写让测试通过的最少代码
    - 禁止：不添加测试未要求的功能
    - 约束：代码必须通过所有测试
    
    阶段 3 - 重构（可选）：
    - 在保持测试通过的前提下优化代码
    - 禁止：不改变外部行为
    
    预期输出：
    - 测试文件路径和测试用例列表
    - 实现文件路径
    - 测试运行结果（必须全部通过）
    
    
    三、常见错误与规避
    
    ❌ 错误 1：范围过大
    
    坏：「修复所有测试」
    好：「修复 src/auth/login.test.ts 中的 3 个失败测试」
    
    
    ❌ 错误 2：上下文不足
    
    坏：「修复竞态条件」
    好：「修复 src/agent/tool-executor.ts 中的竞态条件，错误日志如下：[粘贴日志]」
    
    
    ❌ 错误 3：缺少约束
    
    坏：「优化这个模块」
    好：「优化 src/utils/cache.ts 的性能，只修改该文件，不改变公共 API」
    
    
    ❌ 错误 4：输出模糊
    
    坏：「修好它」
    好：「返回：根因分析 + 修改的文件列表 + 测试通过截图」
    
    
    四、实际调用示例（Hermes Agent）
    
    typescript
    // 单任务 subagent
    delegate_task({
      goal: "修复 src/auth/login.test.ts 中的 3 个失败测试",
      context: `
        背景：CI 报告显示登录模块测试失败
        相关文件：src/auth/login.test.ts, src/auth/login.ts
        错误信息：时间戳比较失败，expected 1703001600 but got 1703001599
        
        约束：
        - 只修改上述两个文件
        - 不修改其他测试
        - 如果是环境配置问题，返回错误信息
        
        预期输出：
        - 根因分析
        - 修复方案
        - 修改的文件列表
        - 测试运行结果
      `,
      toolsets: ["terminal", "file"]
    })
    
    // 并行多 subagent
    delegate_task({
      tasks: [
        {
          goal: "修复 agent-tool-abort.test.ts 的 3 个失败测试",
          context: "错误日志：[粘贴]。只修改该文件和 src/agent/tool-executor.ts",
          toolsets: ["terminal", "file"]
        },
        {
          goal: "修复 batch-completion-behavior.test.ts 的 2 个失败测试",
          context: "错误日志：[粘贴]。只修改该文件和相关实现",
          toolsets: ["terminal", "file"]
        },
        {
          goal: "修复 tool-approval-race-conditions.test.ts 的 1 个失败测试",
          context: "错误日志：[粘贴]。只修改该文件",
          toolsets: ["terminal", "file"]
        }
      ]
    })
    
    
    五、验证清单
    
    启动 subagent 前检查：
    - [ ] 目标是否具体、可衡量？
    - [ ] 上下文是否最小化且充分？
    - [ ] 约束是否明确（文件范围、禁止操作）？
    - [ ] 输出格式是否清晰？
    - [ ] 验证标准是否可执行？
    - [ ] 是否与其他 subagent 有冲突？
    
    完成后检查：
    - [ ] 返回结果是否符合预期格式？
    - [ ] 修改是否在允许范围内？
    - [ ] 测试是否全部通过？
    - [ ] 是否引入新的问题？
