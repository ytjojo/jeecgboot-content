# 决策记录

## D1: 技能定位 — 综合验证而非单项检测

- **时间**：2026-06-19
- **决策**：本技能合并漂移检测（文档 vs 代码）+ 架构审核（代码 vs 架构规范）两个子领域，产出统一验证报告
- **备选方案**：
  - 方案 A：拆分为两个独立技能（`openspec-code-drift-sync` + `openspec-architecture-audit`）
  - 方案 B：合并为一个技能，内部分两个检查阶段
- **选择理由**：用户在 change 完成后需要**一次性**完成所有同步检查工作。两个技能会导致重复读取上下文、两次 subagent 调度，效率低。合并后内部分阶段执行，共享 change 目录上下文
- **影响**：技能文档需要清晰分隔两个阶段；subagent prompt 需要独立编写但共享输入

## D2: 执行方式 — 必须通过 subagent

- **时间**：2026-06-19
- **决策**：审核检查必须通过 subagent 执行，主 agent 仅负责调度和报告汇总
- **理由**：
  - 检查范围大（API 对比 + 字段对齐 + 业务规则 + 设计遵循 + 数据一致 + 架构 7 维度），单次执行 context window 必然溢出
  - 与 `openspec-review-change` 和 `openspec-verify-change` 保持一致
  - subagent 独立上下文避免主 agent 干扰
- **subagent 分工**：
  - 后端漂移检测 agent：prompt-openspec.md 6 维度
  - 前端漂移检测 agent：prompts-frotend.md 5 维度
  - 架构审核 agent：architecture-audit.md 7 维度
  - 报告汇总 agent：合并 3 份 subagent 输出 → 统一报告

## D3: 前后端检测 — 分 agent 执行

- **时间**：2026-06-19
- **决策**：后端和前端漂移检测使用独立 subagent 分派，各自读取不同的代码目录
- **理由**：
  - 后端代码在 `jeecg-boot/`，前端代码在 `jeecgboot-vue3/`，文件路径和检查逻辑完全不同
  - 一次混检容易遗漏或混用规则
  - 前后端 change 在 `change-prd-mapping.yaml` 中配对但不一定同时完成
- **触发条件**：仅当 change 有配对 change 且配对 change 目录存在时才执行跨端检测

## D4: 严重级别 — 沿用现有三级体系

- **时间**：2026-06-19
- **决策**：使用 CRITICAL / WARNING / SUGGESTION 三级（架构审核额外引入 ADVISORY）
- **理由**：与 `openspec-review-change` 和 `openspec-verify-change` 保持一致
- **注意**：架构审核使用 ADVISORY 替代 SUGGESTION，因语义更精确（"建议" vs "建议但更弱"）。在统一报告中将 ADVISORY 合并到 SUGGESTION

## D5: 真相源策略 — 冲突类别驱动

- **时间**：2026-06-19
- **决策**：不采用"文档永远正确"或"代码永远正确"的绝对策略，而是根据冲突类别动态判定
- **策略矩阵**：
  | 冲突类别 | 真相源 | 处理 |
  |---------|--------|------|
  | 表面冲突 | 代码 | 更新文档 |
  | 代码更优 | 代码 | 更新文档 |
  | 文档更优 | 文档 | 修复代码 |
  | 双向缺陷 | 无 | 人工介入 |
  | 业务规则 | 文档 | 修复代码 |
  | 安全约束 | 文档 | 修复代码 |

## D6: 输出产物 — 单文件综合报告

- **时间**：2026-06-19（修订：2026-06-19）
- **决策**：产出**一份**完整报告到 `{changeDir}/`：
  `verify-report-{timestamp}.md` — 漂移检测（后端 + 前端）+ 架构审核 + 门禁判定合并
- **修订理由**：用户要求合并为单文件。漂移和架构两部分用 `## 第一部分` / `## 第二部分` 分隔，阅读时一次性获取全貌，归档时只需追踪一份文件
- **结构**：
  ```
  verify-report-{timestamp}.md
  ├── 门禁判定 + 得分总览
  ├── 第一部分：漂移检测（后端 6 维 + 前端 5 维）
  │   ├── 概览 + 逐维度详情
  │   ├── 漂移类型分布 + 同步方案 + 根因分析
  ├── 第二部分：架构审核（7 维度）
  │   ├── 概览 + 逐维度详情
  │   ├── 架构健康度
  ├── 问题合并统计 + 阻断问题
  └── 修复优先级 + 量化指标 + 后续步骤
  ```

## D7: 版本对应 — 基于 Git Diff

- **时间**：2026-06-19
- **决策**：以 change 的 feature 分支与 base 分支的 `git diff` 确定变更范围
- **理由**：
  - 项目已有明确的分支管理规范（worktree + feature 分支）
  - change 的所有变更都在一个分支上
  - 对比 base 分支可精确获取变更文件列表

## D8: 检测策略 — 默认增量，支持全量

- **时间**：2026-06-19
- **决策**：默认对 change 涉及的变更文件进行增量检测，提供 `--full` 选项进行全量扫描
- **理由**：
  - 日常使用增量足够，速度快
  - 定期审计或重大重构后需要全量扫描
  - 全量扫描可能产生大量报告，不作为默认行为

## D9: 白名单机制 — 注解标记

- **时间**：2026-06-19
- **决策**：使用注释标记 `@drift-ignore` 进行豁免
- **标记方式**：
  ```java
  // @drift-ignore: 实验性功能，代码先行，文档待补充
  ```
  ```yaml
  # @drift-ignore: 以代码为准，docs 待更新
  ```
- **适用范围**：文件级、方法级、字段级

## D10: 技能触发 — 与 openspec-verify-change 互补

- **时间**：2026-06-19
- **决策**：`openspec-verify-change` 负责基础的三维验证（完整性/正确性/一致性），本技能负责**深度**漂移检测和架构审核
- **推荐流程**：`verify-change` → `code-drift-sync` → archive
- **区分**：
  - verify-change: 快速检查，检查 task checkbox、spec requirement 是否实现
  - code-drift-sync: 深度对比，字段级、分支级、架构级

## D11: 输出文件命名 — 时间戳后缀，禁止覆盖

- **时间**：2026-06-19
- **决策**：输出报告文件名必须带时间戳，**绝对禁止覆盖**已有同名文件
- **命名格式**：
  ```
  drift-report-{YYYY-MM-DD-HH-MM}.md
  architecture-audit-report-{YYYY-MM-DD-HH-MM}.md
  verify-report-{YYYY-MM-DD-HH-MM}.md
  ```
- **理由**：
  - 每次运行产出独立报告，保留完整审计轨迹
  - 方便对比多次检测结果，观察漂移趋势
  - 防止误覆盖已有的人工审核记录或修复备注
- **落地规则**：
  - 写入前**必须**检查目标路径是否存在，存在则自动追加时间戳秒级精度（`-HH-MM-SS`）
  - 禁止使用 `drift-report.md` 等无时间戳的固定文件名
  - 报告模板中的文件名引用统一使用 `{timestamp}` 占位符

## D12: 与 Code Review 的边界区分

- **时间**：2026-06-19
- **决策**：本技能与 code review（`/code-review`、`ce-code-review` 等）是**正交关系**，检查目标和检查对象完全不同
- **对比矩阵**：

| 维度 | Code Review | Code-Drift-Sync |
|------|-------------|-----------------|
| **对比目标** | 代码 vs 代码质量标准 | 代码 vs 文档/架构规范 |
| **核心问题** | "这段代码写得对不对、好不好？" | "代码有没有按文档说的做？文档有没有反映代码真实行为？" |
| **关注点** | 逻辑错误、边界条件、性能、安全漏洞、可读性 | API 路径是否一致、字段是否对齐、Scenario 是否覆盖、架构分层是否合规 |
| **输入** | 任意 diff / PR | change 目录的 artifacts（spec/design/tasks） |
| **产出** | Bug 报告、改进建议 | 漂移差异表、同步方案、架构合规报告 |
| **典型发现** | "这里有 N+1 查询""缺少空指针检查" | "spec 要求返回 levelName，但 VO 没有此字段""Controller 直接注入了 Mapper" |
| **触发时机** | 每次提 PR / 每次 commit | change 完成后、归档前 |

- **互补关系**：
  - code review 回答：**代码实现质量如何？**
  - code-drift-sync 回答：**代码与文档是否同步？架构是否合规？**
  - 两者不重复、不替代，change 完成流程中 code review 在前，code-drift-sync 在归档前
- **防止越界**：
  - 本技能**不检查**：逻辑错误、N+1 查询、空指针、算法复杂度、代码风格（这些属于 code review）
  - 本技能**只检查**：文档一致性、字段对齐、Scenario 覆盖、架构分层、模块边界、安全基线
