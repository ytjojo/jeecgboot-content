# 规范文档与代码漂移检测技能需求

## 背景

Agent 开发过程中，文档 → 代码存在漂移偏离性。本技能用于检查文档定义与实际代码的不一致之处，并制定同步策略。

## 漂移类型定义

### 1. 负向漂移（代码落后于文档）
- **定义**：文档已定义，但代码未实现或实现不完整
- **示例**：文档要求支持分页查询，代码只实现了全量返回
- **处理**：代码需要补齐实现

### 2. 正向漂移（代码超越文档）
- **定义**：代码实现了文档未考虑到的场景或优化
- **示例**：文档未提及边界条件处理，代码已实现完善的异常处理
- **处理**：文档需要补充说明

### 3. 冲突漂移（代码与文档不一致）
- **定义**：代码已实现，但与文档定义存在冲突
- **分类处理**：

  | 冲突类别 | 示例 | 处理策略 |
  |---------|------|---------|
  | **表面冲突**（不影响功能） | 变量命名风格差异、API 路径格式不同（`/user` vs `/users`） | **以代码为准**，更新文档 |
  | **代码更优**（实现更完善） | 代码边界条件更清晰、错误处理更完善、性能优化更好 | **以代码为准**，更新文档 |
  | **文档更优**（设计更合理） | 代码实现有缺陷，文档设计更合理 | **以文档为准**，修复代码 |
  | **双向缺陷**（都有问题） | 文档设计有缺陷，代码实现也有问题 | **人工介入**，重新设计 |

### 4. 完成度审查
- **定义**：文档定义了功能，但代码未实现
- **处理**：按代码完成度审查，标记为待实现

## 严重级别定义

| 级别 | 定义 | 示例 | 处理方式 |
|------|------|------|---------|
| **CRITICAL** | 影响核心功能或业务逻辑 | API 参数类型错误、必填字段缺失、业务规则冲突 | 立即阻断，必须修复 |
| **WARNING** | 影响代码质量或可维护性 | 缺少错误处理、命名不规范、缺少注释 | 建议修复，可延后 |
| **SUGGESTION** | 优化建议，不影响功能 | 文档描述可更清晰、代码可进一步优化 | 记录，人工判断 |

## 真相源策略（Source of Truth）

**核心原则**：根据冲突类别动态判定

### 以代码为准的场景
1. **表面冲突**：仅变量名、API 地址等不影响功能实现的差异
2. **代码更优**：代码实现比文档定义更完善，边界条件更清晰
3. **文档未覆盖**：文档未定义但代码已合理实现

### 以文档为准的场景
1. **文档更优**：文档设计更合理，代码实现有缺陷
2. **业务规则**：涉及核心业务逻辑、数据一致性要求
3. **安全约束**：涉及权限控制、数据校验等安全相关

### 人工介入的场景
1. **双向缺陷**：文档和代码都有问题
2. **架构决策**：涉及系统架构、技术选型等重大决策
3. **需求变更**：可能是需求变更导致的漂移


## 最终目标

**规范文档必须反映真实代码行为**

### 核心愿景

建立文档与代码的**单一事实来源**(Single Source of Truth)机制,确保:
- 开发者阅读文档后,能够准确理解代码的实际行为
- 代码变更时,文档同步更新,保持一致性
- 新成员通过文档快速掌握系统,无需依赖口口相传

### 价值主张

| 维度 | 当前痛点 | 目标状态 |
|------|---------|---------|
| **开发效率** | 文档过时导致理解偏差,反复沟通确认 | 文档即真相,减少沟通成本 |
| **代码质量** | 实现与设计脱节,架构约束形同虚设 | 文档驱动开发,架构约束可执行 |
| **维护成本** | 排查问题时需要同时看文档和代码,效率低 | 文档准确反映代码,快速定位问题 |
| **知识传承** | 依赖老员工口述,人员流动导致知识丢失 | 文档沉淀知识,降低人员依赖 |
| **合规审计** | 无法证明实现符合设计规范 | 文档与代码一致,审计可追溯 |

### 衡量标准

#### 定量指标

1. **文档准确率** ≥ 95%
   - 定义: 文档描述与代码实现一致的比例
   - 计算: (一致项数 / 总检查项数) × 100%
   - 目标: CRITICAL 级别漂移 = 0,WARNING 级别漂移 < 5%

2. **漂移检测覆盖率** ≥ 90%
   - 定义: 已检测的文档/代码对占总文档/代码对的比例
   - 计算: (已检测模块数 / 总模块数) × 100%

3. **同步时效** ≤ 24 小时
   - 定义: 代码变更后,文档完成更新的平均时间
   - 目标: CRITICAL 漂移 4 小时内同步,WARNING 漂移 24 小时内同步

4. **漂移复发率** ≤ 10%
   - 定义: 同一模块在 30 天内再次出现漂移的比例
   - 目标: 通过根因分析,降低重复漂移

#### 定性指标

1. **开发者信任度**
   - 开发者愿意仅凭文档理解系统
   - 不再需要"文档不可信,看代码"的口头禅

2. **新人上手速度**
   - 新成员通过文档在 1 周内掌握核心模块
   - 减少对老员工的依赖

3. **审计通过率**
   - 外部审计时,文档与代码一致性检查 100% 通过
   - 无需临时补文档

### 演进路径

#### 阶段 1: 可见性(1-2 个月)
- 建立漂移检测机制
- 生成首份漂移报告
- 团队意识到漂移问题的严重性

#### 阶段 2: 流程化(2-3 个月)
- 将漂移检测集成到 CI/CD
- 建立文档更新流程(代码变更必须同步文档)
- 设定漂移容忍阈值

#### 阶段 3: 自动化(3-6 个月)
- 高置信度漂移自动修复
- 文档生成工具(从代码生成文档)
- 实时漂移监控和告警

#### 阶段 4: 文化化(6-12 个月)
- "文档即代码"成为团队共识
- 文档质量纳入 Code Review 标准
- 漂移率作为团队 KPI

### 成功标准

当以下条件全部满足时,视为目标达成:

- [ ] CRITICAL 级别漂移持续为 0(连续 3 个月)
- [ ] 开发者主动使用文档作为开发依据,而非"被迫"
- [ ] 新人反馈"文档准确,上手快"
- [ ] 审计无需临时补文档,一次通过
- [ ] 漂移检测成为 CI/CD 标准环节,无需人工干预
## 检测范围边界

### 文档类型（输入）
- [ ] PRD（产品需求文档）
- [ ] API 接口文档（OpenAPI/Swagger）
- [ ] 设计文档（技术方案、架构设计）
- [ ] 数据库设计文档（ER 图、表结构说明）
- [ ] 代码注释（JSDoc、JavaDoc）

### 代码类型（输入）
- [ ] Controller 接口定义
- [ ] Service 业务实现
- [ ] Entity/DTO 数据模型
- [ ] 数据库表结构（DDL）
- [ ] 配置文件（application.yml 等）

### 检测粒度
- [ ] 接口级：API 路径、HTTP 方法、参数列表
- [ ] 字段级：字段名、类型、必填性、校验规则
- [ ] 逻辑级：业务规则、边界条件、异常处理
- [ ] 结构级：表结构、索引、关联关系

## 版本对应关系

**问题**：文档和代码都有版本，如何对应？

### 方案 1：基于 Git Commit
- 文档最后修改时间 → 对应最近的 commit
- 代码最后修改时间 → 对应最近的 commit
- 比较两个 commit 之间的差异

### 方案 2：基于版本号
- 文档标注版本号（v1.0, v1.1）
- 代码标注对应版本（通过 tag 或注释）
- 比较同版本号的文档和代码

### 方案 3：基于时间戳
- 文档最后修改时间
- 代码最后修改时间
- 时间早的一方可能落后

## 执行策略

### 策略 1：仅报告
- 生成漂移检测报告
- 人工审核并手动处理
- **适用场景**：项目初期、探索阶段

### 策略 2：自动修复
- 高置信度的简单漂移（命名风格、API 路径）
- 自动生成修复 PR
- **适用场景**：成熟项目、规范明确

### 策略 3：生成 Issue
- 检测到漂移后自动生成 Issue
- 分配给相关负责人
- **适用场景**：团队协作、需要人工确认

### 策略 4：阻断 CI
- CRITICAL 级别漂移禁止合并
- WARNING 级别漂移告警但不阻断
- **适用场景**：严格规范、质量要求高

## 白名单机制

**场景**：有些"漂移"是故意的，需要豁免

### 白名单类型
1. **文件级白名单**：整个文件不参与检测
2. **规则级白名单**：特定规则不检测（如命名风格）
3. **模块级白名单**：特定模块豁免（如实验性功能）

### 白名单标注方式
```yaml
# drift-ignore: 实验性功能，代码先行
# drift-ignore: 文档待更新，已确认以代码为准
```

## 增量 vs 全量策略

### 全量扫描
- **适用场景**：项目初期、重大重构后、定期审计
- **优点**：发现所有历史遗留漂移
- **缺点**：耗时长，可能产生大量报告

### 增量检测
- **适用场景**：每次提交、PR 审查
- **优点**：快速，只检测变更部分
- **缺点**：可能漏掉历史遗留漂移

**建议**：两者结合
- 日常开发：增量检测
- 每周/每月：全量扫描

## 根因追溯

**目标**：发现漂移后，分析为什么会漂移

### 根因分类
1. **文档问题**
   - 文档写错了
   - 文档描述模糊
   - 文档更新不及时

2. **代码问题**
   - 开发没看文档
   - 开发理解偏差
   - 代码重构后未更新文档

3. **流程问题**
   - 文档更新后没通知开发
   - 开发改了代码没更新文档
   - 缺少 Review 环节

### 追溯方式
- 查看 Git 提交历史
- 查看文档修改历史
- 对比时间线

```markdown
  - CRITICAL: 3
  - WARNING: 7
  - SUGGESTION: 5

## 详细列表

### CRITICAL-001: API 参数类型不匹配
- 文档：`POST /api/users`，参数 `age: integer`
- 代码：`age: string`
- 影响：前端传参会报错
- 建议：以文档为准，修复代码

### WARNING-001: 缺少错误处理
- 文档：要求处理用户不存在的情况
- 代码：直接抛异常，未返回友好提示
- 建议：补充错误处理逻辑

## 根因分析
- 文档问题：3 处（20%）

- 流程问题：4 处（27%）
```
