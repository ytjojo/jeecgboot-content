# Loop 工程如何落地
# Loop工程来了——从手写提示词到运行反馈系统
## 核心思想
让 AI 在可验证的循环中，持续逼近目标，直到达成停止条件。

## 循环五步持续迭代流程
整体逻辑：**目标契约 → Agent执行 → 证据反馈 → 停止条件 → 经验回灌**，循环往复直到满足停止条件。

### 1. 目标契约｜先定义完成标准
- 明确目标与范围
- 可量化的验收标准
- 约束与边界条件
- 成功后的交付物形式
> 示例：页面加载 P95 ＜ 1.5s，错误率 ＜ 0.1%，通过全部测试，文档已更新。

### 2. Agent执行｜多工具多实例推进
- 拆解任务，生成计划
- 调用工具（代码、搜索、命令等）
- 多实例并行执行
- 记录思考与行动轨迹
**常用工具**：编辑器、终端、搜索、浏览器、数据库

### 3. 证据反馈｜测试、日志、截图、CI
- 自动化测试结果
- 运行日志与错误堆栈
- 截图/录屏/界面快照
- 性能指标与监控数据
- CI/CD 状态与工件
> 核心原则：证据驱动，用事实说话，避免主观判断。

### 4. 停止条件｜通过、回滚、交给人
- 通过：满足全部验收标准
- 回滚：出现风险或偏离目标
- 交给人：需要人工决策或介入
- 超时：超过最大迭代次数/时间

#### 风险信号（触发回滚或交给人工）
测试失败、错误率上升、性能退化、范围漂移

### 5. 经验回灌｜规则、skills、harness更新
- 成功经验 → 规则/模板沉淀
- 失败教训 → 负面规则/检查点
- 更新 skills（工具用法、最佳实践）
- 更新 harness（测试集、脚手架）
- 知识库与文档同步更新
> 价值：让系统越用越聪明，下一次更快更稳。

## 核心结论
Loop有价值，前提是**能验证**
验证闭环 = 可信度；可信度 = 可规模化

### 一、如何做到“能验证”？
- 清晰的完成标准（可量化）
- 自动化测试与检查（可执行）
- 可观测的证据（可追溯）

### 二、验证信号举例
1. 测试：单测/集成/端到端全部通过
2. 性能：P95、吞吐、资源使用达标
3. 质量：错误率、告警、SLA满足

### 三、关键指标（建议）
- 通过率（向上优化）
- 迭代次数（向下优化）

# Loop不是Cron

## 副标题：关键差异是上下文、工具和反馈
## 一、四类方案对比表
| 维度 | 手动Prompt | Loop | 系统Cron | GitHub Actions |
| ---- | ---- | ---- | ---- | ---- |
| 上下文 | 每次靠人补充<br>容易丢失历史<br>反复解释背景 | 保留会话记忆<br>自动继承上下文<br>理解持续演进 | 冷启动<br>无记忆<br>无任务背景 | 冷启动但可审计<br>无运行时记忆<br>日志可追溯 |
| 工具 | 临时调用<br>手动复制粘贴<br>单次性使用 | 继承MCP与skills<br>多工具按需编排<br>可复用能力资产 | 需要脚本粘合<br>只能调用命令<br>集成成本高 | 天然接CI仓库<br>丰富官方/社区Action<br>生态成熟 |
| 适合任务 | 探索与决策<br>头脑风暴<br>方案评估 | 工作中持续盯状态<br>直到完成或验证通过<br>处理不确定性 | 固定定时任务<br>日志清理<br>备份/巡检 | 云端可靠流水线<br>构建、测试、发布<br>跨环境部署 |
| 风险 | 人被打断<br>注意力切换<br>容易遗忘 | 成本和跑偏<br>长时间运行会消耗<br>需设定边界与预算 | 失去语境<br>出了问题难定位<br>结果不可解释 | 配置重、权限大<br>YAML复杂<br>权限不当有风险 |
| 停止条件 | 人判断<br>手动停止<br>主观标准 | 到期或验证通过<br>多种条件组合<br>可验证为止 | 时间触发<br>按计划执行<br>不关心结果 | CI/审批触发<br>事件/分支/标签<br>或人工审批 |

## 二、核心总结
Loop 的关键价值 = 保留上下文 + 继承工具 + 反馈驱动的停止条件
- ✅ 有记忆，能理解任务变化
- ✅ 会用工具，解决复杂问题
- ✅ 有反馈，知道何时停止
- ✅ 可验证，结果可追溯

## 三、选型判断指南
### 选择前先自问3个问题
1. 需要保留上下文吗？
2. 任务是否会变化？
3. 是否需要自动判断完成？

### 场景对应选型
1. **临时任务**：思路探索、方案讨论、一次性短时任务 → **用 Loop**
2. **持久任务**：需要持续盯状态，直到目标达成 → **用 GitHub Actions**
3. **纯机械任务**：固定时间、固定步骤，无上下文需求 → **用 Cron**



# 团队怎么落地 Loop 工程
## 副标题：先做能改进决策的循环
### 核心目标
Loop工程的目标：让 AI 在可验证的循环中，持续交付可衡量的改进结果。

## 落地六步流程
流程链路：选任务 → 写目标契约 → 接入工具 → 加传感器 → 设停止条件 → 沉淀经验
每一步通过对应**质量门**校验后，方可进入下一环节，最终进入下一轮循环。

### 1. 选任务
适合落地的任务类型：
- CI修复
- 回归验证
- 日志巡检
筛选标准：小而明确、可验证效果

### 2. 写目标契约
明确四要素：
- 输入（给AI什么）
- 输出（要什么结果）
- 通过标准（可量化）
- 禁区（不能做什么）

### 3. 接入工具
可接入工具/系统：
- repo（代码/分支）
- test（测试框架）
- lint（静态检查）
- MCP（工具能力）
- issue（工单系统）

### 4. 加传感器（质量门#1：证据完整才能进入下一步）
自动采集各类可量化证据：
- 测试结果（通过率）
- 截图/录屏（界面/日志）
- 性能指标（耗时/资源）
- 错误栈/日志（可定位）
- 覆盖率/变更范围

### 5. 设停止条件（质量门#2：条件清晰才能结束循环）
触发停止的场景：
- 通过：满足全部验收标准
- 超时：达到时间上限
- 成本上限：消耗可控
- 风险触发：异常/性能退化
- 交给人：需要人工决策

### 6. 沉淀经验
把本轮成果固化为可复用资产，加速下一轮循环：
- 规则：可复用的判定规则
- 模板：目标契约/提示词模板
- skill：可复用的能力片段
- eval案例：成功/失败样本
- 改进点：下次更快更稳

## 三大通用质量门规范
1. **证据规范**
    - 每轮必须留下证据，无证据=无结果
    - 证据可追溯、可复现
    - 优先自动化采集证据
2. **责任规范**
    - 每个循环指定负责人Owner
    - 明确R&R职责范围
    - 负责人对结果、成本全权负责
3. **回滚规范**
    - 失败场景必须支持回滚
    - 具备回滚方案与开关
    - 变更可撤销、可重试
    - 保护主干与核心关键系统

## 三类不适合落地Loop的场景（不要做）
1. **无验收标准的大改造**
    - 范围模糊，无法判断成败
    - 无法自动评估，成本不可控
    - 风险高，回滚难度大
2. **跨多系统高权限操作**
    - 权限链路复杂，影响面不可控
    - 存在安全、合规风险
    - 若必须做：先隔离环境，小范围试点验证
3. **依赖主观品味判断的任务**
    - 无法量化，无自动验证手段
    - 结果高度依赖个人经验
    - 难以复用、规模化推广

## 落地核心原则
小闭环先跑通，再逐步扩大自动化范围



# Loop采用清单
## 副标题：别让自动化变成失控重复
### 核心说明
Loop工程是有边界的自动化控制循环，只有在**可验证、可回滚、可控成本**的场景才值得用。

## 四道准入校验题（依次判断）
### 1. 能自动验证吗？
满足条件：
- 有测试/校验/CI可以判断结果
- 产出有明确的通过标准
- 分支判断：
  - 能 → 继续校验下一条
  - 不能 → 先补测试或人工评审，补齐验证手段或增设人工关卡

### 2. 失败能回滚吗？
满足条件：
- 变更可回滚（代码/配置/数据）
- 影响范围可隔离
- 分支判断：
  - 能 → 继续校验下一条
  - 不能 → 降低权限，限缩范围/只读/沙箱运行

### 3. 成本可控吗？
满足条件：
- 时间/算力/费用可预估
- 有次数或预算上限
- 分支判断：
  - 能 → 继续校验下一条
  - 不能 → 增加次数和预算上限，设置最大轮数/预算上限

### 4. 边界清楚吗？
满足条件：
- 目标/范围/不做什么已明确
- 有验收标准与停止条件
- 分支判断：
  - 能 → ✅ 开Loop，进入受控循环
  - 不能 → 先做计划门，明确目标/范围/验收/风险

## 三大分类场景
### ✅ 适合Loop的场景
1. CI红灯修复：自动定位并修复构建/测试失败
2. 测试补齐：生成/补充单元测并验证通过
3. 日志巡检：异常检测、分类、自动修复建议
4. 文档同步：API/接口/README自动更新
5. 依赖升级：版本检查、升级PR与回归验证

### ⚠️ 需谨慎的场景
- 大范围重构：影响面大、回滚成本高
- 生产权限操作：改动生产/配置/数据
- 安全敏感变更：权限策略/密钥/访问控制
- 纯审美判断：无客观标准，难以验证
> 处理建议：不确定就先小范围试点，用证据说话，再决定是否扩大。

## ⚙️ 开Loop前必须配置项
1. **Owner**：指定负责人，保证可追溯
2. **到期时间**：设定截止时间，避免无限循环
3. **最大轮数**：限制尝试次数，防止失控成本
4. **证据输出**：要求日志/截图/测试/CI结果留存
5. **停止理由**：满足任一即停，交付或回滚
