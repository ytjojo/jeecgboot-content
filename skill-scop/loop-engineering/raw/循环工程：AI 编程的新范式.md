# 循环工程：AI 编程的新范式

## 一、核心概念说明

循环工程（Loop Engineering）代表了与 AI 协作方式的进化：开发者不再是直接向代理（Agent）输入提示词，而是设计一套系统，让系统自动触发、分发任务并验证 AI 的输出。这种模式通过构建一个递归的目标循环，实现代码开发的半自动化至全自动化。

**真正的 Loop 绝不是简单的机械重复或传统的 Cron Job，它是一套带有反馈闭环的工程系统。** 其核心特点包括：

- **具备明确的反馈机制**：循环内必须包含能够说“不”的机制（如自动化测试、类型检查、CI 构建等）。Agent 修改代码后，系统会读取外部验证结果，以此决定是继续、回滚还是交给人类处理，防止错误在循环中被放大。
- **保留会话上下文**：与传统外部脚本每次冷启动不同，现代工具（如 Claude Code 的 `/loop`）在持续存在的会话中运行，能继承上一轮的上下文窗口、工具权限和连接，基于前一轮的观察做动态判断。
- **内置硬停止条件**：为了防止失控，严肃的 Loop 方案都收敛于三个硬性边界：最大迭代次数、无进展检测（跑了但没变化时停止）、以及 Token/美元预算上限。
- **多组件协同架构**：一个完整的 Loop 系统通常由自动化调度、隔离的工作树、技能文件、外部连接器和子代理组成，确保多个 AI 并行工作时不会互相干扰。


## 二、新旧范式对比

### 旧范式：手动提示

**流程**：人类 → 直接向代理输入提示词 ↔ AI 代理交互

### 新范式：循环工程

工程师角色转变为**设计自动化 AI 工作工厂**，整体分为三大模块：

| 模块 | 核心职责 |
| ---- | ---- |
| **自动化与隔离环境（Automations & Worktrees）** | 利用计划任务定时触发工作流；独立工作树环境，避免多个 AI 代理并行时产生文件冲突 |
| **技能固化与工具连接（Skills & Connectors）** | 将项目规范写入 `SKILL.md`，减少重复向 AI 说明项目规则；通过 MCP 插件协议，让 AI 代理可访问数据库、工单、通讯软件等外部真实工具 |
| **职责分离的子代理（Sub-agents）** | 创作者代理负责编写代码；检查者代理独立负责审核校验，保障验证流程客观公正 |


## 三、Loop Engineering 的五大核心要素

一个完整的 Loop 系统由以下五要素构成：

1. **明确的目标（Goal）**  
   不是模糊的“帮我优化代码”，而是可验证的结果定义：测试通过率从 72% 提升到 95%、代码复杂度降低 30%。目标必须是 Agent 自己能判断是否达成的，而不依赖人的主观评估。

2. **上下文管理（Context Management）**  
   Loop 中的上下文不是静态的，而是随迭代动态更新。每轮执行后，哪些信息需要保留、哪些需要压缩、哪些需要遗忘——这套策略决定了 Agent 在长循环中是否越跑越准还是越跑越乱。

3. **可调用的工具（Tool Access）**  
   Agent 在循环中需要调用真实工具：运行测试、读写文件、搜索代码库、调用外部 API。工具链的完整性和权限边界，直接决定 Agent 能解决的问题边界。Claude Code 和 Codex CLI 当前均已具备完整工具调用能力，是构建 Loop 的主流宿主。

4. **对产出的评估（Output Evaluation）**  
   这是 Loop Engineering 与简单循环脚本的本质区别。评估可以是：运行单元测试（客观）、调用另一个 LLM 打分（主观）、对比 diff 判断变更范围（混合）。没有评估机制，循环只会无限执行，无法收敛。

5. **停止标准（Termination Condition）**  
   目标达成时停止，或达到最大迭代次数时优雅退出。停止标准的设计直接影响 Token 消耗和结果质量的平衡——这也是反对者质疑“Loop 会无限烧 Token”的核心争议点。


## 四、工业级 Loop 五大核心骨架 + 记忆层

一套可落地的循环闭环必须包含 5 个基础模块，搭配持久化记忆：

| 模块 | 职责 |
| ---- | ---- |
| **明确可量化目标** | 不模糊描述，设定可验证交付标准（如“所有单元测试 100% 通过、代码无告警”） |
| **上下文调度管理** | 自动过滤冗余信息、留存关键历史结果，解决 AI “金鱼记忆” |
| **可联动外部工具** | 对接 Git、测试工具、数据库、API、IDE 等真实生产环境 |
| **产出自动评估器** | 内置校验逻辑，判断当前结果是否达标 |
| **终止判定标准** | 达标自动停止，不达标自动生成修正指令、进入下一轮迭代 |
| **持久化记忆层** | 跨会话保存任务状态、中间产出，支持中断续跑、多 Agent 协作 |


## 五、Loop 的构成：六块积木

### 1. 自动化（Automations）

或者可以叫心跳、定时任务。它在固定的时间点或者特定事件发生时自动触发，让 Loop 真正成为一个自行运转的“Loop”。

在 Claude Code 里有很多种用法：
- 用 `/loop` 跑定时任务
- 用 cron 做调度
- 用 hook 在 Agent 生命周期关键节点触发
- 或者推到 GitHub Actions，电脑合上了也可以继续跑

**还有一个很重要的模式：`/goal`。** 它不按时间触发，而是一直运行到最开始所制定的标准真正完成为止。每跑完一轮，会有一个独立的 Agent 判断是否完成，Claude Code 和 Codex 都有这个玩法。

如果说自动化负责开启 Loop，那么剩下的几块积木就负责执行工作完成 Loop。

### 2. 工作树（Worktrees）

当开发者同时跑两个以上 Agent 的时候，时常会出现两个 Agent 写同一个文件的撞车问题。

Git worktree 的思路是在同一个仓库历史上切出一个独立的工作目录，一个 Agent 的修改完全碰不到另一个的工作内容，等于给每个 Agent 配了自己独立的工位，互不干扰。

### 3. 技能库（Skills）

Skill 是把开发者对于整个项目的要求，诸如代码风格、规范、架构、踩过的坑等等这些打包展示给 Agent 看的关键功能。Agent 每跑一遍都读一次，无需每次都重新输入。

如果没有 Skill，Loop 每一轮都要从零推导项目该怎么做；有了 Skill，它每一轮都会吸取上一次的经验。

### 4. 插件和连接器（Connectors）

这决定了 Loop 能不能碰到真实的工具。一个只能看文件系统的 Loop 是个小 Loop，能操作工具干活才是真正的 Loop。连接器基于 MCP 协议，让 Agent 能读取 Issue Tracker、查数据库等等。

### 5. 子代理（Sub-agents）

Loop 中非常关键的一步：**把写代码的和检查代码的拆成两个 Agent。** 让写代码的 Agent 给自己打分太容易放水了。第二个独立的 Agent 配不同的指令，甚至有时用不同的模型，专门负责检查其他 Agent 的工作成果。

Loop 运行过程中，让开发者敢放开监管的最大底气就是子 Agent——一个真正信得过的验证者。这也是 `/goal` 命令中会去做的事——让一个独立 Agent 判断目标是否完成。

### 6. 记忆层（Memory）

听起来最简单，实际上是整条流水线的脊椎。一个 Markdown 文件，或一个 Linear 看板，或任何活在单个会话之外、记录什么做了、什么没做的东西。**记忆文件必须存在磁盘上，不能在上下文窗口里——Agent 会忘，文件不会。**

| 技能（Skills） | 固化项目特有背景知识 | Agent Skills（`SKILL.md`） |
| 子代理（Sub-agents） | 拆分创作者与检查者角色 | `.claude/agents/` 任务子代理 |

### Loop Engineering 的五个核心积木（详表）

| 积木 | 技术实现 | 一句话说明 |
| ---- | ---- | ---- |
| 自动化（Automations） | `/loop --schedule`、cron 定时任务 | 让 loop 按时间自动触发，不依赖人坐在电脑前 |
| 工作树（Worktrees） | `--worktree`、独立 git checkout | 多个 Agent 并行工作时互不干扰，各自在独立目录 |
| 技能库（Skills） | `SKILL.md` 文件、`/$` 命令调用 | 项目知识写进技能文件，Agent 不再每次重新猜约定 |
| 插件连接（Connectors） | MCP 协议、接 Issue/DB/Slack | 让 Agent 连真实工具，不只是操作文件系统 |
| 子 Agent（Sub-agents） | maker/checker 分离、审查 Agent 独立评分 | 写代码的和检查的分开，防止自己给自己打满分 |


## 七、运行逻辑（类比 PDCA 循环）

**设定目标 → Agent 执行任务 → 系统自动校验成果 → 不合格自动生成优化指令、重新执行 → 循环往复**，直至满足终止条件后自动收尾、归档结果，全程无需人工介入。


## 八、Loop Engineering 适用场景

### 适合用 Loop 的场景

| 场景 | Loop 价值 | 推荐工具 |
| ---- | ---- | ---- |
| 自动化测试修复 | 无需人工逐条看报错，Agent 自主定位修复 | Claude Code |
| 代码重构迭代 | 分批处理大型仓库，每批验证通过后继续 | Claude Code + Hermes Agent |
| 文档生成与校验 | 生成→检查格式→修正→再检查，直到规范 | Codex CLI |
| 数据清洗流水线 | 处理→验证数据质量→补充缺失→再验证 | 自定义 Agent Loop |
| 企业审批自动化 | 提交→检查合规→修改→再提交 | OpenClaw |
| CI/CD 修复与 PR 监控 | 每几分钟检查一次 PR 构建是否通过，失败则自动修复 | Claude Code |
| 依赖升级与安全漏洞管理 | 自动升级库、修补安全漏洞和管理技术债 | Dependabot 风格 Loop |
| Flaky Tests 修复 | 专门用来维护 CI 的健康度 | Claude Code |
| 长周期文档与数据同步 | 定时监控社交媒体反馈、汇总信息或更新项目进度文档 | 自定义 Loop |

### Loop 的适用边界

| 维度 | 适合用 Loop | 不适合用 Loop |
| ---- | ---- | ---- |
| **成功标准** | 客观可测量：测试通过/不通过、分数达到阈值、输出匹配模板 | 主观判断：“这感觉对吗”、客户会喜欢吗、产品品味如何 |
| **典型任务** | 代码审查自动化、批量代码迁移、SEO 页面生成、运行 evaluation、CI 失败分类 | 产品设计、品牌文案、创业方向决策、需要品味的工作 |
| **团队条件** | 有清晰的 spec、有客观的验证手段、有审计日志需求 | 想法还在孵化、需要反复讨论、目标本身不确定 |


## 九、工程师角色的转变与风险

### 1. 角色转变：从“手持工具”到“设计工厂”

工程师核心工作从手动编写提示词，转变为设计一套能够自动驱动 AI 代理运行的循环系统。

### 2. 三大风险与防线

#### 风险一：认知投降（Cognitive Surrender）

循环运行越顺畅，开发者越容易放弃独立思考。最舒适的姿势往往是最危险的——当循环自动运行时，很容易陷入“不再思考、全盘接受”的状态。设计循环是良药，但如果是为了逃避思考而设计，它就是加速剂。

#### 风险二：理解力负债（Comprehension Debt）

循环跑得越快，如果不人工审查产出代码，开发者对自己项目的理解就会越来越浅。如果不阅读循环生成的代码，你和代码之间的知识鸿沟会越来越大。

#### 风险三：Token 成本失控

无限循环极易导致字符膨胀（Character Bloat），瞬间烧干预算，尤其在产出过程无监督和控制的情况下。就像 OpenClaw 不停的循环执行直接耗费大量 Token。

#### 防线：验证依然是人类的底线

自动化循环可以高效加速产出，但最终代码质量把关、结果验证的核心责任，仍需要工程师承担。**无人值守的循环如果出错，也会无人值守地制造灾难。** 虽然我们分离了验证者，但“完成”终究是一个声明而非数学证明。你的工作依然是确认代码能工作。


## 十、落地建议与避坑指南

如果你准备着手构建自己的第一个 Loop，请记住以下几条实战教训：

1. **从小开始，逐步迭代**  
   不要指望第一天就搭建出复杂的全家桶架构。第一版可以极简：一个定时器 + 一个 Skill + 一个保存状态的 Markdown 文件。跑通后再加入 Checker 机制与外部 Connector。

2. **内置硬停止条件**  
   为了防止失控，严肃的 Loop 必须有明确的边界：最大迭代次数、无进展检测（跑了但没变化时停止）、以及 Token 预算上限。

3. **让沉默成为敌人**  
   Loop 通常在后台静默运行。必须在关键步骤失败时加入通知机制（Webhook 推送至飞书/Slack），并在 Memory 层输出可被快速检索的格式化日志。

4. **警惕 Token 成本与理解力负债**  
   无限循环极易导致字符膨胀，瞬间烧干预算。循环跑得越快，如果不人工审查产出代码，开发者对自己项目的理解就会越来越浅。

5. **考虑 Token 成本（Token Cost）**  
   需考虑不同使用模式下的资源消耗。

6. **保证输出质量（Quality Assurance）**  
   保证输出质量，避免错误累积。


## 十一、如何评估一个 Loop 设计的质量

三个核心指标：

| 指标 | 说明 |
| ---- | ---- |
| **收敛速度** | 平均多少轮达到目标 |
| **Token 效率** | 达成目标消耗的 Token 数 |
| **鲁棒性** | 在异常输入下是否能优雅退出而非死循环 |


## 十二、面临的挑战

| 挑战 | 说明 |
| ---- | ---- |
| **Token 成本高昂** | 无限循环极易导致字符膨胀，瞬间烧干预算 |
| **调试极其困难** | 调试一个已经跑了数十轮的复杂状态机，比修好一个 Prompt 难 10 倍 |
| **理解力负债** | 循环跑得越快，如果不人工审查产出代码，开发者对自己项目的理解就会越来越浅。最终为代码质量把关的依然是人类工程师，事故和责任最终还是要有人来承担 |
