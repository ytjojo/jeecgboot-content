# 需求文档：openspec-fix_all_audit 技能

**版本**: v1  
**创建时间**: 2026-06-28  
**状态**: 需求确认完成，待进入设计阶段

---

## 一、技能目标

自动化修复 OpenSpec 审核文档中指出的问题，覆盖从计划生成到执行修复的完整流程。

### 核心价值
- 减少人工逐条处理审核问题的时间成本
- 自动识别跨 change 依赖关系，避免修复冲突
- 统一修复流程，确保代码质量和文档一致性

---

## 二、输入方式

### 2.1 单 change 模式
指定具体 change 目录路径，如：
```
openspec/changes/channel-24-lifecycle-frontend/
```

### 2.2 批量扫描模式
指定 changes 根目录，自动扫描所有 change：
```
openspec/changes/
```

---

## 三、覆盖的审核文档类型

技能需要解析以下 6 类审核文档：

| 文件模式 | 审核类型 | 问题分级 | 修复目标 |
|---------|---------|---------|---------|
| `review-report-*.md` | 规范文档审核 | BLOCK / FLAG / ADVISORY | 文档（proposal/design/specs/tasks） |
| `drift-report-*.md` | 代码漂移检测 | CRITICAL / WARNING | 代码 + 文档对齐 |
| `verify-report-*.md` | 实现验证 | CRITICAL / WARNING / SUGGESTION | 代码实现 |
| `verify.md` | 实现验证（简版） | 按维度评估 | 代码实现 |
| `verification-review.md` | 验证审核 | 按维度评估 | 代码 + 任务完成度 |
| `backend-issues.md` | 后端遗留问题 | P0 / P1 | 后端 Java 代码 |

### 3.1 问题结构
每个问题包含统一字段：
- **编号**：如 B-01、D-01、C-01
- **位置**：文件路径 + 行号（可选）
- **问题描述**：具体问题说明
- **影响**：不修复的后果
- **修复建议**：推荐的修复方案

---

## 四、子命令定义

### 4.1 plan 子命令
**功能**：扫描审核文档，生成修复计划

**输入**：change 目录路径或 changes 根目录

**输出**：
- 全局 `openspec/changes/fix-plan.md`（扫描根目录时）
- 各 change 目录下的 `fix-plan.md`

**处理逻辑**：
1. 扫描指定目录下的所有审核文档
2. 提取所有问题项（按编号、位置、描述、影响、建议）
3. 识别跨 change 依赖关系（如 Store 命名冲突涉及多个 change）
4. 按依赖关系排序修复项
5. 生成细粒度修复计划（包含具体文件路径、修改内容、验证方式）

### 4.2 fix 子命令
**功能**：执行修复计划

**输入**：已确认的 fix-plan.md

**处理逻辑**：
1. 创建 worktree（所有修复共用一个）
2. 按依赖顺序批量执行修复
3. 文档修复：直接修改 Markdown 文件
4. 代码修复：走 TDD 流程（先写测试 → 实现 → 重构）
5. 自动验证：跑测试、lint
6. 合并已有改动，一起提交
7. 更新 fix-plan.md 状态（done / skipped / failed）
8. 输出回顾复盘报告

## 五、修复范围

### 5.1 文档修复
- proposal.md
- design.md
- specs/*/spec.md
- tasks.md
- 其他 Markdown 文档

### 5.2 代码修复
- **前端**：Vue 组件、TypeScript 文件、Store、API 封装
- **后端**：Java Controller、Service、Mapper、Entity

---

## 六、输出物

### 6.1 fix-plan.md 格式

每个修复项包含：
```markdown
### {问题编号} - {问题简述}

**来源**: {审核文档文件名}  
**位置**: {文件路径}:{行号（可选）}  
**优先级**: {BLOCK/CRITICAL/P0/FLAG/WARNING/...}  
**依赖**: {依赖的其他问题编号，无则填"无"}  

**修复步骤**:
1. {具体步骤 1：文件路径 + 修改内容}
2. {具体步骤 2}
3. ...

**验证方式**:
- {测试命令 / lint 检查 / 人工检查项}

**状态**: pending / done / skipped / failed
```

### 6.2 回顾复盘报告

报告内容包含：
1. **修复统计**
   - 总问题数
   - 已修复数（done）
   - 跳过数（skipped）
   - 失败数（failed）
   - 按优先级分布

2. **常见问题类型分析**
   - 命名冲突类
   - 目录结构类
   - API 缺失类
   - 文档不一致类
   - 代码实现类
   - 其他

3. **改进建议**
   - 针对高频问题的预防措施
   - 流程优化建议
   - 规范更新建议

## 七、执行策略

### 7.1 worktree 管理
- 一个 worktree 放所有修复（文档 + 代码）
- worktree 命名规则由 AGENTS.md 统一管理

### 7.2 代码修复流程
- 走 TDD 流程：先写测试 → 红灯 → 实现 → 绿灯 → 重构
- 自动跑测试、lint
- 合并已有改动，一起提交

### 7.3 文档修复流程
- 直接修改 Markdown 文件
- 与代码修复在同一个 worktree 中

### 7.4 失败处理
- 单个修复项失败时：标记为 failed，继续执行其他项
- 关键依赖项失败时：暂停，等待用户决策

### 7.5 状态更新
- 修复完成后更新 fix-plan.md 中对应项的状态
- 状态值：pending / done / skipped / failed

---

## 八、跨 change 问题处理

### 8.1 依赖识别
- 自动识别跨 change 的依赖关系
- 例如：Store 命名冲突涉及 channel-22 和 channel-24

### 8.2 执行顺序
- 按依赖关系排序：先修上游，再修下游
- 修改 A change 影响 B change 时，自动同步更新

### 8.3 全局汇总
- 扫描根目录时生成全局 fix-plan.md
- 执行时拆分到各 change 独立执行

---

## 九、验收标准

### 9.1 plan 阶段
- [ ] 能正确解析 6 类审核文档
- [ ] 能提取所有问题项的完整信息
- [ ] 能识别跨 change 依赖关系
- [ ] 生成的 fix-plan.md 包含细粒度修复步骤
- [ ] 全局汇总和拆分版本一致

### 9.2 fix 阶段
- [ ] 能创建 worktree 并执行所有修复
- [ ] 文档修复正确修改 Markdown
- [ ] 代码修复走 TDD 流程
- [ ] 自动跑测试、lint 并验证通过
- [ ] 失败项标记正确，不影响其他项
- [ ] fix-plan.md 状态更新正确

### 9.3 收尾阶段
- [ ] 输出回顾复盘报告
- [ ] 报告包含修复统计、问题类型分析、改进建议

## 十、约束与限制

### 10.1 不需要的功能
- fix-plan 不需要修复前后对比示例
- worktree 命名规则不在此技能中指定

### 10.2 遵循的规范
- 遵循 AGENTS.md 中的全局硬规则
- 遵循 worktree 管理规范
- 遵循 TDD 流程规范
