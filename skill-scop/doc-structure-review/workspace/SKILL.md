# 文档一致性审核技能

对文档进行多维度结构化和一致性审查，输出分类分级、附带证据的审核报告。覆盖十一个维度：结构一致性六项 + 内容深度四项 + 决策质量一项。

## 审查维度

审核必须覆盖以下全部十一个维度，不允许遗漏：

### 1. 内部一致性
检查不同章节之间的规则是否存在矛盾或冲突，以及框架层声明是否在具体实现层得到落实。
- 如果一处定义了强制性规则，另一处定义了禁止性规则，两者覆盖的条件集是否存在重叠？
- 同一概念在不同上下文中使用是否一致？
- 表格/列表中的条件是否存在未覆盖的逻辑缺口？
- **约束-模板脱节检查**：提取文档中所有框架级约束/规则声明，逐一验证是否在所有具体模板、示例、检查清单中得到实现；标记"声明了但未落实"的脱节项。

### 2. 术语漂移
检查同一概念在不同章节的称呼是否一致。
- 同一事物在不同地方是否使用了不同的名称？
- 缩写、全称是否混用？
- 中英文术语是否对标统一？

### 3. 结构问题
检查文档的组织结构是否合理，以及各检查项/步骤的逻辑阶段归属是否正确。
- 是否存在前向引用（引用了尚未定义的概念/代号）？
- 章节之间的依赖关系是否清晰？
- 是否缺少必要的上下文说明？
- 检查清单是否与决策树等核心结构对齐？
- **逻辑位置检查**：对文档中的每个检查项/步骤，验证其逻辑阶段归属是否正确——例如"是否存在资源冲突"在启动前无法判断，应属于集成验证阶段而非初始化阶段。

### 4. 歧义
检查是否存在两个读者会做出不同理解的表述。
- 条件语句是否覆盖所有可能情况？
- 阈值/范围定义是否精确？
- 否定式、双重否定是否清晰？
- 符号/标记的含义是否在上下文中一致？

### 5. 重复/冗余
检查内容是否在不同章节中不必要地重复，以及是否存在结构性过度设计。
- 同一规则是否在多处以不同表述出现？
- 是否有可以合并的重复段落？
- 重复内容是否增加维护负担（修改时需同步多处）？
- **结构性冗余检查**：是否存在推测性设计（framework-ahead-of-need）——即在需求明确之前就引入的抽象层或框架？每个抽象/概念是否有足够的消费者支撑其存在？

### 6. 前向引用
检查文档中引用的概念、代号、章节是否先定义后使用。
- Q1/Q2 等代号是否在引用前已定义？
- 章节引用是否正确（编号是否匹配）？
- 阈值引用是否与定义处一致？

### 7. 前提挑战
检查文档的核心前提和假设是否成立、是否被显式声明。系统检查以下五类假设：

| 假设类型 | 检查问题 |
|---------|---------|
| 环境假设 | 文档是否隐式绑定特定平台/框架/工具？这些绑定是否被声明？ |
| 用户行为假设 | 文档是否假设执行者具备某种能力或会以某种方式行动？ |
| 规模假设 | 文档的方案是否只在特定规模下有效（例如假设单会话可以完成所有任务）？ |
| 上下文假设 | 文档对信息传递、历史记录、状态共享做了什么假设？ |
| 认知假设 | 文档是否对差异化的场景一视同仁（例如假设所有情况需要相同的处理策略）？ |

此外还需检查：
- 核心前提（如决策公式的变量）是否可量化、可验证？
- 分类标准（如"硬条件"/"红线"）的划分依据是否合理？
- 如果前提错了，文档的结论是否还有效？
- 是否存在被假设为最优但未经论证的方案选择？

### 8. 边界条件
检查决策逻辑的边界是否完整覆盖，临界情况是否有明确处理。使用**四路径覆盖模型**逐一检查：

| 路径 | 含义 | 检查问题 |
|------|------|---------|
| Happy | 一切正常 | 文档对理想情况的覆盖是否完整？ |
| Nil | 输入为空/不存在 | 目标对象不存在、输入为空时，文档是否有明确处理规则？ |
| Empty | 输入存在但无内容 | 对象存在但为空（空文件、空列表、零结果）时如何处理？ |
| Error | 执行失败 | 操作失败、依赖不可用、重试后仍失败时如何处理？ |

此外还需检查：
- 条件分支是否覆盖所有可能情况？（含等于临界值的情况）
- 阈值定义是否存在模糊区间？
- 互斥条件之间是否存在重叠或盲区？
- 极端场景（任务极小/极大、资源为零等）是否有处理规则？

### 9. 缺失内容
检查文档是否遗漏了重要场景、规则或说明，包括操作指导是否充分。
- 是否存在文档应覆盖但未提及的重要场景？
- 规则系统是否完整（是否每条规则的反面也有处理）？
- 是否存在"隐含知识"——读者必须通过外部信息才能理解的内容？
- 示例和反例是否覆盖了所有关键场景？
- **操作指导充分性**：是否存在只提了一句但缺少操作步骤的关键功能？（例如"支持失败重试"但未说明重试触发条件和次数限制）

### 10. 可操作性
检查文档能否被目标读者无歧义地执行，以及在实际使用中最先在哪里出问题。
- 执行者能否不依赖额外上下文就做出决策？
- 决策步骤是否足够具体？（避免"根据情况判断"等模糊指令）
- 所需的输入信息在执行时是否可获取？
- 是否存在多个读者会做出不同决策的表述？
- **断裂点分析**：模拟文档投入使用后的执行流程，识别最先出问题的位置（第一断裂点），按优先级排序，验证每个断裂点是否有对应的防护或降级机制。
- **验证清单有效性**（若文档包含检查清单）：列出目标场景的常见失败模式，逐一验证清单条目能否捕获这些失败模式；输出"清单遗漏的失败模式"。

### 11. 决策稳健性
检查文档中的关键决策是否经过充分验证，是否具备可逆性和容错性。

对每个关键决策（架构选择、分类标准、优先级规则等）检查：
- **可证伪条件**：在什么条件下这个决策是错的？文档是否给出了识别这种情况的方法？
- **证据质量**：支撑决策的证据有多强？是经验性数据、类比推断、还是纯逻辑推导？
- **逆转成本**：如果决策错了，修改的代价有多高？高逆转成本的决策是否有更强的论证？
- **负载依赖**：有多少其他章节/规则依赖这个决策？高依赖决策是否被特别标注和保护？

> 注意：本维度与「前提挑战」的区别在于——前提挑战检查"假设是否成立"，决策稳健性检查"基于假设做出的选择是否最优、是否可逆"。

## 严重程度定义

| 等级 | 定义 | 示例 |
|------|------|------|
| P0 | 阻断性错误——导致文档无法正确使用 | 决策树分支存在逻辑缺口，读者无法做出决策 |
| P1 | 重要缺陷——可能导致误解或不正确决策 | 关键符号含义在上下文中不一致 |
| P2 | 一般问题——影响阅读体验或可能造成困惑 | 前向引用未定义，术语前后不一致 |
| P3 | 改进建议——不影响理解但可优化 | 内容重复增加维护成本，表述可更精确 |

## 执行方式

### 角色分工

| 角色 | 职责 |
|------|------|
| **主 agent** | 确认文档路径 → 派发 subagent → 接收/产出报告 → 呈现给用户 → 按需修复 |
| **Subagent** | 读取文档 → 识别结构 → 逐维度检查 → 输出结构化报告 |

### 首选路径

通过 `compound-engineering:ce-coherence-reviewer` subagent 执行审查。subagent 加载本 SKILL.md 后自动获得十一个审查维度、输出格式和严重程度定义，prompt 只需补充**目标文档路径**。

### 后备方案（Fallback）

若 subagent 出现以下任一情况，主 agent **必须降级为直接执行**，不得中断审核流程：

- subagent 被用户拒绝
- subagent 返回空结果（仅有"任务已完成"无实质内容）
- subagent 超时
- subagent 返回格式错误无法解析

降级执行时，主 agent 按照下方的「执行流程」和「输出格式」自行完成审核。

### 与项目规则的关联

项目 AGENTS.md 规定"涉及审核/评审/分析/审查/审计/review 操作时，必须通过 subagent 执行"。本 skill 遵循此规则，但同时设定**例外**：当 subagent 失败时，降级为直接执行优于放弃审核——规则为质量服务，不是相反。

## 执行流程

无论主 agent 直接执行还是 subagent 执行，均遵循以下流程：

1. 读取目标文档全文
2. 识别文档结构（章节、表格、列表、决策树等）
3. 逐维度检查，在每个维度下：
   - 识别所有潜在问题
   - 定位到具体章节/行号
   - 提取原文证据
   - 评估严重程度（按上方「严重程度定义」）
4. 按下方「输出格式」模板汇总结构化审核报告

## 输出格式

用中文输出结构化审核报告。每条发现必须包含以下字段：

```
## 发现 N：[简短标题]

- **位置**：第X节（具体章节名称）
- **审查维度**：内部一致性 / 术语漂移 / 结构问题 / 歧义 / 重复/冗余 / 前向引用 / 前提挑战 / 边界条件 / 缺失内容 / 可操作性 / 决策稳健性
- **严重程度**：P0 / P1 / P2 / P3（定义见上方「严重程度定义」章节）
- **问题描述**：清晰说明问题是什么，为什么是问题
- **证据**：引用文档原文（使用 > 格式）
- **建议修复方案**：具体的修改建议
```

报告开头需要包含汇总表格：

| 严重程度 | 数量 | 关键发现 | 建议修复顺序 |
|---|---|---|---|
| P0 | N | ... | 立即 |
| P1 | N | ... | 本次迭代 |
| P2 | N | ... | 下次迭代 |
| P3 | N | ... | 可选 |

如果没有发现问题，明确说"未发现问题"。如果**所有十一个维度**均未发现问题，输出：

```
## 审核结果：未发现问题

经逐维度审查（内部一致性、术语漂移、结构问题、歧义、重复/冗余、前向引用、前提挑战、边界条件、缺失内容、可操作性、决策稳健性），本文档未发现需要修改的问题。
```
