# 文档一致性审核技能

对文档进行多维度结构化和一致性审查，输出分类分级、附带证据的审核报告。覆盖十个维度：结构一致性六项 + 内容深度四项。

## 审查维度

审核必须覆盖以下全部十个维度，不允许遗漏：

### 1. 内部一致性
检查不同章节之间的规则是否存在矛盾或冲突。
- 如果一处定义了强制性规则，另一处定义了禁止性规则，两者覆盖的条件集是否存在重叠？
- 同一概念在不同上下文中使用是否一致？
- 表格/列表中的条件是否存在未覆盖的逻辑缺口？

### 2. 术语漂移
检查同一概念在不同章节的称呼是否一致。
- 同一事物在不同地方是否使用了不同的名称？
- 缩写、全称是否混用？
- 中英文术语是否对标统一？

### 3. 结构问题
检查文档的组织结构是否合理。
- 是否存在前向引用（引用了尚未定义的概念/代号）？
- 章节之间的依赖关系是否清晰？
- 是否缺少必要的上下文说明？
- 检查清单是否与决策树等核心结构对齐？

### 4. 歧义
检查是否存在两个读者会做出不同理解的表述。
- 条件语句是否覆盖所有可能情况？
- 阈值/范围定义是否精确？
- 否定式、双重否定是否清晰？
- 符号/标记的含义是否在上下文中一致？

### 5. 重复/冗余
检查内容是否在不同章节中不必要地重复。
- 同一规则是否在多处以不同表述出现？
- 是否有可以合并的重复段落？
- 重复内容是否增加维护负担（修改时需同步多处）？

### 6. 前向引用
检查文档中引用的概念、代号、章节是否先定义后使用。
- Q1/Q2 等代号是否在引用前已定义？
- 章节引用是否正确（编号是否匹配）？
- 阈值引用是否与定义处一致？

### 7. 前提挑战
检查文档的核心前提和假设是否成立、是否被显式声明。
- 文档依赖了哪些未明确声明的假设？
- 核心前提（如决策公式的变量）是否可量化、可验证？
- 分类标准（如"硬条件"/"红线"）的划分依据是否合理？
- 如果前提错了，文档的结论是否还有效？

### 8. 边界条件
检查决策逻辑的边界是否完整覆盖，临界情况是否有明确处理。
- 条件分支是否覆盖所有可能情况？（含等于临界值的情况）
- 阈值定义是否存在模糊区间？
- 互斥条件之间是否存在重叠或盲区？
- 极端场景（任务极小/极大、资源为零等）是否有处理规则？

### 9. 缺失内容
检查文档是否遗漏了重要场景、规则或说明。
- 是否存在文档应覆盖但未提及的重要场景？
- 规则系统是否完整（是否每条规则的反面也有处理）？
- 是否存在"隐含知识"——读者必须通过外部信息才能理解的内容？
- 示例和反例是否覆盖了所有关键场景？

### 10. 可操作性
检查文档能否被目标读者无歧义地执行。
- 执行者能否不依赖额外上下文就做出决策？
- 决策步骤是否足够具体？（避免"根据情况判断"等模糊指令）
- 所需的输入信息在执行时是否可获取？
- 是否存在多个读者会做出不同决策的表述？

## 严重程度定义

| 等级 | 定义 | 示例 |
|------|------|------|
| P0 | 阻断性错误——导致文档无法正确使用 | 决策树分支存在逻辑缺口，读者无法做出决策 |
| P1 | 重要缺陷——可能导致误解或不正确决策 | 关键符号含义在上下文中不一致 |
| P2 | 一般问题——影响阅读体验或可能造成困惑 | 前向引用未定义，术语前后不一致 |
| P3 | 改进建议——不影响理解但可优化 | 内容重复增加维护成本，表述可更精确 |

## 执行方式

### 角色分工

| 角色 | 职责 |
|------|------|
| **主 agent** | 确认文档路径 → 派发 subagent → 接收/产出报告 → 呈现给用户 → 按需修复 |
| **Subagent** | 读取文档 → 识别结构 → 逐维度检查 → 输出结构化报告 |

### 首选路径

通过 `compound-engineering:ce-coherence-reviewer` subagent 执行审查。subagent 加载本 SKILL.md 后自动获得十个审查维度、输出格式和严重程度定义，prompt 只需补充**目标文档路径**。

### 后备方案（Fallback）

若 subagent 出现以下任一情况，主 agent **必须降级为直接执行**，不得中断审核流程：

- subagent 被用户拒绝
- subagent 返回空结果（仅有"任务已完成"无实质内容）
- subagent 超时
- subagent 返回格式错误无法解析

降级执行时，主 agent 按照下方的「执行流程」和「输出格式」自行完成审核。

### 与项目规则的关联

项目 AGENTS.md 规定"涉及审核/评审/分析/审查/审计/review 操作时，必须通过 subagent 执行"。本 skill 遵循此规则，但同时设定**例外**：当 subagent 失败时，降级为直接执行优于放弃审核——规则为质量服务，不是相反。

## 执行流程

无论主 agent 直接执行还是 subagent 执行，均遵循以下流程：

1. 读取目标文档全文
2. 识别文档结构（章节、表格、列表、决策树等）
3. 逐维度检查，在每个维度下：
   - 识别所有潜在问题
   - 定位到具体章节/行号
   - 提取原文证据
   - 评估严重程度（按上方「严重程度定义」）
4. 按下方「输出格式」模板汇总结构化审核报告

## 输出格式

用中文输出结构化审核报告。每条发现必须包含以下字段：

```
## 发现 N：[简短标题]

- **位置**：第X节（具体章节名称）
- **审查维度**：内部一致性 / 术语漂移 / 结构问题 / 歧义 / 重复/冗余 / 前向引用 / 前提挑战 / 边界条件 / 缺失内容 / 可操作性
- **严重程度**：P0 / P1 / P2 / P3（定义见上方「严重程度定义」章节）
- **问题描述**：清晰说明问题是什么，为什么是问题
- **证据**：引用文档原文（使用 > 格式）
- **建议修复方案**：具体的修改建议
```

报告开头需要包含汇总表格：

| 严重程度 | 数量 | 关键发现 |
|---|---|---|
| P0 | N | ... |
| P1 | N | ... |
| P2 | N | ... |
| P3 | N | ... |

如果没有发现问题，明确说"未发现问题"。如果**所有十个维度**均未发现问题，输出：

```
## 审核结果：未发现问题

经逐维度审查（内部一致性、术语漂移、结构问题、歧义、重复/冗余、前向引用、前提挑战、边界条件、缺失内容、可操作性），本文档未发现需要修改的问题。
```
